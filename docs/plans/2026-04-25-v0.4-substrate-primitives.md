## 1. One-paragraph summary

Three substrate primitives ship as v0.4.0a1: a `Dispatcher` over the existing plan walker, a `fork()` method on the existing `ProjectionAdapter` protocol, and a typed `Provenance` schema with a `causal_history()` walker. No new sub-packages. No new parallel APIs. One new internal file (`plan/_dispatch.py`). **Each primitive is the smallest committable extension to an existing affordance, not a parallel system.** The redesign lesson from the parked design is "the substrate already has 80% of what's needed — find it before adding."

## 2. Why this redesign

The parked design at [`2026-04-25-write-time-synthesis-design.md`](2026-04-25-write-time-synthesis-design.md) tried to ship four substrate primitives + a paper claim (Prop 6) + a vault-side `/vault/why` API in one Phase 2.D for NeSy 2026. Two parallel reviewers (substrate-fit + paper-claim) returned converging "do not ship as written" verdicts. Vault references: `4799908f-151a-42e0-a13d-93bc5360a6d8`, `52bf8ebc-8dfa-4123-ab10-79960df6f9f7`.

Three load-bearing problems, all confirmed in live code on 2026-04-25:

1. **`persistence.plan/_interpret.py` is a depth-first walker** (`walk(node, visitor)`), not an evaluator. `:code` and `:branch` raise `UnimplementedNodeKindError`. The original design assumed an evaluator existed.
2. **`DB.branch(t, assertions)` clones `InMemoryStore` from `as_of(t)` snapshot with zero projector dispatch.** The original design assumed branched DBs would automatically fork their projections.
3. **`Datom.provenance` is `dict = field(default_factory=dict)`** — free-form, untyped, unindexed. The original design assumed a queryable provenance index existed.

Plus one diagnostic finding from the live-code scan that reframes the work:

4. **`fact/projection.py` already ships `ProjectionAdapter` + `rebuild()`** — a pull-based, full-rebuild model whose docstring quotes paper §5.1 verbatim ("the projection is a *disposable* cache. Burn it down, rebuild from the log."). The original design's proposed `Projector.on_assert(datom)` would have been a parallel push-based API competing with this. **Don't add — extend.**

### Pocock #1 ubiquitous-language pin

Three prior meanings of "synthesis" — persistence-os-substrate (the projection function from log to derived state), ai-box-vault-batch (the nightly tier classifier + ripple + backlinks scan), paper §6.3 (the nightly cron job). The substrate concept Π is **renamed to "derivation"** for v0.4+; ai-box-vault keeps "synthesis" as the user-facing batch term. Filename impact: `_interpret.py` → `_walk.py` + new `_dispatch.py` (Phase A).

### Ferrari/brains framing pin

User vision: "build the Ferrari body (substrate); the brains (Opus / GPT / DeepSeek / Kimi / Llama) plug in." Three design implications, load-bearing across all phases:

- Effect-handler dispatch, not LLM-coupled dispatch — the substrate doesn't know who's "thinking"
- Provenance shape is model-agnostic — `:llm/handler-id` (registered handler, swappable) + `:llm/canonical-call` (prompt+output identity hash, stable across models). **No `:provider` field.**
- ai-box-vault's `QdrantProjectionAdapter.fork()` (downstream session) uses Qdrant alias-indirection internally — the substrate doesn't expose alias mechanics, just the `fork()` contract.

## 4. Phase A — Interpreter refactor

### Problem

`plan/_interpret.py` is named for an evaluator but is actually a depth-first walker (`walk(node, visitor)` with optional callback). The effect runtime currently has no way to register per-tag handlers — it would have to subclass the walker or wrap it. Future `:code` execution and `:branch` speculative search (post-NeSy roadmap, Phase 3) need a clean dispatch contract that doesn't bake in their semantics now.

### Move

Split into two files. `_walk.py` keeps the existing `walk()` function unchanged (callers don't break). `_dispatch.py` introduces a `Dispatcher` class with `register(tag, handler)` and `dispatch(node, env) → result`. The dispatcher uses `_walk.walk()` internally for traversal but calls registered handlers per-tag instead of an opaque visitor. `:code` and `:branch` still raise `UnimplementedNodeKindError` (post-NeSy seam).

### Pluggability win

The effect runtime registers `:llm-call`, `:tool-call`, `:transact`, etc. as dispatch handlers. Swapping Opus → DeepSeek is a handler swap, never a code change in the substrate. This is the Ferrari/brains pin operationalised at the dispatch layer.

### Backwards compatibility

- Every existing call to `walk(node, visitor)` keeps working unchanged.
- The `Dispatcher` is purely additive.
- `_interpret.py` is replaced with a one-line shim re-exporting `walk` from `_walk.py` for back-compat (drop in v0.5 after grep-verify zero downstream imports).

### Public API additions

```python
from persistence.plan import Dispatcher

d = Dispatcher()
d.register(":llm-call", my_handler)
result = d.dispatch(plan_root, env={})
```

### Files touched (sequenced)

1. `plan/_interpret.py` → `plan/_walk.py` — rename file, content unchanged
2. `plan/_interpret.py` (NEW) — one-line back-compat shim: `from persistence.plan._walk import walk` (re-export only)
3. `plan/_dispatch.py` (NEW, ~80 lines) — `Dispatcher` class with `register()` + `dispatch()`
4. `plan/__init__.py` — re-export `Dispatcher` for the public API
5. `effect/runtime.py` (~10 lines wiring; opt-in — runtime registers handlers via the new `Dispatcher` for code paths that want it; existing `walk()`-based code paths unchanged)

### Tests

- `tests/plan/test_dispatch.py` (new): handler registration; dispatch order matches walk order; unregistered tag raises clean error; env passes through unchanged; `:code`/`:branch` still raise `UnimplementedNodeKindError`.

### Wall-clock

~3 days.

## 5. Phase B — VANISHED (rationale captured for the record)

The originally-proposed `ProjectorAlias` protocol does not ship in v0.4.0a1. Live-code scan showed:

- The substrate's existing `ProjectionAdapter` + `rebuild(db, adapter)` model is pull-based — projections are FUNCTIONS of the log, not subscribers to writes.
- Re-projection under a new embedding model is therefore: allocate a new sink, call `rebuild(db, new_sink)`, atomically swap, drop old.
- The atomic-swap mechanism (Qdrant `update_aliases`, Kuzu rename-replace) is a backend implementer concern. The substrate's contract is satisfied by `rebuild()` — it doesn't need to know how the swap happens at the storage layer.

ai-box-vault's `QdrantProjectionAdapter.fork()` (Phase C downstream consumer) will use alias-indirection internally. **A backend-implementer guide section in the design doc names the pattern** so future adapter authors don't reinvent it; the substrate API stays narrow.

## 6. Phase C — `ProjectionAdapter.fork()`

### Problem

`DB.branch(t, assertions)` at `db.py:269` clones the in-memory store for the branched DB, but registered projection adapters get nothing. The branched DB has datoms but no vector or graph projection — queries against the branch are crippled. The substrate exposes the gap; the fix is one method on the existing protocol.

### Move

Add ONE method to the existing `ProjectionAdapter`:

```python
class ProjectionAdapter(Protocol):
    def reset(self) -> None: ...
    def apply(self, datom: Datom) -> None: ...
    def fork(self, branch_id: str) -> "ProjectionAdapter": ...   # NEW
```

`fork()` returns a NEW adapter pointing at a fresh sink (new Qdrant collection name, new Kuzu DB path, new in-memory dict). The branched DB calls `rebuild(branched_db, forked_adapter)` to populate the fresh sink from the seed datoms. **The existing `rebuild()` function does the work; `fork()` just allocates the new sink.**

### `fork()` semantics in v0.4.0a1

- **Returns a fresh adapter with empty sink state.** Caller is responsible for calling `rebuild(branched_db, forked_adapter)` to populate the fork before querying through it.
- **`branch_id`** is a caller-provided identifier (UUID, datom hash, opaque string) used by backend implementers to derive collision-free physical target names. The substrate doesn't constrain its shape.
- **Existing adapters that don't implement `fork()`** get a default impl that raises `NotImplementedError("fork() required for branched projection")`. Adapters that don't branch never see this.
- `DictProjection.fork(branch_id)` test impl ships in v0.4.0a1.

### Files touched

- `fact/projection.py` (~15 lines: Protocol method + `DictProjection.fork()` impl + default `NotImplementedError` for adapters that opt out)

### Tests

- `tests/fact/test_projector_fork.py` (new): `DictProjection.fork()` returns fresh empty adapter; rebuild on a forked adapter doesn't touch the parent's sink; existing `DictProjection` adapters without `fork()` raise the documented error.

### Backend conformance prose (for ai-box engineers, NOT enforced by substrate tests)

When implementing `fork()` for `QdrantProjectionAdapter`, `KuzuProjectionAdapter`, etc.:

- `fork()` MUST allocate a fresh physical target (new collection, new DB path) — never share state with the parent.
- The fresh target's name SHOULD encode `branch_id` (e.g. `qdrant_collection_<branch_id>`) for traceability.
- `apply(d)` on the forked adapter MUST write to the new physical target, NOT the parent's.
- Re-projection upgrades (new embedding model) use the same pattern: allocate fresh target, `rebuild(db, fresh_adapter)`, atomically swap (Qdrant `update_aliases`), drop old.
- Conformance is verified in ai-box's own test suite via integration tests against real Qdrant/Kuzu.

### Wall-clock

~3 days.

## 7. Phase D — Typed `Provenance` + `causal_history`

### Problem

`Datom.provenance` at `datom.py:55` is `dict = field(default_factory=dict)` — free-form. But it's already carrying load-bearing structured keys:

- `provenance["source"]` — used by `__post_init__` keyword-strip (datom.py:92-104)
- `provenance["superseded_by_tx"]` — written by `db.py:192` for Phase-1 companion-retracts
- `provenance[":prev-hash"]` — written by `effect/handlers/audit.py:610` for the Merkle chain
- `TX_PLACEHOLDER` sentinel + rewriter at `store.py:386-412` for forward-references

The schema is already structured by convention; it's just untyped. And `fact/db.py:258` already ships `DB.history(e)` — entity-scoped walker over all datoms for `e`. The Phase D work is **type the conventional schema + add a transitive walker over `history(e)`.** No new module.

### Move D.1 — Type-tighten `Datom.provenance`

Replace `provenance: dict` with `provenance: Provenance` where `Provenance` is a `TypedDict(total=False)` with named keys plus an `extra: dict` overflow:

```python
from typing import TypedDict

class Provenance(TypedDict, total=False):
    source: str                          # required by __post_init__ (existing)
    tx_time: str                         # ISO-8601, derived; existing wire shape
    handler_id: str | None               # registered effect-handler name (Ferrari/brains pin)
    canonical_call: str | None           # sha256 of canonical EDN of {model, prompt, tools, params}
    parent_provenance_hash: str | None   # walks the causal DAG backwards (== existing audit :prev-hash)
    superseded_by_tx: int | None         # existing Phase-1 companion-retract key
    extra: dict                          # backend-specific overflow (Qdrant payload bits, etc.)
```

Plus a coercion classmethod `Provenance.from_dict(d: dict) -> Provenance` that lifts known keys and dumps the rest into `extra`. Existing dict callers keep working unchanged via this coercion.

**Equivalence of `parent_provenance_hash` and audit `:prev-hash`:** they ARE the same chain. The audit handler at `effect/handlers/audit.py:610` writes `:prev-hash`; Phase D adds an aliasing layer so both names read through to the same field. Single source of truth.

**Model-agnostic by construction (Ferrari/brains pin):** no `:provider` field, no `:api-key-suffix`. `handler_id` names the registered handler (swappable). `canonical_call` is the prompt-output identity hash.

### Move D.2 — `DB.causal_history(e, max_depth=16)`

Extends `DB.history(e)`. For each datom returned by `history(e)`, follow `parent_provenance_hash` backwards. Returns a `CausalDAG`:

```python
@dataclass(frozen=True)
class CausalDAG:
    seeds: list[Datom]                       # the history(e) result
    parents: dict[str, list[str]]            # datom_id → [parent_datom_id, ...]

def causal_history(self, e: str, max_depth: int = 16) -> CausalDAG:
    """Walk the provenance parent chain from every datom of e.

    Cycle-safe (visited set), depth-bounded. Returns a DAG that
    callers can render as /vault/why or fold into a parent-graph.
    """
```

The walker uses the existing `prev_hash` semantics from `effect/handlers/audit.py`, no new chain mechanism. Cycle detection via visited set + depth cap (default 16) — protects against malformed chains.

### `/vault/why` stays out of substrate

The `/vault/why` ai-box endpoint (Memory Palace concern) is a downstream consumer of `causal_history()`. It calls `db.causal_history(e)` and renders. **Not Phase D scope.**

### Files touched

- `fact/datom.py` (~30 lines: `Provenance` TypedDict + `from_dict` coercion + one-line type swap on the field)
- `fact/wire.py` (~10 lines: known-fields lift in `_provenance_to_wire`/`_provenance_from_wire`, lossless for `extra`)
- `fact/db.py` (~40 lines: `causal_history` method + `CausalDAG` dataclass)

`fact/store.py` is NOT touched — the SQLite encoding goes through `json.dumps(d.provenance)` and `json.loads(row["provenance"])` which work on the typed dict identically (TypedDict serialises as plain dict).

### Tests

- `tests/fact/test_provenance_schema.py` (new): typed `Provenance.from_dict()` roundtrip preserves all keys; canonical form (`_canonical.py` output) unchanged from v0.3.0a1; existing fixtures pass unchanged.
- `tests/fact/test_causal_history.py` (new): chain walks; cycle detection (synthetic cycle in fixture); depth-cap behavior; integration with `audit.py`'s `prev_hash` chain (cross-module test).

### Wall-clock

~3 days.

## 10. Testing strategy

- **Zero-regression rule:** every test green at v0.3.0a1 must stay green at v0.4.0a1 head BEFORE any new test lands. Bisect any regression to fix point.
- **Per-phase unit tests** (Sections 4, 6, 7).
- **Integration test** at end of branch: build a DB with `Dispatcher`-driven effect handlers writing typed `Provenance`; `db.transact()` + `db.branch()` + `forked_adapter = adapter.fork(branch_id)` + `rebuild(branched_db, forked_adapter)`; walk `causal_history` across the branch boundary; assert clean isolation between parent and branch projection sinks.
- **Property tests** (where they earn keep, Hypothesis `max_examples=200`):
    - `Provenance.from_dict` roundtrip preserves canonical hash for arbitrary dict input
    - `causal_history` is cycle-safe under random chain shapes (including self-loops, mutual cycles, deep chains)
    - `Dispatcher` dispatch order == `walk()` traversal order
- **Backend conformance prose** in this doc (§6) — enumerates what ai-box `QdrantProjectionAdapter.fork()` etc. must verify. Not enforced by substrate tests; documented obligation.

## 11. ARIS plan

- **Single gate at end of branch.** Three reviewers from the existing ARIS framework, run in parallel: **R1 Correctness + R2 Rigor + R3 Composability**. **R4 Research explicitly skipped — no paper claim ships in this branch.**
- **R3 Composability brief is extended** to explicitly cover (a) "does this design constrain future paper claim work likely to happen in NeSy 2027 (Prop 6 'Synthesis as Projection')?" (cheap insurance against future regret) and (b) "exercise all new APIs with non-default factory kwargs, not just default-path tests" (closes the composition-path-defect pattern that hit Phase 1 Round 5: W-polish2 → W-polish3 cycle).
- **Pass criteria:** mean ≥ 8.5, min ≥ 7.5 (precedent: v0.2.0a1 / v0.3.0a1 gates). Note: Phase 1 final freeze targeted min ≥ 9.2 — that was for clean-freeze before NeSy paper ship; v0.4.0a1 is engineering-only, the lower min ≥ 7.5 is the right bar.
- **One fix-pass budgeted.** If second-round mean < 8.5, **scope cut** — drop the lowest-rated phase, re-run gate. **No third round.** This is the budget-discipline lesson from the parked design.
- **Distinction from the parked-design review pattern:** that was a *design* review (pre-impl) with two parallel reviewers (substrate-fit + paper-claim) by virtue of the design proposing both substrate primitives and a paper claim. v0.4.0a1's ARIS gate is a *build* review (post-impl), so the standard three-reviewer R1+R2+R3 framework applies.
