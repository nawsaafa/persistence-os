# Persistence OS v0.5.0a1 — Module 5: Txn (transactional commit over the datom log)

> **Design doc** — revised 2026-04-27 after deep code scan + Clojure / Datomic / Python-STM research.
> **Branch:** cuts from `v0.4.0a1` tag (`bce93da`).
> **Hard cutoff:** 2026-05-15 (paper-freeze gate; NeSy abstract 2026-06-09 / paper 2026-06-16).
> **ARIS gate:** R1 (design) + R2 (full surface) + R3 (paper-fitness). R4 skipped (no paper claim).
> **Design constraint:** `PLAN_CANONICAL_VERSION` stays at 1. Zero proposition impact (Prop 1–5 unchanged).

## 0. What changed in this revision

This spec was originally framed as "Clojure-shaped STM" (refs + `dosync` + `commute` + `ensure` + `tx.effect`, ~600–800 LoC, 5 phases). Deep review surfaced four substantive issues:

1. **The framing is misleading.** Datomic itself does not do MVCC compare-on-commit; it serializes writes through a single transactor. Our Store layer (`BEGIN IMMEDIATE` on SQLite, `threading.Lock` on InMemoryStore) is the same. **STM here gives correctness, not throughput.**
2. **Aliased mutation is a Python-specific landmine** that doesn't exist in Clojure (which has persistent datastructures natively). Ref values must be immutable; we mandate `pyrsistent` or frozen built-ins.
3. **`commute` is an optimization, not a semantic.** At our 10–100 tx/sec workload there is no observable contention; `alter` covers everything. Subtle two-phase eager-at-body / re-apply-at-commit semantics are easy to ship buggy.
4. **`ensure` is rare and easy to add later.** Write-skew is a real hazard but no current AI Box surface needs it.

**v0.5.0a1 cut shrinks to ~300–500 LoC, 3 phases.** Commute, Ensure, Atoms, Agents all defer to v0.5.1+. This is a paper-protective release that lands cleanly with comfortable margin.

## 1. Goal & framing

Persistence OS today ships four-of-six Clojure design pillars: immutable values + reified time (`fact`), code-as-data (`plan` parse-side), boundary contracts (`spec`), and algebraic-effect handlers with audit (`effect`), plus counterfactual replay (`replay`) and v0.4.0a1 substrate primitives (Dispatcher, Provenance, fork, causal_history). The two missing pillars are:

1. **Transactional shared state**
2. **Live-programming surface** (REPL — Module 7)

This design covers the first. v0.5.0a1 ships **atomic multi-datom commit + snapshot-read isolation + retry-safe effects** — the correctness primitive that makes Adaptive Trader v2's parallel signal aggregation, GuestFlow's atomic conversation+escalation update, BankabilityAI's memo+status pair, Memory Palace's `/vault/remember`, and Trottinette's quote aggregation all atomic and replayable, without modifying any other module.

**Honest framing — what STM means here:**
- **Like Clojure**: refs + `dosync` API surface; effects-as-intents queued and atomically replayed at commit
- **Like Datomic**: single-writer at the Store, immutable read snapshots, log-as-source-of-truth
- **Unlike Clojure**: no parallel-writer concurrency, no TVal history tuning, no read-locks
- **Unlike Datomic**: explicit retry-on-conflict (because we expose pre-commit reads to user code)
- **What we get**: atomicity, snapshot isolation, retry-safe effects, and counterfactual replay across transactions
- **What we don't get**: throughput beyond what the Store's writer-lock allows (~hundreds–low thousands of tx/sec on SQLite)

**Why this release is paper-protective.** STM is orthogonal to Propositions 1–5 — branch (Prop 1), well-formedness (Prop 2), tamper-evidence (Prop 3), plan AST (Prop 4), and replay no-op (Prop 5) all hold without modification. The next paper revision (v0.9, post-NeSy notification) gets one new sentence noting the v0.5 extension; R3 verifies zero proposition impact.

## 2. Architecture

`persistence.txn` is a single new module added alongside `fact`, `effect`, `spec`, `replay`, and `plan`. It does not modify any existing module. Only `fact.db.DB` gains four new methods (`ref`, `new_ref`, `dosync`, `dosync_decorator`) added in a thin `_db_extension.py` that imports at module init.

**Runtime model:**
- A **`Ref`** is an immutable handle to an entity-id `(eid, db_id_or_branch_id)` with `__hash__` and `__eq__` over `eid`.
- **Ref values must be immutable.** Accepted at write time: `pyrsistent.PMap` / `PVector` / `PSet`, `frozenset`, `tuple`, `str`, `int`, `bool`, `float`, `Decimal`, frozen dataclasses. Rejected with a clear error: `dict`, `list`, mutable dataclass instances. The substrate enforces this at `tx.assoc` and `tx.alter` to prevent the aliased-mutation landmine that Python's lack of persistent datastructures otherwise creates.
- A **`Transaction`** is the object passed into a `dosync` body. It captures:
  - `t_start` — snapshot tx-time captured at body entry via `db._clock()` (frozen for the body so `datetime.now()` calls behave consistently across retries)
  - `read_set` — refs the body has dereferenced
  - `write_set` — pending `tx.assoc` / `tx.alter` writes (dict)
  - `effect_intent_log` — effect calls queued during the body via `tx.effect`
- On exit, the transaction validates against the current log (any datom on `read_set ∪ write_set.keys()` with `tx_time > t_start`?). If clean, it spec-validates the write_set, atomically applies write_set + commit datom as one `db.transact()` call, then replays `effect_intent_log` through the existing handler stack with results pinned to `commit_id`. If dirty, it discards the changeset, increments retry count, re-snapshots, and re-runs the body.

**Five invariants the protocol enforces:**

1. **Snapshot isolation.** Every `tx.deref` reads from `db.as_of(t_start)` — no in-flight writes are visible.
2. **Read-write conflict detection.** Any datom on a read-set entity with `tx_time > t_start` triggers retry. **This naturally handles write-skew without needing `ensure`** (every read is in the read-set).
3. **Atomic multi-datom commit.** `write_set` becomes a single `db.transact()` call → one `BEGIN IMMEDIATE` → one tx-id.
4. **Effect-intent atomicity.** Effects fire only after commit succeeds. If the commit fails, no effect ever fired.
5. **Replay determinism.** Bounded-retry `dosync` blocks are byte-identical across two runs of the same body. Deadline-using transactions get tagged `:txn/non-deterministic-retry true` so replay tooling can flag them.

**Two design wins from the substrate:**

- **Snapshot reads cannot fault.** Clojure's `Ref` carries a 10-cell TVal history ring buffer (`Ref.java:47-71`) because in-memory refs need to support snapshot reads without storage. Our log is unbounded — the value of any ref at any historical `tx_time` is always recoverable. We don't need `minHistory`/`maxHistory`/self-tuning. The log already does the work.
- **No-GIL forward-compatible.** Free-threaded CPython (PEP 703 / 779, 3.13 experimental → 3.14 supported) is real. Every STM data structure mutation in this design is guarded by an explicit lock (`Store._lock` for the log, `Transaction._mutex` for in-flight state) — no reliance on GIL-as-implicit-mutex.

All transactional state changes are themselves datoms. Replay, `branch()`, `fork()`, and `causal_history()` work on transactional state without modification — refs are just entities, transactions are just multi-datom transactions with a coordination protocol on top.

## 4. Data model & datom shape

Two new datom attribute clusters ship in v0.5.0a1, all under the `:persistence.txn/*` namespace. **No changes to existing attributes.** `PLAN_CANONICAL_VERSION` stays at 1.

### 4.1 Per-commit datom (one per successful `dosync`)

```
:persistence.txn/commit-id              UUID,    identity
:persistence.txn/started-at             inst                  (t_start snapshot)
:persistence.txn/committed-at           inst                  (commit tx-time)
:persistence.txn/retry-count            long                  (attempts before success)
:persistence.txn/read-set               [ref :many]           (entity-ids dereferenced)
:persistence.txn/non-deterministic-retry boolean              (true iff deadline= was used)
:persistence.txn/intent-log             [ref :many]           (refs to post-commit audit-chain entries; populated AFTER intents fire)
```

### 4.2 Per-write datoms (one per `tx.assoc` / `tx.alter`)

Regular entity datoms as you'd write through `db.transact()` today, with one extra attribute pinning them to the transaction:

```
:persistence.txn/commit                 ref → commit-id       (which transaction wrote this)
```

### 4.3 Effect-intent envelopes

Not separate datoms during the body. They fire through the existing `effect` handler stack at commit-time as if they were called directly, with their resulting audit-chain entries pinned to the commit-id via a new audit field:

```
:effect/txn-commit                      UUID | nil            (commit-id this effect fired under)
```

This is a single new field on the existing `AuditEntry` schema; it stays nullable for backwards-compat with non-transactional effect calls.

### 4.4 Spec registration

All `:persistence.txn/*` attributes register in the existing `spec` registry at module init, alongside `:persistence.plan/node` etc., to keep parse-don't-validate discipline.

## 5. Public API contract

```python
# Constructors
db.ref(eid: str) -> Ref                                # handle to existing/future entity
db.new_ref(initial: Any | None = None) -> Ref          # uuid7-allocated; optional initial assoc
                                                       # initial value must be immutable

# Transaction entry — both forms
with db.dosync() as tx: ...                            # context-manager sugar
with db.dosync(deadline=5.0) as tx: ...                # opt-in wall-clock deadline
with db.dosync(max_retries=512) as tx: ...             # override default retry cap (256)

@db.dosync                                             # decorator — canonical retryable form
def transfer(tx): ...

@db.dosync(deadline=2.0, max_retries=512)              # decorator with options
def transfer(tx): ...

# Transaction body API
tx.deref(ref: Ref) -> Any                              # snapshot read; adds to read_set
tx.assoc(ref: Ref, value: Any) -> None                 # queue write of a NEW value (replaces)
tx.alter(ref: Ref, fn: Callable, *args) -> Any         # queue write where new = fn(snapshot, *args)
                                                       # adds ref to read_set (read-write conflict)
                                                       # returns the computed new value so the body can use it
tx.effect(op: str, **kwargs) -> None                   # queue effect intent for commit-time fire
tx.now() -> datetime                                   # frozen t_start; deterministic across retries

# Errors (all subclass TxnError)
class TxnRetryExhausted(TxnError): ...                 # bounded retry cap hit
class TxnDeadlineExceeded(TxnError): ...               # wall-clock deadline elapsed
class RefBranchMismatch(TxnError): ...                 # ref from another DB used here
class RefValueNotImmutable(TxnError): ...              # tx.assoc / tx.alter / new_ref called with mutable value
class EffectInIoBlock(TxnError): ...                   # raw effect call inside dosync (use tx.effect instead)
class NestedDosyncNotSupported(TxnError): ...          # nested dosync — explicit error in v0.5.0a1
```

### 5.1 `tx.alter` semantics

`tx.alter(ref, fn, *args)` is Clojure's `alter` — read-then-write. Equivalent to:
```python
v = tx.deref(ref)
new_v = fn(v, *args)
tx.assoc(ref, new_v)
return new_v
```
Adds the ref to `read_set` (so concurrent writes trigger retry). Returns the computed new value so the body can use it. The fn must be deterministic (same inputs → same output) for replay to be byte-identical; this is documented but not statically verified.

### 5.2 Effect intent shape

Effect intents capture `(op, kwargs)` plus the `(handler_id, canonical_call)` Provenance pair from v0.4.0a1. On commit, intents replay through the existing handler stack via `runtime.perform(op, **kwargs)`; on retry they're discarded.

Inside the body, calling raw `effect.perform(...)` (without `tx.effect`) raises `EffectInIoBlock`. Enforcement is via a `ContextVar` set on `dosync` entry — async-task-local for free, no `threading.local` (matches the existing `effect/runtime.py:254` `_active` pattern).

### 5.3 Immutable value enforcement

Accepted at `tx.assoc` / `tx.alter` / `db.new_ref`:
- `pyrsistent.PMap`, `PVector`, `PSet`
- `frozenset`, `tuple`
- `str`, `int`, `bool`, `float`, `Decimal`, `bytes`
- `None`
- frozen dataclass instances (checked via `dataclasses.fields(type(v))[0].metadata` and `getattr(type(v), '__dataclass_params__', None).frozen`)

Rejected with `RefValueNotImmutable("ref values must be immutable; got <type> — use pyrsistent.PMap / PVector / PSet for collections")`:
- `dict`, `list`, `set`
- mutable dataclass instances
- arbitrary objects with `__dict__` mutation possible

Helper: `persistence.txn.freeze(value) -> Any` accepts a dict/list and returns the pyrsistent equivalent for migration ergonomics.

## 6. Conflict-detection & commit protocol

```python
def _run(db, body, max_retries, deadline):
    attempt = 0
    deadline_at = time.monotonic() + deadline if deadline else None
    while True:
        if attempt >= max_retries:
            raise TxnRetryExhausted(...)
        if deadline_at and time.monotonic() >= deadline_at:
            raise TxnDeadlineExceeded(...)

        t_start = db._clock()                       # frozen for the body
        tx = Transaction(db=db, t_start=t_start, attempt=attempt)
        try:
            with _txn_context(tx):                  # ContextVar guard — blocks raw effect.perform()
                body(tx)
        except TxnRetryRequested:
            attempt += 1
            continue
        # Other exceptions escape — no commit, no retry, no effects fire.

        # Commit gate 1: spec validation BEFORE allocating tx-id (Clojure LT:310-314 pattern)
        for ref, value in tx.write_set.items():
            spec.conform(ref.spec_attr, value)      # raises if invalid; no tx burned

        # Commit gate 2: MVCC conflict check
        touched = tx.read_set | set(tx.write_set.keys())
        conflicts = db.query_datoms_since(t_start, entity_ids=touched)
        if conflicts:
            attempt += 1
            continue

        # Apply atomically as one db.transact()
        datoms = tx._build_write_datoms() + [tx._build_commit_datom()]
        db.transact(datoms)                         # single tx, single tx-time

        # Replay effect intents through real handler stack — pinned to commit_id
        for intent in tx.effect_intent_log:
            runtime.perform(intent.op, _txn_commit=tx.commit_id, **intent.kwargs)
        return tx.commit_id
```

**Defaults:**
- `max_retries=256` (was 32 in earlier draft — Clojure's is 10000, but Python is ~10× slower; 256 is a sane middle)
- `deadline=None` (bounded-retry only by default)

**Exceptions:** any exception escapes the `dosync` immediately; no commit, no retry, no effects fire.

**Critical implementation note:** `db.transact()` itself does an O(N) `_find_prior_assert` per fact (db.py:188). A 50-write `dosync` would pay 50 × O(N). **Phase E ships an optimization**: a `db.transact_batch(datoms)` variant that does the prior-assert lookup in one pass. Without this, large transactions degrade quadratically.

## 10. Deferred to post-freeze

Explicitly out of scope for v0.5.0a1.

| Deferred | Target tag | Scope sketch | Why deferred |
|---|---|---|---|
| **`tx.commute` (commutative writes)** | v0.5.1 | Two-phase eager-at-body / re-apply-at-commit semantics; commute-fn registry; starter set (`inc-by`, `sum-into`, `set-union`, `dict-merge-shallow`) | Optimization for contention we don't have; subtle ordering semantics easy to ship buggy |
| **`tx.ensure` (read-set padding)** | v0.5.1 | Commit-time validator pattern (re-read ensured ref, assert tx-time unchanged) | Write-skew already prevented by read-set conflict detection; needed only for refs you depend on but don't deref |
| **Atoms (single-cell CAS)** | v0.5.2 | `db.atom(eid)` + `swap!`-like API; degenerate case of refs with lock-free fast-path | Ergonomic shortcut, not a missing primitive |
| **Agents (async ordered single-cell)** | v0.5.3 | `db.agent(eid)` + `send` / `send-off`; Clojure's async-ordered model | Async model; clean post-paper sprint |
| **Nested `dosync`** | v0.5.4 | v0.5.0a1 raises `NestedDosyncNotSupported`; choice between flatten-into-outer or keep prohibited | Needs design pass |
| **Multi-process STM** | v0.6.x | Postgres SERIALIZABLE-backed; not a Python library, a deployment model | Requires Postgres-backed Store; not load-bearing for any current AI Box surface |
| **`pyrsistent`-free mode** | maybe never | Allow user to opt-in to "I promise my values are immutable" via decorator | YAGNI; pyrsistent dep is small |

### 10.1 Commute sketch (v0.5.1)

```python
counter = db.ref(":metrics/login-count")
db.new_ref(initial=0)
with db.dosync() as tx:
    new_v = tx.commute(counter, lambda v, n: v + n, 1)   # runs eagerly, returns optimistic value
    # at commit: re-applies fn against latest committed value (Clojure LT.doCommute:467-491)
```

Two-phase semantic: fn runs in body so subsequent reads see optimistic value, AND at commit against latest committed. fn must be commutative; substrate does not verify (matches Clojure). Curated registry of pre-approved commute fns ships alongside.

### 10.2 Ensure sketch (v0.5.1)

```python
with db.dosync() as tx:
    bal_a = tx.deref(account_a_ref)       # in read_set
    tx.ensure(account_b_ref)              # in ensure_set; not deref'd, but conflict-checked
    tx.assoc(account_a_ref, bal_a - 100)
```

Implementation: at commit, re-read ensured refs from log, assert tx-time unchanged, retry if not. Works correctly under GIL where Clojure's read-lock approach degrades.

| Aliased mutation of ref values bypasses STM | **Critical** | Mandatory pyrsistent / immutable types at construction; `RefValueNotImmutable` raised loudly on `tx.assoc`/`tx.alter`/`new_ref` with mutable input |
| Concurrency-bug class hides in retry semantics | High | Hypothesis property test for replay byte-identity at `max_examples=200`; INT phase exercises real-thread parallel transactions |
| Effect-intent atomicity violation under retry | High | `ContextVar` guard raises `EffectInIoBlock` on raw-`runtime.perform()` calls inside `dosync`; tests assert "0 intents fire on conflict-failed commit"; audit-chain pinning via `:effect/txn-commit` |
| **InMemoryStore O(N) `since()` makes STM unusable for hot paths** | High | Document explicitly that production deployments require SQLiteStore (or future PostgresStore); InMemoryStore is for tests only. Add benchmark in INT phase showing 100K-datom log on InMemory degrades sub-linearly |
| **db.transact() quadratic in fact count due to per-fact O(N) auto-retraction** | High | Phase E ships `db.transact_batch()` that does the prior-assert lookup in one pass over the log |
| **Long-lived branches with many `dosync` calls leak via copy.deepcopy** | Medium | Document "branches must be short-lived"; add note to §6; `branch_lifecycle.md` doc to write |
| Schedule slip past 2026-05-15 | Low | ~3 weeks margin (today 2026-04-26); fallback: drop `tx.alter` from Phase B → user uses `tx.deref` + `tx.assoc` manually; rare but possible escape valve |
| Paper revision (v0.9) introduces proposition wording drift | Low | R3 passes BEFORE tag; no v0.9 paper revision until post-NeSy notification |

## 16. References (research informing the design)

- **Clojure STM** — `LockingTransaction.java` (lines 25, 74-78, 107-111, 127-159, 192-203, 282-285, 297-301, 310-314, 388-389, 397-422, 438-465, 467-491); `Ref.java:47-71`. The 10-cell TVal history ring buffer we DON'T need.
- **Datomic** — single-writer transactor model (Hickey, "Deconstructing the Database"); `:db/cas` is a guard inside a serialized critical section, not parallel-writer concurrency; throughput caps at hundreds–low thousands of tx/sec on best hardware (Halloway, Clojure/conj 2016).
- **PEP 703 / 779** — free-threaded CPython; design must be no-GIL-safe even on GIL builds.
- **PEP 684** — sub-interpreters; STM lives entirely inside one interpreter, cross-interpreter is message passing.
- **`pyrsistent`** — mature, maintained, the de facto Clojure-on-Python answer for immutable collections.
- **PostgreSQL SERIALIZABLE (SSI)** — Ports & Grittner — the right answer for multi-process STM (deferred to v0.6.x).
- **Harris/Marlow/Peyton-Jones, "Composable Memory Transactions" (PPoPP 2006)** — the `dosync`/`retry` shape Hickey credits, also lineage of our design.
- **Kung & Robinson, "On Optimistic Methods for Concurrency Control" (1981)** — original OCC, our MVCC-on-tx_time model is the bitemporal-log specialization.
