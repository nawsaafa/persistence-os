## Why this release exists

`CHANGELOG-txn.md` ships a "Design-vs-impl narrowings (rev O)" section enumerating
five items that were deferred from v0.5.0a1 to keep the tag inside the paper window.
v0.5.1 closes all five. None changes the propositions, the canonical-form, or the
external API shape; this is an additive / test-strengthening / refactor-cleanup pass.

Goal: ship v0.5.1 inside ~3 days so the paper window keeps ≥15 days of margin.

## Five carry-forwards

### N1 — read-set + intent-log emitted on commit datom provenance

**Status today** (`transaction.py:237-246`, `_db_extension.py:153-162`):

```python
db.transact_batch(
    facts,
    provenance={
        ":persistence.txn/commit-id": commit_id,
        ":persistence.txn/started-at": t_start.isoformat(),
        ":persistence.txn/committed-at": db._clock().isoformat(),
        ":persistence.txn/retry-count": attempt,
        ":persistence.txn/non-deterministic-retry": deadline is not None,
    },
)
```

Two boundary specs are registered (`:persistence.txn/read-set`,
`:persistence.txn/intent-log`) but never emitted on a commit datom. The design
note in `CHANGELOG-txn.md` § rev O says they're reconstructable from
`db.store.since(t_start)` and the audit chain — true, but only operationally; a
direct read would require running a full conflict scan. v0.5.1 emits them inline.

**Change:**

- `read-set` → list of eids (`sorted(r.eid for r in tx.read_set)`).
- `intent-log` → list of `{":op": str, ":kwargs": dict}` dicts (preserve order,
  same shape as `EffectIntent` round-tripped to wire form).

Both keys land under their already-registered colon-keyword names:
`:persistence.txn/read-set`, `:persistence.txn/intent-log`.

**Backwards compatibility:** purely additive on provenance. Existing v0.5.0a1
commits don't carry the keys; readers must `provenance.get(":persistence.txn/read-set")`
and tolerate `None`. The wire-roundtrip canonical-hash test (introduced for v0.4
typed Provenance) keeps passing because `Provenance` is a `total=False` TypedDict
and the test asserts hash-stability of *given* fields, not the absence of new
ones. PLAN_CANONICAL_VERSION stays at 1.

**Risk:** intent kwargs may contain non-EDN-serialisable values (datetime, custom
objects). v0.5.0a1 sidestepped this because intents stayed in-memory. v0.5.1
considered three options: (a) strict validation at `tx.effect()` call-site,
(b) soft-coerce on emit (lossy, silent), (c) strict validation at commit-time
only.

**Choice: (c) — strict at commit-time only.** Three reasons:

1. *Failure point matches blame point.* Intent-log appears on the commit
   datom's provenance, so the invariant is "no non-replayable provenance gets
   committed" — that's a commit-time invariant, not a call-time one. Raising
   at `tx.effect()` forces every caller to learn about EDN-conformance even
   when the intent might never reach commit (body raises, branch discarded).
2. *Keeps `tx.effect` cheap.* Validation at call-time would put a spec lookup
   on the hot path of every intent enqueue; option (c) runs N spec conforms
   once per commit, off the hot path, same total cost.
3. *Blast radius matches the real problem.* Option (a) breaks any caller who
   passes a `datetime` to `tx.effect()` even if the dosync never commits;
   option (c) only breaks committed transactions whose intent-log truly
   couldn't be replayed.

**Validation site:** in `_build_commit_provenance`, conform each intent's
`{op, kwargs}` dict against the registered `:persistence.txn/intent-log`
element shape before serialising into provenance. SpecError abort cleanly
unwinds the lock-held block (no datoms written, no commit_id burned, no
effect intent replay). The ContextVar guard cleanup runs in `finally`.

**Documentation pin (must land in `tx.effect` docstring):** "Validation of
kwargs against `:persistence.txn/intent-log` is deferred to commit time. A
`tx.effect()` call that succeeds may still cause the enclosing dosync to
raise `SpecError` at commit if the kwargs are non-EDN-conformant — failure
mode is commit-time, not call-time. Callers expecting eager validation
should conform their kwargs themselves before calling `tx.effect()`."

### N2 — AuditEntry `:effect/txn-commit` first-class field

**Status today** (`transaction.py:255-256`, `_db_extension.py:167-168`):

```python
for intent in tx.effect_intent_log:
    rt.perform(intent.op, {**intent.kwargs, "_txn_commit": commit_id})
```

The commit-id leaks into kwargs as `_txn_commit`, the audit handler reads it
from there, no schema field. This is brittle: a future kwarg name collision
silently corrupts the audit chain.

**Change:**

- Add `txn_commit: str | None = None` to the `AuditEntry` dataclass (placed
  after `parent`).
- Add `:effect/txn-commit` provenance key in `audit_entry_to_datom` (only when
  set; symmetric with `:episode`).
- Decode the key in `datom_to_audit_entry`.
- Change the `runtime.perform` signature to accept an optional `txn_commit`
  kwarg (NOT inside `args` dict). The plan-A intent replay becomes
  `rt.perform(intent.op, intent.kwargs, txn_commit=commit_id)`.
- For backwards compat: `runtime.perform` still strips a `_txn_commit` key out
  of `args` if present, lifts it to the `txn_commit` field, and emits a
  `DeprecationWarning`. Removed in v0.6.

**Risk:** the runtime signature touches a public-ish surface
(`persistence.effect.runtime.perform`). Need to grep callers — current
plus v0.5.0a1 tests — and confirm none pass `_txn_commit` from outside the
intent-replay path. Spot-check below shows the only producer is
`transaction.py:_run` and `_db_extension.py:_build_dosync_cm`.

### N3 — per-ref attribute spec (`ref.spec_attr`)

**Status today** (`transaction.py:128-148`, `ref.py:55-100`):

```python
WRITE_ATTR = "value"  # ref values stored under this attribute
...
def _spec_validate_writes(write_set: dict) -> None:
    keys = set(_registered_keys())
    spec_key = WRITE_ATTR
    if spec_key not in keys:
        return
    for _ref, value in write_set.items():
        result = _spec_conform(spec_key, value)
        ...
```

Every ref's value is validated against the global `:value` spec key — meaning
either every ref shares one spec or no specs apply. Registering
`:account/balance` separately doesn't fire on writes.

**Change:**

- Add `spec_attr: str = "value"` field to `Ref` dataclass (frozen, slotted).
  Equality stays over `(eid, db_id)` as today (spec_attr does not participate
  in eq/hash; it's a label, not part of identity).
- `db.ref(eid, spec_attr="account/balance")` and
  `db.new_ref(initial=..., spec_attr="account/balance")` accept the kwarg.
- `_spec_validate_writes` switches to per-ref lookup: for each `(ref, value)`,
  if `ref.spec_attr` is registered, conform; else skip. Default `"value"`
  preserves current behaviour for unannotated refs.
- `_build_write_facts` writes `a=ref.spec_attr` (datom strips leading colon, so
  `"account/balance"` lands as-is). Today every fact has `a="value"`; under
  N3 each ref's writes land at its own attribute name.
- `tx.deref` reads `ref.spec_attr` instead of the hard-coded `WRITE_ATTR`.

**Risk:** the Datom store doesn't enforce attribute-name shape beyond the
colon-strip rule. Ref attr names should be valid EDN keyword names (alphanum +
slash + dash) — validate in `__post_init__` of `Ref`. Default `"value"` survives
trivially.

**Backcompat:** existing v0.5.0a1 callers (no `spec_attr` arg) get the default
`"value"` and behave identically. Tests for the global `:value` spec keep
passing.

### N4 — Hypothesis `@given` byte-identity property at `max_examples=200`

**Status today**: `tests/persistence/txn/test_replay_byte_identity.py` is a
deterministic two-run structural comparison (one run, capture, second run,
assert equal). Catches non-determinism that's already on a covered path; misses
issues triggered only by rare value combinations.

**Change:** rewrite the test as a Hypothesis property:

```python
@given(
    transactions=st.lists(
        st.fixed_dictionaries({
            "ref_eid": st.text(min_size=1, max_size=12, alphabet=...),
            "value": st.recursive(st.integers() | st.text(...), ...),
        }),
        min_size=1,
        max_size=10,
    ),
)
@settings(max_examples=200, deadline=None)
def test_replay_byte_identity_property(transactions):
    db1 = DB(...); db2 = DB(...)  # two fresh DBs, identical clock
    apply_transactions(db1, transactions)
    apply_transactions(db2, transactions)
    assert canonicalise_log(db1.store) == canonicalise_log(db2.store)
```

**Risk:** Hypothesis runs are slower; 200 examples × ~50ms ≈ 10s test time. CI
budget allows. If shrinking lands at a pathological transaction list, mark
`xfail` and pin the input — replay byte-identity is a hard correctness invariant
and shrinks should genuinely indicate bugs.

**Decision pin:** keep the deterministic two-run test alongside the Hypothesis
one — it's a fast smoke; the Hypothesis test is the heavyweight backstop.

### N5 — helper-extraction refactor (CM/decorator commit-path dedup)

**Status today**: `_run` (`transaction.py:176-258`) and `_build_dosync_cm`
(`_db_extension.py:112-170`) duplicate ~40 LOC of commit logic:

- `_spec_validate_writes`
- Allocate `commit_id`
- Build `facts` from write_set + commit fact
- `with db.store._lock`: touch-set + `any_datoms_since` + `db.transact_batch`
- Set `tx.commit_id`
- Effect-intent replay

The CM differs from `_run` only in: (a) no retry loop, (b) raises
`TxnRetryExhausted` immediately if `any_datoms_since` returns True.

**Change:** extract `_commit_attempt(tx) -> bool` in `transaction.py`:

```python
def _commit_attempt(tx: Transaction) -> bool:
    """Run the spec-validate / facts-build / lock+conflict-check / transact
    sequence atomically. Returns True if committed, False if the conflict
    check rejected it (caller decides whether to retry or raise).

    On success: sets tx.commit_id and replays effect intents.
    """
    _spec_validate_writes(tx.write_set)
    commit_id = str(_uuid.uuid4())  # noqa: wall-clock
    facts = _build_write_facts(tx) + [_build_commit_fact(tx, commit_id)]
    with tx.db.store._lock:
        touched = {r.eid for r in tx.read_set} | {r.eid for r in tx.write_set}
        if any_datoms_since(tx.db, tx.t_start, touched):
            return False
        tx.db.transact_batch(facts, provenance=_build_commit_provenance(tx, commit_id))
    tx.commit_id = commit_id
    _replay_effect_intents(tx, commit_id)
    return True
```

`_run` calls it inside the retry loop. `_build_dosync_cm` calls it once and
raises `TxnRetryExhausted` on `False`.

`_build_commit_provenance` is the natural home for the N1 read-set/intent-log
addition.

`_replay_effect_intents` is the natural home for the N2 `txn_commit` field
plumbing.

**Net code change:** ~50 LOC removed from the two callers, ~70 LOC added in the
extracted helpers. Behavior unchanged on every existing test.

## Sequencing

Refactor first to consolidate the commit path; semantic changes ride on top:

1. **N5 — helper extraction.** Pure refactor, no test changes expected. Run
   suite green, commit. Forks into one place where N1+N2 will land.
2. **N1 — read-set + intent-log on provenance.** Adds two keys to the
   provenance dict in `_build_commit_provenance`; emits sorted-eids and
   wire-form intents. Adds `tx.effect()` validation against
   `:persistence.txn/intent-log`. New tests assert presence + shape.
3. **N2 — AuditEntry txn_commit field.** Adds dataclass field + wire-form
   round-trip + runtime signature change + back-compat shim. New tests for
   the field, the wire round-trip, and the deprecation warning on
   `_txn_commit` in args.
4. **N3 — per-ref spec_attr.** Adds Ref field + db.ref/new_ref kwarg + per-ref
   validation in `_spec_validate_writes` + per-ref attribute write in
   `_build_write_facts` + per-ref attribute read in `tx.deref`. Validates
   spec_attr shape in Ref.__post_init__. New tests for: default behaviour
   unchanged, custom spec_attr fires correct spec, two refs with different
   spec_attrs validate independently, mixed read/write across spec_attrs.
5. **N4 — Hypothesis property test.** Add new test alongside the existing
   deterministic one. Tune strategy until 200 examples runs in <15s.

Each step ships its own commit. Single subagent per step (parallelism doesn't
buy time on serial-dependent edits). Two-stage review per step (spec compliance
+ code quality) before moving on, per Phase B precedent.
