## 1. One-paragraph summary

Module 3 today ships the parse / walk / dispatch / coerce layers for content-addressed plan ASTs (931 LOC, 8 internal files including the v0.4 `_dispatch.py` seam). Stream A extends Module 3 with the **execution layer**: three new internal files (`_optimize.py`, `_skill_library.py`, `_promotion.py`) that turn a parsed Plan AST into a callable, optimizable, promotable artifact. MIPROv2 is the optimizer of record (DSPy-compatible wrapper); the skill library is content-addressed via the existing fact store; 4-gate promotion is the audit-recorded transition that lets a candidate plan enter the library. The design ships under the `Dispatcher` seam — execution handlers register against the same `:llm-call` / `:tool-call` / `:transact` / `:branch` tags. **Each new file is the smallest committable extension to the existing dispatcher, not a parallel system.**

---

## 2. Why this design

The roadmap pivot ([§3 Strategic frame](2026-04-27-persistence-os-v1.0-roadmap.md)) made Module 3 execution load-bearing for the v1.0 paper claim. Today the substrate parses plans, walks them, dispatches per-tag handlers, and coerces non-EDN attrs through the coercion registry. What it does **not** do:

1. **Execute a plan.** `Dispatcher.dispatch(node, env)` calls registered handlers in walk order but has no notion of "did the plan succeed?" or "what's the result?" — handler results are returned as a flat list. The execution layer needs a result envelope with success/failure semantics + provenance trail.
2. **Optimize a plan.** No prompt or hyperparameter tuning. MIPROv2 is the canonical answer (DSPy ecosystem), but no wrapper exists.
3. **Promote a skill.** No registry of "validated plans I trust." Without this, the skill library claim is vapor.

Plus one structural lesson from the v0.4 substrate-primitives redesign: **find the existing affordance before adding.** The dispatcher seam is already there. Don't add a parallel execution surface — extend the dispatcher.

### What this design does NOT do

- **No new dispatcher.** `Dispatcher` from v0.4 stays. Execution registers handlers against the same registry.
- **No replacement for `_walk.py`.** Walk order semantics (parent-before-children DFS) preserved. Execution layer is `walk → dispatch → result-collect`, not a re-implementation.
- **No coupling to a specific LLM.** `:llm-call` handler routes through `:llm/handler-id` registry; MIPROv2 calls the same registry. Model-pluggable.
- **No new canonical version.** `PLAN_CANONICAL_VERSION = 1` stays. Execution-layer additions are observational (provenance), not structural.
- **No MCTS.** Stream B ships MCTS as `v0.6.5` extension on top of this layer. This is the substrate; MCTS is the search.

| 5 | `src/persistence/plan/__init__.py` | re-exports + `__all__` extension | EXTENDED |

**Execution core (`_execute.py`) is split from optimizer (`_optimize.py`).** This is a code/logic boundary, not a stylistic preference: importing the optimizer pulls DSPy as an optional dep; the execution core must remain importable without DSPy installed (and without DSPy ever entering its import path). `_skill_library.py` and `_promotion.py` import only `_execute.py`, never `_optimize.py`. Test suite enforces: `python -c "import persistence.plan._execute"` succeeds with `dspy-ai` uninstalled.

**Plan-attr key style (key code↔doc rule).** All Plan-AST `Node.attrs` keys are plain strings without leading colon (verified `_ast.py:71-100`: leading `:` raises at construction). `Node.tag` keeps `:` prefix (`:llm-call`, `:transact`, `:branch`). All provenance attr-key examples in this design read as `"plan/optimizer"`, **not** `":plan/optimizer"`. Datom attribute names follow the same convention (`datom.py:114`: `"project/wacc"`-style). No new tag in Plan AST (Stream B handles MCTS via the `:branch` non-leaf path; Stream A adds **none**).

### Out of scope (explicit deferrals)

- **MCTS.** Stream B (`v0.6.5`).
- **Operator approval gate (G4) hard-promotion.** Stream A ships G4 as a **stub** that records the gate result as a fact `("plan/promotion-gate-g4", "stub")` and accepts. Stream D REPL replaces the stub with the `inspect` op's actual approval flow. Documented in [§7](#7-four-gate-promotion-criterion).
- **DSPy module wrappers beyond MIPROv2.** Boostrap, BootstrapFewShotWithRandomSearch, etc. all out of scope; only MIPROv2 ships in v0.6.0a1. Add as needed in v0.6.x point releases.
- **Multi-process skill library backed by Postgres.** Stays in-memory + fact-store-backed in v0.6.0a1. Phase 3 NeSy 2027 promotion.
- **Skill-versioning conflict resolution.** v0.6.0a1 uses content-addressed skill ids — re-registration of an existing skill is a no-op (same hash). Skill renaming / aliasing deferred.

## 4. Execution result envelope

### Problem

`Dispatcher.dispatch(node, env)` returns `list[Any]` — a flat list of handler returns in walk order. No success/failure semantics. No provenance. No way to say "the plan failed at node X."

### Decision

Wrap dispatch in an `execute()` function that returns `ExecutionResult`:

```python
@dataclass(frozen=True, slots=True)
class ExecutionResult:
    plan_id: str                          # content-addressed Plan AST root id
    status: Literal["ok", "failed"]
    leaf_results: tuple[LeafResult, ...]  # one per dispatched node, walk order; frozen
    failure: FailureInfo | None           # populated iff status == "failed"

@dataclass(frozen=True)
class LeafResult:
    node_id: str                       # content-addressed leaf id
    tag: str                           # ":llm-call" etc.
    handler_id: str                    # registered handler key
    result: Any                        # handler return
    # No `duration_ms` in v0.6.0a1 — wall-clock timing in handler code
    # violates the no-time.time() rule. Latency measurement, when needed,
    # routes through a `:clock/now` effect injected at handler boundary.
    # Add in v0.6.x once the effect-clock injection seam is specified.

@dataclass(frozen=True)
class FailureInfo:
    failed_node_id: str
    failed_tag: str
    error_class: str
    error_repr: str                    # __repr__, NOT str() — error_class + reproduction-ready
```

### Where it lives

`src/persistence/plan/_execute.py` — execution core. Always importable, no DSPy dependency. Houses `ExecutionResult` + `LeafResult` + `FailureInfo` + `execute(plan, dispatcher) -> ExecutionResult`. Imported by `_optimize.py`, `_skill_library.py`, `_promotion.py`.

`src/persistence/plan/_optimize.py` — optimizer wrapper. Lazy-imports DSPy at function-call time (kept out of module-import path). Calls `_execute.execute()` against the input plan to compute the baseline score and against the optimized plan to compute the final score. Never imported by `_skill_library.py` or `_promotion.py`.

Re-exported from `src/persistence/plan/__init__.py`:

```python
__all__ = [..., "ExecutionResult", "LeafResult", "FailureInfo", "execute", "optimize"]
```

### Walk-order failure semantics

If any handler raises, dispatch halts at that node, the partial leaf-results list is preserved, `status="failed"`, `failure` populated. Successor nodes are NOT executed (early-stop semantics). Audit log records the partial walk via Module 2's effect runtime — same as today's behavior.

## 5. The `optimize()` API

```python
def optimize(
    plan: Node,
    *,
    training_set: list[TrainingExample],   # canonicalization rule below
    metric: MetricRef,                     # registered metric, NOT a raw callable
    optimizer: str = "miprov2-v1",         # "plan/optimizer" provenance value
    miprov2_kwargs: dict | None = None,
) -> OptimizedPlan: ...
```

```python
@dataclass(frozen=True)
class OptimizedPlan:
    plan: Node                         # the optimized AST
    baseline_plan_id: str              # content-addressed root of input plan
    optimizer: str                     # "plan/optimizer" value
    optimizer_call_hash: str           # see "Canonical hash inputs" below
    score: float                       # final metric value on training_set
    score_baseline: float              # metric on input plan, same training_set
```

### Canonical hash inputs (deterministic by construction)

`optimizer_call_hash` is the SHA-256 of the canonical-JSON of:

```python
{
    "optimizer": optimizer,                         # "miprov2-v1" string
    "training_set_hash": _canonicalize_training_set(training_set),
    "metric": {"id": metric.id, "version": metric.version},
}
```

with three pinned canonicalization rules so the hash is stable across runs and machines:

1. **Training-set canonicalization.** Each `TrainingExample` is `{"inputs": dict, "expected": Any}`. `_canonicalize_training_set` (a) validates a fixed schema (only `inputs` + `expected` keys; `inputs` values are EDN-canonical-serializable per `persistence.spec`); (b) canonicalizes each example via the same canonical-JSON serializer used for audit chains (sorted keys, no whitespace, fixed numeric format); (c) sorts the canonicalized examples lexicographically so input ordering does not affect the hash; (d) returns SHA-256 of the joined byte stream. No Python-callable hashing — only canonical-JSON bytes feed the digest.
2. **Metric registry.** Metrics are registered by `(metric_id: str, version: str)`, not by source hash. `MetricRef = NamedTuple("MetricRef", id=str, version=str)`. The registry (`src/persistence/plan/_metric_registry.py` — small enough to live next to `_execute.py`) maps `(id, version) -> Callable[[ExecutionResult, dict], float]`. Re-registering an `(id, version)` pair raises `ValueError` unless `replace=True` (test-only). The registry is the canonical identity of a metric — caller cannot pass a raw callable into `optimize()`.
3. **Optimizer string.** Pinned name + version (e.g. `"miprov2-v1"`). Bumping MIPROv2 → `"miprov2-v2"` is an explicit string change, not an inferred Python-version-bump.

This makes optimizer calls deterministic-by-record: same inputs → same hash → same provenance. Reproducibility hook for paper §6 numeric tables. Not implementation-defined.

### MIPROv2 integration boundary

`_optimize.py` imports `dspy` at function-call time (NOT module level — keeps Module 3 importable without DSPy installed):

```python
def optimize(...):
    try:
        import dspy
        from dspy.teleprompt import MIPROv2
    except ImportError as e:
        raise OptimizerNotAvailable(
            f"MIPROv2 optimizer requires `dspy`. Install: pip install dspy-ai. ({e})"
        )
    ...
```

DSPy version pin: `dspy-ai >= 2.5.0` (MIPROv2 stable since 2.5). Add as **optional** dependency in `pyproject.toml`:

```toml
[project.optional-dependencies]
optimize = ["dspy-ai >= 2.5.0"]
```

CI suite: install `persistence[optimize]` for the `_optimize.py` test path; baseline suite stays clean of DSPy.

### What MIPROv2 sees

A Plan AST converted to a DSPy `Module` via a thin adapter (`_optimize.py:_plan_to_dspy_module`). The adapter:

1. Walks the AST.
2. For each `:llm-call` node, instantiates a DSPy `Predict` or `ChainOfThought` signature based on the node's attrs.
3. Returns a callable DSPy module whose `forward(**inputs)` runs the dispatcher and returns the final leaf result.

MIPROv2 then optimizes prompts + few-shot examples against the training_set + metric. After convergence, the adapter inverse-converts the optimized DSPy module back to a Plan AST (provenance-tagged with optimizer call hash).

### Provenance pinning

Every node in the optimized AST carries `Node.attrs` keys (plain strings, no leading colon — `_ast.py:71-100` rule):

```python
{
    "plan/optimizer": "miprov2-v1",
    "plan/optimizer-call": "<sha256-hex>",
    "plan/baseline": "<input-plan-id>",
    "plan/training-set-hash": "<sha256-hex>",
    "plan/metric-id": "<metric_id>",
    "plan/metric-version": "<metric_version>",
}
```

These keys live alongside any author-supplied attrs already on the node. They survive the inverse adapter (DSPy-module → Plan AST) by construction — the inverse adapter sets them on every emitted node before `Node(...)` is constructed.

## 6. The skill library

```python
class SkillLibrary:
    """Content-addressed skill registry backed by the fact store.
    Skills are tagged Plan ASTs whose `skill_id` is derived from the
    AST's content hash + a stable namespace prefix.
    """
    def __init__(self, db: DB) -> None: ...

    def register(
        self,
        plan: Node,
        promotion_record: PromotionRecord,
    ) -> str:
        """Register `plan` as a skill. Returns the skill_id.
        Re-registration of an existing skill_id is a no-op (same content hash).
        """

    def lookup(self, skill_id: str) -> tuple[Node, PromotionRecord] | None:
        """O(1) skill_id → (Plan AST, promotion record) lookup.
        Returns None if not registered.
        """

    def list_skills(self, *, tag_filter: str | None = None) -> list[str]:
        """Enumerate all registered skill_ids; optional filter by Plan AST root tag."""
```

### Storage shape

Each skill registration writes 3 datoms to the fact store. `Datom.a` is a plain namespaced string (`datom.py:114` convention — no leading colon):

```python
[
  Datom(e=skill_id, a="skill/plan",              v=plan_id,        ...),  # plan_id is content-addressed Plan AST root
  Datom(e=skill_id, a="skill/promotion-record",  v=promotion_id,   ...),  # reverse-lookup to gates
  Datom(e=skill_id, a="skill/registered-at",     v=clock_now,      ...),  # clock_now value comes from ":clock/now" effect at register-time
]
```

**Identity rule.** `skill_id` is an entity id string in the fact store (the `e` field), not a Plan id. It is derived as `"skill/" + first_16_hex_chars(plan_id)` so it is content-addressed via the Plan id but is its own namespace. The Plan id itself is stored in `v` of the first datom. A skill's content hash IS the skill — re-registering the same Plan returns the same `skill_id` (no-op).

### Why fact-store-backed

- Free for free: existing bitemporal indices, audit chain, replay byte-identity.
- Skill registration shows up in audit log as 3 facts → reconstruct skill library from any historical timestamp via `db.asOf(t)`.
- No new storage backend, no new schema migration.

## 7. Four-gate promotion criterion

A candidate Plan AST `P` enters the skill library iff all four gates pass:

### G1 — Replay byte-identity over held-out trajectories

`P` runs against ≥ N held-out test trajectories (default N=10, configurable). For each trajectory, the gate calls a `ReplayEngine` Protocol satisfying:

```python
@runtime_checkable
class ReplayEngine(Protocol):
    def replay(self, plan: Node, trajectory: Trajectory, /) -> Trajectory: ...
    def compare(self, factual: Trajectory, counterfactual: Trajectory, /) -> dict: ...
```

The Protocol shape allows Stream F's adapter to wrap `persistence.replay.replay/compare` (production wiring against the regulator-replay corpus) without import-order coupling, and lets tests inject deterministic stubs.

**Compare contract.** `compare()` MUST return a dict containing `"divergence_step"`. `None` is the byte-identity signal; any int is the first divergent step index. Any other key (`pnl_delta`, `kl_divergence`, …) is preserved by the engine for downstream analytics but ignored by G1. **A `compare` result missing `divergence_step` raises `TypeError`** (fail-closed against malformed engine wiring) — not silently treated as byte-identity.

**Empty corpus.** `held_out_trajectories=[]` returns False with a `UserWarning` ("no coverage to claim Prop 4") — vacuous truth is not accepted at a correctness gate.

**One divergent byte = G1 fail.** Coverage gate: fewer than `min_count` trajectories also returns False regardless of byte-identity.

Defends Prop 4 (Replay byte-identity): a promoted skill is replay-stable, period.

### G2 — Effect-handler audit-chain unbroken

The gate pulls `AuditEntry` instances from the fact store for the supplied `[start_ms, end_ms]` window via the `audit/` attribute prefix, reconstructs them through `_datom_to_wire_for_audit` (which requires `provenance[":signature"]` — set by `audit_entry_to_datom`'s contract), and delegates to `audit.verify_chain()`. The reconstruction round-trips the canonical `args_hash`, `result_hash`, `prev_hash`, etc. so `verify_chain()` re-derives each entry's content hash and matches it against the stored `id`. **Any chain break = G2 fail.** This catches handlers that emit unrecorded effects (the canonical replay-poisoning bug).

**Empty window.** Zero in-window audit entries returns False with a `UserWarning` ("no audit entries to verify Prop 2") — symmetric with G1's empty-corpus handling. A miscomputed window or a DB with no audit datoms cannot silently pass G2.

**Chain semantics.** `verify_chain()` requires the in-window entries to form a contiguous chain head-first (`prev_hash=None` on entry 0; each subsequent `prev_hash` matches the previous entry's `id`). Insertion order in `db.log()` matches chain order — no extra sort needed. Non-`audit/...` datoms in the window are filtered out by the prefix check.

Defends Prop 2 (effect handler audit invariant).

### G3 — MIPROv2 score improvement vs baseline ≥ ε

`P_optimized.score - P_optimized.score_baseline >= ε` where ε defaults to `0.05` (configurable). Skills that don't improve don't promote.

Defends the substrate's claim that optimization is meaningful — promoted skills are demonstrably better than the baseline plan they came from.

### G4 — Operator approval (REPL `inspect` flow)

**Stream A ships G4 as a stub:** `gate_g4_stub(record) -> {"approved": True, "approver": "stub", "rationale": "Stream A stub — Stream D replaces"}`. The promotion record carries the stub marker explicitly so Stream D's REPL `inspect` op can re-run G4 against any stub-promoted skill.

**Stream D replaces the stub** with: an operator opens REPL `inspect skill_id`, reviews G1/G2/G3 records + the Plan AST diff vs baseline, and either approves (writes a `("plan/promotion-gate-g4", "approved")` datom) or rejects (writes `"rejected"` + reason). Capability-gated: only operator tokens can write G4 facts.

Defends the substrate's claim that promoted skills are operator-vetted, not just metric-vetted.

### Promotion record shape

```python
@dataclass(frozen=True)
class PromotionRecord:
    candidate_plan_id: str
    g1_replay_byte_identity: bool
    g1_held_out_count: int
    g2_audit_chain_verified: bool
    g3_score_delta: float
    g3_score_threshold: float
    g4_approver: str                   # "stub" in Stream A, operator token id in Stream D
    g4_approved: bool
    g4_rationale: str
    promoted_at: int                   # :clock/now millis
    promotion_id: str                  # canonical hash of all gates + candidate_plan_id
```

**Persistence shape (Stream A).** The fact store records the `(skill_id → promotion_id)` link via the `skill/promotion-record` datom in §6, NOT the full PromotionRecord payload. The full record lives on the in-memory `SkillLibrary._records[promotion_id]` cache populated by `register()` and is returned by `lookup()` for byte-identity round-trip; it is **not** content-addressable from the fact store alone. Persistent cross-process reconstruction (i.e. pulling a complete PromotionRecord back after a process restart with the original Python objects) is Phase 3 NeSy 2027 scope (per `_skill_library.py` module docstring). Stream D's REPL `inspect` op can stub this for now by re-running G1/G2 against the recorded `(skill_id, promotion_id)` pair if the in-memory cache is cold; G3/G4 results are rebuildable from `(optimized_score, baseline_score, g4_*)` once those land as first-class audit entries (Stream G expansion).

## 13. Open questions

1. **MIPROv2 metric signature.** Current design: `Callable[[ExecutionResult, dict], float]` where dict is the training-example. Alternate: take a training-set-wide aggregate. Pinning to per-example for v0.6.0a1 (DSPy convention); if real benchmarks need aggregate, defer to v0.6.1.
2. **G3 epsilon default.** Picked `0.05` arbitrarily. Real-world MIPROv2 deltas at convergence often >> 0.05; could go to `0.02` for tighter gating. **Recommend keeping at 0.05** — over-permissive is fine for v0.6.0a1; tighten in v0.6.x once we have eval data.
3. **Skill library namespace.** `skill_id` is `"skill/" + first_16_hex_chars(plan_id)` — entity-id string in the fact store, content-addressed via Plan id. Alternate: human-readable slugs from optimizer training-set name. **Recommend content-addressed only for v0.6.0a1**; human-readable names are an aliasing concern, deferable.

These are not blockers. The design ships under the listed pins; revisions land in v0.6.x point releases.
