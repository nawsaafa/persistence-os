## 1. One-paragraph summary

Module 3 today (after v0.6.0a1) parses, walks, dispatches, executes, optimizes, and 4-gate-promotes a content-addressed Plan AST. **`optimize()` searches the demo + prompt space inside a fixed plan structure.** Stream B adds the **structure** search: a PUCT-style tree-search wrapper that, given an initial Plan AST, a problem (training set + metric), an `Expander` that proposes plan-mutating actions, and an `Evaluator` that scores leaf plans, returns the highest-reward plan it found and feeds it through the existing `optimize()` ŌåÆ `promote()` ŌåÆ `SkillLibrary` pipeline. The whole stream lands as one new internal file `_mcts.py` plus a small surface widening on `__init__.py`. Determinism: every LLM call (expander + evaluator) is recorded as a `:mcts/iteration` provenance datom keyed by `(search_id, iteration_index, plan_id)`, so the search trajectory is reconstructible from the audit log alone ŌĆö defended via the **search-internal Merkle chain** (`mcts/prev-hash`, see ┬¦13), distinct from the effect-handler audit chain (G2). Prop 6 rests on Prop 1 (datom append-only) plus the new Merkle chain, NOT on Prop 2 composition.

## 2. Why this design

The roadmap pivot ([┬¦3 ADR-2](2026-04-27-persistence-os-v1.0-roadmap.md)) reversed the prior `:branch MCTS ŌåÆ Phase 3 NeSy 2027` defer. The deferral was timing-driven, not technical, and the ferrari-first reframing makes the original Phase 3 timing irrelevant. Tree search over content-addressed Plan ASTs is the substrate's strongest **compositional planning** claim ŌĆö without it, the paper is "we have a Plan AST" plus "we can tune prompts inside it" but not "we can search over the space of plan structures." Module 3 execution-without-MCTS leaves the paper one proposition short.

What the existing Module 3 surface lets us assume:
- **Plan AST is content-addressed.** `Node.id` is a deterministic 32-hex sha256 of canonical-JSON (`_ast.py:111-148`). Any plan reachable by any action sequence collapses to the same `Node.id`. This is the affordance MCTS leans on for transposition.
- **`execute()` returns a typed envelope.** `ExecutionResult.status` Ōłł {"ok","failed"} + `LeafResult` per dispatched node + `FailureInfo` on raise (`_execute.py:117-139`). MCTS's deterministic-evaluator path is `execute(plan, dispatcher) ŌåÆ metric_fn(result, expected) ŌåÆ float`.
- **`optimize()` exists.** End-to-end MIPROv2 wrapper with provenance-pinned output (`_optimize.py:458-658`). MCTS terminates by handing the winner to `optimize()` for demo-search inside the chosen structure.
- **`promote()` orchestrates 4 gates.** `_promotion.py:663-851`. MCTS's terminal "register a winner skill" call is `promote(winner, ...) ŌåÆ PromotionRecord` then `SkillLibrary.register(winner, record, ...)`.
- **Fact store gives free provenance.** `db.transact([...], provenance={...})` writes datoms with bitemporal indices + audit chain (`fact/db.py:137`). One `:mcts/iteration` datom per iteration, content-addressed by search-id + iteration index, gives Prop 6 reconstructibility for free.

Plus one structural lesson preserved from v0.4 / v0.6.0a1: **find the existing affordance before adding.** `Dispatcher` + `execute()` + `optimize()` + `promote()` already exist; this stream is the **smallest committable extension** that wires them into a search loop, not a parallel system.

### What this design does NOT do

- **No new dispatcher.** The existing v0.4 `Dispatcher` is what evaluator-path executions run through. MCTS does not register handlers.
- **No replacement for `optimize()`.** MCTS calls `optimize()` on the winning structure as the closing step. The two are orthogonal: MCTS searches structure, `optimize()` searches demos within structure.
- **No new canonical-form.** `PLAN_CANONICAL_VERSION = 1` stays. All datoms added by MCTS are observational provenance, not structural.
- **No `:branch` tag executor.** Roadmap ┬¦3.3 contract option (a) (`:branch` as non-leaf structural tag with a registered dispatcher handler) is the path under which Stream B threads search results back into a runnable plan. **Roadmap ┬¦3.3 amended 2026-04-28 to defer `:branch` handler-registration from v0.6.5 to v0.6.6** ŌĆö see ADR-12 below. v0.6.5 ships `mcts_search()` returning a winning Plan AST; the caller substitutes it structurally into the candidate plan and dispatches as a normal non-leaf `:branch`. v0.6.6 wires the `:branch` handler that delegates to `mcts_search`.
- **No multi-process search.** Single-process, in-memory tree. Postgres-backed distributed MCTS is Phase 3 NeSy 2027 scope.
- **No live operator steering inside the search loop.** Stream D REPL ops (`inspect`, `rewind`, `branch`) operate on the audit log post-search; they do not interrupt an in-flight search.
- **No NeSy / symbolic constraint solver.** The `Expander` is LLM-backed in production but the contract is a Protocol; symbolic expanders are a v0.7+ topic.

## 3. Scope

One new internal file, three new Protocols, two new dataclasses, two new public functions, zero canonical-form change.

| # | File | Public API | Status |
|---|---|---|---|
| 1 | `src/persistence/plan/_mcts.py` | `mcts_search(...)`, `MCTSResult`, `Action`, `Expander`, `Evaluator`, `MCTSConfig` | NEW |
| 2 | `src/persistence/plan/__init__.py` | re-exports + `__all__` extension | EXTENDED |

**Module surface placement.** Single flat file `_mcts.py` next to `_optimize.py` and `_promotion.py`. **Not** a sub-module `persistence.plan.search/` ŌĆö at v0.6.5 size (one search algorithm, one expander Protocol, one evaluator Protocol) the sub-module overhead exceeds the win, and we want the import path symmetry with `_optimize.py` / `_skill_library.py`. If v0.7+ adds beam-search / iterative-widening / non-MCTS variants, **then** lift to `persistence.plan.search/` (ADR-11 below).

**Plan-attr key style (key codeŌåödoc rule, inherited from v0.6.0a1).** All Plan-AST `Node.attrs` keys are plain strings without leading colon (`_ast.py:71-100` rejects colon-prefixed keys at construction time). Provenance attr-key examples in this design read as `"mcts/search-id"`, **not** `":mcts/search-id"`. Datom `a` keys follow the same convention (`datom.py:114`: `"mcts/iteration"`-style ŌĆö bare, no leading colon at the storage boundary). Where the roadmap or sister docs write `:mcts/...` with a leading colon, that is the **wire form**; the on-disk Datom slot strips the colon. **Wire-form attribute keys after the namespace prefix use `kebab-case`, not `snake_case`** ŌĆö examples: `mcts/iter-index`, `mcts/winner-plan-id`, `mcts/prev-hash`, `mcts/inputs-hash`, `mcts/error-class`, `mcts/started-at`, `mcts/finished-at`, `mcts/terminated-by`. The `mcts/` prefix itself is `lowercase + slash`. This matches the `audit/...` and `skill/...` namespaces in the existing repo.

### Out of scope (explicit deferrals)

- **Distributed search.** Single-process. Multi-worker MCTS over Postgres-backed shared state ŌåÆ Phase 3 NeSy 2027.
- **Adaptive c_puct.** Static `c_puct` (default `1.4`) with one ADR (ADR-3) below. Adaptive schedules ŌåÆ v0.7+.
- **Custom rollout policies.** No "policy network" between leaf and evaluator. Evaluator is called directly on each newly expanded leaf ŌĆö i.e. **rollout depth = 1**. The Plan AST is the policy; MIPROv2 inside `optimize()` is the policy refinement step.
- **Symbolic / constraint-based expanders.** Out-of-scope concept; the expander Protocol shape is open enough that v0.7+ symbolic expanders satisfy it without a contract change.
- **Operator-in-the-loop steering inside a search.** Stream D REPL ops post-search only.
- **MCTS-aware reward shaping.** No multi-objective reward, no Pareto front. Single scalar reward in [0,1] (or higher-is-better float ŌĆö the substrate doesn't enforce a bound, but evaluator implementations should normalize).

## 4. Search-tree node identity vs Plan AST node identity

### Problem

`MCTSNode` carries search statistics (`visits`, `total_value`, prior, parent pointer, child dict). Plan `Node` carries content (tag, attrs, children) and content-addressed `Node.id`. They are **not the same object** ŌĆö one search tree may visit the same plan twice via different action sequences (transposition), and one plan node may appear in multiple search trees over the same problem.

### Decision (ADR-1)

**Search-tree nodes are keyed by Plan `Node.id`.** A transposition table maps `plan.id ŌåÆ MCTSNode`; reaching the same plan by two different action sequences updates the **same** `MCTSNode`'s statistics. This is the "graph MCTS" or "transposition-aware MCTS" variant ŌĆö standard in modern AlphaZero-style implementations and the natural fit for content-addressed plans.

```python
@dataclass(slots=True)  # NOT frozen ŌĆö visits/total_value/children dict-membership-mutable
class MCTSNode:
    plan_id: str                        # Plan AST content hash (Node.id)
    visits: int = 0                     # mutable: incremented per BACKUP
    total_value: float = 0.0            # mutable: accumulated per BACKUP
    # NOTE: no `prior` field on the node ŌĆö priors are per-edge (softmax
    # normalized per-parent), stored on MCTSEdge.prior. A node reached via
    # multiple parents has multiple priors, one per incoming edge.
    children: dict[str, "MCTSEdge"] = field(default_factory=dict)
                                        # mutable: dict-membership grows during EXPAND
    # No parent pointer ŌĆö the search tree is a DAG over plan ids, NOT a
    # tree. Backup walks the *path* taken in the current iteration, not
    # parent edges. This is the right call for transposition correctness:
    # a node reached via two paths must back up only the path actually
    # traversed in this iteration.
    is_terminal: bool = False           # mutable: flipped True iff Expander returned 0
                                        #   actions OR Evaluator returned a sentinel-terminal score

    @property
    def q_value(self) -> float:
        return 0.0 if self.visits == 0 else self.total_value / self.visits
```

**Mutable fields, explicit:** `visits`, `total_value`, `is_terminal`, plus dict-membership of `children`. Static post-construction: `plan_id`. `slots=True` keeps memory footprint small under transposition tables of thousands of plans.

```python
@dataclass(slots=True)
class MCTSEdge:
    """One outgoing edge from MCTSNode. Keyed by action canonical hash.

    NOT `frozen=True` ŌĆö `visits_through_edge` and `total_value_through_edge`
    are mutated in BACKUP. The static fields (`action_hash`, `action`,
    `child_plan_id`, `prior`) are de-facto immutable post-construction;
    the search loop only writes to the two counters.
    """
    action_hash: str                    # canonical-JSON sha256 of Action payload
    action: Action                      # the mutation itself, kept on the edge
    child_plan_id: str                  # Plan id reached by applying action
    prior: float                        # P(s,a) softmax prior from the expander; lives on
                                        # the edge because softmax-normalization is per-parent
    visits_through_edge: int = 0        # N(s,a) ŌĆö distinct from MCTSNode.visits
    total_value_through_edge: float = 0.0
```

**Why edge-statistics distinct from node-statistics.** PUCT uses `N(s,a)` (visits along an edge) and `Q(s,a)` (mean value along an edge), NOT just `N(s)` and `Q(s)`. With transposition, these can differ: a child node may have `N(s')=10` from two parent edges of `N(s_a, a_a)=4` and `N(s_b, a_b)=6`. The edge dataclass keeps the math correct.

**Consequence.** "Same plan reached by different action sequences" is **one MCTSNode**. The action-sequence multiplicity lives on the **edges**, where it belongs. Backup updates `MCTSEdge.visits_through_edge` + `MCTSNode.visits` along the traversed path; both rise by 1 per iteration. (This matches Lan & Yi 2022 "Graph-Aware MCTS" and AlphaZero's transposition handling.)

## 5. Action space

### Decision (ADR-2)

The minimal viable action space for v0.6.5 is **three action kinds**, chosen to give compositional reach without exploding the branching factor:

```python
@dataclass(frozen=True)
class SubstituteLeafAction:
    """Replace a leaf node in the plan with a different leaf.

    Most common LLM-proposable mutation. The replacement leaf must be
    content-addressed (a `Node`), so it round-trips canonical-JSON.
    """
    target_path: tuple[int, ...]        # e.g. (0, 2) = root.children[0].children[2]
    new_leaf: Node                      # tag, attrs, children=()

@dataclass(frozen=True)
class AddStepAction:
    """Insert a new child at `target_path` index `at`.

    Restricted to non-leaf parents (`children` is a tuple, but leaves are
    `children=()` ŌĆö adding a child to a leaf would change its kind, which
    is structurally invalid).
    """
    target_path: tuple[int, ...]
    at: int                             # insertion index in parent.children
    new_child: Node

@dataclass(frozen=True)
class ComposeWithSkillAction:
    """Wrap the subtree at `target_path` inside a registered SkillLibrary skill.

    The skill_id is looked up via SkillLibrary.lookup(skill_id); its
    Plan AST becomes the new parent, with the original subtree spliced
    in as the skill's first child. Only skills with a Plan AST whose
    root tag accepts non-leaf children may be composed (we enforce
    structurally ŌĆö see ┬¦6 invariants).
    """
    target_path: tuple[int, ...]
    skill_id: str                       # "skill/<16-hex>" form
```

Union type:

```python
Action = SubstituteLeafAction | AddStepAction | ComposeWithSkillAction
```

**Action discrimination is `isinstance`-strict, not duck-typed.** A third-party dataclass with matching field names but no inheritance from one of the three subclasses is rejected at the applicator boundary with `raise ValueError(f"unknown action kind: {type(action).__name__}")`. The applicator dispatch is:

```python
def apply_action(plan: Node, action: Action, ...) -> Node:
    if isinstance(action, SubstituteLeafAction):
        return _apply_substitute_leaf(plan, action)
    if isinstance(action, AddStepAction):
        return _apply_add_step(plan, action)
    if isinstance(action, ComposeWithSkillAction):
        return _apply_compose_with_skill(plan, action, skill_library)
    raise ValueError(f"unknown action kind: {type(action).__name__}")
```

**Why `isinstance` not `match`:** Python 3.10+ structural pattern matching would dispatch on field shapes, not nominal type, and an attacker (or accidental third-party subclass with identical fields) would silently route. `isinstance` enforces the nominal-type contract loud and clear. **Test pin:** `test_action_isinstance_strict_dispatch.py` constructs a `dataclass(frozen=True)` with matching `target_path` + `new_leaf` fields but no inheritance, asserts `apply_action` raises `ValueError`. Third-party action kinds must subclass one of the three or wait for v0.7+'s expanded action union.

### What's deferred

| Action | Defer reason |
|---|---|
| `RemoveStepAction` | Symmetry with `AddStep` is tempting, but a remove can break dispatcher-handler invariants (e.g. a `:seq` with zero children is structurally fine but semantically vacuous). Re-add in v0.7 once we have eval data showing remove-actions improve search. |
| `SwapMetricAction` | Metric is a stream-level constant, not part of the candidate plan. Swapping metric mid-search invalidates Q-values. If the user wants a different metric, run a different search. |
| `LLMProposedFreeFormAction` | An expander returning "the LLM said do X to get plan Y" is already encoded by `SubstituteLeaf(target_path=..., new_leaf=Y)`. We do NOT need a separate kind for it. The expander's job is to *propose* `(action, prior)` pairs from any source. |
| `ReorderChildrenAction` | Tuple permutation. Useful but increases branching factor; defer. |

### Rationale

Three action kinds give the search **full reach over the v0.6.0a1 substrate**: any plan reachable by repeatedly substituting leaves + adding subtrees + composing skills is reachable by a finite SubstituteLeaf+AddStep+ComposeWithSkill sequence from the initial plan. This is sufficient for the paper claim (Prop 6: provenance reconstructibility, NOT completeness over the plan-AST space ŌĆö the paper does not claim MCTS reaches every plan, only that what it does reach is reconstructible).

### Action canonical hash

`_action_hash(action)` reuses `Node.id` for nested Nodes ŌĆö `dataclasses.asdict` recurses through `Node.children` field-by-field, which would diverge from `Node.id`'s canonical rule. Compose the hash from `Node.id` directly:

```python
def _action_hash(action: Action) -> str:
    """SHA-256 hex of canonical-JSON of the action's structural payload.

    Used as the dict key for MCTSNode.children. Reuses `Node.id` (32-hex
    sha256 prefix; see `_ast.py:111-148`) for nested Node payloads ŌĆö this
    keeps the action-hash recipe aligned with the Plan AST's content-
    addressing rule. Two actions with the same target_path AND same
    new_leaf.id / new_child.id / skill_id collapse to the same hash ŌåÆ
    same edge.

    Canonical-JSON params match `Node.id`'s recipe: sort_keys=True,
    separators=(",",":"), allow_nan=False. Reuses the canonical-JSON
    helper exposed from `_ast.py` (no parallel implementation).
    """
    if isinstance(action, SubstituteLeafAction):
        payload = {
            "kind": "SubstituteLeafAction",
            "target_path": list(action.target_path),
            "new_leaf_id": action.new_leaf.id,
        }
    elif isinstance(action, AddStepAction):
        payload = {
            "kind": "AddStepAction",
            "target_path": list(action.target_path),
            "at": action.at,
            "new_child_id": action.new_child.id,
        }
    elif isinstance(action, ComposeWithSkillAction):
        payload = {
            "kind": "ComposeWithSkillAction",
            "target_path": list(action.target_path),
            "skill_id": action.skill_id,
        }
    else:
        raise ValueError(f"unknown action kind: {type(action).__name__}")
    encoded = json.dumps(payload, sort_keys=True, separators=(",", ":"), allow_nan=False)
    return hashlib.sha256(encoded.encode("utf-8")).hexdigest()
```

**Cross-reference:** `src/persistence/plan/_ast.py:111-148` is the source of truth for the Node-canonical-JSON recipe. Action-hash is one consumer; `Node.id` itself is another. Any change to the canonical rule lifts both.

## 6. Action application + plan invariants

### Decision (ADR-3 ŌĆö applicator contract)

`apply_action(plan: Node, action: Action) -> Node` is a pure function (in `_mcts.py`, private) that:

1. Resolves `action.target_path` against `plan` (raises `IndexError` if path is invalid).
2. Constructs the new `Node` tree by structural recursion, leaving untouched subtrees object-identical.
3. Returns the new root.

**Plan invariants preserved by every applicator branch:**
- Tag prefix rule: `Node.tag` always starts with `:`.
- Attr-key rule: no leading-colon attr keys.
- Children type: `tuple[Node, ...]`.
- Coercion: any new attrs the applicator emits are JSON-native (str/int/float/bool/None) ŌĆö no coercion-required types added by the search layer.

If the applicator constructs an invalid `Node` (e.g. SubstituteLeaf into a path that doesn't exist), **the `Node(...)` constructor raises**, and MCTS treats this as a **rejected action** (does not enter the tree, does not update statistics). Logged via `:mcts/iteration` datom with `phase="reject"` and `reason="plan_construction_raised"` (see ┬¦13 for the full reject-datom shape).

### Structural validation

```python
def _is_valid_action_for_plan(plan: Node, action: Action) -> bool:
    """Cheap up-front check the expander can call to filter invalid actions
    before we count them against the prior budget.

    Substrate-level: path resolves, the tag-tree shape is consistent.
    NOT semantic: 'is this a good action?' is the evaluator's job.
    """
    try:
        _resolve_path(plan, action.target_path)
    except IndexError:
        return False
    if isinstance(action, AddStepAction):
        parent = _resolve_path(plan, action.target_path)
        if not parent.children and parent.tag in _LEAF_REQUIRED_TAGS:
            return False
    return True
```

`_LEAF_REQUIRED_TAGS` defaults to `frozenset()` (empty) for v0.6.5; the dispatcher seam already enforces tag-leaf invariants at execute-time. We expose the constant for forward-compat with v0.7+ tag families.

### Plan-depth cap (`MAX_PLAN_DEPTH = 32`)

`ComposeWithSkillAction` can blow the plan depth up exponentially: skill A wraps skill A which wraps skill A ŌĆö and a single applicator step constructs a 10MB Node tree, blowing memory and stalling canonical-JSON. **Module-level constant `MAX_PLAN_DEPTH = 32`.** `apply_action` enforces this on output:

```python
def _plan_depth(plan: Node) -> int:
    if not plan.children:
        return 1
    return 1 + max(_plan_depth(c) for c in plan.children)


def apply_action(plan: Node, action: Action, ...) -> Node:
    # ComposeWithSkillAction rejects upfront if the skill plan is already
    # MAX_PLAN_DEPTH/2 = 16 deep ŌĆö composing it under the candidate plan
    # would exceed MAX_PLAN_DEPTH on the next iteration. Cheap pre-check
    # before constructing the output Node.
    if isinstance(action, ComposeWithSkillAction):
        skill_plan = ...  # SkillLibrary.lookup(action.skill_id)
        if _plan_depth(skill_plan) > MAX_PLAN_DEPTH // 2:
            raise PlanDepthExceeded(
                f"ComposeWithSkillAction: skill {action.skill_id!r} plan "
                f"depth {_plan_depth(skill_plan)} > MAX_PLAN_DEPTH/2"
            )
    new_plan = ...  # construct
    if _plan_depth(new_plan) > MAX_PLAN_DEPTH:
        raise PlanDepthExceeded(
            f"apply_action produced a plan of depth {_plan_depth(new_plan)} "
            f"> MAX_PLAN_DEPTH={MAX_PLAN_DEPTH}"
        )
    return new_plan
```

`PlanDepthExceeded` raises during `apply_action`; MCTS catches it the same way it catches `Node(...)`-constructor raises and emits a `phase="reject"` datom with `reason="plan_too_deep"`. The action is not added to the tree; statistics not updated. **Test pin:** `test_mcts_compose_skill_depth_blowup_rejected.py` (see ┬¦18).

`32` is generous: real DSPy programs are Ōēż 8 deep in practice; AlphaZero-style search trees are wider, not deeper. The cap protects against pathological recursion, not real workloads.

## 7. PUCT formula + hyperparameters

### Decision (ADR-4)

Standard AlphaZero PUCT:

```
UCB(s, a) = Q(s, a) + c_puct * P(s, a) * sqrt(parent.visits) / (1 + N(s, a))
```

Where:
- `Q(s, a) = MCTSEdge.total_value_through_edge / max(1, MCTSEdge.visits_through_edge)` ŌĆö mean reward along the edge. Returns `0.0` for unvisited edges (cold-start; the prior carries the exploration weight).
- `P(s, a) = MCTSEdge.prior` ŌĆö softmax-normalized prior emitted by the `Expander` at expansion time and stored on the edge (priors are per-parent, hence per-edge). Re-normalization is the expander's job, not MCTS's.
- `parent.visits = MCTSNode.visits` of the parent (NOT the edge's `visits_through_edge`).
- `N(s, a) = MCTSEdge.visits_through_edge`.
- `c_puct` ŌĆö exploration constant. **Default `1.4`.** Rationale: `1.4 Ōēł sqrt(2)` is the classic UCB1 constant; AlphaZero used 1.0ŌĆō4.0 with 1.0 most common but 1.4 is the median across the Stockfish/Leela/AlphaZero literature for "balanced" exploration. We expose it as a `MCTSConfig.c_puct: float = 1.4` knob; calling-site override is one keyword arg.

### Cold-start prior

When a child is first added (expansion phase), its `MCTSEdge.prior` is set from the expander's softmax. **Cold-start `Q(s,a) = 0.0`** ŌĆö the edge's value is unknown, and the prior is what drives the first selection. No "first-visit value bonus" ŌĆö that's an FPU (First Play Urgency) variant that shifts behavior in subtle ways; not worth the complexity for v0.6.5.

### Selection rule

```python
def _select_child(node: MCTSNode, c_puct: float) -> MCTSEdge | None:
    if not node.children:
        return None
    parent_visits = node.visits
    sqrt_parent = math.sqrt(parent_visits) if parent_visits > 0 else 1.0
    best_edge: MCTSEdge | None = None
    best_score = -math.inf
    for edge in node.children.values():
        n_sa = edge.visits_through_edge
        q = (edge.total_value_through_edge / n_sa) if n_sa > 0 else 0.0
        u = c_puct * edge.prior * sqrt_parent / (1 + n_sa)
        score = q + u
        if score > best_score:
            best_score = score
            best_edge = edge
    return best_edge
```

**Tie-breaking.** Stable: first edge in dict-insertion order wins on equal score. Dict insertion order is preserved in Python 3.7+. Determinism-preserving.

## 8. Expander Protocol

### Decision (ADR-5)

```python
@runtime_checkable
class Expander(Protocol):
    """Proposes (action, prior) pairs at expansion time.

    Implementations are typically LLM-backed (an LLM judges the plan
    and proposes mutations + softmax-normalized priors), but the
    Protocol shape is open: deterministic stubs satisfy it for tests,
    symbolic expanders satisfy it for v0.7+, and a 'always propose
    skill X' fixture satisfies it for closed-loop integration tests.

    The expander MUST return softmax-normalized priors (sum to 1.0).
    MCTS does NOT re-normalize. This keeps the contract crisp: the
    expander owns the policy distribution; MCTS owns the tree.

    Empty iterable = terminal node. The MCTSNode is marked is_terminal,
    no further selection visits it (selection short-circuits on
    is_terminal nodes ŌĆö see ┬¦10).
    """

    def propose(
        self,
        plan: Node,
        *,
        k: int,                         # caller's beam width hint; expander may return Ōēż k
    ) -> Sequence[tuple[Action, float]]: ...
```

**Why `Sequence` (not `Iterable`).** Single-pass generators interact badly with the prior-sum-to-1.0 normalization assertion (the loop has to materialize them anyway, and a generator consumed for normalization can't be re-iterated for tree population). `Sequence` is the minimal correct contract; `tuple` and `list` both satisfy it.

**Prior normalization assertion.** On every cache miss MCTS asserts:

```python
assert abs(sum(prior for _, prior in proposals) - 1.0) < _PRIOR_TOL
```

where `_PRIOR_TOL = 1e-6` (module-level constant; see ┬¦15 module-globals). Failure raises `ExpanderContractError("Expander.propose returned priors summing to {s}, not 1.0 ┬▒ _PRIOR_TOL")` ŌĆö this is an expander-contract violation, NOT a search-runtime issue. Empty proposals are exempt (terminal-node signal; see ┬¦10).

**Caching.** Plans are content-addressed; expander calls are pure functions of `plan.id` (modulo the LLM's nondeterminism, which is what the audit chain captures). MCTS keeps a `_expander_cache: dict[str, tuple[tuple[Action, float], ...]]` ŌĆö `plan.id ŌåÆ frozen list of proposals`. **Cache hit ŌćÆ no LLM call ŌćÆ no `:mcts/iteration` provenance datom.** Cache miss ŌćÆ one LLM call ŌćÆ one `:mcts/iteration` datom with `phase="expand"`.

### Two expanders ship in v0.6.5

- `LLMExpander` ŌĆö production wiring. Constructor takes a `provider: Callable[[Node, int], Sequence[tuple[Action, float]]]` closure that the caller wires to whichever LLM dispatch path is appropriate (Anthropic / Bedrock / etc.). MCTS does NOT own the registry seam. Body of the LLM prompt (composed inside `provider`) is structured: plan EDN + "propose K mutations as `(kind, target_path, payload)` triples + softmax priors." Response parsing is a JSON-mode adapter ŌĆö **no free-form parsing**.
- `_StaticExpander` (test-only) ŌĆö fabricated stub. Fixed proposal list per plan.id.

```python

    def __init__(
        self,
        provider: Callable[[Node, int], Sequence[tuple[Action, float]]],
    ) -> None: ...

        return self._provider(plan, k)


class _StaticExpander:

This matches ┬¦12's "caller orchestrates" principle and keeps `_mcts.py` test-isolated from the LLM-handler stack.

## 9. Evaluator Protocol

### Decision (ADR-6)

```python
@runtime_checkable
class Evaluator(Protocol):
    """Returns a scalar reward for a leaf plan.

    Two natural sources, both satisfy the Protocol:

      (a) DeterministicExecuteEvaluator ŌĆö runs `execute(plan, dispatcher,
          env=example["inputs"])` per training example, collects
          ExecutionResult, hands (result, expected) to the metric_fn,
          returns the mean. Same shape as optimize()'s _mean_score
          helper. Determinism inherits from execute() + dispatcher
          determinism.

      (b) LLMJudgeEvaluator ŌĆö wraps a `provider: Callable[[Node], float]`
          closure supplied by the caller (the closure dispatches to
          Anthropic / Bedrock / etc; no MCTS-owned registry). NON-
          deterministic; cached by plan.id (next paragraph).

    The substrate ships ONE evaluator contract; both flavors satisfy
    it. Pluggable strategy is over.
    """

    def evaluate(self, plan: Node) -> float: ...
```

**Return type is exactly `float` ŌĆö never `None`.** A None return from a real evaluator is a Protocol violation: rejected at the boundary via a `phase="reject"` datom (`reason="evaluator_returned_none"`), expansion not kept in the tree, search continues. The unambiguous "evaluator raised vs returned" signal is `error_class != null` in the iteration datom ŌĆö see ┬¦13. Datom `output_canonical_value` for `phase="evaluate"` is therefore always a float; no None overload.

**Caching by `plan.id` is essential.** Plans are content-addressed; evaluator calls collapse to `plan.id ŌåÆ float`. MCTS keeps `_evaluator_cache: dict[str, float]` and asks the evaluator only on cache miss. **Cache hit ŌćÆ no `:mcts/iteration` datom with `phase="evaluate"`. Cache miss ŌćÆ one datom with `phase="evaluate"` recording `plan_id` + `score`.**

### Two evaluators ship in v0.6.5

- `DeterministicExecuteEvaluator(training_set, metric, dispatcher)` ŌĆö production-ready under the v0.6.0a1 wiring. Reuses `lookup_metric` + `execute`. Deterministic given dispatcher determinism (which v0.6.0a1's no-`time.time()` discipline guarantees).
- `LLMJudgeEvaluator(provider)` ŌĆö production wiring through a `provider: Callable[[Node], float]` closure (caller composes; no MCTS-owned registry). **Caches by `plan.id` so repeated visits to the same plan don't re-call the LLM.**

```python
class LLMJudgeEvaluator:
    """Production wiring; provider closure is the caller's concern."""
    def __init__(
        self,
        provider: Callable[[Node], float],
    ) -> None: ...

    def evaluate(self, plan: Node) -> float:
        return self._provider(plan)


class _StaticEvaluator:
    """Test-only stub. Pinned signature, mirrors `_StaticExpander`."""
    def __init__(
        self,
        scores: dict[str, float],
        *,
        on_unknown: Literal["zero", "raise"] = "zero",
    ) -> None: ...

    def evaluate(self, plan: Node) -> float: ...
```

**No `:mcts/evaluator` registry indirection in v0.6.5** ŌĆö same shape as the expander side. Production callers compose a closure that dispatches to their LLM judge stack. Mirrors ┬¦12 "caller orchestrates" principle.

### Failure handling

If the evaluator raises, MCTS treats this as a **rejected leaf**:
- The expansion is not kept in the tree.
- A `:mcts/iteration` datom is written with `phase="evaluate"` + `error_class` + `error_repr` (mirroring `FailureInfo`'s shape from `_execute.py`).
- The iteration counter advances; the search continues.

This is symmetric with v0.6.0a1's per-leaf failure capture in `execute()`.

### NaN / Inf defense (G1)

The evaluator's return value is rejected at the boundary if `not math.isfinite(score)` ŌĆö i.e. NaN, +Inf, -Inf all reject. Rejection emits a `phase="reject"` datom with `reason="evaluator_returned_non_finite"` and `output_canonical_value={"plan_id": ..., "raw_score_repr": repr(score)}`. The leaf's expansion is not kept; **BACKUP is skipped for that iteration**; the iteration counter advances; search continues. This protects PUCT arithmetic (`q + u` would propagate NaN through every ancestor's Q estimate, irreversibly poisoning the tree).

Symmetric with the "evaluator raises" path: same `phase="reject"` datom shape, just a different `reason` field.

## 10. Determinism + replayability

### Problem

v0.6.0a1's whole point is byte-identical replay (Prop 4). MCTS introduces non-determinism: LLM expander, LLM evaluator. **What does "replayable MCTS" mean?**

### Decision (ADR-7)

Replayable MCTS = **the audit log uniquely determines the search trajectory**. Three options were on the table:

| Option | Approach | Verdict |
|---|---|---|
| (a) Cache all LLM calls keyed by plan.id + record the cache. | Persist `_expander_cache` + `_evaluator_cache` to fact store; on replay, hit the cache instead of calling the LLM. | **Selected as the primary path.** Aligns with v0.6.0a1's hash-keyed audit cache discipline. |
| (b) Record action sequence; on replay, re-walk the sequence ignoring the tree. | Cheap to record but discards the tree state; transposition-correctness lost on replay. Replay produces the same winner but not the same Q-values. | Rejected ŌĆö Prop 6 wants the search trajectory reconstructible, not just the winner. |
| (c) `:mcts/iteration` provenance datom per iteration capturing inputs + outputs. | Strongest ŌĆö every iteration's selection / expansion / evaluation / backup is a fact; replay re-derives the tree from the log. | **Selected as the primary path, layered on top of (a).** Both (a) and (c) ship; (c) is the substrate-level claim, (a) is the operational caching. |

**The combined path:**
1. Every LLM call (expander + evaluator cache miss) writes a `:mcts/iteration` datom with `phase Ōłł {"expand","evaluate","reject"}` recording `plan_id`, full-payload `output` (proposals incl. action_payload, or score, or rejection record), `mcts/prev-hash` (search-internal Merkle link; see ┬¦13), and a content hash of the inputs.
2. On replay, MCTS reconstructs the tree by scanning `:mcts/iteration` datoms with the matching `mcts/search-id`, in iteration-index order. Cache hits on cache-miss-recorded datoms by `plan_id`.
3. The search-internal Merkle chain (`mcts/prev-hash`) makes the iteration datoms tamper-evident ŌĆö reordering or fabricating iterations invalidates downstream `prev-hash` values. **This is independent of Prop 2 (effect-handler audit chain); see ┬¦13's audit-chain context note.**

Honest framing: **the cache is NOT a sufficient statistic on its own**. The full sufficient statistic for replay is **the audit log + the deterministic search loop together**. The audit log records all non-deterministic inputs (LLM expander/evaluator cache-miss outputs); the search loop is a pure function of those plus the config (PUCT tie-breaking is dict-insertion-order-stable; `max(...)` is stable under Python's tie semantics; `MCTSEdge` allocation order is determined by proposal iteration order). Defends Prop 6 under that honest composition. The replay-from-datoms-alone byte-identity property is the load-bearing pin (B-INT step 7, ┬¦18).

### Determinism-pin contract

For a given `(initial_plan_id, search_id, seed, expander_responses, evaluator_responses)` tuple, the search trajectory is byte-identical across two runs. **`MCTSConfig.seed` has NO consumer in v0.6.5** ŌĆö the field is reserved for v0.7+ features that genuinely need an RNG:
- **Dirichlet noise on the root prior** (AlphaZero-style root-exploration), deferred to v0.7+ per ┬¦24 OQ-5.
- **Shuffle-style tie-break for equal-prior proposals** if a future expander variant returns ties; v0.6.5's `_StaticExpander` and `LLMExpander` do not.

In v0.6.5, **MCTSEdge insertion order on first expansion is determined by the expander's proposal-iteration order** (the order tuples appear in the `Sequence` returned by `expander.propose`), NOT by the seed. The seed is validated in `MCTSConfig.__post_init__` (must be `int`, not `bool`) and recorded in `mcts/config-hash` so a v0.7+ replay against a v0.6.5 audit log will see a stable seed value. **Setting `seed` to a non-default value in v0.6.5 has zero observable effect on the search trajectory.** Documented at config-field level + tested in `test_mcts_config_validation.py` ("seed accepts only `int`, rejects `bool`").

Test-side, this is a `Hypothesis @given` property: same expander stub + same evaluator stub + same config + same initial plan ŌåÆ byte-identical `MCTSResult.tree_dump`.

## 11. Termination

### Decision (ADR-8)

Six termination reasons total (4 budget-based config knobs + 2 search-runtime conditions), **OR-combined** (whichever fires first): `max_iter`, `max_unique_plans`, `simple_regret`, `wall_clock`, `exhausted` (root has no proposals at iter 0), `all_evaluations_failed` (every evaluator call raised). The 4 config knobs:

```python
@dataclass(frozen=True, slots=True)
class MCTSConfig:
    max_iter: int = 200                 # hard ceiling on total iterations
    max_unique_plans: int = 64          # stop after this many distinct plan ids visited
    simple_regret_threshold: float | None = None
                                        # if set, stop when (best - second-best) Q-value
                                        # at the root exceeds this threshold for K
                                        # consecutive iterations
    simple_regret_window: int = 5       # K above
    wall_clock_budget_ms: int | None = None
                                        # optional; routes through :clock/now effect, not
                                        # time.time() ŌĆö repo discipline. None = no clock budget.
    c_puct: float = 1.4
    expander_k: int = 4                 # default beam width per expand
    seed: int = 0                       # determinism seed

    def __post_init__(self) -> None:
        # W1.x analog (Stream A G4 bool-coercion fix). bool is a subclass of int in
        # Python; isinstance(True, int) is True. Reject bool *before* numeric checks.
        for fld in ("c_puct", "max_iter", "max_unique_plans", "expander_k",
                    "simple_regret_window"):
            v = getattr(self, fld)
            if isinstance(v, bool) or not isinstance(v, (int, float)) or v <= 0:
                raise ValueError(
                    f"MCTSConfig.{fld} must be a positive number, got {v!r}"
                )
        if self.simple_regret_threshold is not None:
            srt = self.simple_regret_threshold
            if isinstance(srt, bool) or not isinstance(srt, (int, float)) or srt < 0:
                raise ValueError(
                    f"MCTSConfig.simple_regret_threshold must be None or a "
                    f"non-negative number, got {srt!r}"
                )
        if self.wall_clock_budget_ms is not None:
            wcb = self.wall_clock_budget_ms
            if isinstance(wcb, bool) or not isinstance(wcb, int) or wcb <= 0:
                raise ValueError(
                    f"MCTSConfig.wall_clock_budget_ms must be None or a positive "
                    f"int, got {wcb!r}"
                )
        if not isinstance(self.seed, int) or isinstance(self.seed, bool):
            raise ValueError(f"MCTSConfig.seed must be an int, got {self.seed!r}")
```

**Vacuous-truth + bool-coercion guards.** `simple_regret_threshold=False` would slip past a naive `> 0.0` check (Python: `False == 0` and `False > 0` is `False` so the threshold-never-fires; `True == 1` so `True > 0.5` is `True` accidentally). Stream A's W1.B G4 fix-pass landed an identical `isinstance(_, bool)` check on the gate's bool-ness check ŌĆö same anti-pattern, same fix. `max_unique_plans=0` and `max_iter=0` are rejected as non-positive (would produce vacuous `iter_count=0` "search"). `expander_k=0` rejected as non-positive (technically allowed by the math but always-pointless ŌĆö fail loudly).

**Defaults rationale:**
- `max_iter=200` ŌĆö empirical AlphaZero-style work shows diminishing returns past ~200 iters at small branching factor. Cheap to bump.
- `max_unique_plans=64` ŌĆö content-addressing makes "we've seen this plan" cheap to count; 64 distinct structures is a reasonable upper bound for the problem class v0.6.5 targets.
- `simple_regret_threshold=None` (off by default) ŌĆö adaptive termination is finicky; opt-in.
- `wall_clock_budget_ms=None` ŌĆö wall-clock budgets violate the no-`time.time()` discipline by default. The field is in the config dataclass for forward-compat, but the v0.6.5 search loop ignores it (deferred to v0.7+; see ┬¦15 signature note). Setting a positive value on v0.6.5 has no runtime effect ŌĆö the corresponding termination reason `"wall_clock"` is never fired.

The first termination signal observed wins; the search exits gracefully and `mcts_search()` returns a `MCTSResult` with `terminated_by Ōłł {"max_iter","max_unique_plans","simple_regret","wall_clock","exhausted","all_evaluations_failed"}`.

**Edge case ŌĆö `simple_regret` with `len(root.children) < 2`.** The `(best - second-best)` gap is undefined when the root has 0 or 1 child. **Statement: `simple_regret` termination NEVER FIRES under those conditions** ŌĆö `_termination_reached()` short-circuits (`if len(root.children) < 2: simple_regret_signal = False`). One of `max_iter`, `max_unique_plans`, `wall_clock`, or `exhausted` will terminate instead. This avoids a hidden divide-by-zero / index-error and keeps the user-facing contract crisp ("simple_regret needs Ōēź 2 root children to compute").

**`simple_regret` Q-gap is computed on visits-sorted top-2 root edges.** The "best - second-best" gap is computed by sorting `root.children.values()` by `visits_through_edge` descending, then taking `top[0].q_value() - top[1].q_value()`. Visits is the AlphaZero / MCTS convention for posterior commitment ("most-visited is most-confident"; matches ┬¦16 winner-edge selection); Q-sorted alternative would compute "regret on point estimate" which is not what the convergence literature proves. Test pin (in `test_mcts_simple_regret_with_fewer_than_two_root_children.py` and a sibling `test_mcts_simple_regret_visits_sorted.py`): construct a fixture where Q-sorted top-2 differ from visits-sorted top-2, assert the `simple_regret` signal uses visits ordering.

## 12. Skill-library closed loop

### Decision (ADR-9)

`mcts_search()` is **pure search**. It returns the winning Plan AST + the search tree summary. **It does NOT call `optimize()` or `promote()`.** The caller orchestrates the closed loop.

Rationale:
- Single-responsibility: search is one thing, optimization is another, promotion is a third.
- Test isolation: search tests don't need DSPy installed.
- Caller flexibility: a developer running search interactively wants the winner returned; a CI-style closed-loop wants `optimize` + `promote` + `register`. The substrate exposes both layers.

The roadmap-mandated closed loop is documented in `mcts_search()`'s docstring as the canonical post-search recipe:

```python
result = mcts_search(initial_plan, expander=..., evaluator=..., config=...)
optimized = optimize(result.winner, training_set=..., metric=..., dispatcher=...)
record = promote(optimized.plan, db=db, optimized_score=optimized.score, baseline_score=optimized.score_baseline, ...)
skill_id = SkillLibrary(db).register(optimized.plan, record, registered_at_ms=...)
```

### "Use existing skill" Action shape

`ComposeWithSkillAction(target_path, skill_id)` (┬¦5) is how MCTS pulls promoted skills back into the search prior. The expander chooses which skills are candidates ŌĆö the production `LLMExpander` reads `SkillLibrary.list_skills()` and offers the top-K (by some heuristic, e.g. recency-weighted by `skill/registered-at`) as composition candidates alongside SubstituteLeaf / AddStep proposals.

The applicator resolves `ComposeWithSkillAction` via:

```python
def _apply_compose_with_skill(plan: Node, action: ComposeWithSkillAction, skill_library: SkillLibrary) -> Node:
    looked_up = skill_library.lookup(action.skill_id)
    if looked_up is None:
        raise ValueError(f"ComposeWithSkillAction: skill_id {action.skill_id!r} not registered")
    skill_plan, _record = looked_up
    subtree = _resolve_path(plan, action.target_path)
    new_skill_root = Node(
        tag=skill_plan.tag,
        attrs=skill_plan.attrs,
        children=(subtree, *skill_plan.children),
    )
    return _replace_at_path(plan, action.target_path, new_skill_root)
```

`mcts_search()` accepts an optional `skill_library: SkillLibrary | None = None` arg; if `None`, `ComposeWithSkillAction` proposals from the expander are silently rejected (logged in `:mcts/iteration` with `phase="reject", reason="skill_not_registered"` ŌĆö same reason class as the unregistered-skill_id case, since "no skill library" subsumes it).

## 13. Provenance datom shape

### Decision (ADR-10)

Two new datom attribute namespaces, both under `mcts/`. **Every search emits one `:mcts/search` summary datom + one `:mcts/iteration` datom per LLM-touching iteration plus one `:mcts/iteration` per rejected action (`phase="reject"`).**

**Audit-chain context (CRITICAL ŌĆö closes R2.M2 category-error MAJOR).** The MCTS provenance datoms form a **separate Merkle chain** over Module-1 fact-store datoms, threaded by `mcts/prev-hash` (defined below). They are **NOT** participants in G2's effect-handler audit chain (`gate_g2_audit_chain` in `src/persistence/plan/_promotion.py`, which filters `a.startswith(_AUDIT_ATTR_PREFIX)` ŌĆö `_AUDIT_ATTR_PREFIX = "audit/"` ŌĆö and requires `provenance[":signature"]`). MCTS datoms use the `mcts/` prefix specifically to opt OUT of G2's window scan. Prop 6's reconstructibility claim rests on:

1. Prop 1 (datom append-only) ŌĆö Module 1 invariant; the chain cannot be rewritten in place.
2. The search-internal Merkle chain via `mcts/prev-hash` ŌĆö tampering with any iteration datom invalidates all subsequent `mcts/prev-hash` values.

This is **distinct from** Prop 2 (audit-chain unbrokenness on the effect-handler chain), which Stream B does NOT compose with. Lifting MCTS provenance into the effect-handler audit chain (i.e. running every iteration through an `mcts.iteration` effect handler) would be a v1.0+ refactor with substantial scope (handler registration + signature-key plumbing + G2 widening); v0.6.5 ships orthogonally.

#### `mcts/search` (one per `mcts_search()` call)

```python
[
  Datom(e=search_id, a="mcts/initial-plan-id", v=initial_plan.id, ...),
  Datom(e=search_id, a="mcts/winner-plan-id",  v=winner_plan.id,  ...),
  Datom(e=search_id, a="mcts/config-hash",     v=config_canonical_hash, ...),
  Datom(e=search_id, a="mcts/iter-count",      v=actual_iter_count, ...),
  Datom(e=search_id, a="mcts/terminated-by",   v=termination_reason, ...),
  Datom(e=search_id, a="mcts/started-at",      v=started_at_ms, ...),
  Datom(e=search_id, a="mcts/finished-at",     v=finished_at_ms, ...),
]
```

`search_id` is content-addressed:

```python
search_id = "mcts/" + sha256(canonical_json({
    "initial_plan_id": initial_plan.id,
    "config_hash": config_canonical_hash,
    "started_at": started_at_ms,
}))[:16]
```

The `search_id` is in its own namespace (`mcts/<16-hex>`), parallel with `skill_id`'s `skill/<16-hex>`. The 16-hex slice is consistent with `_skill_library.py`'s `_SKILL_ID_HEX_WIDTH = 16`.

#### `mcts/iteration` (one per LLM-touching iteration AND one per rejected action)

```python
[
  Datom(e=iter_id, a="mcts/search-id",     v=search_id, ...),
  Datom(e=iter_id, a="mcts/iter-index",    v=iter_index, ...),         # 0-based MCTS iter counter
  Datom(e=iter_id, a="mcts/phase",         v=phase, ...),               # "expand" | "evaluate" | "reject"
  Datom(e=iter_id, a="mcts/plan-id",       v=plan_id, ...),
  Datom(e=iter_id, a="mcts/inputs-hash",   v=inputs_canonical_hash, ...),
  Datom(e=iter_id, a="mcts/output",        v=output_canonical_value, ...),
  Datom(e=iter_id, a="mcts/prev-hash",     v=prev_iter_content_hash, ...),  # search-internal Merkle link
  Datom(e=iter_id, a="mcts/error-class",   v=error_class_or_None, ...),     # only on raises
  Datom(e=iter_id, a="mcts/error-repr",    v=error_repr_or_None, ...),      # only on raises
]
```

`iter_id`:

```python
iter_id = "mcts-iter/" + sha256(canonical_json({
    "search_id": search_id,
    "iter_index": iter_index,
    "phase": phase,
    "plan_id": plan_id,
    "action_hash": action_hash_or_None,   # disambiguates multiple rejects in one iter
}))[:16]
```

#### `mcts/prev-hash` ŌĆö the search-internal Merkle chain

Every `:mcts/iteration` datom carries a `mcts/prev-hash` slot:

```python
prev_iter_content_hash = sha256(canonical_json(prev_iteration_datom_content)).hexdigest()
```

where `prev_iteration_datom_content` is the canonical-JSON of the **immediately preceding** iteration datom's `(search_id, iter_index, phase, plan_id, inputs_hash, output_canonical_value, error_class, error_repr)` slots (everything except `prev-hash` itself, to avoid the chicken-and-egg). The FIRST iteration datum's `prev-hash` = `sha256(canonical_json(:mcts/search-datom-content))` where the search-datom content is the canonical-JSON of the seven `mcts/search` slots above (excluding `winner-plan-id` and `iter-count` which are written at search end ŌĆö for the chain we use only the up-front-known slots: initial-plan-id, config-hash, started-at).

**Property:** tampering with any iteration datom (or the search datom) invalidates all subsequent `mcts/prev-hash` values. Reconstruction can re-compute every `prev-hash` from the recorded slots and assert match against what's stored ŌĆö any mismatch = tamper detected. **This is the search-internal Merkle chain, distinct from G2's effect-handler audit chain.**

**Prop 6's `parent_provenance_hash` term denotes `mcts/prev-hash`** ŌĆö the paper Prop 6 wording resolves to this slot specifically.

#### `output_canonical_value` shape

- `phase="expand"` ŌćÆ list of **full-payload** proposal records:
  ```python
  [
    {
      "action_hash": str,                # _action_hash(action)
      "action_kind": str,                # one of: "SubstituteLeafAction" / "AddStepAction" / "ComposeWithSkillAction"
      "action_payload": {                # canonical JSON of the action's structural fields
        "target_path": [int, ...],
        # plus, depending on kind:
        "new_leaf_id": str,              # for SubstituteLeafAction ŌĆö sha256 hash of new_leaf
        "new_leaf_canonical": dict,      # for SubstituteLeafAction ŌĆö FULL canonical-JSON of
                                         #   new_leaf Node ({"tag":..., "attrs":..., "children":[...]});
                                         #   recorded in addition to the hash so a freshly-synthesized
                                         #   Node can be reconstructed from this datom alone (the
                                         #   bytes of new_leaf are otherwise nowhere in the audit log).
        "new_child_id": str,             # for AddStepAction ŌĆö sha256 hash of new_child
        "new_child_canonical": dict,     # for AddStepAction ŌĆö FULL canonical-JSON of new_child Node
                                         #   (same justification as new_leaf_canonical above).
        "at": int,                       # for AddStepAction
        "skill_id": str,                 # for ComposeWithSkillAction (no canonical Node needed ŌĆö
                                         #   the skill plan is recoverable from SkillLibrary lookup
                                         #   on replay; skill_id is itself content-addressed).
      },
      "prior": float,
    },
    ...
  ]
  ```
  **Invariant (load-bearing for Prop 6):** for any Action whose payload contains a freshly-synthesized Node (i.e. a Node not already present in the search tree ŌĆö the production `LLMExpander` typically synthesizes fresh `new_leaf` / `new_child` Nodes on every cache-miss expand), the **full canonical-JSON of that Node MUST be recorded in the proposal entry** under `new_leaf_canonical` / `new_child_canonical`. Recording only `new_leaf_id` / `new_child_id` (the hashes) is **insufficient** because Node bytes are not otherwise recoverable from the audit log ŌĆö the hash is one-way. This invariant closes R2.M4 ("action-payload stores hash-not-bytes; novel Node leaves unrecoverable from audit log alone"). Canonical-JSON for both the recorded `_canonical` slot and the hash recipe is the single helper exposed from `_ast.py:111-148` (no parallel implementation).

  This is the resolution of OQ-7 (now closed; see ┬¦24): the action payload is recorded in full, including canonical Node bytes for synthesized Nodes, so an Action object is fully reconstructible from the datom alone ŌĆö no dependence on "the next iteration's plan_id" (which would be ambiguous under transposition: two distinct actions may produce the same child plan_id) and no dependence on any in-process search-tree state.

#### Transaction granularity (closes Theme D2)

**One MCTS iteration = exactly one `db.transact()` call**, containing all datoms emitted for that iteration:
- On iteration 0 only: the `mcts/search` summary datom (initial fields) + the iteration's own datoms.
- On every iteration: the `mcts/iteration` datoms for that iteration's expand / evaluate / reject phases.

Synthetic transaction time: `t = started_at_ms + iter_index` (NOT `time.time()` ŌĆö repo's no-wall-clock discipline. `iter_index` is the **MCTS iteration counter**, advanced on every iteration including cache hits, NOT the cache-miss sequence number).

**Within-transaction ordering.** Within one `db.transact()` call (one MCTS iteration), datoms are emitted in this order: (a) the iteration's primary `phase` datom (one of `expand` / `evaluate` / `reject`), then (b) any inline `phase="reject"` datoms produced during proposal application by `_populate_children` ŌĆö emitted **in proposal-iteration order** (the same order the expander returned them). `mcts/prev-hash` chains every datom in that emission order: the primary's `prev-hash` links to the prior iteration's terminal datom; each subsequent reject's `prev-hash` links to the immediately preceding datom in the same transaction. This pins replay's lex-sort key `(iter-index, phase, action_hash)` against a single deterministic order across record and replay.

**On replay**, the iter-6 cache hit is RE-DERIVED ŌĆö the replay loop re-runs `mcts_search` with the same config + the recorded caches. Iteration 6 hits cache for both expand (already populated from iter-5's `phase="expand"` output, **including the materialized `new_leaf_canonical` / `new_child_canonical` Node bytes** per the ┬¦13 replay-procedure step 4) and evaluate (already populated from iter-5's `phase="evaluate"` output), produces zero LLM calls, zero new datoms. The iter_index counter is the **MCTS iteration counter** ŌĆö it advances on every iteration including cache hits, so replay's `iter_index=6` matches original's `iter_index=6` even though no datom was written. Replay verifies "iteration 6 was a cache hit" by observing the absence of an `iter-index=6` datom under this `search_id` and asserting both caches were populated by then.

#### Replay procedure

**Cache-hit iterations are NOT recorded.** Only cache-miss LLM calls produce datoms. Replay reconstructs the tree by:

1. Scan `:mcts/search` for `search_id`.
2. Scan `:mcts/iteration` for that `search_id`; sort lexicographically by `(iter-index, phase, action_hash)`.
3. **Verify the Merkle chain:** re-compute every `mcts/prev-hash` from the recorded slots; assert each matches what's stored. Any mismatch = tamper detected, replay fails loudly with `MCTSReplayChainBroken`.
4. **Materialize Action objects from canonical Node bytes.** Before populating the expander cache for any plan `P`, walk each `phase="expand"` proposal record under `output_canonical_value` and reconstruct the Action object: deserialize `new_leaf_canonical` / `new_child_canonical` back into a `Node` via the same canonical-JSON contract used to compute `Node.id` (the helper exposed from `_ast.py:111-148`), then construct the matching `SubstituteLeafAction(target_path, new_leaf=Node(...))` / `AddStepAction(target_path, at, new_child=Node(...))` / `ComposeWithSkillAction(target_path, skill_id)`. **Sanity check:** assert `reconstructed_node.id == new_leaf_id` (resp. `new_child_id`) byte-equality; on mismatch raise `ValueError("proposal canonical Node hash mismatch")` immediately. This is the load-bearing step for Prop 6 ŌĆö a freshly-synthesized Node `Y` (e.g. `LLMExpander` invented a new `:leaf/predict` with a novel prompt on iteration K) is recoverable here only because `new_leaf_canonical` recorded its bytes; the hash alone is one-way.
5. Pre-populate `_expander_cache` + `_evaluator_cache` from the **materialized** Actions (step 4) + the recorded scores (`phase="expand"` outputs ŌåÆ expander cache as `tuple[(Action, float), ...]`, `phase="evaluate"` outputs ŌåÆ evaluator cache as `dict[plan_id, float]`).
6. Re-run the search loop with the same config + the populated caches, with **stub-LLM-handlers that fail loudly on any cache miss** (`raise MCTSReplayCacheMiss(...)`). **`MCTSReplayCacheMiss` typing:** raised exclusively by test-only `_ReplayLoudExpander` / `_ReplayLoudEvaluator` stubs (defined in the B-INT integration test). The production `mcts_search()` loop never raises `MCTSReplayCacheMiss` directly ŌĆö production cache-miss is the normal path that calls the real LLM-backed expander/evaluator. The exception class lives in the test module, not in `_mcts.py`'s public surface.
7. Assert `MCTSResult.tree_dump` byte-equality against the original.

**Property under fixed `MCTSConfig` + deterministic stub expander/evaluator:** replay-from-datoms-alone produces a byte-identical `tree_dump`.

This is the load-bearing Prop 6 test ŌĆö see ┬¦18 B-INT step 7.

Defends Prop 6: full search trajectory reconstructible from the audit log alone.

| Mode | What happens | Provenance | Raises? |
|---|---|---|---|
| Evaluator raises | Iteration recorded with `error_class` / `error_repr`; expansion not kept in tree; iteration counter advances. | `:mcts/iteration` w/ `phase="evaluate"`, error fields populated. | No ŌĆö search continues. |
| Expander returns 0 candidates for a non-root plan | MCTSNode is marked `is_terminal=True`; subsequent selection skips it (selection looks for non-terminal children, falls back to backing up the rollout immediately). | `:mcts/iteration` w/ `phase="expand"`, output = `[]`. | No. |
| Expander returns 0 candidates for the root plan on iter 0 | Root has no children to select; search exits immediately with `terminated_by="exhausted"`, winner=initial_plan. | `:mcts/iteration` w/ `phase="expand"` recorded; `:mcts/search` w/ `terminated_by="exhausted"`. | No. |
| All proposed actions produce invalid plans | Each action_application raise is logged as a rejected action (`:mcts/iteration` w/ `phase="reject"`, `reason="plan_construction_raised"` or `"plan_too_deep"`). | `c_puct` Ōēż 0 | `MCTSConfig.__post_init__` raises `ValueError("c_puct must be > 0")`. | None ŌĆö config-time check. | Yes. |

|---|---|---|---|---|
| `P0` | `P1` | `aa00...` | 7 | 0.62 |
| `P0` | `P2` | `bb11...` | 5 | 0.51 |
| `P1` | `P3` | `cc22...` | 3 | 0.79 |

### MCTS loop pseudocode

```python
def mcts_search(...):
    config = config or MCTSConfig()
    config.__post_init__()  # config validation
    transposition: dict[str, MCTSNode] = {}
    expander_cache: dict[str, tuple[tuple[Action, float], ...]] = {}
    evaluator_cache: dict[str, float] = {}

    root = MCTSNode(plan_id=initial_plan.id)
    transposition[initial_plan.id] = root
    plan_by_id: dict[str, Node] = {initial_plan.id: initial_plan}

    search_id = _derive_search_id(initial_plan.id, config, started_at_ms)
    # Emit :mcts/search "started" provenance up front (idempotent on
    # winner-plan-id update at the end).

    iter_index = 0
    prev_hash = _hash_search_datom_content(search_id, initial_plan.id, config_hash, started_at_ms)
    # ^ initial Merkle anchor: the FIRST iteration datom's prev-hash
    # links to the search-summary datom's content hash.

    while not _termination_reached(root, iter_index, config, ...):
        path: list[tuple[MCTSNode, MCTSEdge | None]] = []
        pending_datoms: list[Datom] = []  # accumulated for one db.transact() per iteration

        # 1. SELECT
        node = root
        while node.children and not node.is_terminal:
            edge = _select_child(node, config.c_puct)
            if edge is None:
                break
            path.append((node, edge))
            node = transposition[edge.child_plan_id]

        # 2. EXPAND (if non-terminal & unexpanded)
        if not node.is_terminal and not node.children:
            proposals = expander_cache.get(node.plan_id)
            if proposals is None:
                proposals = tuple(expander.propose(plan_by_id[node.plan_id], k=config.expander_k))
                if proposals and abs(sum(p for _, p in proposals) - 1.0) >= _PRIOR_TOL:
                    raise ExpanderContractError(...)
                expander_cache[node.plan_id] = proposals
                expand_datoms, prev_hash = _build_iteration_datoms(
                    search_id, iter_index, "expand",
                    plan_id=node.plan_id, output=proposals, prev_hash=prev_hash,
                )
                pending_datoms.extend(expand_datoms)
            if not proposals:
                node.is_terminal = True
            else:
                # _populate_children may emit phase="reject" datoms for
                # actions that fail apply_action ŌĆö each appended to
                # pending_datoms with its own prev-hash chained off
                # the prior emitted datom in this iteration.
                reject_datoms, prev_hash = _populate_children(
                    node, proposals, plan_by_id, transposition,
                    skill_library, search_id, iter_index, prev_hash,
                )
                pending_datoms.extend(reject_datoms)

        # 3. EVALUATE (the leaf reached)
        leaf_plan_id = node.plan_id
        score = evaluator_cache.get(leaf_plan_id)
        if score is None:
            try:
                score = evaluator.evaluate(plan_by_id[leaf_plan_id])
            except Exception as exc:
                reject_datom, prev_hash = _build_iteration_datoms(
                    search_id, iter_index, "reject",
                    plan_id=leaf_plan_id,
                    output={"reason": "evaluator_raised"},
                    error_class=type(exc).__name__, error_repr=repr(exc),
                    prev_hash=prev_hash,
                )
                pending_datoms.extend(reject_datom)
                if db is not None:
                    db.transact(pending_datoms, ...)  # one txn for this iter
                iter_index += 1
                continue
            if not math.isfinite(score):
                reject_datom, prev_hash = _build_iteration_datoms(
                    search_id, iter_index, "reject",
                    plan_id=leaf_plan_id,
                    output={"reason": "evaluator_returned_non_finite",
                            "raw_score_repr": repr(score)},
                    prev_hash=prev_hash,
                )
                pending_datoms.extend(reject_datom)
                if db is not None:
                    db.transact(pending_datoms, ...)
                iter_index += 1
                continue
            evaluator_cache[leaf_plan_id] = score
            eval_datom, prev_hash = _build_iteration_datoms(
                search_id, iter_index, "evaluate",
                plan_id=leaf_plan_id, output=score, prev_hash=prev_hash,
            )
            pending_datoms.extend(eval_datom)

        # 4. BACKUP along the traversed path
        for parent_node, edge in path:
            parent_node.visits += 1
            if edge is not None:
                edge.visits_through_edge += 1
                edge.total_value_through_edge += score
        node.visits += 1   # the leaf itself (Invariant 5: leaf is the
                           # parent-plan that was just expanded; evaluator
                           # ran on parent, not on newly-added children)

        # Single db.transact() per iteration (Theme D2 transactional contract):
        if db is not None and pending_datoms:
            db.transact(pending_datoms, t=started_at_ms + iter_index, ...)

        iter_index += 1

    # Pick winner: root edge with max visits (AlphaZero convention) OR max Q
    # (we use max visits ŌĆö most-visited is most-confident).
    winner_edge = max(root.children.values(), key=lambda e: e.visits_through_edge, default=None)
    winner_plan = plan_by_id[winner_edge.child_plan_id] if winner_edge else initial_plan
    ...
```

### Invariants (runtime-checkable)

1. **Plan-id stability.** For every `MCTSNode n` and every `MCTSEdge e` with `e.child_plan_id = n.plan_id`, `plan_by_id[n.plan_id].id == n.plan_id` (content-addressing identity).
2. **Visit conservation (three exact equalities, no approximations).**
   - **Root case.** `root.visits == count of `:mcts/iteration` datoms with `phase="evaluate"` AND `error_class IS NULL` under this `search_id`` ŌĆö every iteration that reaches BACKUP backs up through the root, and BACKUP is reached iff EVALUATE returned a finite float (no raise, no NaN/Inf, no None). The audit-log scan is the observable counter; we deliberately do NOT introduce a new public `_completed_iter_count` field on `MCTSResult` to expose this ŌĆö the audit log already records it. Iterations that aborted in EVALUATE-raise / NaN-reject / None-reject before BACKUP show up as `phase="reject"` datoms (not `phase="evaluate"`) and are correctly excluded from this count.
   - **Interior case.** For every non-root MCTSNode `n` reached via incoming edges `E_in` (the set of edges in the DAG with `child_plan_id == n.plan_id`): `n.visits == sum(e.visits_through_edge for e in E_in)`. Visits at an interior node = total transit count along all incoming edges.
   - **Leaf-of-path case.** On any iteration whose selection path ends at `n` (i.e. SELECT terminated at `n` because `n.children` was empty or `n.is_terminal`): `n.visits` increments by exactly 1 for that iteration in addition to any increments from being on a longer path. (Equivalent: in BACKUP, the leaf node receives a `+1` visit independently of the edge increments along the traversed path.)

   Each case has a dedicated test in `test_mcts_visit_conservation.py` (┬¦18 below).
3. **Cache-hit invariant.** `expander_cache.keys() ŌŖå {visited plan ids that triggered an expand}`; `evaluator_cache.keys() ŌŖå {visited plan ids that triggered an evaluate}`.
4. **Determinism invariant.** Two `mcts_search()` calls with identical (initial_plan, config, expander_responses, evaluator_responses, started_at_ms) produce equal `MCTSResult.tree_dump`.
5. **Selection-of-leaf-on-expansion (where the evaluator runs).** On any iteration where EXPAND fires (the SELECTed leaf has children added to it), **the evaluator is called on the JUST-EXPANDED PARENT plan**, not on its newly-added children. The newly-added children are evaluated **only** if a subsequent iteration's SELECT picks one of them. Justification: evaluator expense (one LLM call per evaluate); **rollout depth = 1** (per ┬¦3 deferrals ŌĆö no policy network between leaf and evaluator). To be precise: rollout depth = 1 means the evaluator runs at the SELECTed-and-expanded leaf, **not after a random rollout from there** (no random-policy walk between expansion and evaluation; the leaf IS the rollout terminus). Test pin: `test_mcts_evaluator_called_on_parent_at_expansion.py` (┬¦18) constructs a fixture where SELECT terminates at root, EXPAND adds 3 children, asserts the evaluator was called exactly once with the root plan (NOT with any child).
6. **Exactly one `phase="evaluate"` datom per iteration.** Each MCTS iteration that completes BACKUP emits **exactly one** `:mcts/iteration` datom with `phase="evaluate"`. Iterations that abort before BACKUP (evaluator raised, returned NaN/Inf, or returned None) emit **zero** `phase="evaluate"` datoms and **one** `phase="reject"` datom with the corresponding `reason`. Iterations that hit `evaluator_cache` emit **zero** `phase="evaluate"` datoms (cache-miss-only recording, ADR-10) but still complete BACKUP. Test pin: `test_mcts_provenance_datom_count.py` (┬¦18) asserts: `count(phase="evaluate" datoms) == count(distinct plan_ids that triggered evaluator-cache-miss-and-finite-return)`.

(1), (2), (5), and (6) are property-test-checkable. (3) is a debug assertion. (4) is the load-bearing pin test.

### ADR-1 ŌĆö Search-tree node identity = Plan AST `Node.id` (transposition table)
**Options.** (a) Plan-id-keyed transposition table (graph MCTS). (b) Distinct MCTSNode per action sequence (pure tree).
**Decision.** (a). Plans are content-addressed; the `Node.id` is the natural search-state key. Transposition gives statistical efficiency at no correctness cost.
**Rationale.** Same plan reached via different sequences shares Q-value learning. Edge-keyed `MCTSEdge.visits_through_edge` keeps path-specific PUCT correct (┬¦4 above).
**Consequences.** Backup walks the *path traversed this iteration*, not parent pointers. The MCTS object graph is a DAG, not a tree. No parent pointer on `MCTSNode`.

### ADR-2 ŌĆö Action space = Substitute / AddStep / ComposeWithSkill
**Options.** (a) Three action kinds (selected). (b) Five+ kinds including Remove, Reorder, SwapMetric. (c) Free-form LLM mutation as one kind.
**Decision.** (a) ŌĆö minimal viable.
**Rationale.** Three kinds cover compositional reach without exploding the branching factor. Defer Remove (semantic risk), SwapMetric (out-of-bounds), Reorder (branching cost). Free-form mutations are encoded as SubstituteLeaf + AddStep.
**Consequences.** v0.7+ may add Remove / Reorder; the Action union widens, the dispatch shape stays.

### ADR-3 ŌĆö Action applicator is pure + structural; invalidity = rejected
**Options.** (a) Pure applicator, raise on invalid path; (b) applicator returns Optional[Node].
**Decision.** (a). `Node(...)`'s constructor already raises on invariants; we let it.
**Rationale.** "Action produced an invalid plan" is a real signal ŌĆö the audit log records it via a `phase="reject"` datom (┬¦13). Optional return would hide failure modes.
**Consequences.** Every applicator branch has a try/except in MCTS that converts `ValueError` / `IndexError` to a rejected-action provenance datom.

### ADR-4 ŌĆö PUCT formula + `c_puct = 1.4` default
**Options.** (a) `c_puct = 1.0` (AlphaZero classic), (b) `c_puct = 1.4 Ōēł ŌłÜ2` (UCB1 classic), (c) adaptive schedule.
**Decision.** (b). Static, configurable.
**Rationale.** `1.4` is the median of the AlphaZero / Stockfish / Leela literature for "balanced" exploration; `ŌłÜ2` is the closed-form UCB1 value. Adaptive schedules add complexity without v0.6.5-scope value.
**Consequences.** Single hyperparam knob; v0.7+ adaptive variants don't break the contract.

### ADR-5 ŌĆö Expander Protocol returns softmax-normalized priors
**Options.** (a) Expander returns normalized priors; (b) MCTS re-normalizes; (c) Expander returns logits, MCTS does softmax.
**Decision.** (a). Expander owns the policy distribution.
**Rationale.** Crisp contract boundary. MCTS is pure tree machinery; the policy is the expander's responsibility.
**Consequences.** Test stubs must softmax-normalize; documented in Protocol docstring + asserted in `test_expander_contract.py`.

### ADR-6 ŌĆö Evaluator Protocol takes plan, returns float; cached by `plan.id`
**Options.** (a) Single Protocol, two ship-time implementations (DeterministicExecute + LLMJudge); (b) two distinct Protocols.
**Decision.** (a). Pluggable strategy via Protocol satisfaction, not type union.
**Rationale.** `plan.id`-keyed caching makes the deterministic vs LLM choice transparent to the MCTS loop. Both sources are content-addressed pure functions of the plan.
**Consequences.** `evaluator_cache[plan.id]` is the hot path; cache-miss cost is the only difference between deterministic and LLM evaluators.

### ADR-7 ŌĆö Replayability via `:mcts/iteration` provenance datoms + LLM cache + deterministic search loop
**Options.** (a) Cache + record (selected). (b) Action-sequence-only recording. (c) Per-iteration provenance datoms.
**Decision.** (a) and (c) combined, **plus the deterministic search loop**.
**Rationale.** Honest framing: the cache is NOT a sufficient statistic on its own. Replay correctness depends on multiple non-cache invariants ŌĆö PUCT tie-breaking by dict-insertion order, `MCTSEdge` allocation order, `path` list build order, `max(...)` stability under ties. The full sufficient statistic is **the audit log + the deterministic search loop** together. The audit log records all non-deterministic inputs (LLM expander/evaluator outputs); the search loop is a pure function of those plus the config. Defends Prop 6 under that honest composition.
**Consequences.** Every cache-miss LLM call writes one datom; replay re-derives the tree from the log via the same `mcts_search()` code path with stub-LLM-handlers that fail loudly on cache miss. **Property under fixed `MCTSConfig` + deterministic stub expander/evaluator: replay-from-datoms-alone produces a byte-identical `tree_dump`.** This property is the load-bearing pin (B-INT step 7, ┬¦18).

### ADR-8 ŌĆö Termination: 6-reason union (`max_iter | max_unique_plans | simple_regret | wall_clock | exhausted | all_evaluations_failed`), OR-combined
**Options.** (a) `max_iter` only; (b) multi-signal OR.
**Decision.** (b). 4 budget-based config knobs (`max_iter`, `max_unique_plans`, `simple_regret`, `wall_clock`) plus 2 search-runtime conditions (`exhausted` for "root has no proposals at iter 0", `all_evaluations_failed` for "every evaluator call raised").
**Rationale.** Different problem classes want different termination shapes. `max_iter` is the minimum viable; the others opt-in. The two runtime-condition reasons are not opt-in ŌĆö they fire when the corresponding pathological state is reached.
**Consequences.** `MCTSResult.terminated_by` records which fired; one ARIS R2 readability concern (multi-signal exit logic) is mitigated by a single `_termination_reached()` helper. `wall_clock` is a no-op in v0.6.5 (deferred per ┬¦15) but stays in the union for forward-compat.

### ADR-9 ŌĆö Skill-library closed loop is the caller's recipe, not `mcts_search()`'s
**Options.** (a) `mcts_search()` calls `optimize` + `promote` + `register` internally; (b) `mcts_search()` returns the winner, caller orchestrates.
**Decision.** (b).
**Rationale.** Single-responsibility. Test isolation. Caller flexibility (interactive vs CI). DSPy stays out of MCTS's import path.
**Consequences.** Documentation burden ŌĆö the canonical recipe lives in `mcts_search()` docstring + the closed-loop integration test (B-INT). No silent loss of optimization quality vs the alternative.

### ADR-10 ŌĆö Provenance: `mcts/search` (one) + `mcts/iteration` (cache-miss-only)
**Options.** (a) Per-iteration regardless of cache; (b) cache-miss only.
**Decision.** (b).
**Rationale.** Cache hits are deterministic from cache-miss recordings; re-recording them on every visit bloats the log without information gain. Replay reconstructs them by walking the cache-miss datoms.
**Consequences.** `:mcts/iteration` row count = total LLM calls, not total MCTS iterations. Test assertions distinguish.

### ADR-11 ŌĆö Module surface: flat `_mcts.py`, not `persistence.plan.search/`
**Options.** (a) Flat file; (b) sub-module.
**Decision.** (a).
**Rationale.** v0.6.5 ships one search algorithm. Sub-module overhead exceeds benefit. Lift to `persistence.plan.search/` when v0.7+ adds beam-search / iterative-widening / non-MCTS variants.
**Consequences.** Future v0.7+ refactor to sub-module is a single `git mv` + `__init__.py` re-export update; no public-API break.

### ADR-12 ŌĆö `:branch` dispatcher-handler registration deferred to v0.6.6
**Options.** (a) Register `:branch` handler in v0.6.5 (roadmap ┬¦3.3 option (a) full closure); (b) ship MCTS orthogonally with structural substitution and defer the handler to v0.6.6.
**Decision.** (b). **Roadmap ┬¦3.3 amended 2026-04-28 to defer `:branch` handler-registration from v0.6.5 to v0.6.6; this design ships under that amendment.** See `docs/plans/2026-04-27-persistence-os-v1.0-roadmap.md` ┬¦3.3 amendment block for the contract change.
**Rationale.** MCTS produces a winning Plan AST that the caller substitutes structurally. The dispatcher-handler-for-`:branch` use case is "execute the search as part of execute()" ŌĆö strictly larger scope than v0.6.5 needs. Stream B's paper claim (Prop 6 reconstructibility) is fully defensible without the `:branch` handler ŌĆö Prop 6 rests on the search-internal Merkle chain (`mcts/prev-hash`), not on dispatcher integration.
**Consequences.** v0.6.5 ships `mcts_search()` returning a complete root plan; the caller manually substitutes the winner subtree into a parent Plan AST and dispatches as a normal non-leaf `:branch`. v0.6.6 wires the `:branch` handler that delegates to `mcts_search`, closing roadmap ┬¦3.3 option (a). No canonical-form change between v0.6.5 and v0.6.6 (`PLAN_CANONICAL_VERSION` stays at 1).

Scan `db` for all `:mcts/iteration` datoms with the recorded `search_id`; sort by `(iter-index, phase, action_hash)`; verify every `mcts/prev-hash` chains correctly. **Materialize Action objects from the recorded canonical Node bytes** per ┬¦13 replay-procedure step 4: for every `phase="expand"` proposal, deserialize `new_leaf_canonical` / `new_child_canonical` back into `Node` objects, construct the corresponding `SubstituteLeafAction` / `AddStepAction` / `ComposeWithSkillAction`, and **assert `reconstructed_node.id == new_leaf_id` (resp. `new_child_id`) byte-equality** ŌĆö this is the explicit Prop 6 verification that the audit log alone (no in-process state) suffices to recover Action bytes. Reconstruct `expander_cache` (now keyed by `plan_id` to `tuple[(Action, float), ...]` with materialized Actions) + `evaluator_cache` (`plan_id` to `float`). Re-run `mcts_search(initial_plan, expander=_ReplayLoudExpander, evaluator=_ReplayLoudEvaluator, started_at_ms=<original>, config=<original>, db=None)` where the replay-loud stubs raise `MCTSReplayCacheMiss(plan_id)` on any call (i.e. fail loudly if the recorded caches are insufficient). Assert: `result_replay.tree_dump == result.tree_dump` (byte-identical). **Without this step Stream B does not validate its paper claim.**

## 24. Open questions

1. **`c_puct` default ŌĆö `1.4` vs `1.0`.** Picked `1.4` per UCB1 closed-form (`ŌłÜ2 Ōēł 1.414`). AlphaZero / Leela used `1.0` most commonly. **Recommend keeping `1.4`** ŌĆö closed-form rationale is stronger than literature consensus; tune in v0.7+ once we have eval data. (Stockfish is NOT a citable precedent here ŌĆö it uses alpha-beta search, not PUCT; only Leela Chess Zero / AlphaZero are PUCT-family.)
2. **Winner selection ŌĆö most-visited vs highest-Q.** Picked most-visited (AlphaZero convention; "most confident"). Alternate: highest-Q (greedy on point estimate). **Recommend keeping most-visited** ŌĆö same reasoning as `c_puct`; well-precedented.
3. **`expander_k = 4` default.** Arbitrary. Real LLM expanders may want 8 or 16. **Recommend keeping `4`** for v0.6.5; bump in v0.7+ once we observe LLM-expansion cost.
4. **Skill-library composition heuristic.** `LLMExpander` reads `SkillLibrary.list_skills()` and offers top-K ŌĆö by what heuristic? Recency? Promotion-record G3 score? Random sample? **Recommend recency-weighted** for v0.6.5 (`skill/registered-at` is already a datom value); tune later. Documented in `LLMExpander` docstring.
5. **Dirichlet-noise on root prior (AlphaZero convention).** Adds exploration at the root. Defer to v0.7+ ŌĆö adds determinism complexity (need a seed-driven RNG threading through MCTS) and Stream B doesn't strictly need it for Prop 6.
6. **`:mcts/search` datom ŌĆö should it carry the full config payload or just `config_hash`?** Picked `config_hash` only (one datom value, content-addressed). Alternate: 8-12 separate datoms for individual config fields. **Recommend `config_hash` only** for v0.6.5; if Stream H's paper section wants per-field provenance, expand in v0.6.6.

(OQ-7 closed in ┬¦13 body ŌĆö `:mcts/iteration` `phase="expand"` `output_canonical_value` records full action_payload per proposal; resolution superseded the hash-only alternative.)

These are not blockers. The design ships under the listed pins; revisions land in v0.6.x point releases.
