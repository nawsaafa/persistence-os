## 1. One-paragraph summary

Module 3 today (after v0.6.0a1) parses, walks, dispatches, executes, optimizes, and 4-gate-promotes a content-addressed Plan AST. **`optimize()` searches the demo + prompt space inside a fixed plan structure.** Stream B adds the **structure** search: a PUCT-style tree-search wrapper that, given an initial Plan AST, a problem (training set + metric), an `Expander` that proposes plan-mutating actions, and an `Evaluator` that scores leaf plans, returns the highest-reward plan it found and feeds it through the existing `optimize()` → `promote()` → `SkillLibrary` pipeline. The whole stream lands as one new internal file `_mcts.py` plus a small surface widening on `__init__.py`. Determinism: every LLM call (expander + evaluator) is recorded as a `:mcts/iteration` provenance datom keyed by `(search_id, iteration_index, plan_id)`, so the search trajectory is reconstructible from the audit log alone — same Prop 1 / Prop 2 / Prop 4 composition that v0.6.0a1 inherits, lifted to the search layer.

## 2. Why this design

The roadmap pivot ([§3 ADR-2](2026-04-27-persistence-os-v1.0-roadmap.md)) reversed the prior `:branch MCTS → Phase 3 NeSy 2027` defer. The deferral was timing-driven, not technical, and the ferrari-first reframing makes the original Phase 3 timing irrelevant. Tree search over content-addressed Plan ASTs is the substrate's strongest **compositional planning** claim — without it, the paper is "we have a Plan AST" plus "we can tune prompts inside it" but not "we can search over the space of plan structures." Module 3 execution-without-MCTS leaves the paper one proposition short.

What the existing Module 3 surface lets us assume:
- **Plan AST is content-addressed.** `Node.id` is a deterministic 32-hex sha256 of canonical-JSON (`_ast.py:111-148`). Any plan reachable by any action sequence collapses to the same `Node.id`. This is the affordance MCTS leans on for transposition.
- **`execute()` returns a typed envelope.** `ExecutionResult.status` ∈ {"ok","failed"} + `LeafResult` per dispatched node + `FailureInfo` on raise (`_execute.py:117-139`). MCTS's deterministic-evaluator path is `execute(plan, dispatcher) → metric_fn(result, expected) → float`.
- **`optimize()` exists.** End-to-end MIPROv2 wrapper with provenance-pinned output (`_optimize.py:458-658`). MCTS terminates by handing the winner to `optimize()` for demo-search inside the chosen structure.
- **`promote()` orchestrates 4 gates.** `_promotion.py:663-851`. MCTS's terminal "register a winner skill" call is `promote(winner, ...) → PromotionRecord` then `SkillLibrary.register(winner, record, ...)`.
- **Fact store gives free provenance.** `db.transact([...], provenance={...})` writes datoms with bitemporal indices + audit chain (`fact/db.py:137`). One `:mcts/iteration` datom per iteration, content-addressed by search-id + iteration index, gives Prop 6 reconstructibility for free.

Plus one structural lesson preserved from v0.4 / v0.6.0a1: **find the existing affordance before adding.** `Dispatcher` + `execute()` + `optimize()` + `promote()` already exist; this stream is the **smallest committable extension** that wires them into a search loop, not a parallel system.

### What this design does NOT do

- **No new dispatcher.** The existing v0.4 `Dispatcher` is what evaluator-path executions run through. MCTS does not register handlers.
- **No replacement for `optimize()`.** MCTS calls `optimize()` on the winning structure as the closing step. The two are orthogonal: MCTS searches structure, `optimize()` searches demos within structure.
- **No new canonical-form.** `PLAN_CANONICAL_VERSION = 1` stays. All datoms added by MCTS are observational provenance, not structural.
- **No `:branch` tag executor.** Roadmap §3.3 contract option (a) (`:branch` as non-leaf structural tag with a registered dispatcher handler) is the path under which Stream B threads search results back into a runnable plan. **MCTS does not need to register a `:branch` handler in v0.6.5** — the search emits a winning subtree which the caller substitutes into the candidate Plan AST and dispatches as a normal non-leaf `:branch`. The handler-registration option (a) work is a separate ticket; v0.6.5 ships orthogonally to it.
- **No multi-process search.** Single-process, in-memory tree. Postgres-backed distributed MCTS is Phase 3 NeSy 2027 scope.
- **No live operator steering inside the search loop.** Stream D REPL ops (`inspect`, `rewind`, `branch`) operate on the audit log post-search; they do not interrupt an in-flight search.
- **No NeSy / symbolic constraint solver.** The `Expander` is LLM-backed in production but the contract is a Protocol; symbolic expanders are a v0.7+ topic.

## 3. Scope

One new internal file, three new Protocols, two new dataclasses, two new public functions, zero canonical-form change.

| # | File | Public API | Status |
|---|---|---|---|
| 1 | `src/persistence/plan/_mcts.py` | `mcts_search(...)`, `MCTSResult`, `Action`, `Expander`, `Evaluator`, `MCTSConfig` | NEW |
| 2 | `src/persistence/plan/__init__.py` | re-exports + `__all__` extension | EXTENDED |

**Module surface placement.** Single flat file `_mcts.py` next to `_optimize.py` and `_promotion.py`. **Not** a sub-module `persistence.plan.search/` — at v0.6.5 size (one search algorithm, one expander Protocol, one evaluator Protocol) the sub-module overhead exceeds the win, and we want the import path symmetry with `_optimize.py` / `_skill_library.py`. If v0.7+ adds beam-search / iterative-widening / non-MCTS variants, **then** lift to `persistence.plan.search/` (ADR-11 below).

**Plan-attr key style (key code↔doc rule, inherited from v0.6.0a1).** All Plan-AST `Node.attrs` keys are plain strings without leading colon (`_ast.py:71-100` rejects colon-prefixed keys at construction time). Provenance attr-key examples in this design read as `"mcts/search-id"`, **not** `":mcts/search-id"`. Datom `a` keys follow the same convention (`datom.py:114`: `"mcts/iteration"`-style — bare, no leading colon at the storage boundary). Where the roadmap or sister docs write `:mcts/...` with a leading colon, that is the **wire form**; the on-disk Datom slot strips the colon.

### Out of scope (explicit deferrals)

- **Distributed search.** Single-process. Multi-worker MCTS over Postgres-backed shared state → Phase 3 NeSy 2027.
- **Adaptive c_puct.** Static `c_puct` (default `1.4`) with one ADR (ADR-3) below. Adaptive schedules → v0.7+.
- **Custom rollout policies.** No "policy network" between leaf and evaluator. Evaluator is called directly on each newly expanded leaf — i.e. **rollout depth = 1**. The Plan AST is the policy; MIPROv2 inside `optimize()` is the policy refinement step.
- **Symbolic / constraint-based expanders.** Out-of-scope concept; the expander Protocol shape is open enough that v0.7+ symbolic expanders satisfy it without a contract change.
- **Operator-in-the-loop steering inside a search.** Stream D REPL ops post-search only.
- **MCTS-aware reward shaping.** No multi-objective reward, no Pareto front. Single scalar reward in [0,1] (or higher-is-better float — the substrate doesn't enforce a bound, but evaluator implementations should normalize).

## 4. Search-tree node identity vs Plan AST node identity

### Problem

`MCTSNode` carries search statistics (`visits`, `total_value`, prior, parent pointer, child dict). Plan `Node` carries content (tag, attrs, children) and content-addressed `Node.id`. They are **not the same object** — one search tree may visit the same plan twice via different action sequences (transposition), and one plan node may appear in multiple search trees over the same problem.

### Decision (ADR-1)

**Search-tree nodes are keyed by Plan `Node.id`.** A transposition table maps `plan.id → MCTSNode`; reaching the same plan by two different action sequences updates the **same** `MCTSNode`'s statistics. This is the "graph MCTS" or "transposition-aware MCTS" variant — standard in modern AlphaZero-style implementations and the natural fit for content-addressed plans.

```python
@dataclass
class MCTSNode:
    plan_id: str                        # Plan AST content hash (Node.id)
    visits: int = 0
    total_value: float = 0.0
    prior: float = 0.0                  # P(s,a) from parent's expansion, set on first visit
    children: dict[str, "MCTSEdge"] = field(default_factory=dict)
    # No parent pointer — the search tree is a DAG over plan ids, NOT a
    # tree. Backup walks the *path* taken in the current iteration, not
    # parent edges. This is the right call for transposition correctness:
    # a node reached via two paths must back up only the path actually
    # traversed in this iteration.
    is_terminal: bool = False           # True iff Expander returned 0 actions OR
                                        #   Evaluator returned a sentinel-terminal score
    @property
    def q_value(self) -> float:
        return 0.0 if self.visits == 0 else self.total_value / self.visits
```

```python
@dataclass(frozen=True)
class MCTSEdge:
    """One outgoing edge from MCTSNode. Keyed by action canonical hash."""
    action_hash: str                    # canonical-JSON sha256 of Action payload
    action: Action                      # the mutation itself, kept on the edge
    child_plan_id: str                  # Plan id reached by applying action
    visits_through_edge: int = 0        # N(s,a) — distinct from MCTSNode.visits
    total_value_through_edge: float = 0.0
```

**Why edge-statistics distinct from node-statistics.** PUCT uses `N(s,a)` (visits along an edge) and `Q(s,a)` (mean value along an edge), NOT just `N(s)` and `Q(s)`. With transposition, these can differ: a child node may have `N(s')=10` from two parent edges of `N(s_a, a_a)=4` and `N(s_b, a_b)=6`. The edge dataclass keeps the math correct.

**Consequence.** "Same plan reached by different action sequences" is **one MCTSNode**. The action-sequence multiplicity lives on the **edges**, where it belongs. Backup updates `MCTSEdge.visits_through_edge` + `MCTSNode.visits` along the traversed path; both rise by 1 per iteration. (This matches Lan & Yi 2022 "Graph-Aware MCTS" and AlphaZero's transposition handling.)

## 5. Action space

### Decision (ADR-2)

The minimal viable action space for v0.6.5 is **three action kinds**, chosen to give compositional reach without exploding the branching factor:

```python
@dataclass(frozen=True)
class SubstituteLeafAction:
    """Replace a leaf node in the plan with a different leaf.

    Most common LLM-proposable mutation. The replacement leaf must be
    content-addressed (a `Node`), so it round-trips canonical-JSON.
    """
    target_path: tuple[int, ...]        # e.g. (0, 2) = root.children[0].children[2]
    new_leaf: Node                      # tag, attrs, children=()

@dataclass(frozen=True)
class AddStepAction:
    """Insert a new child at `target_path` index `at`.

    Restricted to non-leaf parents (`children` is a tuple, but leaves are
    `children=()` — adding a child to a leaf would change its kind, which
    is structurally invalid).
    """
    target_path: tuple[int, ...]
    at: int                             # insertion index in parent.children
    new_child: Node

@dataclass(frozen=True)
class ComposeWithSkillAction:
    """Wrap the subtree at `target_path` inside a registered SkillLibrary skill.

    The skill_id is looked up via SkillLibrary.lookup(skill_id); its
    Plan AST becomes the new parent, with the original subtree spliced
    in as the skill's first child. Only skills with a Plan AST whose
    root tag accepts non-leaf children may be composed (we enforce
    structurally — see §6 invariants).
    """
    target_path: tuple[int, ...]
    skill_id: str                       # "skill/<16-hex>" form
```

Union type:

```python
Action = SubstituteLeafAction | AddStepAction | ComposeWithSkillAction
```

### What's deferred

| Action | Defer reason |
|---|---|
| `RemoveStepAction` | Symmetry with `AddStep` is tempting, but a remove can break dispatcher-handler invariants (e.g. a `:seq` with zero children is structurally fine but semantically vacuous). Re-add in v0.7 once we have eval data showing remove-actions improve search. |
| `SwapMetricAction` | Metric is a stream-level constant, not part of the candidate plan. Swapping metric mid-search invalidates Q-values. If the user wants a different metric, run a different search. |
| `LLMProposedFreeFormAction` | An expander returning "the LLM said do X to get plan Y" is already encoded by `SubstituteLeaf(target_path=..., new_leaf=Y)`. We do NOT need a separate kind for it. The expander's job is to *propose* `(action, prior)` pairs from any source. |
| `ReorderChildrenAction` | Tuple permutation. Useful but increases branching factor; defer. |

### Rationale

Three action kinds give the search **full reach over the v0.6.0a1 substrate**: any plan reachable by repeatedly substituting leaves + adding subtrees + composing skills is reachable by a finite SubstituteLeaf+AddStep+ComposeWithSkill sequence from the initial plan. This is sufficient for the paper claim (Prop 6: provenance reconstructibility, NOT completeness over the plan-AST space — the paper does not claim MCTS reaches every plan, only that what it does reach is reconstructible).

### Action canonical hash

```python
def _action_hash(action: Action) -> str:
    """SHA-256 hex of canonical-JSON of the action's structural payload.

    Used as the dict key for MCTSNode.children. Same canonical-JSON
    rule as Node.id (sort_keys, separators=(",",":"),
    allow_nan=False). Two actions with the same target_path AND same
    new_leaf/new_child/skill_id collapse to the same hash → same edge.
    """
    payload = {"kind": action.__class__.__name__, **asdict(action)}
    # Recursively canonicalize Node payloads via the same rule used by
    # Node.id — actions carry Nodes, and Node round-trips canonicalize.
    encoded = json.dumps(payload, sort_keys=True, separators=(",",":"), allow_nan=False)
    return hashlib.sha256(encoded.encode("utf-8")).hexdigest()
```

## 6. Action application + plan invariants

### Decision (ADR-3 — applicator contract)

`apply_action(plan: Node, action: Action) -> Node` is a pure function (in `_mcts.py`, private) that:

1. Resolves `action.target_path` against `plan` (raises `IndexError` if path is invalid).
2. Constructs the new `Node` tree by structural recursion, leaving untouched subtrees object-identical.
3. Returns the new root.

**Plan invariants preserved by every applicator branch:**
- Tag prefix rule: `Node.tag` always starts with `:`.
- Attr-key rule: no leading-colon attr keys.
- Children type: `tuple[Node, ...]`.
- Coercion: any new attrs the applicator emits are JSON-native (str/int/float/bool/None) — no coercion-required types added by the search layer.

If the applicator constructs an invalid `Node` (e.g. SubstituteLeaf into a path that doesn't exist), **the `Node(...)` constructor raises**, and MCTS treats this as a **rejected action** (does not enter the tree, does not update statistics). Logged via `:mcts/iteration` datom with an `"action_rejected": True` flag (see §8).

### Structural validation

```python
def _is_valid_action_for_plan(plan: Node, action: Action) -> bool:
    """Cheap up-front check the expander can call to filter invalid actions
    before we count them against the prior budget.

    Substrate-level: path resolves, the tag-tree shape is consistent.
    NOT semantic: 'is this a good action?' is the evaluator's job.
    """
    try:
        _resolve_path(plan, action.target_path)
    except IndexError:
        return False
    if isinstance(action, AddStepAction):
        parent = _resolve_path(plan, action.target_path)
        if not parent.children and parent.tag in _LEAF_REQUIRED_TAGS:
            return False
    return True
```

`_LEAF_REQUIRED_TAGS` defaults to `frozenset()` (empty) for v0.6.5; the dispatcher seam already enforces tag-leaf invariants at execute-time. We expose the constant for forward-compat with v0.7+ tag families.

## 7. PUCT formula + hyperparameters

### Decision (ADR-4)

Standard AlphaZero PUCT:

```
UCB(s, a) = Q(s, a) + c_puct * P(s, a) * sqrt(parent.visits) / (1 + N(s, a))
```

Where:
- `Q(s, a) = MCTSEdge.total_value_through_edge / max(1, MCTSEdge.visits_through_edge)` — mean reward along the edge. Returns `0.0` for unvisited edges (cold-start; the prior carries the exploration weight).
- `P(s, a) = MCTSEdge.action.prior` — softmax-normalized prior emitted by the `Expander` at expansion time. Re-normalization is the expander's job, not MCTS's.
- `parent.visits = MCTSNode.visits` of the parent (NOT the edge's `visits_through_edge`).
- `N(s, a) = MCTSEdge.visits_through_edge`.
- `c_puct` — exploration constant. **Default `1.4`.** Rationale: `1.4 ≈ sqrt(2)` is the classic UCB1 constant; AlphaZero used 1.0–4.0 with 1.0 most common but 1.4 is the median across the Stockfish/Leela/AlphaZero literature for "balanced" exploration. We expose it as a `MCTSConfig.c_puct: float = 1.4` knob; calling-site override is one keyword arg.

### Cold-start prior

When a child is first added (expansion phase), its `MCTSEdge.prior` is set from the expander's softmax. **Cold-start `Q(s,a) = 0.0`** — the edge's value is unknown, and the prior is what drives the first selection. No "first-visit value bonus" — that's an FPU (First Play Urgency) variant that shifts behavior in subtle ways; not worth the complexity for v0.6.5.

### Selection rule

```python
def _select_child(node: MCTSNode, c_puct: float) -> MCTSEdge | None:
    if not node.children:
        return None
    parent_visits = node.visits
    sqrt_parent = math.sqrt(parent_visits) if parent_visits > 0 else 1.0
    best_edge: MCTSEdge | None = None
    best_score = -math.inf
    for edge in node.children.values():
        n_sa = edge.visits_through_edge
        q = (edge.total_value_through_edge / n_sa) if n_sa > 0 else 0.0
        u = c_puct * edge.prior * sqrt_parent / (1 + n_sa)
        score = q + u
        if score > best_score:
            best_score = score
            best_edge = edge
    return best_edge
```

**Tie-breaking.** Stable: first edge in dict-insertion order wins on equal score. Dict insertion order is preserved in Python 3.7+. Determinism-preserving.

## 8. Expander Protocol

### Decision (ADR-5)

```python
@runtime_checkable
class Expander(Protocol):
    """Proposes (action, prior) pairs at expansion time.

    Implementations are typically LLM-backed (an LLM judges the plan
    and proposes mutations + softmax-normalized priors), but the
    Protocol shape is open: deterministic stubs satisfy it for tests,
    symbolic expanders satisfy it for v0.7+, and a 'always propose
    skill X' fixture satisfies it for closed-loop integration tests.

    The expander MUST return softmax-normalized priors (sum to 1.0).
    MCTS does NOT re-normalize. This keeps the contract crisp: the
    expander owns the policy distribution; MCTS owns the tree.

    Empty iterable = terminal node. The MCTSNode is marked is_terminal,
    no further selection visits it (selection short-circuits on
    is_terminal nodes — see §10).
    """

    def propose(
        self,
        plan: Node,
        *,
        k: int,                         # caller's beam width hint; expander may return ≤ k
    ) -> Iterable[tuple[Action, float]]: ...
```

**Caching.** Plans are content-addressed; expander calls are pure functions of `plan.id` (modulo the LLM's nondeterminism, which is what the audit chain captures). MCTS keeps a `_expander_cache: dict[str, tuple[tuple[Action, float], ...]]` — `plan.id → frozen list of proposals`. **Cache hit ⇒ no LLM call ⇒ no `:mcts/iteration` provenance datom.** Cache miss ⇒ one LLM call ⇒ one `:mcts/iteration` datom with `phase="expand"`.

### Two expanders ship in v0.6.5

- `LLMExpander` — production wiring. Calls a registered `:mcts/expander` handler-id via the same registry style as Module 2's `:llm/handler-id` (ADR-9 below). Body of the LLM prompt is structured: plan EDN + "propose K mutations as `(kind, target_path, payload)` triples + softmax priors." Response parsing is a JSON-mode adapter — **no free-form parsing**.
- `_StaticExpander` (test-only) — fabricated stub. Fixed proposal list per plan.id.

Both ship in `_mcts.py`; the LLM expander imports its provider through the existing `:llm/handler-id` indirection so the model is pluggable.

## 9. Evaluator Protocol

### Decision (ADR-6)

```python
@runtime_checkable
class Evaluator(Protocol):
    """Returns a scalar reward for a leaf plan.

    Two natural sources, both satisfy the Protocol:

      (a) DeterministicExecuteEvaluator — runs `execute(plan, dispatcher,
          env=example["inputs"])` per training example, collects
          ExecutionResult, hands (result, expected) to the metric_fn,
          returns the mean. Same shape as optimize()'s _mean_score
          helper. Determinism inherits from execute() + dispatcher
          determinism.

      (b) LLMJudgeEvaluator — calls a registered :mcts/evaluator
          handler-id with the plan + a held-out dev example, parses a
          0-1 score. NON-deterministic; cached by plan.id (next
          paragraph).

    The substrate ships ONE evaluator contract; both flavors satisfy
    it. Pluggable strategy is over.
    """

    def evaluate(self, plan: Node) -> float: ...
```

**Caching by `plan.id` is essential.** Plans are content-addressed; evaluator calls collapse to `plan.id → float`. MCTS keeps `_evaluator_cache: dict[str, float]` and asks the evaluator only on cache miss. **Cache hit ⇒ no `:mcts/iteration` datom with `phase="evaluate"`. Cache miss ⇒ one datom with `phase="evaluate"` recording `plan_id` + `score`.**

### Two evaluators ship in v0.6.5

- `DeterministicExecuteEvaluator(training_set, metric, dispatcher)` — production-ready under the v0.6.0a1 wiring. Reuses `lookup_metric` + `execute`. Deterministic given dispatcher determinism (which v0.6.0a1's no-`time.time()` discipline guarantees).
- `LLMJudgeEvaluator(handler_id, dev_example)` — production wiring through the registry seam. **Caches by `plan.id` so repeated visits to the same plan don't re-call the LLM.**

### Failure handling

If the evaluator raises, MCTS treats this as a **rejected leaf**:
- The expansion is not kept in the tree.
- A `:mcts/iteration` datom is written with `phase="evaluate"` + `error_class` + `error_repr` (mirroring `FailureInfo`'s shape from `_execute.py`).
- The iteration counter advances; the search continues.

This is symmetric with v0.6.0a1's per-leaf failure capture in `execute()`.

## 10. Determinism + replayability

### Problem

v0.6.0a1's whole point is byte-identical replay (Prop 4). MCTS introduces non-determinism: LLM expander, LLM evaluator. **What does "replayable MCTS" mean?**

### Decision (ADR-7)

Replayable MCTS = **the audit log uniquely determines the search trajectory**. Three options were on the table:

| Option | Approach | Verdict |
|---|---|---|
| (a) Cache all LLM calls keyed by plan.id + record the cache. | Persist `_expander_cache` + `_evaluator_cache` to fact store; on replay, hit the cache instead of calling the LLM. | **Selected as the primary path.** Aligns with v0.6.0a1's hash-keyed audit cache discipline. |
| (b) Record action sequence; on replay, re-walk the sequence ignoring the tree. | Cheap to record but discards the tree state; transposition-correctness lost on replay. Replay produces the same winner but not the same Q-values. | Rejected — Prop 6 wants the search trajectory reconstructible, not just the winner. |
| (c) `:mcts/iteration` provenance datom per iteration capturing inputs + outputs. | Strongest — every iteration's selection / expansion / evaluation / backup is a fact; replay re-derives the tree from the log. | **Selected as the primary path, layered on top of (a).** Both (a) and (c) ship; (c) is the substrate-level claim, (a) is the operational caching. |

**The combined path:**
1. Every LLM call (expander + evaluator cache miss) writes a `:mcts/iteration` datom with `phase ∈ {"expand","evaluate"}` recording `plan_id`, `output` (the proposals or score), and a content hash of the inputs.
2. On replay, MCTS reconstructs the tree by scanning `:mcts/iteration` datoms with the matching `mcts/search-id`, in iteration-index order. Cache hits on cache miss-recorded datoms by `plan_id`.
3. The audit chain (Module 2) makes the iteration datoms tamper-evident; reordering or fabricating iterations breaks Prop 2.

Combined: the audit log is a sufficient statistic for the entire search. Defends Prop 6.

### Determinism-pin contract

For a given `(initial_plan_id, search_id, seed, expander_responses, evaluator_responses)` tuple, the search trajectory is byte-identical across two runs. The seed governs:
- Tie-breaking (insertion-order is already deterministic, but explicit seed for shuffle-style operations like Dirichlet noise on the root prior, **deferred to v0.7+**).
- The action-hash → MCTSEdge insertion order on first expansion.

Test-side, this is a `Hypothesis @given` property: same expander stub + same evaluator stub + same config + same initial plan → byte-identical `MCTSResult.tree_dump`.

## 11. Termination

### Decision (ADR-8)

Three termination signals, **OR-combined** (whichever fires first):

```python
@dataclass(frozen=True, slots=True)
class MCTSConfig:
    max_iter: int = 200                 # hard ceiling on total iterations
    max_unique_plans: int = 64          # stop after this many distinct plan ids visited
    simple_regret_threshold: float | None = None
                                        # if set, stop when (best - second-best) Q-value
                                        # at the root exceeds this threshold for K
                                        # consecutive iterations
    simple_regret_window: int = 5       # K above
    wall_clock_budget_ms: int | None = None
                                        # optional; routes through :clock/now effect, not
                                        # time.time() — repo discipline. None = no clock budget.
    c_puct: float = 1.4
    expander_k: int = 4                 # default beam width per expand
    seed: int = 0                       # determinism seed
```

**Defaults rationale:**
- `max_iter=200` — empirical AlphaZero-style work shows diminishing returns past ~200 iters at small branching factor. Cheap to bump.
- `max_unique_plans=64` — content-addressing makes "we've seen this plan" cheap to count; 64 distinct structures is a reasonable upper bound for the problem class v0.6.5 targets.
- `simple_regret_threshold=None` (off by default) — adaptive termination is finicky; opt-in.
- `wall_clock_budget_ms=None` — wall-clock budgets violate the no-`time.time()` discipline by default. The hook routes through `:clock/now` for callers who want it; opt-in.

The first termination signal observed wins; the search exits gracefully and `mcts_search()` returns a `MCTSResult` with `terminated_by ∈ {"max_iter","max_unique_plans","simple_regret","wall_clock"}`.

## 12. Skill-library closed loop

### Decision (ADR-9)

`mcts_search()` is **pure search**. It returns the winning Plan AST + the search tree summary. **It does NOT call `optimize()` or `promote()`.** The caller orchestrates the closed loop.

Rationale:
- Single-responsibility: search is one thing, optimization is another, promotion is a third.
- Test isolation: search tests don't need DSPy installed.
- Caller flexibility: a developer running search interactively wants the winner returned; a CI-style closed-loop wants `optimize` + `promote` + `register`. The substrate exposes both layers.

The roadmap-mandated closed loop is documented in `mcts_search()`'s docstring as the canonical post-search recipe:

```python
result = mcts_search(initial_plan, expander=..., evaluator=..., config=...)
optimized = optimize(result.winner, training_set=..., metric=..., dispatcher=...)
record = promote(optimized.plan, db=db, optimized_score=optimized.score, baseline_score=optimized.score_baseline, ...)
skill_id = SkillLibrary(db).register(optimized.plan, record, registered_at_ms=...)
```

### "Use existing skill" Action shape

`ComposeWithSkillAction(target_path, skill_id)` (§5) is how MCTS pulls promoted skills back into the search prior. The expander chooses which skills are candidates — the production `LLMExpander` reads `SkillLibrary.list_skills()` and offers the top-K (by some heuristic, e.g. recency-weighted by `skill/registered-at`) as composition candidates alongside SubstituteLeaf / AddStep proposals.

The applicator resolves `ComposeWithSkillAction` via:

```python
def _apply_compose_with_skill(plan: Node, action: ComposeWithSkillAction, skill_library: SkillLibrary) -> Node:
    looked_up = skill_library.lookup(action.skill_id)
    if looked_up is None:
        raise ValueError(f"ComposeWithSkillAction: skill_id {action.skill_id!r} not registered")
    skill_plan, _record = looked_up
    subtree = _resolve_path(plan, action.target_path)
    new_skill_root = Node(
        tag=skill_plan.tag,
        attrs=skill_plan.attrs,
        children=(subtree, *skill_plan.children),
    )
    return _replace_at_path(plan, action.target_path, new_skill_root)
```

`mcts_search()` accepts an optional `skill_library: SkillLibrary | None = None` arg; if `None`, `ComposeWithSkillAction` proposals from the expander are silently rejected (logged in `:mcts/iteration` with `action_rejected=True, reason="no skill_library"`).

## 13. Provenance datom shape

### Decision (ADR-10)

Two new datom attribute namespaces, both under `mcts/`. **Every search emits one `:mcts/search` summary datom + one `:mcts/iteration` datom per LLM-touching iteration.**

#### `mcts/search` (one per `mcts_search()` call)

```python
[
  Datom(e=search_id, a="mcts/initial-plan-id", v=initial_plan.id, ...),
  Datom(e=search_id, a="mcts/winner-plan-id",  v=winner_plan.id,  ...),
  Datom(e=search_id, a="mcts/config-hash",     v=config_canonical_hash, ...),
  Datom(e=search_id, a="mcts/iter-count",      v=actual_iter_count, ...),
  Datom(e=search_id, a="mcts/terminated-by",   v=termination_reason, ...),
  Datom(e=search_id, a="mcts/started-at",      v=started_at_ms, ...),
  Datom(e=search_id, a="mcts/finished-at",     v=finished_at_ms, ...),
]
```

`search_id` is content-addressed:

```python
search_id = "mcts/" + sha256(canonical_json({
    "initial_plan_id": initial_plan.id,
    "config_hash": config_canonical_hash,
    "started_at": started_at_ms,
}))[:16]
```

The `search_id` is in its own namespace (`mcts/<16-hex>`), parallel with `skill_id`'s `skill/<16-hex>`. The 16-hex slice is consistent with `_skill_library.py`'s `_SKILL_ID_HEX_WIDTH = 16`.

#### `mcts/iteration` (one per LLM-touching iteration)

```python
[
  Datom(e=iter_id, a="mcts/search-id",     v=search_id, ...),
  Datom(e=iter_id, a="mcts/iter-index",    v=iter_index, ...),         # 0-based
  Datom(e=iter_id, a="mcts/phase",         v=phase, ...),               # "expand" | "evaluate"
  Datom(e=iter_id, a="mcts/plan-id",       v=plan_id, ...),
  Datom(e=iter_id, a="mcts/inputs-hash",   v=inputs_canonical_hash, ...),
  Datom(e=iter_id, a="mcts/output",        v=output_canonical_value, ...),
  Datom(e=iter_id, a="mcts/error-class",   v=error_class_or_None, ...), # only on raises
  Datom(e=iter_id, a="mcts/error-repr",    v=error_repr_or_None, ...),  # only on raises
]
```

`iter_id`:

```python
iter_id = "mcts-iter/" + sha256(canonical_json({
    "search_id": search_id,
    "iter_index": iter_index,
    "phase": phase,
    "plan_id": plan_id,
}))[:16]
```

`output_canonical_value` shape:
- `phase="expand"` ⇒ list of `{"action_hash": str, "action_kind": str, "prior": float}` (proposals); the action payload itself is reconstructible from `action_kind` + the next iteration's plan_id, so we don't duplicate it.
- `phase="evaluate"` ⇒ float (the score) or `None` if the evaluator raised.

**Cache-hit iterations are NOT recorded.** Only cache-miss LLM calls produce datoms. Replay reconstructs the tree by:
1. Scan `:mcts/search` for `search_id`.
2. Scan `:mcts/iteration` for that `search_id`, sort by `iter-index`.
3. Pre-populate `_expander_cache` + `_evaluator_cache` from the recorded outputs.
4. Re-run the search loop with the same config + the populated caches; every LLM call hits cache.

Defends Prop 6: full search trajectory reconstructible from the audit log.

## 14. Failure-mode contracts

| Mode | What happens | Provenance | Raises? |
|---|---|---|---|
| Evaluator raises | Iteration recorded with `error_class` / `error_repr`; expansion not kept in tree; iteration counter advances. | `:mcts/iteration` w/ `phase="evaluate"`, error fields populated. | No — search continues. |
| Expander returns 0 candidates for a non-root plan | MCTSNode is marked `is_terminal=True`; subsequent selection skips it (selection looks for non-terminal children, falls back to backing up the rollout immediately). | `:mcts/iteration` w/ `phase="expand"`, output = `[]`. | No. |
| Expander returns 0 candidates for the root plan on iter 0 | Root has no children to select; search exits immediately with `terminated_by="exhausted"`, winner=initial_plan. | `:mcts/iteration` w/ `phase="expand"` recorded; `:mcts/search` w/ `terminated_by="exhausted"`. | No. |
| All proposed actions produce invalid plans | Each action_application raise is logged as a rejected action (`:mcts/iteration` extra field `action_rejected=True`, `reason=<exc class>`). If ALL actions reject, the node is marked `is_terminal=True`. | One datom per rejected action attempt (cheap; rare). | No. |
| `ComposeWithSkillAction` for an unregistered `skill_id` | Action rejected with `reason="skill_not_registered"`. | One `action_rejected` datom. | No. |
| Empty `training_set` to `DeterministicExecuteEvaluator` | Evaluator returns `0.0` for every plan; the search degenerates to prior-driven selection. **`UserWarning` emitted at evaluator-construction** ("MCTS evaluator built over empty training set — no signal to discriminate plans"). Symmetric with G1 / G2's empty-corpus warnings. | No iteration-level effect; warning only. | No. |
| `MetricNotRegistered` from evaluator | Re-raised at `mcts_search()` boundary (caller's contract is broken — same posture as `optimize()`). | None — search aborts. | Yes (`MetricNotRegistered`). |
| `OptimizerNotAvailable` (DSPy missing) | N/A — `mcts_search()` does not import DSPy. The closed-loop caller's `optimize()` call may raise; that's the caller's contract. | None at MCTS layer. | No (at MCTS layer). |
| `c_puct` ≤ 0 | `MCTSConfig.__post_init__` raises `ValueError("c_puct must be > 0")`. | None — config-time check. | Yes. |
| `max_iter` ≤ 0 | `MCTSConfig.__post_init__` raises `ValueError`. | None. | Yes. |

**Vacuous-truth analogues** (mirroring v0.6.0a1's W1.C / W1.A patterns):
- Empty training_set → `UserWarning` at evaluator build (above).
- Zero iterations completed (e.g. `max_iter=0`, which is rejected at config validation) — never reachable.
- All evaluator calls failed → `MCTSResult.winner = initial_plan` + `terminated_by="all_evaluations_failed"` + `UserWarning`.

## 16. Internal contracts

### MCTS loop pseudocode

```python
def mcts_search(...):
    config = config or MCTSConfig()
    config.__post_init__()  # config validation
    transposition: dict[str, MCTSNode] = {}
    expander_cache: dict[str, tuple[tuple[Action, float], ...]] = {}
    evaluator_cache: dict[str, float] = {}

    root = MCTSNode(plan_id=initial_plan.id)
    transposition[initial_plan.id] = root
    plan_by_id: dict[str, Node] = {initial_plan.id: initial_plan}

    search_id = _derive_search_id(initial_plan.id, config, started_at_ms)
    # Emit :mcts/search "started" provenance up front (idempotent on
    # winner-plan-id update at the end).

    iter_index = 0
    while not _termination_reached(...):
        path: list[tuple[MCTSNode, MCTSEdge | None]] = []

        # 1. SELECT
        node = root
        while node.children and not node.is_terminal:
            edge = _select_child(node, config.c_puct)
            if edge is None:
                break
            path.append((node, edge))
            node = transposition[edge.child_plan_id]

        # 2. EXPAND (if non-terminal & unexpanded)
        if not node.is_terminal and not node.children:
            proposals = expander_cache.get(node.plan_id)
            if proposals is None:
                proposals = tuple(expander.propose(plan_by_id[node.plan_id], k=config.expander_k))
                expander_cache[node.plan_id] = proposals
                _emit_iteration_datom(db, search_id, iter_index, "expand",
                                      plan_id=node.plan_id, output=proposals)
            if not proposals:
                node.is_terminal = True
            else:
                _populate_children(node, proposals, plan_by_id, transposition,
                                   skill_library, db, search_id, iter_index)

        # 3. EVALUATE (the leaf reached)
        leaf_plan_id = node.plan_id
        score = evaluator_cache.get(leaf_plan_id)
        if score is None:
            try:
                score = evaluator.evaluate(plan_by_id[leaf_plan_id])
            except Exception as exc:
                _emit_iteration_datom(db, search_id, iter_index, "evaluate",
                                      plan_id=leaf_plan_id, output=None,
                                      error_class=type(exc).__name__,
                                      error_repr=repr(exc))
                iter_index += 1
                continue
            evaluator_cache[leaf_plan_id] = score
            _emit_iteration_datom(db, search_id, iter_index, "evaluate",
                                  plan_id=leaf_plan_id, output=score)

        # 4. BACKUP along the traversed path
        for parent_node, edge in path:
            parent_node.visits += 1
            if edge is not None:
                edge.visits_through_edge += 1
                edge.total_value_through_edge += score
        node.visits += 1   # the leaf itself

        iter_index += 1

    # Pick winner: root edge with max visits (AlphaZero convention) OR max Q
    # (we use max visits — most-visited is most-confident).
    winner_edge = max(root.children.values(), key=lambda e: e.visits_through_edge, default=None)
    winner_plan = plan_by_id[winner_edge.child_plan_id] if winner_edge else initial_plan
    ...
```

### Invariants (runtime-checkable)

1. **Plan-id stability.** For every `MCTSNode n` and every `MCTSEdge e` with `e.child_plan_id = n.plan_id`, `plan_by_id[n.plan_id].id == n.plan_id` (content-addressing identity).
2. **Visit conservation.** `MCTSNode.visits == sum(edge.visits_through_edge for edge in node.children.values()) + (1 if leaf else 0)`. (Approximate; loose at the root where the iteration counter is the source of truth.)
3. **Cache-hit invariant.** `expander_cache.keys() ⊆ {visited plan ids that triggered an expand}`; `evaluator_cache.keys() ⊆ {visited plan ids that triggered an evaluate}`.
4. **Determinism invariant.** Two `mcts_search()` calls with identical (initial_plan, config, expander_responses, evaluator_responses, started_at_ms) produce equal `MCTSResult.tree_dump`.

(1) and (2) are property-test-checkable. (3) is a debug assertion. (4) is the load-bearing pin test.

## 17. ADRs

### ADR-1 — Search-tree node identity = Plan AST `Node.id` (transposition table)
**Options.** (a) Plan-id-keyed transposition table (graph MCTS). (b) Distinct MCTSNode per action sequence (pure tree).
**Decision.** (a). Plans are content-addressed; the `Node.id` is the natural search-state key. Transposition gives statistical efficiency at no correctness cost.
**Rationale.** Same plan reached via different sequences shares Q-value learning. Edge-keyed `MCTSEdge.visits_through_edge` keeps path-specific PUCT correct (§4 above).
**Consequences.** Backup walks the *path traversed this iteration*, not parent pointers. The MCTS object graph is a DAG, not a tree. No parent pointer on `MCTSNode`.

### ADR-2 — Action space = Substitute / AddStep / ComposeWithSkill
**Options.** (a) Three action kinds (selected). (b) Five+ kinds including Remove, Reorder, SwapMetric. (c) Free-form LLM mutation as one kind.
**Decision.** (a) — minimal viable.
**Rationale.** Three kinds cover compositional reach without exploding the branching factor. Defer Remove (semantic risk), SwapMetric (out-of-bounds), Reorder (branching cost). Free-form mutations are encoded as SubstituteLeaf + AddStep.
**Consequences.** v0.7+ may add Remove / Reorder; the Action union widens, the dispatch shape stays.

### ADR-3 — Action applicator is pure + structural; invalidity = rejected
**Options.** (a) Pure applicator, raise on invalid path; (b) applicator returns Optional[Node].
**Decision.** (a). `Node(...)`'s constructor already raises on invariants; we let it.
**Rationale.** "Action produced an invalid plan" is a real signal — the audit log records it via `action_rejected=True`. Optional return would hide failure modes.
**Consequences.** Every applicator branch has a try/except in MCTS that converts `ValueError` / `IndexError` to a rejected-action provenance datom.

### ADR-4 — PUCT formula + `c_puct = 1.4` default
**Options.** (a) `c_puct = 1.0` (AlphaZero classic), (b) `c_puct = 1.4 ≈ √2` (UCB1 classic), (c) adaptive schedule.
**Decision.** (b). Static, configurable.
**Rationale.** `1.4` is the median of the AlphaZero / Stockfish / Leela literature for "balanced" exploration; `√2` is the closed-form UCB1 value. Adaptive schedules add complexity without v0.6.5-scope value.
**Consequences.** Single hyperparam knob; v0.7+ adaptive variants don't break the contract.

### ADR-5 — Expander Protocol returns softmax-normalized priors
**Options.** (a) Expander returns normalized priors; (b) MCTS re-normalizes; (c) Expander returns logits, MCTS does softmax.
**Decision.** (a). Expander owns the policy distribution.
**Rationale.** Crisp contract boundary. MCTS is pure tree machinery; the policy is the expander's responsibility.
**Consequences.** Test stubs must softmax-normalize; documented in Protocol docstring + asserted in `test_expander_contract.py`.

### ADR-6 — Evaluator Protocol takes plan, returns float; cached by `plan.id`
**Options.** (a) Single Protocol, two ship-time implementations (DeterministicExecute + LLMJudge); (b) two distinct Protocols.
**Decision.** (a). Pluggable strategy via Protocol satisfaction, not type union.
**Rationale.** `plan.id`-keyed caching makes the deterministic vs LLM choice transparent to the MCTS loop. Both sources are content-addressed pure functions of the plan.
**Consequences.** `evaluator_cache[plan.id]` is the hot path; cache-miss cost is the only difference between deterministic and LLM evaluators.

### ADR-7 — Replayability via `:mcts/iteration` provenance datoms + LLM cache
**Options.** (a) Cache + record (selected). (b) Action-sequence-only recording. (c) Per-iteration provenance datoms.
**Decision.** (a) and (c) combined.
**Rationale.** Audit log = sufficient statistic for the search trajectory. Defends Prop 6.
**Consequences.** Every cache-miss LLM call writes one datom; replay re-derives the tree from the log.

### ADR-8 — Termination: `max_iter | max_unique_plans | simple_regret | wall_clock`, OR-combined
**Options.** (a) `max_iter` only; (b) multi-signal OR.
**Decision.** (b).
**Rationale.** Different problem classes want different termination shapes. `max_iter` is the minimum viable; the others opt-in.
**Consequences.** `MCTSResult.terminated_by` records which fired; one ARIS R2 readability concern (multi-signal exit logic) is mitigated by a single `_termination_reached()` helper.

### ADR-9 — Skill-library closed loop is the caller's recipe, not `mcts_search()`'s
**Options.** (a) `mcts_search()` calls `optimize` + `promote` + `register` internally; (b) `mcts_search()` returns the winner, caller orchestrates.
**Decision.** (b).
**Rationale.** Single-responsibility. Test isolation. Caller flexibility (interactive vs CI). DSPy stays out of MCTS's import path.
**Consequences.** Documentation burden — the canonical recipe lives in `mcts_search()` docstring + the closed-loop integration test (B-INT). No silent loss of optimization quality vs the alternative.

### ADR-10 — Provenance: `mcts/search` (one) + `mcts/iteration` (cache-miss-only)
**Options.** (a) Per-iteration regardless of cache; (b) cache-miss only.
**Decision.** (b).
**Rationale.** Cache hits are deterministic from cache-miss recordings; re-recording them on every visit bloats the log without information gain. Replay reconstructs them by walking the cache-miss datoms.
**Consequences.** `:mcts/iteration` row count = total LLM calls, not total MCTS iterations. Test assertions distinguish.

### ADR-11 — Module surface: flat `_mcts.py`, not `persistence.plan.search/`
**Options.** (a) Flat file; (b) sub-module.
**Decision.** (a).
**Rationale.** v0.6.5 ships one search algorithm. Sub-module overhead exceeds benefit. Lift to `persistence.plan.search/` when v0.7+ adds beam-search / iterative-widening / non-MCTS variants.
**Consequences.** Future v0.7+ refactor to sub-module is a single `git mv` + `__init__.py` re-export update; no public-API break.

### ADR-12 — `:branch` dispatcher-handler registration is NOT in v0.6.5 scope
**Options.** (a) Register `:branch` handler in v0.6.5 (roadmap §3.3 option (a) full closure); (b) ship MCTS orthogonally and treat `:branch` handler as a separate ticket.
**Decision.** (b).
**Rationale.** MCTS produces a winning Plan AST that the caller substitutes structurally. The dispatcher-handler-for-`:branch` use case is "execute the search as part of execute()" — strictly larger scope than v0.6.5 needs. Ship the search; defer the inline-execution wiring.
**Consequences.** Stream B's paper claim (Prop 6 reconstructibility) is fully defensible without the `:branch` handler. The handler can be a small follow-up in v0.6.6 if needed for Stream H paper section §5.X.

## 24. Open questions

1. **`c_puct` default — `1.4` vs `1.0`.** Picked `1.4` per UCB1 closed-form. AlphaZero used `1.0` most commonly. **Recommend keeping `1.4`** — closed-form rationale is stronger than literature consensus; tune in v0.7+ once we have eval data.
2. **Winner selection — most-visited vs highest-Q.** Picked most-visited (AlphaZero convention; "most confident"). Alternate: highest-Q (greedy on point estimate). **Recommend keeping most-visited** — same reasoning as `c_puct`; well-precedented.
3. **`expander_k = 4` default.** Arbitrary. Real LLM expanders may want 8 or 16. **Recommend keeping `4`** for v0.6.5; bump in v0.7+ once we observe LLM-expansion cost.
4. **Skill-library composition heuristic.** `LLMExpander` reads `SkillLibrary.list_skills()` and offers top-K — by what heuristic? Recency? Promotion-record G3 score? Random sample? **Recommend recency-weighted** for v0.6.5 (`skill/registered-at` is already a datom value); tune later. Documented in `LLMExpander` docstring.
5. **Dirichlet-noise on root prior (AlphaZero convention).** Adds exploration at the root. Defer to v0.7+ — adds determinism complexity (need a seed-driven RNG threading through MCTS) and Stream B doesn't strictly need it for Prop 6.
6. **`:mcts/search` datom — should it carry the full config payload or just `config_hash`?** Picked `config_hash` only (one datom value, content-addressed). Alternate: 8-12 separate datoms for individual config fields. **Recommend `config_hash` only** for v0.6.5; if Stream H's paper section wants per-field provenance, expand in v0.6.6.
7. **`:mcts/iteration` `output` field for `phase="expand"` — full proposals or just hashes?** Picked full proposals as canonical-JSON. Concern: bloats the log per expand. Alternate: hash-only with proposals reconstructible from cache. **Recommend full proposals** — replay-from-log-alone is the Prop 6 claim; a hash-only form requires the cache to be persistent, which it isn't in v0.6.5.

These are not blockers. The design ships under the listed pins; revisions land in v0.6.x point releases.
