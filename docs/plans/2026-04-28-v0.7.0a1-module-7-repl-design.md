## 1. Summary

Module 7 today is a stub line in `src/persistence/__init__.py` (`repl — live production inspection [stub]`). Stream D ships Module 7 as a capability-gated WebSocket server + a zero-build browser console UI that exposes four operations — `inspect`, `edit`, `rewind`, `branch` — over the existing substrate. Every op is JSON-RPC 2.0 over a WebSocket frame, every op is gated by a capability token bound to a session, and every op (including read-only `inspect`) emits exactly one audit-chained `:repl/op` entry through `make_audit_handler` so that REPL traffic is indistinguishable in form from a programmatic write. The four ops map 1:1 to existing primitives — `inspect` reads via `db.as_of(t)` + `db.entity(e)` + `db.causal_history(e)` + `verify_chain` over an audit window; `rewind` mutates a per-session view-cursor only; `branch` calls `db.branch(t, [])` to fork; `edit` calls `db.transact([...])` under capability check. UI is one HTML file + one vanilla-JS module served from `src/persistence/repl/ui/`. Determinism contract is single-line: `inspect` and `rewind` are pure functions of `(view-cursor, store-state)` plus a write of one `:repl/op` audit entry; `edit` and `branch` write through the same `db.transact` path that programmatic callers use, so byte-identity replay holds. **Ships 2026-05-16; ~1100 LOC + ~240 tests + UI bundle.**

## 2. Strategic frame

### Why Module 7 ships full surface

The roadmap §3 ADR-1 (locked 2026-04-27) reversed the prior "REPL is headless-only / UI deferred to Phase 2.E" decision. That deferral was paper-driven — the "live steering" claim was treated as a forward-looking promise that did not require a runnable artifact to defend formally. Three things changed at the v1.0 pivot:

1. **The substrate's positioning against MemGPT / Mem0 / Zep / Memento depends on the write-time + steering-time + replay-time triad.** Those systems are read-only or read-mostly memory layers; persistence-os is the only one with a substrate that lets an operator inspect, edit, rewind, and branch a live trajectory through capability-gated audit-chained ops. Without a UI the proposition reads aspirational; with a UI the demo is the proof.
2. **The four ops map 1:1 to existing primitives.** `db.as_of`, `db.entity`, `db.causal_history`, `db.transact`, `db.branch` are all shipped (see `src/persistence/fact/db.py:416-523`). The `make_audit_handler` factory + `verify_chain` are shipped (`src/persistence/effect/handlers/audit.py:380-407, 357`). Module 7 is a thin RPC seam over those primitives, not new substrate logic.
3. **Stream A's G4 promotion gate ships as a stub that Stream D replaces.** `_promotion.py:476` explicitly notes `"Stream D (Module 7 REPL): an operator-token capability check on …"`. Without Stream D's `inspect` + `edit` ops, the G4 stub stays in v1.0.0, weakening the paper claim that "promoted skills are operator-vetted." Stream D unblocks the stub-removal precondition that Stream G's pre-v1.0.0 integration test will assert.

### Where Stream D sits in the v1.0 roadmap

**Module surface placement.** Sub-package `persistence.repl/` with private leading-underscore files mirrors the `persistence.plan/` and `persistence.txn/` layout. The UI lives at `src/persistence/repl/ui/` so it ships inside the wheel; serving it is `WSServer`'s concern (one `aiohttp` static handler attached to the same `host:port`). **No separate UI package, no separate frontend repo, no build step.** ADR-6 below pins zero-build vanilla JS.

**Datom-attr key style (key code↔doc rule, inherited from Stream A + Stream B + Stream C).** `Datom.a` slots written by REPL ops follow the repo convention: bare namespaced kebab-case, no leading colon at storage. Examples: `repl/session-id`, `repl/op-kind`, `repl/cap-token-id`, `repl/args-hash`, `repl/view-cursor-tx`, `repl/view-cursor-vt`. Wire-form `AuditEntry.op` keeps the leading colon (`:repl/op`) because that is the EDN-keyword form `audit_entry_to_datom` enforces (see `src/persistence/effect/handlers/audit.py:92-96`). The `repl/` prefix opts REPL writes out of any other namespace's reserved-prefix scans.

### Out of scope (explicit deferrals)

- **Multi-user concurrency beyond capability tokens.** A single REPL server instance accepts many concurrent sessions, each with its own token; sessions do not coordinate, do not see each other's view-cursors, and do not get notified of each other's edits except via the shared audit tail. **No multi-user lock manager, no presence indicator, no operator-to-operator chat.** Phase 3 NeSy 2027 if anyone wants it.
- **Persistent sessions across server restart.** Sessions live in process memory; a server restart drops all view-cursors. Tokens that were valid pre-restart remain valid post-restart (token store is fact-store-backed; see ADR-3), but the session needs to re-establish its view-cursor from `(tt, vt)` args on the next `inspect` call. Sticky session-id resumption across restart → v0.7.x.
- **OAuth / SSO / federation.** Tokens are issued by a bootstrap CLI or an env-var-loaded config file (ADR-3). No third-party identity provider integration. Phase 3.
- **Web-based plan editor.** The UI exposes the four ops as JSON-RPC commands typed into a console; it does NOT render a Plan AST tree, allow drag-drop plan composition, or visualize search trees. Plan-editor UI is v0.7.x or later.
- **Subscriptions to anything other than the audit tail.** v0.7.0a1 has exactly one server-pushed event stream — the audit tail (every new `AuditEntry` after subscription start). No per-entity subscriptions, no plan-execution event streams, no MCTS-rollout streams. Per-topic subscriptions are an OQ deferred to v0.7.x (§11 OQ-1).
- **REPL-side caching of view-cursor reads.** Every `inspect` call re-runs the underlying `db.as_of(t).entity(e)` query. No memoization in the session. The fact-store query is fast enough at v0.6.5 store sizes that adding a cache layer would be premature optimization, and the cache would itself need invalidation logic that complicates the audit-trace contract. Add in v0.7.x if profiling justifies it.
- **Browser-side TLS termination.** WS server listens on plaintext `ws://localhost:<port>` by default; production callers wrap behind a reverse proxy (nginx / Caddy / Traefik) that handles TLS. Module 7's threat model assumes loopback-only or LAN-only deployment for v0.7.0a1; public-internet exposure requires the reverse-proxy story (documented in `src/persistence/repl/__init__.py` module docstring).
- **CSRF / SameSite hardening for the bundled HTML page.** The bundled `index.html` is intended to be opened directly in a browser against a `localhost` WS port; it does not set cookies, does not embed in iframes, and does not accept third-party HTTP origins. Hardening for embedding is v0.7.x.
- **Plan-AST inspection beyond raw datom dump.** `inspect` on a plan_id returns the raw datoms; reified Plan AST tree rendering (with collapsed children, attr keys, etc.) is a UI-only concern that v0.7.0a1 does not ship. The console output is canonical-JSON pretty-printed; a tree renderer is OQ-3.

### ADR-1 — Transport: WebSocket (RFC 6455), not SSE / HTTP polling

**Status:** ACCEPTED.

**Context.** Module 7 needs bidirectional traffic: client → server for the four ops (request/response), server → client for the audit-tail subscription (server-pushed events). Three transports were on the table:

- **WebSocket (RFC 6455).** Bidirectional from frame zero. One TCP connection per session. Mature library support (Python `websockets` 12.x, browser-native API).
- **Server-Sent Events (SSE).** One-way server → client. Would need a separate POST-based RPC channel for the four ops. Two transports per session, two failure modes, two reconnect stories.
- **HTTP long-polling.** Bidirectional emulation. Adds round-trip latency on every server-pushed audit event. Not viable for "live audit tail" UX.

**Decision.** WebSocket over loopback. One TCP connection per session; client opens `ws://localhost:<port>/repl`, presents capability token in the first frame (auth handshake; ADR-3), then exchanges JSON-RPC frames bidirectionally for the lifetime of the session. The server pushes `notification` frames for audit events; the client sends `request` frames for ops.

**Consequences.**
- Single TCP connection state to manage. Auth replay attacks (ADR-3 detail) only need to defend the handshake frame, not every request.
- No HTTP framework dependency. The `aiohttp` static handler for `index.html` + the `websockets` server live in the same `asyncio` event loop and share the same listening port (via `aiohttp`'s `WebSocketResponse` integration).
- Browser-native `WebSocket` API is the only client-side runtime cost. Zero JS dependencies.
- Loopback-only by default; LAN/public exposure is a reverse-proxy concern documented in the module docstring.

### ADR-2 — Message protocol: JSON-RPC 2.0 envelope, not custom

**Status:** ACCEPTED.

**Context.** Three protocol shapes were considered:

- **JSON-RPC 2.0.** Standardized envelope (`{"jsonrpc": "2.0", "method": "...", "params": {...}, "id": <int|str|null>}`), well-known error code conventions, batch support, notification support (no `id` = no response expected). Library support in browser (none needed; trivial to write) and Python (`jsonrpcserver` exists but we hand-roll for tighter control).
- **Custom envelope.** Smaller wire footprint, but every reviewer + downstream consumer would need to read a spec. Reinventing the wheel for a stream that ships under a 7-day window.
- **gRPC over WS.** Type-safety win but a build-step + protobuf dependency, which contradicts ADR-6's zero-build UI principle.

**Decision.** JSON-RPC 2.0. Method names are `repl/inspect`, `repl/edit`, `repl/rewind`, `repl/branch`. Params are op-specific objects (§5 schemas). Error codes follow the JSON-RPC convention (`-32700` parse, `-32600` invalid request, `-32601` method not found, `-32602` invalid params, `-32603` internal error) plus a substrate-defined band (`-32000 .. -32099` reserved by JSON-RPC for application errors; we use `-32001` capability denied, `-32002` retroactive correction, `-32003` plan construction error, `-32004` audit-chain integrity broken at op execution; ADR-9 below).

**Consequences.**
- Reviewers reading the protocol spec recognize JSON-RPC 2.0 immediately; zero new vocabulary.
- Server-pushed audit-tail uses JSON-RPC notifications (no `id`); clients distinguish by `id` presence.
- Method-name namespace `repl/` mirrors the `Datom.a` and `AuditEntry.op` namespacing convention.
- Batch requests (JSON-RPC array form) are accepted by the parser but treated as one logical client intent — each batched call still emits its own `:repl/op` AuditEntry. This avoids "one batch = one audit entry" ambiguity.

### ADR-3 — Capability token format: opaque random 256-bit, fact-store-backed, env/config bootstrap

**Status:** ACCEPTED.

**Context.** The roadmap §3 ADR-1 decision pins capability gating as mandatory ("WebSocket must require an operator-issued capability token, no anonymous access"). Three token shapes were considered:

- **JWT (signed).** Self-validating, no server-side state. But: revocation requires a denylist (server-side state again), and key rotation drifts non-revoked tokens silently. Plus a JWT library dependency.
- **Opaque random token + server-side registry.** Server holds `token_id → CapabilitySet` mapping. Revocation = remove from registry. Trivial to validate, trivial to revoke. State lives in the substrate's existing fact store (free bitemporal indices + audit chain).
- **OAuth 2.0 bearer.** Requires an identity provider; out of scope (§3).

**Decision.** Opaque random 256-bit tokens, base64url-encoded (43 chars). Token registry is fact-store-backed: each issued token writes three datoms with `e = token_id`:

```python
[
  Datom(e=token_id, a="repl/cap-set",      v=<json-encoded CapabilitySet>, ...),
  Datom(e=token_id, a="repl/issued-at",    v=<:clock/now millis>,           ...),
  Datom(e=token_id, a="repl/issued-by",    v=<bootstrap-issuer-string>,     ...),
]
```

Validation: `validate_token(token_str) → CapabilitySet | None` does a single `db.entity(token_id_from_hash(token_str))` lookup; missing or revoked (a `repl/revoked-at` datom present) returns `None`. Revocation: `revoke_token(token_id)` writes one `("repl/revoked-at", :clock/now)` datom; subsequent validation rejects.

**Bootstrap (issuance).** Two paths, both out-of-band relative to the running REPL server:

1. **Bootstrap CLI:** `python -m persistence.repl.cli issue --caps "inspect:read,rewind:any" --label "ops-readonly"` writes the three datoms above against the configured `DB` and prints the token to stdout. Operator copies it into a secrets store (1Password, Vault, etc.) and ships to the consumer out of band.
2. **Env-var loading:** at REPL server startup, `PERSISTENCE_REPL_BOOTSTRAP_TOKENS=<path-to-json>` points to a JSON file like `[{"caps": ["inspect:read"], "label": "ci"}, ...]`. The startup hook reads, issues each token, prints them once to stderr, and starts the server. Used for development and CI; production deployments use the CLI path.

**Capability set shape:**

```python
@dataclass(frozen=True)
class Capability:
    op: Literal["inspect", "edit", "rewind", "branch"]
    qualifier: str   # "read", "write", "any", "fork", per ADR table below

@dataclass(frozen=True)
class CapabilitySet:
    caps: frozenset[Capability]
    expires_at_ms: int | None     # None = no expiry (rare; only bootstrap tokens)
    label: str                    # human-readable, audit-visible
```

Defined `(op, qualifier)` pairs (closed set in v0.7.0a1):

| op | qualifier | meaning |
|---|---|---|
| inspect | read | read-only entity / audit / plan inspection |
| edit | write | submit a `db.transact(...)` payload through `repl/edit` |
| rewind | any | move the session view-cursor to any `(tt, vt)` |
| branch | fork | call `db.branch(t, [])` to fork the store |

Future qualifiers (e.g. `inspect:audit-only`, `edit:retract-only`, `branch:read-only-fork`) are forward-compat — `_caps.py` validates the closed set in v0.7.0a1 and raises `UnknownCapability` on unrecognized pairs.

**Token presentation.** Client opens WS, sends one auth-handshake frame as the first message: `{"jsonrpc": "2.0", "method": "repl/auth", "params": {"token": "<base64url>"}, "id": 0}`. Server validates; on success replies `{"jsonrpc": "2.0", "result": {"session_id": "<uuid>", "caps": [...]}, "id": 0}`; on failure replies a `-32001` error and closes the WS. **All subsequent frames are bound to that session_id**; the WS connection IS the auth context.

**Consequences.**
- No JWT library dependency. ~80 LOC token logic.
- Token issuance is itself audit-chained (`:repl/auth` AuditEntry on the issuing transact through `make_audit_handler` — bootstrap CLI runs under an audit handler stack the same as production).
- Revocation is one datom write; takes effect on the next `validate_token` call.
- Tokens are content-addressed via `token_id = sha256(token_str)[:16]` so the token itself is never stored, only its hash. `validate_token` recomputes the hash from the presented string.
- Threat model bounds: an attacker who reads the fact store sees only token hashes, not the raw tokens. An attacker who reads the in-flight WS frame sees the raw token; loopback-only deployment is the mitigation for v0.7.0a1.

### ADR-4 — Op auditing: every op emits one `:repl/op` AuditEntry, chained

**Status:** ACCEPTED.

**Context.** The roadmap §3 ADR-1 makes "audit-traceable" mandatory: every REPL op (including read-only `inspect`) must be recorded in the audit chain. Three placements were considered:

- (a) **Audit only writes** (`edit`, `branch`). Reads (`inspect`, `rewind`) emit no audit. **Rejected** — violates the "no off-the-record peeking" property; an attacker with `inspect:read` could probe entity state without leaving a trace.
- (b) **Audit all four ops, but route reads through a separate non-Merkle log.** Two audit chains, two verify paths, two replay stories. **Rejected** — bifurcates the audit-chain hash continuity claim.
- (c) **All four ops emit one `:repl/op` AuditEntry through `make_audit_handler` into the same Merkle chain.** Reads and writes are uniform; `verify_chain` covers REPL traffic identically to programmatic traffic.

**Decision.** Option (c). Every op invocation runs through a small wrapper:

```python
def emit_repl_op_audit(
    db: DB,
    session: Session,
    op_kind: Literal["inspect", "edit", "rewind", "branch"],
    args_canonical: dict,
    result_summary: dict,
    *,
    runtime: Runtime,
) -> AuditEntry:
    """Wrap the op in a :repl/op effect performed through the runtime's
    audit handler stack. Returns the AuditEntry for caller introspection.

    The handler stack is the SAME stack programmatic callers use — no
    REPL-private audit chain. `runtime` is the active Runtime context; the
    audit handler is presumed installed (caller's contract; the WSServer
    constructs it at startup).
    """
    return runtime.named_perform(
        "audit",
        ":repl/op",
        {
            "session_id": session.id,
            "cap_token_id": session.cap_token_id,
            "op_kind": op_kind,
            "args_hash": canonical_hash(args_canonical),
            "view_cursor_tx_ms": session.view_cursor_tx_ms,
            "view_cursor_vt_ms": session.view_cursor_vt_ms,
            "result_summary": result_summary,  # bounded; never the full read payload
        },
    )
```

`result_summary` is bounded — for `inspect`, it's `{"entity_count": N, "datom_count": M, "audit_count": K}`; for `edit`, it's `{"datoms_written": N, "tx": <tx-id>}`; for `rewind`, it's `{"new_cursor_tx_ms": ..., "new_cursor_vt_ms": ...}`; for `branch`, it's `{"branched_db_id": "<uuid>", "base_tx_ms": ...}`. The full payload is NOT captured in the audit entry — that would be a privacy hazard for `inspect`-on-PII flows. The args_hash + the deterministic `(view-cursor, store-state)` view at audit time is sufficient to replay the read.

**Consequences.**
- One `:repl/op` AuditEntry per op invocation. `verify_chain` over the full log (programmatic + REPL) returns `True` — the chain is one Merkle log, not two.
- Reads are audit-chained: an attacker with `inspect:read` can read entity state but cannot do so silently. The audit entry records `(session_id, cap_token_id, op_kind, args_hash, view_cursor)` — sufficient to reconstruct what was inspected.
- `result_summary` is bounded so PII in inspected entities does not leak into the audit log itself.
- The `:repl/op` op-name follows the `AuditEntry.__post_init__` invariants (`src/persistence/effect/handlers/audit.py:84-106`): leading colon, ≤ one slash, no literal dots. Pinned at construction.

### ADR-5 — View-cursor semantics: per-session, decoupled from branch

**Status:** ACCEPTED.

**Context.** The `rewind` op moves the session's view of the substrate to a prior `(tt, vt)` coordinate. Three semantic options were on the table:

- (a) **Per-session view-cursor; `rewind` mutates session state only; `branch` calls `db.branch(t, [])` and the session re-binds to the branched DB.** Reads after `rewind` go through `db.as_of(view_cursor.tx).entity(e)` against the original DB. Reads after `branch` go through the branched DB at its head.
- (b) **Global view-cursor.** All sessions share one cursor. **Rejected** — operators step on each other's views; defeats the multi-session story.
- (c) **`rewind` mutates the underlying store.** **Rejected** — `rewind` would imply a destructive rewrite of the bitemporal store, contradicting Prop 1 (datom append-only).

**Decision.** Option (a). View-cursor lives on `Session`:

```python
@dataclass
class Session:
    id: str                                # uuid4
    cap_token_id: str                      # 16-hex from token registry
    cap_set: CapabilitySet                 # frozen at auth time
    db: DB                                 # the active DB (may be a branched fork)
    view_cursor_tx_ms: int | None          # transaction-time cursor; None = head
    view_cursor_vt_ms: int | None          # valid-time cursor; None = head
    parent_session_id: str | None          # set if this session was created by `branch`
    created_at_ms: int
```

- `rewind(args)` sets `view_cursor_tx_ms` and/or `view_cursor_vt_ms` to the supplied coordinate; emits `:repl/op` audit; returns the updated cursor. **Does NOT mutate `db`.**
- `inspect(args)` reads through `db.as_of(view_cursor_tx_ms).entity(e)` (or `db.as_of_valid(view_cursor_vt_ms)` depending on the request shape — see §5 schemas). If both cursors are `None`, reads at head.
- `branch(args)` calls `db.branch(t=view_cursor_tx_ms or now, assertions=[])` — same primitive Stream A's tests use. Returns a new session bound to the branched DB. The original session's `db` is unchanged. The new session's `parent_session_id` records the parent. `view_cursor_*` on the new session is reset to `None` (head of the branched DB). Capability set is inherited from the parent session.
- `edit(args)` calls `parent_db.transact([...])` if the session is on a branched DB or main DB. Writes ALWAYS land in the session's current `db` — never silently in main if the session is on a branch.

**Cross-cursor + branch interaction.** If `view_cursor_tx_ms` is set and the operator calls `branch`, the branch fork point is the cursor, not the head. This makes "rewind to bug, fork to fix" a one-liner: `rewind {tx_ms: T} → branch {} → edit {datoms: [...]}`.

**Consequences.**
- `rewind` is a pure session-state mutation + an audit emission; no store side-effect.
- `branch` is one `db.branch(t, [])` call that returns a new session; the parent session is unaffected.
- A session on a branched DB is structurally isolated from main; `edit` on the branched session never leaks into main. Promotion of a branch's writes back to main is OUT of scope for v0.7.0a1 (would be a `db.fork()` / merge primitive that doesn't yet exist; ADR's open-question OQ-4).
- The view-cursor IS the session's REPL state. Server restart drops it (§3 deferral).

### ADR-6 — UI bundling: zero-build vanilla JS + single HTML file

**Status:** ACCEPTED.

**Context.** The roadmap §3 ADR-1 strongly prefers "vanilla JS or tiny dep" over a SPA framework. Three options were considered:

- (a) **Zero-build vanilla JS + single HTML.** Browser-native `WebSocket` + `fetch`. ES module imports for `repl.js` so the file loads without bundling. Single HTML file with inline `<script type="module" src="/repl.js">`. **Selected.**
- (b) **htmx.** Tiny dep (~14kb), no build step, but htmx is HTTP-oriented; WS support is via an extension that complicates the JSON-RPC surface.
- (c) **React + Vite build step.** Modern DX, but adds a `package.json`, a build artifact, and a frontend-CI surface. Out of proportion to a 7-day stream.

**Decision.** Option (a). Files:

- `src/persistence/repl/ui/index.html` — one HTML page, ~80 lines. Two-pane layout: left pane = command input + scrollable output history; right pane = scrollable audit-tail. Loaded directly via `ws://localhost:<port>/`.
- `src/persistence/repl/ui/repl.js` — one ES module, ~250 lines. WebSocket client, JSON-RPC request/response correlator, audit-tail subscriber, command parser (`inspect entity foo`, `rewind tx 12345`, etc.), output renderer (canonical-JSON pretty-print).
- `src/persistence/repl/ui/repl.css` — ~60 lines of vanilla CSS. No framework, no SCSS.

**Wireframe (text):**

```
┌────────────────────────────────────────────────┬──────────────────────────┐
│ persistence-os repl  session: <uuid>  caps: …  │ audit tail (live)       │
├────────────────────────────────────────────────┼──────────────────────────┤
│ > inspect entity sku-123                       │ 12:01:03 :repl/op edit  │
│ {                                              │   sess=… tx=42          │
│   "name": "widget",                            │ 12:01:04 :llm/call      │
│   "wacc": 0.087                                │   args-hash=abc…        │
│ }                                              │ 12:01:05 :repl/op rewind│
│ > rewind tx_ms 1727000000000                   │   sess=… cursor=tx#41   │
│ cursor: tx_ms=1727000000000 vt_ms=null         │ 12:01:06 :repl/op       │
│ > _                                            │   inspect sess=…        │
│                                                │ ...                      │
└────────────────────────────────────────────────┴──────────────────────────┘
```

**Consequences.**
- `pip install persistence-os` ships the UI in the wheel. No separate frontend build, no separate deployment.
- Browser-native APIs only. Tested in Chromium 120+ and Firefox 120+ (the two evergreen targets that ship `WebSocket` + ES modules).
- XSS surface: audit-tail rendering uses `textContent` not `innerHTML` for any field that originates from substrate data (entity values, args_hash, error messages); pinned in the JS module + tested via a payload with `<script>` tags in the entity value (§9 risks).
- No `package.json`, no `node_modules`, no Vite config. Reviewer can read the entire UI in one pass.

### ADR-7 — Confirmation flow for `edit`: capability + per-request `confirm: true`

**Status:** ACCEPTED.

**Context.** `edit` is the only op that mutates the substrate. Three flows were considered:

- (a) **Single-step with capability.** Operator with `edit:write` token can submit any `db.transact` payload; one frame in, one frame out. Lowest friction.
- (b) **Two-step propose-confirm.** `edit-propose` returns a proposal hash + a preview of the resulting datoms; `edit-confirm` references the hash and commits. Two audit entries per write.
- (c) **Single-step with per-request `confirm: true` flag.** Operator with `edit:write` AND a typed `confirm: true` field in the request can write; missing or `false` returns a `-32001`-class error with a preview of what would have been written. One audit entry per write; one rejection audit per non-confirmed attempt.

**Decision.** Option (c). The request body for `edit` requires:

```json
{
  "jsonrpc": "2.0",
  "method": "repl/edit",
  "params": {
    "datoms": [{"e": "...", "a": "...", "v": "...", "valid_from": "..."}],
    "provenance": {"reason": "operator-correction"},
    "confirm": true,
    "force_retroactive": false
  },
  "id": 7
}
```

If `confirm` is missing or `false`, server returns:

```json
{
  "jsonrpc": "2.0",
  "error": {
    "code": -32001,
    "message": "edit requires confirm: true",
    "data": {
      "preview": {"datoms": [...], "estimated_tx_ms": ..., "would_retract": [...]}
    }
  },
  "id": 7
}
```

The preview computes the auto-retraction set (companion retracts `db.transact` would emit; `src/persistence/fact/db.py:184-234`) without writing. The rejection emits a `:repl/op` audit entry with `op_kind="edit-rejected"` so abandoned-edit attempts are also chained.

**Consequences.**
- One audit entry per actual write (option a's audit shape) + one audit entry per rejected attempt (option c's safety net).
- The preview is the substrate's auto-retraction logic, NOT a re-implementation; the preview function is `_db_extension.preview_transact(facts) → PreviewResult` — a refactor of `transact()` that stops before `allocate_and_append`. Adds ~30 LOC to `db.py`; tested via byte-identity property (preview + actual transact produce the same datom shapes modulo `tx`).
- Operators can't accidentally write by typo; `confirm: true` is intentional.
- Hostile clients with valid `edit:write` can still write; capability is the load-bearing gate, `confirm` is the second factor against fat-finger ops, not against malicious operators.

### ADR-8 — Subscription / event-stream semantics: one stream per session, audit-tail only

**Status:** ACCEPTED.

**Context.** The UI's right pane shows a live audit tail. Three subscription models were considered:

- (a) **One stream per session, server-pushes every new AuditEntry after subscription start.** Simplest. No filtering on the server; client renders all.
- (b) **Per-topic subscriptions** (e.g. `audit:repl/op`, `audit:effect/llm-call`). Server-side filter; lower bandwidth.
- (c) **No subscriptions; client polls.** Wastes round trips; defeats the "live" UX.

**Decision.** Option (a) for v0.7.0a1. The session opts into audit-tail by sending `repl/subscribe-audit` with no params; server responds with the subscription_id and starts pushing notifications. Notifications follow JSON-RPC notification shape (no `id`):

```json
{
  "jsonrpc": "2.0",
  "method": "repl/audit-event",
  "params": {
    "audit_id": "...",
    "op": ":repl/op",
    "verdict": "ok",
    "recorded_at": 1727000000.000,
    "principal": {"session_id": "...", "cap_token_id": "..."},
    "summary": {"op_kind": "inspect", "result_summary": {...}}
  }
}
```

The push is fire-and-forget — server does not retain undelivered events past a small ring buffer (configurable, default 256 entries). If the client falls behind, the server emits one `repl/audit-event-overflow` notification and drops the oldest. The client may then call `repl/inspect` with `kind: "audit-window"` to backfill from the fact store directly.

**Per-topic filtering** (option b) is OQ-1; the wire format and server-side hook are forward-compat — `repl/subscribe-audit` accepts an optional `filter` param that v0.7.0a1 rejects with `-32602` "filter parameter not supported in this version" if non-null.

**Consequences.**
- Subscription state is in-process per session; server restart drops it.
- Bandwidth at v0.6.5 store sizes is fine; one AuditEntry is a few hundred bytes; even sustained 100 events/sec is < 50KB/sec.
- Forward-compat hook (filter param) is documented and tested for rejection in v0.7.0a1; v0.7.x can flip the rejection to acceptance without a wire-format change.

### ADR-9 — Error envelope shape: JSON-RPC error object + substrate error codes

**Status:** ACCEPTED.

**Context.** JSON-RPC 2.0 reserves `-32700..-32603` for protocol errors; `-32000..-32099` is the application-error band. We need a stable mapping from substrate exceptions to error codes so the UI and downstream clients can render meaningfully.

**Decision.** Closed code table (v0.7.0a1):

| Code | Class | Trigger |
|---|---|---|
| -32700 | parse error | malformed JSON |
| -32600 | invalid request | not JSON-RPC 2.0 |
| -32601 | method not found | unknown `method` |
| -32602 | invalid params | params shape mismatch (per-op schema; §5) |
| -32603 | internal error | unhandled server exception |
| -32001 | capability denied | session caps don't grant the requested op qualifier; OR `edit` without `confirm: true` |
| -32002 | retroactive correction | `db.transact` raised `RetroactiveCorrectionError`; `data` carries the `force_retroactive: true` retry hint |
| -32003 | plan construction error | `Node(...)` raised during an inspect on a Plan AST datom |
| -32004 | audit chain integrity broken | `verify_chain` returned False on the inspected window; signals substrate corruption |
| -32005 | view cursor invalid | `rewind` to a coordinate before the store's earliest datom |
| -32006 | session expired | cap token expired between auth and op |

**Error envelope:**

```json
{
  "jsonrpc": "2.0",
  "error": {
    "code": -32001,
    "message": "capability denied: edit:write required",
    "data": {
      "session_caps": ["inspect:read", "rewind:any"],
      "required": "edit:write"
    }
  },
  "id": 12
}
```

The `data` object is op-specific; closed-set per the table above. The UI renders `error.message` in red and `error.data` as collapsed JSON.

**Consequences.**
- Closed code set in v0.7.0a1; new codes require a CHANGELOG bump.
- All errors are themselves audited via `op_kind="<op>-rejected"` (§ADR-4) so the audit chain captures intent regardless of outcome.

### ADR-10 — Async runtime: asyncio + websockets, no threading

**Status:** ACCEPTED.

**Context.** Python's `websockets` library is `asyncio`-native. `db.transact` is synchronous and may block on store I/O (SQLite path). Three runtime shapes were considered:

- (a) **Pure asyncio with `loop.run_in_executor` for blocking calls.** Modern, single event loop, `asyncio.Lock` for session-state mutations.
- (b) **asyncio main + threadpool worker per session.** Per-session thread for `db.transact` calls. Adds GIL contention without proportionate benefit at v0.7.0a1 traffic levels.
- (c) **gevent / sync.** Mature but mixing with `websockets` (asyncio) is awkward; no win.

**Decision.** Option (a). One asyncio event loop in `WSServer.serve()`; `db.transact` and `db.entity` calls are wrapped in `loop.run_in_executor(None, ...)` so the loop is never blocked. Per-session state (`Session.view_cursor_*`) is mutated under an `asyncio.Lock` held only inside the op handler, never across awaits.

**Consequences.**
- Single Python process, no thread-per-session. ~50 concurrent sessions at v0.7.0a1 traffic levels is comfortable.
- `db.transact` running in the default executor means up to `os.cpu_count()` parallel writes; the substrate's `Store.allocate_and_append` is already serialized internally (`src/persistence/fact/store.py`), so concurrent calls queue.
- `asyncio.Lock` per session prevents the "rewind mid-inspect produces inconsistent results" race.
- Test runtime is `pytest-asyncio` for the `_ws.py` tests; `_ops.py` tests are pure sync over a `Session` fixture.

## 5. Op contracts

### 5.1 `repl/inspect`

Reads the bitemporal store at the session's view-cursor (or at an explicitly-supplied coordinate that overrides the cursor for this call only).

**Required capability:** `inspect:read`.

**Request schema:**

```json
{
  "jsonrpc": "2.0",
  "method": "repl/inspect",
  "params": {
    "kind": "entity" | "audit-window" | "plan" | "causal-history",
    "args": {
      // for kind=entity:
      "entity_id": "<string>",
      "tx_ms": <int|null>,           // override; null = use session cursor
      "vt_ms": <int|null>,
      // for kind=audit-window:
      "start_ms": <int>,
      "end_ms": <int>,
      "op_filter": "<:audit-op|null>",
      // for kind=plan:
      "plan_id": "<string>",
      // for kind=causal-history:
      "entity_id": "<string>",
      "max_depth": <int>             // currently 1; v0.7.x extends
    }
  },
  "id": <int>
}
```

**Response (success):**

```json
{
  "jsonrpc": "2.0",
  "result": {
    "kind": "entity",
    "data": {"name": "widget", "wacc": 0.087},
    "view_cursor": {"tx_ms": 1727000000000, "vt_ms": null},
    "audit_id": "<the :repl/op AuditEntry id>"
  },
  "id": <int>
}
```

**Response (error):** see ADR-9 codes; relevant: `-32001` cap denied, `-32004` chain broken, `-32005` cursor invalid.

**Audit fields emitted (`:repl/op`):**

```python
{
  "session_id": "...",
  "cap_token_id": "...",
  "op_kind": "inspect",
  "args_hash": "<sha256 of canonical-JSON params>",
  "view_cursor_tx_ms": <int|null>,
  "view_cursor_vt_ms": <int|null>,
  "result_summary": {"kind": "entity", "datom_count": N, "audit_count": K}
}
```

**Worked example.** Operator with `inspect:read` runs `inspect entity sku-123` after a `rewind` to a known-good tx:

```json
// → request
{"jsonrpc":"2.0","method":"repl/inspect","params":{"kind":"entity","args":{"entity_id":"sku-123"}},"id":1}
// ← response
{"jsonrpc":"2.0","result":{"kind":"entity","data":{"name":"widget","wacc":0.087},"view_cursor":{"tx_ms":1727000000000,"vt_ms":null},"audit_id":"abc12345"},"id":1}
```

```json
}
```

**Response (success):**

```json
{
  "jsonrpc": "2.0",

    "tx": 12345,
```

## 6. Capability model

### 6.1 Token format

Opaque random 256-bit, base64url-encoded (43 chars). Generated via `secrets.token_urlsafe(32)`. The raw token is NEVER persisted; only `token_id = sha256(token_str)[:16]` lands in the fact store.

### 6.2 Issuance

Two paths:

**Bootstrap CLI** (`python -m persistence.repl.cli issue`):

```bash
$ python -m persistence.repl.cli issue \
    --caps "inspect:read,rewind:any" \
    --label "ops-readonly" \
    --expires-in-days 30 \
    --db-path /var/lib/persistence-os/store.db

issued: persistence.repl/eyJhbGciOiJI...      # 43-char base64url
token_id: a1b2c3d4e5f67890
caps: inspect:read, rewind:any
expires_at: 2026-05-28T12:00:00Z
```

**Env-var bootstrap.** `PERSISTENCE_REPL_BOOTSTRAP_TOKENS=/path/to/tokens.json` at REPL server startup. `tokens.json`:

```json
[
  {"caps": ["inspect:read"], "label": "ci-readonly", "expires_in_days": 7},
  {"caps": ["edit:write", "inspect:read", "rewind:any", "branch:fork"], "label": "operator-full", "expires_in_days": 30}
]
```

Server reads, issues each token, prints them once to stderr, then starts. **Tokens printed at startup are the only place the raw token surfaces in production** — operator must capture them from the log + ship to the consumer out of band.

### 6.3 Validation

```python
def validate_token(db: DB, token_str: str, *, now_ms: int) -> CapabilitySet | None:
    token_id = hashlib.sha256(token_str.encode()).hexdigest()[:16]
    entity = db.entity(token_id)
    if not entity:
        return None
    if entity.get("repl/revoked-at") is not None:
        return None
    expires = entity.get("repl/expires-at-ms")
    if expires is not None and now_ms >= expires:
        return None
    cap_set = CapabilitySet.from_json(entity["repl/cap-set"])
    return cap_set
```

Single `db.entity` call. Latency is O(history-length-of-token-id) under the current `DBView.entity` projection — since each token is a separate `e`, history is bounded by 3-4 datoms (issue + optional revoke + optional expire-bump).

### 6.4 Revocation

```python
def revoke_token(db: DB, token_id: str, *, now_ms: int, by: str) -> DB:
    return db.transact([
        {"e": token_id, "a": "repl/revoked-at", "v": now_ms, "valid_from": ...},
        {"e": token_id, "a": "repl/revoked-by", "v": by, "valid_from": ...},
    ], provenance={"source": "repl/revoke", "operator": by})
```

Takes effect on next `validate_token`. Active sessions on the revoked token continue until their next op attempt, which will fail with `-32006`.

### 6.5 Worked examples

**Read-only token (least-privilege).**

```python
caps = CapabilitySet(
    caps=frozenset({Capability(op="inspect", qualifier="read")}),
    expires_at_ms=1730000000000,
    label="ops-readonly",
)
```

Holder can `repl/inspect` only. `repl/rewind`, `repl/edit`, `repl/branch` all return `-32001`.

**Full-edit token.**

```python
caps = CapabilitySet(
    caps=frozenset({
        Capability(op="inspect", qualifier="read"),
        Capability(op="edit",    qualifier="write"),
        Capability(op="rewind",  qualifier="any"),
        Capability(op="branch",  qualifier="fork"),
    }),
    expires_at_ms=1730000000000,
    label="operator-full",
)
```

Holder can do everything; `edit` still requires the per-request `confirm: true` flag (ADR-7).

### 7.4 Minimum-viable UI behavior

- On load, `repl.js` reads `?token=<base64url>` from the URL fragment (NEVER from the query string — fragment doesn't hit server logs or HTTP referrers), opens a WS to the same origin, sends `repl/auth`.
- On `auth` success, the UI enables the command input and begins rendering audit-tail notifications.
- Commands typed into the input are parsed by a small grammar (`COMMAND ARG_KEY=ARG_VAL ...`), translated to JSON-RPC, sent.
- Responses are pretty-printed canonical-JSON in the left pane; errors render in red with `error.message` + `error.data`.
- Audit-tail push notifications append a one-line summary to the right pane (timestamp + `op` + summary fields). Click to expand to full canonical-JSON.
- All substrate-originating string fields are rendered via `textContent`, never `innerHTML`. Pinned in code review + tested via XSS payload.

## 8. Reproducibility / determinism contract

### 8.1 What "deterministic" means for a REPL

The REPL is **non-deterministic in user input** (humans type at unpredictable times, in unpredictable orders) but **deterministic in substrate response**. For any fixed sequence of `(session_id, op_kind, args_canonical, view_cursor_at_op_time)` tuples, the substrate produces a byte-identical sequence of:

1. `:repl/op` AuditEntries (with chained `prev_hash`).
2. Underlying `db.transact` writes (for `edit` ops) — same as programmatic callers.
3. Read responses for `inspect` (modulo wall-clock-derived `recorded_at` on AuditEntries, which is captured via `:clock/now` effect and is replay-pinable).

### 8.2 Byte-identity claim (mirrors Stream A Prop-4 framing + Stream B Prop-6 framing)

> *Module 7 byte-identity contract.* For any REPL session trace `T = [(op_kind_i, args_canonical_i, view_cursor_i)]`, replaying `T` against the same initial DB state under a replay-clock handler reproduces a byte-identical sequence of `:repl/op` AuditEntries, byte-identical `db.transact` outputs for any `edit` ops, and byte-identical read responses for any `inspect` ops.

**Defense outline:**
- Each `:repl/op` AuditEntry's content hash is `_content_hash(canonicalised_content)` (`src/persistence/effect/handlers/audit.py:308-312`); inputs to that hash are `(op, args_hash, principal, recorded_at, ...)`. `recorded_at` comes from `:clock/now`; replay-clock pins it. `principal.session_id` is replay-pinable since session_id is a function of `(token_id, replay-clock-millis-of-auth)`.
- `args_hash` is `canonical_hash(args_canonical)` over canonical JSON; deterministic by construction.
- `db.transact` writes inside `edit` go through the same path Stream A's tests already prove byte-identical (`tests/integration/test_v0_6_plan_execution.py`).
- `inspect` reads are pure functions of `(view_cursor, store-state)`; no LLM, no random.
- `verify_chain(audit_entries_in_replay) == verify_chain(audit_entries_in_original) == True`. The chain extends; `prev_hash` continuity holds.

### 8.3 What the REPL CANNOT replay byte-identically

- Wall-clock latency (`AuditEntry.latency_ms`) is captured but not replayed; replay records its own latencies.
- The order of concurrent sessions is not deterministic without serialization. v0.7.0a1 does NOT make a multi-session-byte-identity claim — only single-session traces.

## 11. Open questions

### OQ-1 — Per-topic subscription filtering (deferred)

**Question.** Should `repl/subscribe-audit` accept a `filter` param in v0.7.0a1 that lets clients subscribe to only `:repl/op` events, only `:effect/llm-call` events, etc.?

**Default for v0.7.0a1.** No. The wire format reserves the param (server returns `-32602` if non-null) but the implementation is forward-compat. Filtering by `op` prefix is one `if event.op.startswith(filter_prefix):` line in v0.7.x; no point shipping a feature the audit-tail bandwidth doesn't justify yet.

### OQ-2 — `inspect` on Plan AST: raw datoms vs reified node tree

**Question.** When `inspect` is called with `kind="plan"` and a `plan_id`, should it return the raw `Datom` rows for the entity, or should it walk the Plan AST and return a reified node tree (tag, attrs, children)?

**Default for v0.7.0a1.** Raw datoms. The reified-tree renderer is a UI concern, and the UI in v0.7.0a1 is a console — pretty-printing canonical JSON is good enough for the operator workflow. Reified-tree rendering is OQ-3 deferred to v0.7.x once the UI evolves. Adding a server-side reify is a 5-line addition (`Node.from_datoms(datoms) → Node`); we just don't ship it in v0.7.0a1.

### OQ-3 — Plan AST tree renderer in UI

**Question.** Should the UI render Plan AST trees as collapsible JSON-tree views or as text dumps?

**Default for v0.7.0a1.** Text dump (canonical JSON pretty-print). The console UX is "operator types a command, gets a response"; a JSON-tree component adds ~200 LOC of vanilla JS that does not load-bear for the v1.0 paper. Defer.

### OQ-4 — Can `edit` synthesize Effect intents, or only datom writes?

**Question.** Should `repl/edit` allow operators to submit an Effect intent (e.g. "perform `:llm/call` with these args, audit-chain it") in addition to raw datom writes?

**Default for v0.7.0a1.** Datom writes only. Effect intent submission widens the threat surface dramatically (an operator could synthesize fake LLM responses into the audit chain) and the feature is not load-bearing for any v1.0 paper claim. Operators who need to inject test effects can use the substrate's programmatic API directly. Defer to v0.7.x with a separate `effect:perform` capability that is NOT granted by `edit:write`.

### OQ-5 — `branch` from a non-main current cursor

**Question.** What is the behavior when the operator is already on a branched session and calls `branch` again — does this fork from the branched DB (creating a chain of branches) or from main?

**Default for v0.7.0a1.** Forks from the current session's DB, whatever that is (main or branched). Chains of branches are allowed. The audit entry's `parent_session_id` plus the `result_summary.branched_db_id` gives the chain its lineage. **No "merge a branch back to main" primitive in v0.7.0a1** — the substrate doesn't have one yet (`db.branch` returns a new isolated store; `db.fork()` referenced in the roadmap does not yet exist as a public API). Cross-branch reconciliation is Phase 3 NeSy 2027 scope. Documented in the `_ops.py` module docstring + the §5.4 worked example.

### OQ-6 — Token expiry policy default

**Question.** Should default-issued tokens have an expiry, and if so what?

**Default for v0.7.0a1.** Yes — bootstrap CLI defaults to `--expires-in-days 30`. Env-var bootstrap respects whatever each token's `expires_in_days` JSON field says. Tokens with `expires_at_ms = None` (no expiry) require an explicit `--no-expiry` flag and are flagged in the audit log via a `repl/no-expiry` provenance field. Justification: 30 days is short enough that operators rotate tokens regularly, long enough that day-to-day workflows aren't disrupted.

These are not blockers. The design ships under the listed defaults; revisions land in v0.7.x point releases.

## 12. Risks

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Capability bypass — a code path that calls an op without going through `validate_token` + cap-check | Low | substrate-claim-breaking | Single-entry-point invariant: all op handlers in `_ops.py` accept a `Session` (which carries `cap_set`); WS layer is the only thing that constructs `Session`s, and only after `validate_token` + cap-check. Unit-test pin: `test_session_cannot_be_constructed_without_validated_token.py`. |
| WS auth replay — attacker captures the auth handshake frame, replays from another connection | Medium (LAN), Low (loopback) | session hijack | Loopback-only default; LAN/public via reverse-proxy with TLS (documented). Session_id binding to the WS connection prevents cross-WS replay even on the same token. v0.7.x can add nonce-based handshake. |
| UI XSS in audit-tail rendering — a substrate value containing `<script>` tags renders as HTML | Medium | privilege escalation | `textContent`-only rule pinned in `repl.js` + tested via XSS-payload integration test (`test_repl_ui_xss_pinned.py` constructs a Datom with `v="<img src=x onerror=alert(1)>"`, drives the UI, asserts no script execution). |
| Deadlock with txn `dosync` — REPL holds an open WS during a long `dosync` block in the substrate | Low | hang | `db.transact` calls in `_ops.py` go through `loop.run_in_executor`; never block the asyncio loop. `dosync` is a substrate-internal coordination, not a REPL session state — REPL never holds a `Transaction` handle across awaits. |
| Drift between REPL `view_cursor` and store advancement — operator rewinds, store advances, operator's cursor now references a tx_ms that was "head" at rewind time but is now mid-history | Low | surprise, not corruption | Documented behavior: cursor is absolute, not relative-to-head. Reads at `tx_ms = T` always return the as_of(T) view, regardless of how many tx have landed since. UI displays both cursor and current head when they differ. |
| Token leak via UI fragment URL — operator copies a `?token=...` URL into a chat | Medium | session hijack | URL fragment (`#token=...`) NOT query string — fragment is never sent in HTTP referrers, never logged in server access logs. UI strips fragment from `window.location` immediately after read. Documented. v0.7.x can add a one-time-use redemption code that expires after first WS-auth. |
| Long-running `inspect` on `causal-history` over deep histories blocks the executor pool | Low | latency degradation | `max_depth` capped at 1 in v0.7.0a1 (single-level walk, matches `db.causal_history` v0.4 contract). Multi-level walks are an OQ. |
| `verify_chain` failure surfaces as `-32004` — operator sees substrate-corrupted error | Very low (substrate bug) | substrate trust | Treated as a substrate-level emergency: WS server logs the audit_id + the chain break, returns `-32004` to the operator. The chain-break event itself emits a `:repl/op` AuditEntry with `op_kind="inspect-chain-broken"` — meta-recorded. Stream G's R4 review re-runs `verify_chain` over the full v1.0.0 audit log as a release-gate. |
| Audit-tail ring buffer overflow loses events for a slow client | Medium | UX degradation, not security | `repl/audit-event-overflow` notification tells the client; client backfills via `repl/inspect kind="audit-window"`. Buffer size configurable. |
| WebSocket lib version drift breaks auth handshake | Low | install break | Pin `websockets >= 12.0, < 14.0` in `pyproject.toml` optional `[repl]` extras. Re-pin per release. |
| Cap-set forward-compat — v0.7.x adds a new `(op, qualifier)` and v0.7.0a1 tokens silently grant it | Low | privilege escalation across versions | Closed-set validation in `_caps.py`: unknown qualifiers raise `UnknownCapability` at `validate_token`. v0.7.x must explicitly bump the closed set; a v0.7.0a1 token does NOT grant a v0.7.x-introduced qualifier. Pinned via `test_caps_forward_compat_strict.py`. |
