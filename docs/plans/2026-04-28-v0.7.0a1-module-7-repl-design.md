## 1. Summary

Module 7 today is a stub line in `src/persistence/__init__.py` (`repl — live production inspection [stub]`). Stream D ships Module 7 as a capability-gated WebSocket server + a zero-build browser console UI that exposes four operations — `inspect`, `edit`, `rewind`, `branch` — over the existing substrate. Every op is JSON-RPC 2.0 over a WebSocket frame, every op is gated by a capability token bound to a session, and every op (including read-only `inspect`) emits exactly one audit-chained `:repl/op` entry through `make_audit_handler` so that REPL traffic is indistinguishable in form from a programmatic write. The four ops map 1:1 to existing primitives — `inspect` reads via `db.as_of(t)` + `db.entity(e)` + `db.causal_history(e)` + `verify_chain` over an audit window; `rewind` mutates a per-session view-cursor only; `branch` calls `db.branch(t, [])` to fork; `edit` calls `db.transact([...])` under capability check. UI is one HTML file + one vanilla-JS module served from `src/persistence/repl/ui/`. Determinism contract is single-line: `inspect` and `rewind` are pure functions of `(view-cursor, store-state)` plus a write of one `:repl/op` audit entry; `edit` and `branch` write through the same `db.transact` path that programmatic callers use, so byte-identity replay holds. **Ships 2026-05-16; ~1100 LOC + ~240 tests + UI bundle.**

## 2. Strategic frame

### Why Module 7 ships full surface

The roadmap §3 ADR-1 (locked 2026-04-27) reversed the prior "REPL is headless-only / UI deferred to Phase 2.E" decision. That deferral was paper-driven — the "live steering" claim was treated as a forward-looking promise that did not require a runnable artifact to defend formally. Three things changed at the v1.0 pivot:

1. **The substrate's positioning against MemGPT / Mem0 / Zep / Memento depends on the write-time + steering-time + replay-time triad.** Those systems are read-only or read-mostly memory layers; persistence-os is the only one with a substrate that lets an operator inspect, edit, rewind, and branch a live trajectory through capability-gated audit-chained ops. Without a UI the proposition reads aspirational; with a UI the demo is the proof.
2. **The four ops map 1:1 to existing primitives.** `db.as_of`, `db.as_of_valid`, `db.causal_history`, `db.transact`, `db.branch` (all on `DB`); `DBView.entity` (entity projection on a `DBView`) — all shipped on `main` at `fcd1894` (see `src/persistence/fact/db.py:416-535`). The `make_audit_handler` factory + `verify_chain` are shipped (`src/persistence/effect/handlers/audit.py:380-407, 357`). Module 7 is a thin RPC seam over those primitives, not new substrate logic.
3. **Stream A's G4 promotion gate ships as a stub that Stream D replaces.** `_promotion.py:476` explicitly notes `"Stream D (Module 7 REPL): an operator-token capability check on …"`. Without Stream D's `inspect` + `edit` ops, the G4 stub stays in v1.0.0, weakening the paper claim that "promoted skills are operator-vetted." Stream D unblocks the stub-removal precondition that Stream G's pre-v1.0.0 integration test will assert.

   **G4 plumbing (concrete default).** Stream A's `promote(...)` accepts `g4_fn: Callable[[], dict]` (`_promotion.py:675`); the stub returns `{"approved": True, "approver": "stub", "rationale": ...}`. Stream D replaces the stub by an operator's `repl/edit` writing a single approval datom against a well-known token entity, plus a callable that reads it:

   ```python
   # operator side (one repl/edit op):
   #   datoms=[{"e": token_id,
   #            "a": "plan/promotion-gate-g4",
   #            "v": {"approved": True,
   #                  "approver": token_id,
   #                  "plan_hash": plan_hash,
   #                  "rationale": "<operator note>"},
   #            "valid_from": <now-iso>}]

   # caller side (passed into promote(...) at the call site):
   def g4_fn() -> dict:
       entity = db.as_of(_now()).entity(token_id)
       record = entity.get("plan/promotion-gate-g4") or {}
       if record.get("plan_hash") != plan_hash:
           return {"approved": False, "approver": token_id,
                   "rationale": "plan_hash mismatch"}
       return record
   ```

   The callable closes over `(db, token_id, plan_hash)` from the caller's scope — matches the no-arg `() -> dict` contract `promote()` already enforces. Stream G's integration test asserts no record has `approver == "stub"`.

**Datom-attr key style (key code↔doc rule, inherited from Stream A + Stream B + Stream C).** `Datom.a` slots written by REPL ops follow the repo convention: bare namespaced kebab-case, no leading colon at storage. Examples: `repl/session-id`, `repl/op-kind`, `repl/cap-token-id`, `repl/args-hash`, `repl/view-cursor-tx`, `repl/view-cursor-vt`. Wire-form `AuditEntry.op` keeps the leading colon (`:repl/op`) because that is the EDN-keyword form `audit_entry_to_datom` enforces (see `src/persistence/effect/handlers/audit.py:92-96`). The `repl/` prefix opts REPL writes out of any other namespace's reserved-prefix scans.

### Out of scope (explicit deferrals)

- **Multi-user concurrency beyond capability tokens.** A single REPL server instance accepts many concurrent sessions, each with its own token; sessions do not coordinate, do not see each other's view-cursors, and do not get notified of each other's edits except via the shared audit tail. **No multi-user lock manager, no presence indicator, no operator-to-operator chat.** Phase 3 NeSy 2027 if anyone wants it.
- **Persistent sessions across server restart.** Sessions live in process memory; a server restart drops all view-cursors. Tokens that were valid pre-restart remain valid post-restart (token store is fact-store-backed; see ADR-3), but the session needs to re-establish its view-cursor from `(tt, vt)` args on the next `inspect` call. Sticky session-id resumption across restart → v0.7.x. **Note (W2.C):** The audit ring is the only hot cache that drops on restart; the underlying `:repl/op` datoms persist in the fact store via `audit_entry_to_datom` + `db.transact` (ADR-4 + ADR-8). A reconnecting client can backfill the audit history via `repl/inspect kind="audit-window"` against the durable store. Session state (view-cursor, subscription) is dropped; AUDIT state is not.
- **OAuth / SSO / federation.** Tokens are issued by a bootstrap CLI or an env-var-loaded config file (ADR-3). No third-party identity provider integration. Phase 3.
- **Web-based plan editor.** The UI exposes the four ops as JSON-RPC commands typed into a console; it does NOT render a Plan AST tree, allow drag-drop plan composition, or visualize search trees. Plan-editor UI is v0.7.x or later.
- **Subscriptions to anything other than the audit tail.** v0.7.0a1 has exactly one server-pushed event stream — the audit tail (every new `AuditEntry` after subscription start). No per-entity subscriptions, no plan-execution event streams, no MCTS-rollout streams. Per-topic subscriptions are an OQ deferred to v0.7.x (§11 OQ-1).
- **REPL-side caching of view-cursor reads.** Every `inspect` call re-runs the underlying `db.as_of(t).entity(e)` query. No memoization in the session. The fact-store query is fast enough at v0.6.5 store sizes that adding a cache layer would be premature optimization, and the cache would itself need invalidation logic that complicates the audit-trace contract. Add in v0.7.x if profiling justifies it.
- **Browser-side TLS termination.** WS server listens on plaintext `ws://localhost:<port>` by default; production callers wrap behind a reverse proxy (nginx / Caddy / Traefik) that handles TLS. Module 7's threat model assumes loopback-only or LAN-only deployment for v0.7.0a1; public-internet exposure requires the reverse-proxy story (documented in `src/persistence/repl/__init__.py` module docstring).
- **CSRF / SameSite hardening for the bundled HTML page.** The bundled `index.html` is intended to be opened directly in a browser against a `localhost` WS port; it does not set cookies, does not embed in iframes, and does not accept third-party HTTP origins. Hardening for embedding is v0.7.x.
- **Plan-AST inspection beyond raw datom dump.** `inspect` on a plan_id returns the raw datoms; reified Plan AST tree rendering (with collapsed children, attr keys, etc.) is a UI-only concern that v0.7.0a1 does not ship. The console output is canonical-JSON pretty-printed; a tree renderer is OQ-3.

### ADR-1 — Transport: WebSocket (RFC 6455), not SSE / HTTP polling

**Status:** ACCEPTED.

**Context.** Module 7 needs bidirectional traffic: client → server for the four ops (request/response), server → client for the audit-tail subscription (server-pushed events). Three transports were on the table:

- **WebSocket (RFC 6455).** Bidirectional from frame zero. One TCP connection per session. Mature library support (browser-native `WebSocket` API; on the server, both Python `websockets` 12.x and `aiohttp.web.WebSocketResponse` are mature and asyncio-native).
- **Server-Sent Events (SSE).** One-way server → client. Would need a separate POST-based RPC channel for the four ops. Two transports per session, two failure modes, two reconnect stories.
- **HTTP long-polling.** Bidirectional emulation. Adds round-trip latency on every server-pushed audit event. Not viable for "live audit tail" UX.

**Decision.** WebSocket over loopback, served by **`aiohttp.web.WebSocketResponse`** (NOT the standalone `websockets` library). One TCP connection per session; client opens `ws://localhost:<port>/ws`, presents capability token in the first frame (auth handshake; ADR-3), then exchanges JSON-RPC frames bidirectionally for the lifetime of the session. The server pushes `notification` frames for audit events; the client sends `request` frames for ops.

**Rationale for `aiohttp` over `websockets`.** The same process must also serve the static UI bundle (ADR-6: `index.html` + `repl.js` + `repl.css`). `aiohttp` ships HTTP routing + WebSocket upgrade + static-file handler in a single library + single event loop, so we avoid (a) wiring a separate HTTP server next to a separate WS server, (b) coordinating two listening sockets, (c) carrying two libraries' release cadences. `aiohttp` is mature, asyncio-native, and already commonly available in test infrastructure.

**Consequences.**
- Single TCP connection state to manage. Auth replay attacks (ADR-3 detail) only need to defend the handshake frame, not every request.
- Single library dependency. `aiohttp.web.Application` registers two routes — `GET /` → static `index.html` + sibling assets; `GET /ws` → upgrade to `aiohttp.web.WebSocketResponse`. One event loop, one listening port.
- Browser-native `WebSocket` API is the only client-side runtime cost. Zero JS dependencies.
- Loopback-only by default; LAN/public exposure is a reverse-proxy concern documented in the module docstring.

### ADR-2 — Message protocol: JSON-RPC 2.0 envelope, not custom

**Status:** ACCEPTED.

**Context.** Three protocol shapes were considered:

- **JSON-RPC 2.0.** Standardized envelope (`{"jsonrpc": "2.0", "method": "...", "params": {...}, "id": <int|str|null>}`), well-known error code conventions, batch support, notification support (no `id` = no response expected). Library support in browser (none needed; trivial to write) and Python (`jsonrpcserver` exists but we hand-roll for tighter control).
- **Custom envelope.** Smaller wire footprint, but every reviewer + downstream consumer would need to read a spec. Reinventing the wheel for a stream that ships under a 7-day window.
- **gRPC over WS.** Type-safety win but a build-step + protobuf dependency, which contradicts ADR-6's zero-build UI principle.

**Decision.** JSON-RPC 2.0. Method names are `repl/inspect`, `repl/edit`, `repl/rewind`, `repl/branch`. Params are op-specific objects (§5 schemas). Error codes follow the JSON-RPC convention (`-32700` parse, `-32600` invalid request, `-32601` method not found, `-32602` invalid params, `-32603` internal error) plus a substrate-defined band (`-32000 .. -32099` reserved by JSON-RPC for application errors; we use `-32001` capability denied, `-32002` retroactive correction, `-32003` plan construction error, `-32004` audit-chain integrity broken at op execution; ADR-9 below).

**Consequences.**
- Reviewers reading the protocol spec recognize JSON-RPC 2.0 immediately; zero new vocabulary.
- Server-pushed audit-tail uses JSON-RPC notifications (no `id`); clients distinguish by `id` presence.
- Method-name namespace `repl/` mirrors the `Datom.a` and `AuditEntry.op` namespacing convention.
- Batch requests (JSON-RPC array form) are accepted by the parser but treated as one logical client intent — each batched call still emits its own `:repl/op` AuditEntry. This avoids "one batch = one audit entry" ambiguity.

### ADR-3 — Capability token format: opaque random 256-bit, fact-store-backed, env/config bootstrap

**Status:** ACCEPTED.

**Context.** The roadmap §3 ADR-1 decision pins capability gating as mandatory ("WebSocket must require an operator-issued capability token, no anonymous access"). Three token shapes were considered:

- **JWT (signed).** Self-validating, no server-side state. But: revocation requires a denylist (server-side state again), and key rotation drifts non-revoked tokens silently. Plus a JWT library dependency.
- **Opaque random token + server-side registry.** Server holds `token_id → CapabilitySet` mapping. Revocation = remove from registry. Trivial to validate, trivial to revoke. State lives in the substrate's existing fact store (free bitemporal indices + audit chain).
- **OAuth 2.0 bearer.** Requires an identity provider; out of scope (§3).

**Decision.** Opaque random 256-bit tokens, base64url-encoded (43 chars). Token registry is fact-store-backed: each issued token writes three datoms with `e = token_id`:

```python
[
  Datom(e=token_id, a="repl/cap-set",      v=canonical_dumps(cs.to_canonical()), ...),  # canonical JSON string
  Datom(e=token_id, a="repl/issued-at",    v=<:clock/now ISO-8601 string>,              ...),
  Datom(e=token_id, a="repl/issued-by",    v=<bootstrap-issuer-string>,                 ...),
]
```

The `repl/cap-set` `v` is exactly **one canonical JSON string** (see "Canonical JSON encoding" below) — one datom per token, not one datom per Capability. Validation: `validate_token(token_str) → CapabilitySet | None` does a single `db.as_of(_now()).entity(_token_id(token_str))` lookup; missing or revoked (a `repl/revoked-at` datom present) returns `None`. Revocation: `revoke_token(token_id)` writes one `("repl/revoked-at", :clock/now)` datom; subsequent validation rejects.

**Future normalisation (MINOR-2).** v0.7.0a1 stores `cap-set` as a single canonical-JSON string in `Datom.v`. v0.7.x may normalise into one datom per `(op, qualifier)` for queryability (e.g., "find all tokens that grant `edit:write`"); the current single-string shape is sufficient for v0.7.0a1 R1+R2 gates and keeps the entity history shallow (~3-4 datoms per token). Migration would be additive — new datoms appended at the v0.7.x bump tx, old single-string datoms remain valid until expiry.

**Bootstrap (issuance).** Two paths, both out-of-band relative to the running REPL server:

1. **Bootstrap CLI:** `python -m persistence.repl.cli issue --caps "inspect:read,rewind:any" --label "ops-readonly"` writes the three datoms above against the configured `DB` and prints the token to stdout. Operator copies it into a secrets store (1Password, Vault, etc.) and ships to the consumer out of band.
2. **Env-var loading:** at REPL server startup, `PERSISTENCE_REPL_BOOTSTRAP_TOKENS=<path-to-json>` points to a JSON file like `[{"caps": ["inspect:read"], "label": "ci"}, ...]`. The startup hook reads, issues each token, prints them once to stderr, and starts the server. Used for development and CI; production deployments use the CLI path.

**Capability set shape:**

```python
@dataclass(frozen=True)
class Capability:
    op: Literal["inspect", "edit", "rewind", "branch"]
    qualifier: str   # "read", "write", "any", "fork", per ADR table below

@dataclass(frozen=True)
class CapabilitySet:
    caps: frozenset[Capability]
    expires_at: datetime | None   # None = no expiry (rare; only bootstrap tokens)
    label: str                    # human-readable, audit-visible
```

**Canonical JSON encoding (W1.F pin).** `CapabilitySet` serializes via a stable, sorted, whitespace-free shape that is itself a function of the set contents (so `cs.to_canonical()` is referentially transparent and `canonical_dumps(cs.to_canonical())` is byte-identical for equal `cs`):

```python
def to_canonical(cs: CapabilitySet) -> dict:
    return {
        "caps": sorted(
            [{"op": c.op, "qualifier": c.qualifier} for c in cs.caps],
            key=lambda d: (d["op"], d["qualifier"]),
        ),
        "expires_at_iso": cs.expires_at.isoformat() if cs.expires_at else None,
        "label": cs.label,
    }
```

Serialized via `persistence.effect.canonical.canonical_dumps` (sorted keys, no whitespace). The fact-store datom carries this canonical string in `v`:

```python
Datom(e=token_id, a="repl/cap-set", v=canonical_dumps(cs.to_canonical()), ...)
```

Round-trip: `CapabilitySet.from_canonical_json(canonical_dumps(cs.to_canonical())) == cs` for any well-formed `cs` — pinned by a Hypothesis property test in Task D1 (§10).

Defined `(op, qualifier)` pairs (closed set in v0.7.0a1):

| op | qualifier | meaning |
|---|---|---|
| inspect | read | read-only entity / audit / plan inspection |
| edit | write | submit a `db.transact(...)` payload through `repl/edit` |
| rewind | any | move the session view-cursor to any `(tt, vt)` |
| branch | fork | call `db.branch(t, [])` to fork the store |

Future qualifiers (e.g. `inspect:audit-only`, `edit:retract-only`, `branch:read-only-fork`) are forward-compat — `_caps.py` validates the closed set in v0.7.0a1 and raises `UnknownCapability` on unrecognized pairs.

**Token presentation.** Client opens WS to `ws://localhost:<port>/ws`, sends one auth-handshake frame as the first message: `{"jsonrpc": "2.0", "method": "repl/auth", "params": {"token": "<base64url>"}, "id": 0}`. Server validates; on success replies `{"jsonrpc": "2.0", "result": {"session_id": "<deterministic-16-hex>", "caps": [...]}, "id": 0}` — `session_id = sha256(token_id + ":" + auth_clock_iso)[:16]` (W1.E); on failure replies a `-32001` error and closes the WS. **All subsequent frames are bound to that session_id**; the WS connection IS the auth context.

**Consequences.**
- No JWT library dependency. ~80 LOC token logic.
- Token issuance is itself audit-chained (`:repl/auth` AuditEntry on the issuing transact through `make_audit_handler` — bootstrap CLI runs under an audit handler stack the same as production).
- Revocation is one datom write; takes effect on the next `validate_token` call.
- Tokens are content-addressed via the `_token_id` helper (defined once in `_caps.py`):

  ```python
  def _token_id(token_str: str) -> str:
      return hashlib.sha256(token_str.encode()).hexdigest()[:16]
  ```

  so the token itself is never stored, only its 16-hex hash. `validate_token` recomputes the hash from the presented string. **Single name throughout (W2 MINOR-7):** `_token_id` is the only spelling in the design; `token_id_from_hash` and ad-hoc `hashlib.sha256(token_str.encode()).hexdigest()[:16]` inline calls are folded into `_token_id(token_str)` everywhere.
- Threat model bounds: an attacker who reads the fact store sees only token hashes, not the raw tokens. An attacker who reads the in-flight WS frame sees the raw token; loopback-only deployment is the mitigation for v0.7.0a1.

### ADR-4 — Op auditing: every op emits one `:repl/op` AuditEntry, chained

**Status:** ACCEPTED.

**Context.** The roadmap §3 ADR-1 makes "audit-traceable" mandatory: every REPL op (including read-only `inspect`) must be recorded in the audit chain. Three placements were considered:

- (a) **Audit only writes** (`edit`, `branch`). Reads (`inspect`, `rewind`) emit no audit. **Rejected** — violates the "no off-the-record peeking" property; an attacker with `inspect:read` could probe entity state without leaving a trace.
- (b) **Audit all four ops, but route reads through a separate non-Merkle log.** Two audit chains, two verify paths, two replay stories. **Rejected** — bifurcates the audit-chain hash continuity claim.
- (c) **All four ops emit one `:repl/op` AuditEntry through `make_audit_handler` into the same Merkle chain.** Reads and writes are uniform; `verify_chain` covers REPL traffic identically to programmatic traffic.

**Decision.** Option (c). Every op invocation appends one `AuditEntry` (op = `:repl/op`) to a caller-owned `entries: list[AuditEntry]`.

### Handler stack composition (direct path, NOT named_perform)

Stream B's `:mcts/iteration` writes go through `db.transact` directly — not through `make_audit_handler`. Stream D mirrors that pattern: REPL ops construct `AuditEntry` instances via `_content_hash` + `_canonicalise_content` (`audit.py:308, 313`) and append to a `WSServer._audit_entries: list[AuditEntry]` ring buffer. **NO use of `make_audit_handler` and NO `runtime.named_perform("audit", ":repl/op", …)`** — that handler factory is for stateful `:llm/call`-class wraps where the audit clause itself observes the wrapped effect's verdict + latency. REPL ops have no wrapped inner effect; the verdict is a local computation against the cap-set.

The "direct path" simplifies replay (no leaf-handler invariant to defend) and keeps the chain shape identical to a programmatic-write chain. `verify_chain` (`audit.py:357`) covers REPL traffic identically.

**Critical design constraint — REPL fields ride on `principal`.** The `AuditEntry` dataclass (`audit.py:40-69`) has a fixed canonical field set: `id`, `prev_hash`, `op`, `args_hash`, `verdict`, `latency_ms`, `recorded_at`, `result_hash`, `error`, `policy_id`, `handler_chain`, `principal`, `run_id`, `parent`, `txn_commit`. `verify_chain` (`audit.py:357-372`) recomputes `_content_hash(entry.to_dict())` against exactly those fields. **REPL-specific fields (`op_kind`, `view_cursor_tx_time_iso`, `view_cursor_vt_iso`) MUST NOT appear as flat top-level keys** — if they did, the helper-side hash would include them but `verify_chain`'s recomputed hash (against the dataclass-only shape) would not, and every `:repl/op` entry would fail Merkle verification. The fix: stuff REPL-specific fields into `principal` (which IS a canonical dataclass field, typed `dict[str, Any]`), compute `content` via `_canonicalise_content` over the canonical field set, and construct the entry from that exact content. The hash then matches `verify_chain`'s recomputation exactly.

```python
def emit_repl_op_audit(
    entries: list[AuditEntry],
    session: Session,
    op_kind: str,
    args: dict,
    *,
    verdict: str,             # canonical: "ok" | "error" | "deny" | "deny-silently" | "require-approval"
    latency_ms: int,          # AuditEntry.latency_ms is `int` per audit.py:54
    result_hash: str | None = None,
    error: str | None = None,
    view_cursor_tx_time_iso: str | None = None,
    view_cursor_vt_iso: str | None = None,
) -> AuditEntry:
    """Construct one ``:repl/op`` AuditEntry, append to ``entries``, return it.

    REPL-specific fields ride on ``principal`` (a canonical dataclass field)
    so the entry's content_hash is computed over exactly the same canonical
    field set ``verify_chain`` recomputes — no hash mismatch.

    No ``runtime.named_perform`` call. No ``make_audit_handler`` stack. The
    caller (WSServer) owns the ``entries`` list; ``prev_hash`` threads from
    the previous entry's id.
    """
    prev_hash = entries[-1].id if entries else None
    principal = {
        "token_id": session.cap_token_id,
        "session_id": session.session_id,
        "op_kind": op_kind,                        # bare op name: "inspect"|"edit"|"rewind"|"branch"|"auth"
        "view_cursor_tx_time_iso": view_cursor_tx_time_iso,
        "view_cursor_vt_iso": view_cursor_vt_iso,
    }
    content = {
        "prev_hash": prev_hash,
        "op": ":repl/op",
        "args_hash": canonical_hash(args),
        "verdict": verdict,                        # canonical substrate set; see below
        "latency_ms": latency_ms,
        "recorded_at": session.clock().timestamp(),  # session.clock is Callable[[], datetime]
        "result_hash": result_hash,
        "error": error,
        "policy_id": None,
        "handler_chain": (),
        "principal": principal,
        "run_id": None,
        "parent": prev_hash,
    }
    canonical_content = _canonicalise_content(content)
    content_hash_value = _content_hash(canonical_content)
    entry = AuditEntry(id=content_hash_value, **canonical_content)
    # AuditEntry.__post_init__ enforces the ``:repl/op`` leading-colon /
    # ≤-one-slash / no-literal-dot invariants on ``op``.
    entries.append(entry)
    return entry
```

**Walk-through.** Building a 2-entry chain `[e1, e2]` with this helper and calling `verify_chain([e1, e2])` returns `True`: each entry's `id` was computed by `_content_hash(_canonicalise_content(content))` over exactly the canonical `AuditEntry` field set, which is what `verify_chain` recomputes via `entry.to_dict()` minus `id`. `e2.prev_hash == e1.id` by construction (line `prev_hash = entries[-1].id if entries else None`), so the chain pointer holds. `principal` carries `op_kind` + cursors as dict members rather than top-level fields, so adding new REPL-specific fields in v0.7.x is a `principal` schema change, not a hash-shape change.

**Verdict vocabulary (canonical).** `verdict` is one of the substrate's canonical set: `"ok" | "error" | "deny" | "deny-silently" | "require-approval"` (`audit.py:53` docstring; `effect/verdicts.py` `_verdict_as_edn` / `_verdict_as_python` round-trip). REPL maps:

- success path → `"ok"`
- capability-denied / `request_hash` mismatch / stale-cursor edit / `repl/auth` reject → `"deny"`
- internal error / `verify_chain` False / token JSON corrupt → `"error"`

`op_kind` is the **bare op name** (`"inspect" | "edit" | "rewind" | "branch" | "auth"`); the rejection sub-class is implied by `verdict`. So an attempted-and-denied edit is `op=":repl/op"`, `verdict="deny"`, `principal.op_kind="edit"` — no `"edit-rejected"` sub-label. This is information-preserving: `(op_kind, verdict)` carries the same signal as the prior `op_kind="edit-rejected"` shape, with simpler vocabulary and zero collision against the substrate's `_verdict_as_edn` self-conform.

`canonical_hash` here is `persistence.effect.canonical.canonical_hash` (re-exported at `audit.py:29`, defined in `effect/canonical.py`).

`result_summary` (kept on the session for response shaping but NOT placed in the audit entry's content body): for `inspect`, `{"entity_count": N, "datom_count": M, "audit_count": K}`; for `edit`, `{"datoms_written": N, "tx": <tx-id>}`; for `rewind`, `{"new_cursor_tx_time_iso": ..., "new_cursor_vt_iso": ...}`; for `branch`, `{"branched_db_id": "<uuid>", "fork_at_tx_time_iso": ...}`. The full payload is NOT captured in the audit entry — that would be a privacy hazard for `inspect`-on-PII flows. The `args_hash` + the deterministic `(view-cursor, store-state)` view at audit time is sufficient to replay the read. The non-sensitive shape summary may be hashed into the canonical `result_hash` field if the caller passes it — that field IS a canonical `AuditEntry` slot.

**Persistence path.** After `emit_repl_op_audit` returns, the caller converts the entry into datoms via `audit_entry_to_datom(entry)` (`audit.py:612-718`) and commits them with `db.transact(datoms)`. The in-memory `WSServer._audit_entries` list is then a **hot cache** for the WS-tail subscription (ADR-8); durability lives in the fact store, identical to every other audit-chained op. This closes the audit-tail-overflow backfill loop (ADR-8): clients backfill via `repl/inspect kind="audit-window"` against the fact store, which holds the durably-persisted `:repl/op` datoms.

**Consequences.**
- One `:repl/op` AuditEntry per op invocation, constructed directly (no `make_audit_handler` stack). `verify_chain` over the full log (programmatic + REPL) returns `True` — the chain is one Merkle log, not two — because REPL fields ride on `principal` instead of forging a non-canonical flat field set.
- Reads are audit-chained: an attacker with `inspect:read` can read entity state but cannot do so silently. The audit entry records `(session_id, cap_token_id, op_kind, args_hash, view_cursor)` (the last three on `principal`) — sufficient to reconstruct what was inspected.
- `result_summary` is held only on the session for response shaping; it does NOT land in `AuditEntry.principal` or `AuditEntry.args_hash` content, so PII in inspected entities does not leak into the audit log itself.
- The `:repl/op` op-name follows the `AuditEntry.__post_init__` invariants (`src/persistence/effect/handlers/audit.py:84-106`): leading colon, ≤ one slash, no literal dots. Pinned at construction.
- `AuditEntry.op` is **always** `:repl/op`. The bare op name (`"inspect"`, `"edit"`, `"rewind"`, `"branch"`, `"auth"`) lives in `principal["op_kind"]`. The `verdict` field is the substrate canonical set `{"ok", "error", "deny", "deny-silently", "require-approval"}`; REPL uses `"ok"` for success, `"deny"` for capability/cursor/hash rejections, `"error"` for internal failures (corrupt token JSON, `verify_chain` False, etc.).
- `args_hash` uses `persistence.effect.canonical.canonical_hash` (re-exported at `audit.py:29`, defined in `effect/canonical.py`) — same canonical-JSON path used everywhere else in the substrate, so REPL's args_hash is byte-identical to programmatic-call args_hash for equal arg dicts.
- `verify_chain([e1, ..., en])` over a pure-REPL trace returns `True` by construction (the helper computes the hash via the same `_canonicalise_content` + `_content_hash` path `verify_chain` recomputes against `entry.to_dict()`).

### ADR-5 — View-cursor semantics: per-session, decoupled from branch

**Status:** ACCEPTED.

**Context.** The `rewind` op moves the session's view of the substrate to a prior `(tt, vt)` coordinate. Three semantic options were on the table:

- (a) **Per-session view-cursor; `rewind` mutates session state only; `branch` calls `db.branch(t, [])` and the session re-binds to the branched DB.** Reads after `rewind` go through `db.as_of(view_cursor.tx).entity(e)` against the original DB. Reads after `branch` go through the branched DB at its head.
- (b) **Global view-cursor.** All sessions share one cursor. **Rejected** — operators step on each other's views; defeats the multi-session story.
- (c) **`rewind` mutates the underlying store.** **Rejected** — `rewind` would imply a destructive rewrite of the bitemporal store, contradicting Prop 1 (datom append-only).

**Decision.** Option (a). View-cursor lives on `Session`:

```python
from collections.abc import Callable
from datetime import datetime

@dataclass
class Session:
    session_id: str                        # deterministic: sha256(token_id + ":" + auth_clock_iso)[:16]
    cap_token_id: str                      # 16-hex from token registry
    cap_set: CapabilitySet                 # frozen at auth time
    db: DB                                 # the active DB (may be a branched fork)
    view_cursor_tx_time_iso: str | None    # transaction-time cursor (ISO-8601); None = head
    view_cursor_vt_iso: str | None         # valid-time cursor (ISO-8601); None = head
    parent_session_id: str | None          # set if this session was created by `branch`
    parent_chain_depth: int                # 0 for main-rooted; ≥ 1 for branched-of-branched (max_branch_depth = 16)
    auth_clock_iso: str                    # ISO-8601 of :clock/now at auth time; load-bearing for replay-pinable session_id
    clock: Callable[[], datetime]          # the runtime_clock passed to WSServer.serve; `clock()` → datetime
    created_at_iso: str                    # ISO-8601
```

`clock` is the substrate's `:clock/now`-pinned `Callable[[], datetime]` injected via `WSServer.serve(host, port, db, *, runtime_clock)`; under replay it is substituted by the captured-clock function so `session.clock().timestamp()` reproduces every `:repl/op` AuditEntry's `recorded_at` byte-identically. See §6.2 for the auth-time wiring.

- `rewind(args)` sets `view_cursor_tx_time_iso` and/or `view_cursor_vt_iso` to the supplied coordinate; emits `:repl/op` audit; returns the updated cursor. **Does NOT mutate `db`.**
- `inspect(args)` reads through `db.as_of(datetime.fromisoformat(view_cursor_tx_time_iso)).entity(e)` (or `db.as_of_valid(datetime.fromisoformat(view_cursor_vt_iso))` depending on the request shape — see §5 schemas). If both cursors are `None`, reads at head. The `datetime.fromisoformat(...)` conversion is the boundary between the wire vocabulary (ISO strings) and the substrate (`datetime` only — `_coerce_dt` at `fact/db.py:594-601`).
- `branch(args)` calls `db.branch(t=datetime.fromisoformat(view_cursor_tx_time_iso) if set else _now(), assertions=[])` — same primitive Stream A's tests use. Returns a new session bound to the branched DB. The original session's `db` is unchanged. The new session's `parent_session_id` records the parent; `parent_chain_depth = parent.parent_chain_depth + 1`. `view_cursor_*` on the new session is reset to `None` (head of the branched DB). Capability set is inherited from the parent session **verbatim, including `expires_at`**. **Invariant (MINOR-6):** if the parent's `cap_set.expires_at` is past at branch time, server rejects `repl/branch` with `-32006 session expired` BEFORE forking — preventing an operator from extending an expired token's lifetime indefinitely by chain-branching. **Also: if `parent_chain_depth >= max_branch_depth` (default 16), server rejects with `-32007 branch depth exceeded`** (see ADR-9 + OQ-5).
- `edit(args)` calls `parent_db.transact([...])` if the session is on a branched DB or main DB. Writes ALWAYS land in the session's current `db` — never silently in main if the session is on a branch.

**Cross-cursor + branch interaction.** If `view_cursor_tx_time_iso` is set and the operator calls `branch`, the branch fork point is the cursor, not the head. This makes "rewind to bug, fork to fix" a one-liner: `rewind {tx_time_iso: T} → branch {} → edit {datoms: [...]}`.

**Consequences.**
- `rewind` is a pure session-state mutation + an audit emission; no store side-effect.
- `branch` is one `db.branch(t, [])` call that returns a new session; the parent session is unaffected.
- A session on a branched DB is structurally isolated from main; `edit` on the branched session never leaks into main. Promotion of a branch's writes back to main is OUT of scope for v0.7.0a1 (would be a `db.fork()` / merge primitive that doesn't yet exist; ADR's open-question OQ-4).
- The view-cursor IS the session's REPL state. Server restart drops it (§3 deferral).

### ADR-7 — Two-step propose-confirm flow without server-side preview

**Status:** ACCEPTED.

**Context.** `edit` is the only op that mutates the substrate. Three flows were considered:

- (a) **Single-step with capability.** Operator with `edit:write` token can submit any `db.transact` payload; one frame in, one frame out. Lowest friction.
- (b) **Two-step propose-confirm with server-computed preview.** `edit-propose` returns a proposal hash + a server-computed preview of the resulting auto-retractions; `edit-confirm` references the hash and commits. Requires extending `db.py` with a `preview_transact` function that runs the auto-retraction logic without writing. ~30 LOC of substrate-source change inside the new package's blast radius — violates §3's "zero substrate-source change outside the new package" claim.
- (c) **Two-step propose-confirm without server-side preview.** Server returns the request body verbatim on missing `confirm`, with a `request_hash` for the operator to verify. Operator inspects the request locally, then re-issues with `confirm: true` AND the same `request_hash`. NO server-side preview computation, NO `db.py` extension. The operator accepts that "the preview is the input, not the output" — which is honest, since the auto-retractions are a deterministic function of the input + current store state, and the operator can run an `inspect` ahead of the `edit` to see the current state.

**Decision.** Option (c). The request body for `edit` (first step):

```json
{
  "jsonrpc": "2.0",
  "method": "repl/edit",
  "params": {
    "datoms": [{"e": "...", "a": "...", "v": "...", "valid_from": "..."}],
    "provenance": {"reason": "operator-correction"},
    "force_retroactive": false
  },
  "id": 7
}
```

Missing `confirm` → server returns `requires_confirmation` envelope:

```json
{
  "jsonrpc": "2.0",
  "result": {
    "requires_confirmation": true,
    "request_hash": "<canonical_hash of params with confirm field stripped>",
    "echoed_request": {
      "datoms": [...],
      "provenance": {...},
      "force_retroactive": false
    },
    "message": "re-issue with confirm: true and the same request_hash to commit"
  },
  "id": 7
}
```

Operator inspects `echoed_request` (or runs an `inspect` to see what state would auto-retract), then re-issues with both `confirm: true` and `request_hash` matching. Server recomputes `canonical_hash(params with confirm stripped)`; mismatch → `-32001` "request_hash mismatch" rejection. Match → `db.transact(...)` runs and the success response shape returns.

`request_hash = canonical_hash({"datoms": ..., "provenance": ..., "force_retroactive": ...})` (the params dict minus `confirm` and `request_hash` themselves). Computed via `persistence.effect.canonical.canonical_hash` — same path used everywhere else.

The rejection (missing-confirm path) emits a `:repl/op` audit entry with `principal.op_kind="edit"`, `verdict="deny"` so abandoned-edit attempts are still chained. The bare op name stays; the rejection sub-class is implied by `verdict="deny"` (the canonical substrate verdict for capability/cursor/hash-mismatch denials). See ADR-4 verdict-mapping table.

**Consequences.**
- **Zero substrate-source change.** No `preview_transact` extension in `db.py`. §3's "zero substrate-source change outside the new package" claim is preserved.
- One audit entry per actual write + one audit entry per rejected attempt.
- Operators can't accidentally write by typo; the second-step `confirm: true` + matching `request_hash` is intentional.
- The "preview" is the input not the output — operators wanting to know what auto-retractions will fire run an `inspect` against current state ahead of the `edit`. The `request_hash` ties the second step to the exact bytes the operator approved.
- Hostile clients with valid `edit:write` can still write; capability is the load-bearing gate, `confirm + request_hash` is the second factor against fat-finger ops, not against malicious operators.

### ADR-8 — Subscription / event-stream semantics: one stream per session, audit-tail only

**Status:** ACCEPTED.

**Context.** The UI's right pane shows a live audit tail. Three subscription models were considered:

- (a) **One stream per session, server-pushes every new AuditEntry after subscription start.** Simplest. No filtering on the server; client renders all.
- (b) **Per-topic subscriptions** (e.g. `audit:repl/op`, `audit:effect/llm-call`). Server-side filter; lower bandwidth.
- (c) **No subscriptions; client polls.** Wastes round trips; defeats the "live" UX.

**Decision.** Option (a) for v0.7.0a1. The session opts into audit-tail by sending `repl/subscribe-audit` with no params; server responds with the subscription_id and starts pushing notifications. Notifications follow JSON-RPC notification shape (no `id`):

```json
{
  "jsonrpc": "2.0",
  "method": "repl/audit-event",
  "params": {
    "audit_id": "...",
    "op": ":repl/op",
    "verdict": "ok",
    "recorded_at": 1727000000.000,
    "principal": {"session_id": "...", "cap_token_id": "..."},
    "summary": {"op_kind": "inspect", "result_summary": {...}}
  }
}
```

**Two-tier audit storage (W2.C — ring is hot cache, fact store is durable).** Every REPL op's `:repl/op` AuditEntry is **both** (a) appended to the per-session in-memory ring (default 256 entries; serves the WS-tail push) AND (b) persisted via `audit_entry_to_datom(entry)` + `db.transact(datoms)` to the fact store (durable). The ring is a hot-cache for the WS-tail subscription; the fact store is the source of truth.

If a slow client falls behind and the per-session ring overflows, the server emits one `repl/audit-event-overflow` notification with the dropped tx-window (`{"dropped_first_tx_iso": "...", "dropped_last_tx_iso": "...", "dropped_count": N}`) and the client backfills via `repl/inspect kind="audit-window"` against the fact store, which holds the durably-persisted REPL entries. **The previously-broken backfill path (where `:repl/op` lived only in `WSServer._audit_entries` while `inspect kind="audit-window"` queried `db`) is closed: the two stores are no longer disjoint — every `:repl/op` lands in `db` via `audit_entry_to_datom` so `inspect kind="audit-window"` returns it.**

**Server-restart durability win.** Pre-W2.C, server restart dropped every `:repl/op` because it lived only in the in-memory ring. Post-W2.C, server restart drops only the in-memory hot cache; the underlying datoms persist in the fact store. A reconnecting client can replay the audit history from any prior session via `inspect kind="audit-window"` even after a restart. (Sticky session-id resumption is still v0.7.x — the SESSION drops; the AUDIT does not.)

**Per-topic filtering** (option b) is OQ-1; the wire format and server-side hook are forward-compat — `repl/subscribe-audit` accepts an optional `filter` param that v0.7.0a1 rejects with `-32602` "filter parameter not supported in this version" if non-null.

**Consequences.**
- Subscription state is in-process per session; server restart drops it.
- Bandwidth at v0.6.5 store sizes is fine; one AuditEntry is a few hundred bytes; even sustained 100 events/sec is < 50KB/sec.
- Forward-compat hook (filter param) is documented and tested for rejection in v0.7.0a1; v0.7.x can flip the rejection to acceptance without a wire-format change.

### ADR-9 — Error envelope shape: JSON-RPC error object + substrate error codes

**Status:** ACCEPTED.

**Context.** JSON-RPC 2.0 reserves `-32700..-32603` for protocol errors; `-32000..-32099` is the application-error band. We need a stable mapping from substrate exceptions to error codes so the UI and downstream clients can render meaningfully.

**Decision.** Closed code table (v0.7.0a1) with corresponding canonical verdict (W2.B substrate-vocabulary mapping):

| Code | Class | Trigger | Verdict |
|---|---|---|---|
| -32700 | parse error | malformed JSON | `error` |
| -32600 | invalid request | not JSON-RPC 2.0 | `error` |
| -32601 | method not found | unknown `method` | `error` |
| -32602 | invalid params | params shape mismatch (per-op schema; §5). Sub-cases include `unknown filter` and `filter not supported in this version` (cross-ref ADR-8 forward-compat hook). | `error` |
| -32603 | internal error | unhandled server exception | `error` |
| -32001 | capability denied | session caps don't grant the requested op qualifier; OR `edit` without `confirm: true` + matching `request_hash` (ADR-7) | `deny` |
| -32002 | retroactive correction | `db.transact` raised `RetroactiveCorrectionError`; `data` carries the `force_retroactive: true` retry hint | `error` |
| -32003 | plan construction error | `Node(...)` raised during an inspect on a Plan AST datom | `error` |
| -32004 | audit chain integrity broken | `verify_chain` returned False on the inspected window; signals substrate corruption | `error` |
| -32005 | view cursor invalid | `rewind` to a coordinate before the store's earliest datom | `error` |
| -32006 | session expired | cap token expired between auth and op | `deny` |
| -32007 | branch depth exceeded | `repl/branch` from a session whose `parent_chain_depth >= max_branch_depth` (default 16) — see OQ-5 | `deny` |
| -32008 | stale-cursor edit not allowed | `repl/edit` with `view_cursor_tx_time_iso` set and the operator has not branched first — see §5.2 | `deny` |

The Verdict column is the canonical substrate value emitted on the `:repl/op` AuditEntry's `verdict` field for any op invocation that returned that error code. `principal.op_kind` stays the bare op name (`"inspect" | "edit" | "rewind" | "branch" | "auth"`) regardless of error code. Total error codes: **13** (5 protocol-reserved + 8 application). §9.2 R2 test count line `per-error-code × 13` matches this row count.

**Error envelope:**

```json
{
  "jsonrpc": "2.0",
  "error": {
    "code": -32001,
    "message": "capability denied: edit:write required",
    "data": {
      "session_caps": ["inspect:read", "rewind:any"],
      "required": "edit:write"
    }
  },
  "id": 12
}
```

The `data` object is op-specific; closed-set per the table above. The UI renders `error.message` in red and `error.data` as collapsed JSON.

**Consequences.**
- Closed code set in v0.7.0a1; new codes require a CHANGELOG bump.
- All errors are themselves audited via `verdict ∈ {"deny", "error"}` and `principal.op_kind = <bare op name>` (§ADR-4 + the verdict column in the table above) so the audit chain captures intent regardless of outcome. The canonical substrate verdict set is `{"ok", "error", "deny", "deny-silently", "require-approval"}`; REPL uses `"ok"` for success, `"deny"` for capability/cursor/hash rejections, `"error"` for internal failures (corrupt token JSON, `verify_chain` False, RetroactiveCorrectionError, etc.). `principal.op_kind` is the bare op name — rejection sub-class is implied by `verdict`.

### ADR-10 — Async runtime: asyncio + aiohttp, no threading

**Status:** ACCEPTED.

**Context.** `aiohttp.web` is `asyncio`-native, ships HTTP routing + WS upgrade + static-file handler in one library. `db.transact` is synchronous and may block on store I/O (SQLite path). Three runtime shapes were considered:

- (a) **Pure asyncio with `loop.run_in_executor` for blocking calls.** Modern, single event loop, `asyncio.Lock` for session-state mutations.
- (b) **asyncio main + threadpool worker per session.** Per-session thread for `db.transact` calls. Adds GIL contention without proportionate benefit at v0.7.0a1 traffic levels.
- (c) **gevent / sync.** Mature but mixing with `aiohttp` (asyncio) is awkward; no win.

**Decision.** Option (a). One asyncio event loop in `WSServer.serve()`; `aiohttp.web.Application` registers `GET /` (static) + `GET /ws` (WS upgrade); `db.transact` and `DBView.entity` calls are wrapped in `loop.run_in_executor(None, ...)` so the loop is never blocked. Per-session state (cursor + chain head) is mutated under an `asyncio.Lock` held only inside the op handler, never across awaits.

**Library pin (v0.7.0a1).** `aiohttp >= 3.9.0, < 4.0.0`, listed under `[project.optional-dependencies]` `repl` extra in `pyproject.toml`. **NO `websockets` library dependency.** `aiohttp` is already commonly available in test infrastructure as a transitive dep; declaring it as a direct REPL extra makes the dependency explicit.

**Consequences.**
- Single Python process, no thread-per-session, single library for HTTP + WS. ~50 concurrent sessions at v0.7.0a1 traffic levels is comfortable.
- `db.transact` running in the default executor means up to `os.cpu_count()` parallel writes; the substrate's `Store.allocate_and_append` is already serialized internally (`src/persistence/fact/store.py`), so concurrent calls queue.
- `asyncio.Lock` per session prevents the "rewind mid-inspect produces inconsistent results" race.
- Test runtime is `pytest-asyncio` for the `_ws.py` tests against an in-process `aiohttp.test_utils.TestServer`; `_ops.py` tests are pure sync over a `Session` fixture.

## 5. Op contracts

### 5.0 Time vocabulary

The REPL wire vocabulary uses **ISO-8601 datetime strings** for all temporal cursors:

- `view_cursor_tx_time_iso: str` — slices on `tx_time` via `db.as_of(<datetime>)`. **Distinct from `Datom.tx`** (the integer transaction id; never appears on the REPL wire).
- `view_cursor_vt_iso: str` — slices on `valid_time` via `db.as_of_valid(<datetime>)`.
- `auth_clock_iso: str` — captured `:clock/now` at auth time; load-bearing for replay-pinable `session_id` (§6.2).
- `recorded_at` on `:repl/op` AuditEntries — `float` epoch seconds per the canonical `AuditEntry.recorded_at` slot (W2.A: REPL uses the same canonical field set as every other audit-chained op; ISO conversion happens at the wire boundary in `to_edn` via `_recorded_at_to_inst`, not in the canonicalised content).

Conversion: when needed for client display, `int(datetime.fromisoformat(s).timestamp() * 1000)` yields milliseconds since epoch.

**Substrate boundary.** `db.as_of`, `db.as_of_valid`, `db.branch` accept `datetime` only via `_coerce_dt` (`fact/db.py:594-601` — accepts `datetime` or ISO string, NOT `int`). Server uses `datetime.fromisoformat(s)` at every op-boundary and passes a `datetime` to the substrate. The wire never carries millisecond integers; the substrate never sees ISO strings except via `_coerce_dt`'s string branch.

**Worked example (operator → server → substrate flow).**

```
operator types:    > rewind tx_time_iso=2026-04-28T12:00:00+00:00
client sends:      {"method":"repl/rewind",
                    "params":{"tx_time_iso":"2026-04-28T12:00:00+00:00"},
                    "id":3}
server step 1:     dt = datetime.fromisoformat("2026-04-28T12:00:00+00:00")
                   # validation: must be tz-aware (matches db.py _coerce_dt rule).
server step 2:     session.view_cursor_tx_time_iso = "2026-04-28T12:00:00+00:00"
                   # ISO string stored on Session, NOT a datetime — keeps Session
                   # JSON-serialisable for replay traces.
server step 3:     # next inspect:
                   db.as_of(datetime.fromisoformat(session.view_cursor_tx_time_iso))
                     .entity(eid)
```

This vocabulary choice (option **a — ISO-8601 datetime strings end-to-end**) was preferred over millisecond integers because it (1) matches `db.transact`'s ISO `valid_from` convention, (2) disambiguates from `Datom.tx`, (3) keeps Session traces JSON-serialisable without a custom encoder. Replay capture pins the auth-time clock as ISO and substitutes deterministically.

### 5.1 `repl/inspect`

Reads the bitemporal store at the session's view-cursor (or at an explicitly-supplied coordinate that overrides the cursor for this call only).

**Required capability:** `inspect:read`.

**Request schema:**

```json
{
  "jsonrpc": "2.0",
  "method": "repl/inspect",
  "params": {
    "kind": "entity" | "audit-window" | "plan" | "causal-history",
    "args": {
      // for kind=entity:
      "entity_id": "<string>",
      "tx_time_iso": "<ISO-8601 string|null>",   // override; null = use session cursor
      "vt_iso": "<ISO-8601 string|null>",
      // for kind=audit-window:
      "start_iso": "<ISO-8601 string>",
      "end_iso": "<ISO-8601 string>",
      "op_filter": "<:audit-op|null>",
      // for kind=plan:
      "plan_id": "<string>",
      // for kind=causal-history:
      "entity_id": "<string>",
      "max_depth": <int>             // currently 1; v0.7.x extends
    }
  },
  "id": <int>
}
```

**Response (success):**

```json
{
  "jsonrpc": "2.0",
  "result": {
    "kind": "entity",
    "data": {"name": "widget", "wacc": 0.087},
    "view_cursor": {"tx_time_iso": "2026-04-28T12:00:00+00:00", "vt_iso": null},
    "audit_id": "<the :repl/op AuditEntry id>"
  },
  "id": <int>
}
```

For `kind="audit-window"`, the `data` slot is a list of `AuditEntry`-shaped dicts (per the wire form via `audit_entry_to_datom` + `datom_to_audit_entry`) covering both programmatic AND REPL-originated entries in the requested `[start_iso, end_iso]` window. Optional `op_filter` narrows by `AuditEntry.op` (e.g., `":repl/op"` returns only REPL traffic, `":llm/call"` returns only LLM-call traffic). This is the canonical backfill path used by clients after `repl/audit-event-overflow` and after server restart (W2.C — `:repl/op` entries persist via `audit_entry_to_datom` so they appear here regardless of in-memory ring state).

**Response (error):** see ADR-9 codes; relevant: `-32001` cap denied, `-32004` chain broken, `-32005` cursor invalid.

**Audit fields emitted (`:repl/op`):**

```text
op            = ":repl/op"
args_hash     = canonical_hash({"kind": "entity", "args": {...}})
verdict       = "ok"   # canonical substrate set; "deny" on cap-reject, "error" on internal
latency_ms    = <int>          # AuditEntry.latency_ms is `int` per audit.py:54
recorded_at   = session.clock().timestamp()  # float
result_hash   = canonical_hash({"kind": "entity", "datom_count": N, "audit_count": K})  # nullable; non-PII shape summary only
principal     = {
    "token_id": "...",
    "session_id": "...",
    "op_kind": "inspect",                   # bare op name
    "view_cursor_tx_time_iso": "<ISO-8601|null>",
    "view_cursor_vt_iso": "<ISO-8601|null>",
}
```

`result_summary` (the per-kind shape `{"kind": "entity", "datom_count": N, "audit_count": K}`) is held only on the session for response shaping; if the caller decides to pin its hash into the canonical `result_hash` slot, that's a deterministic non-PII summary. The full payload is never written to the audit log.

**Worked example.** Operator with `inspect:read` runs `inspect entity sku-123` after a `rewind` to a known-good tx:

```json
// → request
{"jsonrpc":"2.0","method":"repl/inspect","params":{"kind":"entity","args":{"entity_id":"sku-123"}},"id":1}
// ← response
{"jsonrpc":"2.0","result":{"kind":"entity","data":{"name":"widget","wacc":0.087},"view_cursor":{"tx_time_iso":"2026-04-28T12:00:00+00:00","vt_iso":null},"audit_id":"abc12345"},"id":1}
```

### 5.2 `repl/edit`

Submits a `db.transact(...)` payload through the substrate's normal write path. Two-step propose-confirm flow (ADR-7).

**Required capabilities:** `edit:write`. Plus a second-step request carrying `confirm: true` AND a matching `request_hash`.

**Edit-on-stale-cursor invariant.** `edit` always writes at HEAD regardless of view-cursor. If the session has a non-null `view_cursor_tx_time_iso` and the operator has not branched first, the server rejects with `-32008 stale-cursor edit not allowed`. To write at a past coordinate the operator must `repl/branch` first, which forks a new session at the cursor, and then `edit` against that branched session (whose head IS the fork point).

**Request schema (step 1 — propose):**

```json
{
  "jsonrpc": "2.0",
  "method": "repl/edit",
  "params": {
    "datoms": [
      {"e": "<eid>", "a": "<attr>", "v": <any>, "valid_from": "<isoformat>", "op": "assert"}
    ],
    "provenance": {"reason": "...", "operator_note": "..."},
    "force_retroactive": false
  },
  "id": <int>
}
```

**Response (step 1 — requires confirmation):**

```json
{
  "jsonrpc": "2.0",
  "result": {
    "requires_confirmation": true,
    "request_hash": "<canonical_hash of params with confirm and request_hash stripped>",
    "echoed_request": {
      "datoms": [...],
      "provenance": {...},
      "force_retroactive": false
    },
    "message": "re-issue with confirm: true and the same request_hash to commit"
  },
  "id": <int>
}
```

**Request schema (step 2 — confirm):**

```json
{
  "jsonrpc": "2.0",
  "method": "repl/edit",
  "params": {
    "datoms": [...],
    "provenance": {...},
    "force_retroactive": false,
    "confirm": true,
    "request_hash": "<must match step-1 hash exactly>"
  },
  "id": <int>
}
```

**Response (step 2 — success):**

```json
{
  "jsonrpc": "2.0",
  "result": {
    "tx": 12345,
    "datoms_written": 3,
    "auto_retracted": 1,
    "audit_id": "<the :repl/op entry>"
  },
  "id": <int>
}
```

**Response (error):** `-32001` cap denied or `request_hash` mismatch, `-32002` retroactive correction, `-32008` stale-cursor, `-32602` invalid params.

**Audit fields emitted:**

```text
op            = ":repl/op"
args_hash     = canonical_hash({"datoms": [...], "provenance": {...}, "force_retroactive": false})
verdict       = "ok"            # success commit; "deny" on cap/request_hash/stale-cursor reject;
                                # "error" on RetroactiveCorrectionError or other internal
latency_ms    = <int>          # int per audit.py:54; dominated by db.transact + auto-retraction time
recorded_at   = session.clock().timestamp()
result_hash   = canonical_hash({"tx": 12345, "datoms_written": 3, "auto_retracted": 1})  # nullable; null on reject
principal     = {
    "token_id": "...",
    "session_id": "...",
    "op_kind": "edit",                       # bare op name; rejection sub-class implied by verdict
    "view_cursor_tx_time_iso": "<ISO-8601|null>",
    "view_cursor_vt_iso": "<ISO-8601|null>",
}
```

The step-1 propose path (missing `confirm`) and the cap-denied / `request_hash`-mismatch / stale-cursor paths all emit a `:repl/op` AuditEntry with `verdict="deny"` and `principal.op_kind="edit"` — the bare op name stays, the rejection sub-class is implied by `verdict`. The `db.transact(...)` call (success path) ALSO emits its own datom-level provenance via the standard path; the `:repl/op` AuditEntry sits adjacent in the audit chain, linking the REPL session to the underlying tx via the canonical `result_hash` (which hashes `{"tx": ..., "datoms_written": ..., "auto_retracted": ...}`).

**Worked example (step 1 → step 2).**

```json
// → step 1 (propose)
{"jsonrpc":"2.0","method":"repl/edit","params":{"datoms":[{"e":"sku-123","a":"wacc","v":0.092,"valid_from":"2026-04-28T12:00:00Z"}],"provenance":{"reason":"operator-correction","ticket":"OPS-4421"}},"id":2}
// ← step 1 response
{"jsonrpc":"2.0","result":{"requires_confirmation":true,"request_hash":"a1b2c3...","echoed_request":{...},"message":"re-issue with confirm: true and the same request_hash to commit"},"id":2}

// → step 2 (confirm)
{"jsonrpc":"2.0","method":"repl/edit","params":{"datoms":[{"e":"sku-123","a":"wacc","v":0.092,"valid_from":"2026-04-28T12:00:00Z"}],"provenance":{"reason":"operator-correction","ticket":"OPS-4421"},"confirm":true,"request_hash":"a1b2c3..."},"id":3}
// ← step 2 response (success)
{"jsonrpc":"2.0","result":{"tx":12346,"datoms_written":2,"auto_retracted":1,"audit_id":"def67890"},"id":3}
```

### 5.3 `repl/rewind`

Sets the session's view-cursor.

**Required capability:** `rewind:any`.

**Request schema:**

```json
{
  "jsonrpc": "2.0",
  "method": "repl/rewind",
  "params": {
    "tx_time_iso": "<ISO-8601 string|null>",
    "vt_iso": "<ISO-8601 string|null>"
  },
  "id": <int>
}
```

`tx_time_iso` or `vt_iso` (or both, or neither — `{}` resets to head) define the new cursor. `null` for a slot means "leave unchanged." Server runs `datetime.fromisoformat(s)` at the op-boundary and passes a `datetime` to `db.as_of` / `db.as_of_valid` (per `_coerce_dt` at `fact/db.py:594-601`).

**Response (success):**

```json
{
  "jsonrpc": "2.0",
  "result": {
    "view_cursor": {"tx_time_iso": "2026-04-28T12:00:00+00:00", "vt_iso": null},
    "audit_id": "<the :repl/op entry>"
  },
  "id": <int>
}
```

**Response (error):** `-32001` cap denied, `-32005` view cursor invalid (before earliest datom).

**Audit fields emitted:**

```text
op            = ":repl/op"
args_hash     = canonical_hash({"tx_time_iso": "...", "vt_iso": "..."})
verdict       = "ok"   # "deny" on cap-reject; "error" on -32005 (cursor before earliest)
latency_ms    = <int>          # AuditEntry.latency_ms is `int` per audit.py:54
recorded_at   = session.clock().timestamp()
result_hash   = canonical_hash({"new_cursor_tx_time_iso": "...", "new_cursor_vt_iso": "..."})
principal     = {
    "token_id": "...",
    "session_id": "...",
    "op_kind": "rewind",                    # bare op name
    "view_cursor_tx_time_iso": "<ISO-8601|null, BEFORE the rewind>",
    "view_cursor_vt_iso": "<ISO-8601|null, BEFORE the rewind>",
}
```

The `view_cursor_*` slots in `principal` capture the cursor value at audit time (i.e. *before* the rewind takes effect on `Session`), giving the audit log the "rewind from X to Y" delta when paired with `result_hash`. Replay re-derives both endpoints from `(args_hash, result_hash, principal.view_cursor_*)`.

### 5.4 `repl/branch`

Forks a new session bound to a `db.branch(t, [])` of the current session's DB.

**Required capability:** `branch:fork`.

**Request schema:**

```json
{
  "jsonrpc": "2.0",
  "method": "repl/branch",
  "params": {
    "fork_at_tx_time_iso": "<ISO-8601 string|null>",   // null = use session view-cursor or head
    "label": "what-if-wacc-correction"
  },
  "id": <int>
}
```

**Response (success):**

```json
{
  "jsonrpc": "2.0",
  "result": {
    "branched_session_id": "<deterministic 16-hex>",
    "branched_db_id": "<uuid>",
    "fork_at_tx_time_iso": "2026-04-28T12:00:00+00:00",
    "parent_chain_depth": 1,
    "audit_id": "<the :repl/op entry>"
  },
  "id": <int>
}
```

The client must then issue a fresh `repl/auth` (or a `repl/switch-session` op deferred to v0.7.x) to bind the WS to the branched session. In v0.7.0a1, the operator opens a second WS connection and authenticates with the same token (which now grants access to the branched DB through the `branched_session_id` returned).

**Response (error):** `-32001` cap denied; `-32007` branch depth exceeded (when `parent_chain_depth >= max_branch_depth`, default 16 — see OQ-5).

**Audit fields emitted:**

```text
op            = ":repl/op"
args_hash     = canonical_hash({"fork_at_tx_time_iso": "...", "label": "..."})
verdict       = "ok"   # "deny" on cap-reject; also "deny" on -32007 branch depth exceeded
latency_ms    = <int>          # int per audit.py:54; dominated by db.branch snapshot copy
recorded_at   = session.clock().timestamp()
result_hash   = canonical_hash({
    "branched_session_id": "...",
    "branched_db_id": "...",
    "fork_at_tx_time_iso": "...",
    "parent_chain_depth": <int>,
})
principal     = {
    "token_id": "...",
    "session_id": "...",                    # parent session
    "op_kind": "branch",                    # bare op name
    "view_cursor_tx_time_iso": "<ISO-8601|null>",
    "view_cursor_vt_iso": "<ISO-8601|null>",
}
```

The `parent_session_id` of the new branched session is the `principal.session_id` field of this entry; downstream tooling reconstructs the branch lineage by walking `result_hash` back to `principal.session_id` chain.

## 6. Capability model

### 6.1 Token format

Opaque random 256-bit, base64url-encoded (43 chars). Generated via `secrets.token_urlsafe(32)`. **NOT a JWT** — despite the superficial resemblance, tokens carry no claims and are not signed; they are pure random bytes whose `token_id` is the lookup key for a server-side fact-store entity. The raw token is NEVER persisted; only `token_id = _token_id(token_str) = sha256(token_str.encode()).hexdigest()[:16]` lands in the fact store.

### 6.2 Issuance

Two paths:

**Bootstrap CLI** (`python -m persistence.repl.cli issue`):

```bash
$ python -m persistence.repl.cli issue \
    --caps "inspect:read,rewind:any" \
    --label "ops-readonly" \
    --expires-in-days 30 \
    --db-path /var/lib/persistence-os/store.db

issued: persistence.repl/Xy7p9NqK4mJzL2sQwR8tVuFb6cE1aH3dYgKjMnTpZ_E   # 43-char base64url; opaque random bytes via secrets.token_urlsafe(32), NOT a JWT
token_id: a1b2c3d4e5f67890
caps: inspect:read, rewind:any
expires_at: 2026-05-28T12:00:00+00:00
```

**Env-var bootstrap.** `PERSISTENCE_REPL_BOOTSTRAP_TOKENS=/path/to/tokens.json` at REPL server startup. `tokens.json`:

```json
[
  {"caps": ["inspect:read"], "label": "ci-readonly", "expires_in_days": 7},
  {"caps": ["edit:write", "inspect:read", "rewind:any", "branch:fork"], "label": "operator-full", "expires_in_days": 30}
]
```

Server reads, issues each token, prints them once to stderr, then starts. **Tokens printed at startup are the only place the raw token surfaces in production** — operator must capture them from the log + ship to the consumer out of band.

**Deterministic `session_id` (W1.E pin).** When the auth handshake succeeds, the server constructs:

```python
# runtime_clock is the Callable[[], datetime] passed to WSServer.serve(...)
# (W2.D wiring); under replay it is the substrate's :clock/now-pinned function.
auth_clock_iso = runtime_clock().isoformat()
session_id = hashlib.sha256(
    f"{token_id}:{auth_clock_iso}".encode()
).hexdigest()[:16]
session = Session(
    session_id=session_id,
    cap_token_id=token_id,
    cap_set=cap_set,
    db=db,
    view_cursor_tx_time_iso=None,
    view_cursor_vt_iso=None,
    parent_session_id=None,
    parent_chain_depth=0,
    auth_clock_iso=auth_clock_iso,
    clock=runtime_clock,                                 # store the same Callable for subsequent ops
    created_at_iso=auth_clock_iso,
)
```

**`Session.clock` field (W2.D).** Every `Session` carries `clock: Callable[[], datetime]` set to the same `runtime_clock` injected at server startup via `WSServer.serve(host, port, db, *, runtime_clock)`. Subsequent `emit_repl_op_audit` calls compute `recorded_at = session.clock().timestamp()` (canonical `AuditEntry.recorded_at` is `float`). Under replay, `runtime_clock` is substituted with the captured-clock function; `session.clock()` reproduces every wall-clock value byte-identically. Mirrors how Module 5 Txn captures wall-clock under replay.

`auth_clock_iso` comes from `runtime_clock()` (i.e. `:clock/now` under the audit handler in production) — replay-clock pins it. Both `(token_id, auth_clock_iso)` are captured in the auth-time `:repl/op` AuditEntry's `principal` slot; replay substitutes the captured clock value, regenerating the same `session_id`. **`uuid4()` is NOT used for `session_id`** — it would defeat replay-pinability. A unit test in Task D1 / D2 ("deterministic session_id under replay clock") asserts `session_id` equality across two runs that differ only in wall clock.

### 6.3 Validation

```python
def validate_token(db: DB, token_str: str, *, now: datetime) -> CapabilitySet | None:
    token_id = _token_id(token_str)              # single helper, W2 MINOR-7
    # the as_of(_now()) seam captures wall-clock at validation time;
    # replay substitutes the captured clock value.
    entity = db.as_of(now).entity(token_id)
    if not entity:
        return None
    if entity.get("repl/revoked-at") is not None:
        return None
    expires_iso = entity.get("repl/expires-at-iso")
    if expires_iso is not None and now >= datetime.fromisoformat(expires_iso):
        return None
    cap_set = CapabilitySet.from_canonical_json(entity["repl/cap-set"])
    return cap_set
```

Single `db.as_of(now).entity(token_id)` call. Latency is O(history-length-of-token-id) under the current `DBView.entity` projection — since each token is a separate `e`, history is bounded by 3-4 datoms (issue + optional revoke + optional expire-bump). `_now()` returns a `datetime` from `:clock/now` under the audit handler; replay-clock pins it.

### 6.4 Revocation

```python
def revoke_token(db: DB, token_id: str, *, now: datetime, by: str) -> DB:
    now_iso = now.isoformat()
    return db.transact([
        {"e": token_id, "a": "repl/revoked-at-iso", "v": now_iso, "valid_from": now_iso},
        {"e": token_id, "a": "repl/revoked-by",     "v": by,      "valid_from": now_iso},
    ], provenance={"source": "repl/revoke", "operator": by})
```

**Mid-op semantics (MINOR-3 pin).** Revocation takes effect on the **next** `validate_token` call; in-flight ops are not interrupted; this is documented operator behavior, not a security bug because tokens issue capabilities for whole sessions, not per-op. Active sessions on the revoked token continue until their next op attempt, which will fail with `-32006`. For mid-op cancellation, the operator must close the WS connection — which the server interprets as session teardown without further validation. Phase 3 NeSy 2027 may add per-op cap rechecks; v0.7.0a1 does not.

**Branch inheritance invariant (MINOR-6).** A branched session inherits the parent session's `cap_set` verbatim, including `expires_at`. If the parent's `expires_at` is past at branch time, the server rejects `repl/branch` with `-32006 session expired` BEFORE forking — preventing chain-branching from extending an expired token's lifetime. The branched session's effective expiry is therefore always ≤ the parent's. Mirror semantics also hold for revocation: revocation of the parent token's `cap_token_id` propagates to all branched-of-branched sessions on the next `validate_token` call against any of them.

### 6.5 Worked examples

**Read-only token (least-privilege).**

```python
caps = CapabilitySet(
    caps=frozenset({Capability(op="inspect", qualifier="read")}),

```

### 7.4 Minimum-viable UI behavior

- On load, `repl.js` reads `#token=<base64url>` from the URL fragment (NEVER from the query string — fragment doesn't hit server logs or HTTP referrers), opens a WS to the same origin, sends `repl/auth`. **Immediately after read, the UI strips the fragment from `window.location` AND calls `history.replaceState({}, '', '/')`** so the fragment is not retained in browser back-button history.
- On `auth` success, the UI enables the command input and begins rendering audit-tail notifications.
- Commands typed into the input are parsed by a small grammar (`COMMAND ARG_KEY=ARG_VAL ...`), translated to JSON-RPC, sent.
- Responses are pretty-printed canonical-JSON in the left pane; errors render in red with `error.message` + `error.data`.
- Audit-tail push notifications append a one-line summary to the right pane (timestamp + `op` + summary fields). Click to expand to full canonical-JSON.
- All substrate-originating string fields are rendered via `textContent`, never `innerHTML`. Pinned in code review + tested via XSS payload.

## 8. Reproducibility / determinism contract

### 8.1 What "deterministic" means for a REPL

The REPL is **non-deterministic in user input** (humans type at unpredictable times, in unpredictable orders) but **deterministic in substrate response**. For any fixed sequence of `(session_id, op_kind, args_canonical, view_cursor_at_op_time)` tuples, the substrate produces a byte-identical sequence of:

1. `:repl/op` AuditEntries (with chained `prev_hash`).
2. Underlying `db.transact` writes (for `edit` ops) — same as programmatic callers.
3. Read responses for `inspect` (modulo wall-clock-derived `recorded_at` on AuditEntries, which is captured via `:clock/now` effect and is replay-pinable).

### 8.2 Byte-identity claim (mirrors Stream A Prop-4 framing + Stream B Prop-6 framing)

> *Module 7 byte-identity contract.* For any REPL session trace `T = [(op_kind_i, args_canonical_i, view_cursor_i)]`, replaying `T` against the same initial DB state under a replay-clock handler reproduces a byte-identical sequence of `:repl/op` AuditEntries, byte-identical `db.transact` outputs for any `edit` ops, and byte-identical read responses for any `inspect` ops.

**Defense outline:**
- Wire vocabulary is **ISO-8601 strings** end-to-end (§5.0). `recorded_at` on the AuditEntry is the float `session.clock().timestamp()` (canonical `AuditEntry.recorded_at` slot is `float`); the same clock value is also captured as ISO into `principal.view_cursor_*` slots when relevant. Replay-clock pins both forms; no millisecond-int ambiguity.
- Each `:repl/op` AuditEntry's content hash is `_content_hash(_canonicalise_content(content))` (`src/persistence/effect/handlers/audit.py:308-354`). Inputs to that hash are exactly the canonical `AuditEntry` field set: `(prev_hash, op=":repl/op", args_hash, verdict, latency_ms, recorded_at, result_hash, error, policy_id, handler_chain, principal, run_id, parent, txn_commit?)`. **REPL-specific fields (`op_kind`, `view_cursor_*`) ride on `principal` (W2.A), so `verify_chain`'s recomputation against `entry.to_dict()` matches the helper-side hash exactly** — there is no flat-payload-vs-dataclass shape mismatch; chain-continuity holds for pure-REPL traces and mixed REPL+programmatic traces.
- **`session.clock` is the runtime_clock passed to `WSServer.serve(host, port, db, *, runtime_clock)`** (W2.D) — under replay this is the substrate's `:clock/now`-pinned function; `auth_clock_iso = runtime_clock().isoformat()` captured at auth time pins the seed; subsequent `session.clock()` calls produce wall-clock values that are themselves captured per-AuditEntry's `recorded_at`. Replay re-runs the whole sequence under the same captured-clock function and reproduces every `recorded_at` byte-identically.
- **`session_id` is deterministic in `(token_id, auth_clock_iso)`.** Both are captured in the auth-time `:repl/op` AuditEntry's `principal` slot; replay substitutes the captured clock value, regenerating the same `session_id`. No `uuid4()` non-determinism on the principal field.
- `args_hash` is `canonical_hash(args_canonical)` over canonical JSON; deterministic by construction.
- `db.transact` writes inside `edit` go through the same path Stream A's tests already prove byte-identical (`tests/integration/test_v0_6_plan_execution.py`). REPL-emitted AuditEntries are also persisted via `audit_entry_to_datom` + `db.transact` (W2.C), so they ride the same byte-identity proof.
- `inspect` reads are pure functions of `(view_cursor, store-state)`; no LLM, no random.
- `verify_chain(audit_entries_in_replay) == verify_chain(audit_entries_in_original) == True`. The chain extends; `prev_hash` continuity holds because `prev_hash = entries[-1].id if entries else None` is the construction rule on every `emit_repl_op_audit` call.

### 8.3 What the REPL CANNOT replay byte-identically

- Wall-clock latency (`AuditEntry.latency_ms`) is captured but not replayed; replay records its own latencies.
- The order of concurrent sessions is not deterministic without serialization. v0.7.0a1 does NOT make a multi-session-byte-identity claim — only single-session traces.

Filtering by `op` prefix is one `if event.op.startswith(filter_prefix):` line in v0.7.x; no point shipping a feature the audit-tail bandwidth doesn't justify yet.

### OQ-2 — `inspect` on Plan AST: raw datoms vs reified node tree

**Question.** When `inspect` is called with `kind="plan"` and a `plan_id`, should it return the raw `Datom` rows for the entity, or should it walk the Plan AST and return a reified node tree (tag, attrs, children)?

**Default for v0.7.0a1.** Raw datoms. The reified-tree renderer is a UI concern, and the UI in v0.7.0a1 is a console — pretty-printing canonical JSON is good enough for the operator workflow. Reified-tree rendering is OQ-3 deferred to v0.7.x once the UI evolves. Adding a server-side reify is a 5-line addition (`Node.from_datoms(datoms) → Node`); we just don't ship it in v0.7.0a1.

### OQ-3 — Plan AST tree renderer in UI

**Question.** Should the UI render Plan AST trees as collapsible JSON-tree views or as text dumps?

**Default for v0.7.0a1.** Text dump (canonical JSON pretty-print). The console UX is "operator types a command, gets a response"; a JSON-tree component adds ~200 LOC of vanilla JS that does not load-bear for the v1.0 paper. Defer.

### OQ-4 — Can `edit` synthesize Effect intents, or only datom writes?

**Question.** Should `repl/edit` allow operators to submit an Effect intent (e.g. "perform `:llm/call` with these args, audit-chain it") in addition to raw datom writes?

**Default for v0.7.0a1.** Datom writes only. Effect intent submission widens the threat surface dramatically (an operator could synthesize fake LLM responses into the audit chain) and the feature is not load-bearing for any v1.0 paper claim. Operators who need to inject test effects can use the substrate's programmatic API directly. Defer to v0.7.x with a separate `effect:perform` capability that is NOT granted by `edit:write`.

### OQ-5 — `branch` from a non-main current cursor

**Question.** What is the behavior when the operator is already on a branched session and calls `branch` again — does this fork from the branched DB (creating a chain of branches) or from main?

**Default for v0.7.0a1.** Forks from the current session's DB, whatever that is (main or branched). Chains of branches are allowed **up to `max_branch_depth` (default 16)**. Beyond that, server rejects `repl/branch` with the new error code `-32007 branch depth exceeded` (§ADR-9). The audit entry's `parent_session_id` plus the `result_summary.branched_db_id` plus the `parent_chain_depth` integer gives the chain its lineage. **No "merge a branch back to main" primitive in v0.7.0a1** — the substrate doesn't have one yet (`db.branch` returns a new isolated store; `db.fork()` referenced in the roadmap does not yet exist as a public API). Cross-branch reconciliation is Phase 3 NeSy 2027 scope. Documented in the `_ops.py` module docstring + the §5.4 worked example. **D5 ships 5 max-depth enforcement tests** (§9.2 test count).

### OQ-6 — Token expiry policy default

**Question.** Should default-issued tokens have an expiry, and if so what?

**Default for v0.7.0a1.** Yes — bootstrap CLI defaults to `--expires-in-days 30`. Env-var bootstrap respects whatever each token's `expires_in_days` JSON field says. Tokens with `expires_at = None` (no expiry; the canonical-JSON field is `expires_at_iso: null`) require an explicit `--no-expiry` flag and are flagged in the audit log via a `repl/no-expiry` provenance field. Justification: 30 days is short enough that operators rotate tokens regularly, long enough that day-to-day workflows aren't disrupted.

These are not blockers. The design ships under the listed defaults; revisions land in v0.7.x point releases.

## 12. Risks

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Capability bypass — a code path that calls an op without going through `validate_token` + cap-check | Low | substrate-claim-breaking | Single-entry-point invariant: all op handlers in `_ops.py` accept a `Session` (which carries `cap_set`); WS layer is the only thing that constructs `Session`s, and only after `validate_token` + cap-check. Unit-test pin: `test_session_cannot_be_constructed_without_validated_token.py`. |
| WS auth replay — attacker captures the auth handshake frame, replays from another connection | Medium (LAN), Low (loopback) | session hijack | Loopback-only default; LAN/public via reverse-proxy with TLS (documented). Session_id binding to the WS connection prevents cross-WS replay even on the same token. v0.7.x can add nonce-based handshake. |
| UI XSS in audit-tail rendering — a substrate value containing `<script>` tags renders as HTML | Medium | privilege escalation | `textContent`-only rule pinned in `repl.js` + tested via XSS-payload integration test (`test_repl_ui_xss_pinned.py` constructs a Datom with `v="<img src=x onerror=alert(1)>"`, drives the UI, asserts no script execution). |
| Deadlock with txn `dosync` — REPL holds an open WS during a long `dosync` block in the substrate | Low | hang | `db.transact` calls in `_ops.py` go through `loop.run_in_executor`; never block the asyncio loop. `dosync` is a substrate-internal coordination, not a REPL session state — REPL never holds a `Transaction` handle across awaits. |
| Drift between REPL `view_cursor` and store advancement — operator rewinds, store advances, operator's cursor now references a `tx_time_iso` that was "head" at rewind time but is now mid-history | Low | surprise, not corruption | Documented behavior: cursor is absolute, not relative-to-head. Reads at `tx_time_iso = T` always return the `as_of(datetime.fromisoformat(T))` view, regardless of how many tx have landed since. UI displays both cursor and current head when they differ. |
| Token leak via UI fragment URL — operator copies a `#token=...` URL into a chat | Medium | session hijack | URL fragment (`#token=...`) NOT query string — fragment is never sent in HTTP referrers, never logged in server access logs. UI strips fragment from `window.location` immediately after read AND calls `history.replaceState({}, '', '/')` so the fragment is not in browser back-button history (MINOR-7). Documented. v0.7.x can add a one-time-use redemption code that expires after first WS-auth. |
| `edit` accidental write — operator typo or copy-paste error commits unintended datoms | Medium | data integrity | Two-step propose-confirm flow (ADR-7): step 1 returns request body verbatim with `request_hash`; step 2 requires `confirm: true` AND matching `request_hash`. Server-side preview NOT computed (preview is the input not the output) — operator runs `inspect` ahead of `edit` to see what auto-retracts. Hostile clients with valid `edit:write` can still write; capability is the load-bearing gate, `confirm + request_hash` is the second factor against fat-finger ops. |
| Long-running `inspect` on `causal-history` over deep histories blocks the executor pool | Low | latency degradation | `max_depth` capped at 1 in v0.7.0a1 (single-level walk, matches `db.causal_history` v0.4 contract). Multi-level walks are an OQ. |
| `verify_chain` failure surfaces as `-32004` — operator sees substrate-corrupted error | Very low (substrate bug) | substrate trust | Treated as a substrate-level emergency: WS server logs the audit_id + the chain break, returns `-32004` to the operator. The chain-break event itself emits a `:repl/op` AuditEntry with `principal.op_kind="inspect"`, `verdict="error"`, and `error="audit chain integrity broken at <audit_id>"` — meta-recorded against the canonical substrate verdict set. Stream G's R4 review re-runs `verify_chain` over the full v1.0.0 audit log as a release-gate. |
| Audit-tail ring buffer overflow loses events for a slow client | Medium | UX degradation, not security | **Per-session ring buffer (256 entries each)** so slow subscribers don't starve others; total memory bounded at `N_sessions * 256 * sizeof(AuditEntry)`. Per-session FIFO eviction. On overflow, server emits a `repl/audit-event-overflow` notification with the dropped tx-window so client can backfill via `repl/inspect kind="audit-window"`. Buffer size configurable. (MINOR-9) |
| `aiohttp` lib version drift breaks auth handshake or static handler | Low | install break | Pin `aiohttp >= 3.9.0, < 4.0.0` in `pyproject.toml` optional `[repl]` extras (W1.G — switched from `websockets` library). Re-pin per release. |
| Cap-set forward-compat — v0.7.x adds a new `(op, qualifier)` and v0.7.0a1 tokens silently grant it | Low | privilege escalation across versions | Closed-set validation in `_caps.py`: unknown qualifiers raise `UnknownCapability` at `validate_token`. v0.7.x must explicitly bump the closed set; a v0.7.0a1 token does NOT grant a v0.7.x-introduced qualifier. Pinned via `test_caps_forward_compat_strict.py`. |
| Capability-revocation-mid-op — operator revokes a token while a session has an in-flight op | Low | UX surprise, not security | Revocation takes effect on the **next** `validate_token` call; in-flight ops are not interrupted; this is documented operator behavior, not a security bug because tokens issue capabilities for whole sessions, not per-op (MINOR-3). For mid-op cancellation, operator must close the WS connection (server treats close as session teardown). |
