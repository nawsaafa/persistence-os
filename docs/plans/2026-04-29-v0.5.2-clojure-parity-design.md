### N6 — `tx.alter` byte-identity Hypothesis property

**Status today** (v0.5.1 N4 closure at `df1a3ec` per `CHANGELOG-txn.md:14-15`):

The Hypothesis `@given` byte-identity property at `max_examples=200` exists in `tests/persistence/txn/test_replay_byte_identity_property.py`, but the registered strategy enumerates only single-shot `tx.assoc` transactions. The N4 closure note (`CHANGELOG-txn.md:14-15`) and the v0.5.0a1 rev-O entry (`CHANGELOG-txn.md:84-86`) both defer `tx.alter` and `tx.effect` coverage to v0.5.2 (*"Hypothesis property covers single-shot `assoc` transactions; `tx.alter` / `tx.effect` byte-identity coverage deferred to v0.5.2."*).

**Byte-identity scope (precise definition).** The existing property test deliberately filters out commit datoms in its `_run_one` helper (`tests/persistence/txn/test_replay_byte_identity_property.py:43-49`):

```python
return [
    (d.e, d.a, d.v, d.op, d.valid_from)
    for d in db.log()
    if not d.a.startswith("persistence.txn/")
]
```

Commit datoms carry per-run UUIDs (`commit_id = str(_uuid.uuid4())` at `transaction.py:315`) so they are intrinsically not byte-identical across runs. The clock is frozen via `clock=lambda: fixed_t` in the fixture so `valid_from` IS deterministic for user-write datoms. **N6 + N7 inherit this exact scope** — assertion is structural equality of the user-write log under frozen clock + filtered commit datoms. No claim is made about commit_id, started-at/committed-at provenance, or per-run UUIDs.

**Change:**

Extend the v0.5.1 strategy in `test_replay_byte_identity_property.py` to draw `tx.alter` operations alongside `tx.assoc`:

```python
# tests/persistence/txn/test_replay_byte_identity_property.py
@st.composite
def _alter_op(draw):
    return ("alter", draw(_eid_st()), draw(_alter_fn_id_st()))

@st.composite
def _assoc_op(draw):
    return ("assoc", draw(_eid_st()), draw(_immutable_value_st()))

_op_strategy = st.one_of(_assoc_op(), _alter_op())
```

`_alter_fn_id_st()` draws *string identifiers* from a curated table of pure module-level functions (`{"int_inc": lambda v: v + 1, "pmap_set": lambda v: v.set("k", 1), ...}`). The strategy never synthesises closures — it draws keys, the test body resolves keys to module-level fns. This sidesteps the entire "Hypothesis can't shrink across closure boundaries" failure mode.

**Why curated `alter_fn` is correct (not a hedge):**

1. The byte-identity invariant is over the *filtered structural projection* defined in the "Byte-identity scope" block above (user-write log + projected audit-chain tuples; commit datoms and per-run UUIDs excluded), not over the `alter_fn` itself. The fn runs inside the body, sees the same snapshot, produces the same value across runs. Any curated set whose values are deterministic given snapshot + input suffices.
2. Free-closure synthesis would force serialisation of the closure into the strategy state and add a category of "the closure isn't preserved across Hypothesis shrink steps" failures unrelated to STM byte-identity. Curated table-of-fn-ids cleanly avoids this.

**Acceptance:** Hypothesis property runs at `max_examples=200`, finds zero divergence (under the filtered structural-comparison scope above) between two `dosync` runs of the same body when both `assoc` and `alter` ops are interleaved. CI gate: `pytest tests/persistence/txn/test_replay_byte_identity_property.py -k alter` passes.

### N7 — `tx.effect` byte-identity Hypothesis property

**Status today:** same N4 v0.5.1 deferral note. Effect intents queued via `tx.effect` are committed as part of the commit datom's `:persistence.txn/intent-log` provenance (closed in N1 at `df1a3ec`). The byte-identity property must therefore also cover the case where the body emits effect intents.

**Change:**

Add `("effect", op, kwargs)` as a third op shape in the strategy. @st.composite
def _effect_op(draw):

    kwargs = draw(st.dictionaries(
        keys=st.text(_alpha, min_size=1, max_size=8),

    return ("effect", op, kwargs)

### N8 — `Ref.spec_attr` EDN-keyword regex tightening

**Status today** (`src/persistence/txn/ref.py:18`):

```python
_SPEC_ATTR_PATTERN = re.compile(r"[A-Za-z0-9][A-Za-z0-9./_-]*")
```

This was the v0.5.1 N3 shipping cut — accepts any alphanum + `./_-` with leading alphanum (`fullmatch`-anchored at the call site, `ref.py:55-57`). It rejects whitespace, leading colon, empty string, and (incidentally) all leading non-alphanum chars including `-`, `.`, `+`, `:`, `#`, `/` (the three motivating bugs at N3 design time were the first three; the rest are a happy by-product of the leading `[A-Za-z0-9]` anchor). It does NOT reject:

- `0/foo` — leading digit on a namespace segment (EDN spec: name must start with non-digit symbol char; current regex's leading `[A-Za-z0-9]` lets digits through)
- `foo/bar/baz` — multiple `/` (EDN spec: at most one `/` separating namespace from name; current regex's body class `[A-Za-z0-9./_-]` includes `/`)
- `foo/` — trailing empty segment after `/` (same body-class issue)

It also REJECTS some shapes that EDN actually permits, like `-foo`, `.foo`, `+bar` (valid EDN keyword names with leading symbol-char). v0.5.2 N8 simultaneously **gains** the missing rejections (multi-`/`, leading-digit, trailing-empty-segment) AND **loosens** the over-rejections to admit valid EDN-shaped names.

**Change:**

Tighten to a Datomic-compatible EDN keyword-name grammar (per the EDN spec at https://github.com/edn-format/edn — Symbols + Keywords sections). EDN keyword names follow symbol rules after the leading `:` is stripped at the Datom-store boundary. The spec has one tricky rule: **if the first char of a segment is `-`, `+`, or `.`, the second char (if any) must be non-numeric** — this disambiguates the symbol path from the number path in the bare-symbol case. Keywords inherit the rule even though the leading `:` already disambiguates, because EDN names round-trip through symbol parsers.

```python
# Standard segment head: alpha or symbol char (NOT the special trio -,+,.)
_STANDARD_HEAD = r"[A-Za-z*!_?$%&=<>]"
# Body: any alphanumeric + symbol chars (digits OK)
_BODY = r"[A-Za-z0-9*+!\-_?$%&=<>.#:]"
# A segment is either a standard head + body* OR a special-leader (-,+,.)
# with a negative lookahead asserting the next char (if any) is non-digit,
# then body*. The negative lookahead `(?![0-9])` succeeds at end-of-string
# (no next char, vacuously not-digit) so single-char `-`/`+`/`.` are valid.
_SEG = (
    rf"(?:{_STANDARD_HEAD}{_BODY}*"
    rf"|[\-+.](?![0-9]){_BODY}*)"
)
_SPEC_ATTR_PATTERN = re.compile(rf"^{_SEG}(?:/{_SEG})?$")
```

Rejections gained (each motivated by a real failure mode):

| Reject | Reason |
|---|---|
| `0/foo`, `42name` | Leading digit on a segment — EDN spec forbids; Clojure's reader would parse as a number-prefix |
| `foo/bar/baz` | Multiple `/` — EDN spec allows at most one (namespace separator) |
| `/foo`, `foo/` | Empty namespace or empty name segment |
| `-1`, `+42`, `.5` | Leading `-`/`+`/`.` followed by digit — EDN second-char rule (would parse as a number) |
| (still rejected from v0.5.1 N3) `:foo`, `#foo`, `foo bar`, `` (empty) | Leading sigils, whitespace, empty |

Newly **permitted** (loosening relative to v0.5.1 N3):

| Permit | Justification |
|---|---|
| `foo123`, `count42`, `version2` | Trailing digits — common Datomic convention |
| `-foo`, `+bar`, `.baz` | Leading symbol-chars where the second char is non-digit (EDN-valid; current v0.5.1 regex over-rejects) |
| `-`, `+`, `.` (single chars) | EDN-valid as bare symbols (vacuously pass the second-char rule) |
| `app.user/email`, `db/cas` | Standard namespaced-keyword shape (already worked in v0.5.1) |
| `foo:bar` | Internal `:` — Datomic-style nested namespacing seen in user fact corpora; the Datom store strips only the leading `:` |

**Backwards compatibility:**

Every existing Ref in v0.5.0a1 and v0.5.1 use sites uses `spec_attr="value"` (the default) or one of `"count"`, `"balance"`, `"score"`, `"text"`, `"items"` (test fixtures). All of these match the tighter regex. Pre-flight: `grep -rn "spec_attr=" src/ tests/` produces 0 user-defined non-default values that would now reject. Public API surface unchanged — only ValueError messages are more specific.

**Acceptance:** `Ref(eid="e1", db_id="d1", spec_attr="0/foo")` raises ValueError with message naming the EDN-spec rule violated. Existing 931 + 7 xfailed test suite stays green.

### F1 — Atoms (single-cell CAS over datom-refs)

**What lands:**

```python
counter_atom = db.atom(":metrics/login-count", initial=0)
counter_atom.deref()                          # → 0
counter_atom.swap(lambda v: v + 1)            # → 1, writer-lock-protected CAS retry on contention
counter_atom.compare_and_set(1, 2)            # → True
counter_atom.reset(0)                         # → 0
```

**Surface:**

- `Atom` — frozen dataclass: `(eid: str, db_id: str)`. Distinct type from `Ref` to keep `swap!` discoverable; carries no `spec_attr` (atoms write under fixed `:value` attribute).
- `db.atom(eid, *, initial=...)` — constructor; raises if eid already exists with `:value` datom.
- `Atom.deref()` — single-datom log read; no transaction context required.
- `Atom.swap(fn)` — read-modify-write loop; up to `max_retries=256` (mirrors `dosync`); raises `AtomCASExhausted`.
- `Atom.compare_and_set(old, new)` — single-shot guard; returns `bool`.
- `Atom.reset(value)` — unconditional write.

**Why it's not just a `Ref` shortcut:**

Refs require entering a `dosync` body for any read or write. Atoms are the "I want CAS without a transaction" case — the ergonomic fast-path Clojure `Atom.java` ships.

**CAS mechanism (substrate-correct).** `db.transact_batch` does NOT carry a `:if` precondition shape today; constructing one would expand `fact.DB`'s public surface beyond v0.5.2's charter. Atoms therefore implement CAS with the primitive that already exists — the Store writer-lock — using the same idiom v0.5.1's commit retry uses (`transaction.py` retry loop):

```python
def swap(self, fn):
    for attempt in range(self._max_retries):
        with self._db.store._lock:                    # acquire writer-lock
            current = self._read_latest_value()        # tip of log
            new_value = fn(current)
            # Re-read inside the lock to confirm no writer interleaved
            # between read and the transact() call (lock guarantees this,
            # but the explicit re-read makes the intent grep-able).
            self._db.transact([{
                "e": self._eid, "a": self._spec_attr, "v": new_value,
                "op": ":assert", "valid_from": self._db._clock(),
            }])
            return new_value
    raise AtomCASExhausted(...)
```

Atoms ride the same Store writer-lock that Refs do — the difference is purely API ergonomics (no `dosync` body), not concurrency. **Public substrate change required:** none. (The reach into `db.store._lock` mirrors how `_commit_attempt` at `src/persistence/txn/transaction.py:325` already takes the lock for its check+transact gate; both live inside the txn module and are not new public API.)

**Intentional Clojure-parity deviation — Atom-in-dosync prohibition.** Clojure permits atoms to be `swap!`'d inside a `dosync` body (the swap is not transactional and runs immediately). v0.5.2 instead **raises** `AtomInDosyncProhibited` if an atom op is invoked under the `_in_dosync` ContextVar. Rationale: persistence-os's `dosync` body is the intent-queue + retry domain (per `EffectInIoBlock`); allowing atom CAS inside would punch a non-replayable hole in the audit chain (atom writes wouldn't appear in commit-id provenance). This is a deliberate substrate choice, NOT an oversight — flagged here so the deviation isn't silently inherited as Clojure-faithful.

**No new datom attributes.** Atoms write under `:value` like single-spec Refs. Audit-chain pinning via the existing `:effect/source` field — value `:txn/atom-cas` distinguishes from `:txn/ref-write`.

**Acceptance:** 5+ tests at `tests/persistence/txn/test_atoms.py` covering swap-success, swap-retry-under-contention, compare_and_set true/false paths, reset, and atom-inside-dosync prohibition.

### F2 — `tx.ensure` (read-set padding)

**What lands:**

```python
with db.dosync() as tx:
    bal_a = tx.deref(account_a_ref)        # in read_set
    tx.ensure(account_b_ref)               # in ensure_set; not deref'd, but conflict-checked
    tx.assoc(account_a_ref, bal_a - 100)
```

**Why minimal:** `Transaction` already maintains `read_set`. `tx.ensure` extends it with `ensure_set: set[Ref]` and the conflict-detection pass at commit unions both sets. ~30 lines of code in `transaction.py`, ~30 lines of test coverage, plus the entry in `__init__.py`.

**Surface:**

- `Transaction.ensure(ref: Ref) -> Any` — adds ref to `ensure_set` AND returns the snapshot value (matches Clojure `LockingTransaction.ensure(ref)` which returns the deref'd value so `ensure` is a strict superset of `deref` for ergonomic chained reads). Internally calls `tx.deref(ref)` then adds the ref to `ensure_set` so any subsequent body code that ignores the return value still gets the conflict-padding behavior.
- Conflict detection at commit reads `read_set | ensure_set` against `db.store.since(t_start)` — if any datom on either set has `tx_time > t_start`, retry.
- Provenance: ensure_set list emitted as `:persistence.txn/ensure-set` on the commit datom (mirrors v0.5.1 N1 `:persistence.txn/read-set`).

**Boundary spec:** `:persistence.txn/ensure-set` registered as `seq_of(_eid_str_spec)`.

**Acceptance:** classical bank-transfer write-skew test (Alice and Bob each transfer half their joint balance to a third party in parallel; without `ensure`, both succeed and overdraw; with `ensure` on the partner's account, one retries and overdraws is prevented). Plus 3 unit tests covering ensure-set conflict detection.

### F3 — `tx.commute` (commutative writes + curated registry)

**What lands:**

```python
counter_ref = db.new_ref(initial=0, spec_attr="count")
db.new_ref(initial=pmap(), spec_attr="tags")

with db.dosync() as tx:
    new_count = tx.commute(counter_ref, "inc-by", 1)        # eager-at-body, returns optimistic value
    tx.commute(tags_ref, "set-union", {"alpha", "beta"})    # commutative; reorders safe
```

**Two-phase semantic** (matches Clojure `LockingTransaction.doCommute:467-491`):

1. **At body call site:** `tx.commute(ref, fn_id, *args)` looks up `fn_id` in the curated registry, applies fn against the *optimistic* committed-or-locally-written value, returns the new value (so subsequent body reads see the optimistic write). Adds an entry `(ref, fn_id, args)` to `commute_log`.
2. **At commit:** for each entry, re-read the ref's *latest committed* value (after taking the writer-lock), re-apply fn, and write the result. The result of step 1 is *discarded* — only step 2's value reaches the log.

**Why the eager-then-reapply dance is correct:** `read_set` is NOT extended on `tx.commute` — that's the entire point of `commute` vs `alter`. Two parallel transactions calling `tx.commute(counter, "inc-by", 1)` both succeed with no retry and produce a final counter incremented by 2. (`alter` would force one to retry.) Soundness depends on `fn` being commutative: `f(f(v, a), b) == f(f(v, b), a)` for all `v, a, b`. The substrate does NOT verify this (matches Clojure) — the registry is curated to functions known to be commutative.

**Curated registry (initial cut).** Three of the four are unconditionally commutative; one ships with documented caveats:

| `fn_id` | Commutativity | Implementation |
|---|---|---|
| `"inc-by"` | Unconditionally commutative | `lambda v, n: v + n` |
| `"sum-into"` | Unconditionally commutative | `lambda v, n: v + n` |
| `"set-union"` | Unconditionally commutative | `lambda v, items: v \| frozenset(items)` |
| `"dict-merge-shallow"` | **Commutative ONLY on disjoint key sets**; on overlap, deterministic last-write-wins per commit-order. Caller must guarantee disjoint keys OR explicitly accept the LWW commit-order resolution. Implemented as `lambda v, m: v.update(m)`. The test suite asserts both branches (commutative on disjoint keys, deterministic-but-not-commutative on overlap). |

**Why dict-merge-shallow ships despite the caveat.** Dropping it would force callers to fall back to `tx.alter` (which forces retries under any concurrent write), defeating the whole point of `commute`. The disjoint-key case is the dominant real workload (e.g., Adaptive Trader v2 signal aggregator: each thread writes a distinct signal-id key into a shared signals map). The non-disjoint case is opt-in and explicitly documented at the registry table + at the `register_commute` docstring.

User-defined commute fns live behind `register_commute(fn_id, fn)` which raises in production mode and is unlocked only by the test sentinel env var (mirrors `register_coercion` at `src/persistence/plan/_coerce.py:103-152`). This keeps cross-host determinism: two hosts replaying a `:commute` log entry must agree on the fn lookup.

**Intra-txn semantics (specified to remove ambiguity).**

- **Multiple `tx.commute` calls on same ref**: composed in body-order; eager value at body call N is `fn(eager_value_at_N-1, *args)`. At commit, all entries replay in body-order against the latest committed value.
- **`tx.commute` followed by `tx.assoc`/`tx.alter` on same ref**: the explicit write WINS — commute log entries on that ref are dropped at commit; the explicit write is the single fact emitted. (Matches Clojure: `set` after `commute` overrides.)
- **`tx.assoc`/`tx.alter` followed by `tx.commute` on same ref**: both apply; the body-eager value seen by the commute is the just-set value; at commit, the explicit write fact emits FIRST, then the commute reapplies against the explicit-write's value (which equals the latest committed since they're committed atomically).
- **`tx.deref` after `tx.commute` on same ref**: returns the optimistic body-eager value (the most recent commute application); does NOT add the ref to `read_set` more than once.

**Surface:**

- `Transaction.commute(ref: Ref, fn_id: str, *args) -> Any` — public API.
- `_commute_registry: dict[str, Callable]` in `txn/_commute.py` — module-level dict, frozen at import time, runtime mutation gated by `PERSISTENCE_TXN_ALLOW_RUNTIME_REGISTRATION` env var.
- `register_commute(fn_id, fn, *, replace=False)` — test-only registration helper (raises `RuntimeError` in production).
- Boundary spec: `:persistence.txn/commute-log` — `seq_of(keys({":ref": _eid_str_spec, ":fn-id": str_, ":args": _edn_value_spec}))`.

**Provenance:** commute-log emitted as `:persistence.txn/commute-log` on the commit datom. Replay reads commute-log, looks up each fn-id in the registry, re-applies fn against the snapshot value at commit time → byte-identical commit value.

**Acceptance:**

- Parallel-commute test: two threads each call `tx.commute(counter, "inc-by", 1)` 100 times against same counter; final value = 200, zero retries (non-deterministic order, but commutative result).
- Replay byte-identity test: a commute-log committed at t1, replayed against the same store at t2, produces structurally-identical user-write log + audit-chain projection (under the same filtered scope as N6/N7 — commit datoms and per-run UUIDs excluded; clock frozen).
- Registry-locked test: `register_commute("evil", lambda v, x: v - x)` raises `RuntimeError` without sentinel env var.
- 4 fn-id test: each curated fn-id has a 3-test slice covering identity, associativity-witness, and disjoint-key edge cases.
