## Decision 1 — `step_id` resolution

`step_id` IS `Node.id` (the 32-hex content-address from `_ast.ID_HEX_WIDTH`).

**Rationale:** the AST already content-addresses every node. Inventing a
positional path string (e.g. `"0.1.2"`) would duplicate state and would
collide on its own with re-rooted subtrees produced by edit ops (every
edit changes the parent's id, so every existing path gets invalidated
after one edit anyway). Using `Node.id` directly:

- Aligns with `:plan/edit` audit datom keys: `before_op_hash` /
  `after_op_hash` are already `Node.id`s, so `step_id` keeping the same
  type is one fewer abstraction.
- Aligns with ADR-6: "every edit is a `:plan/edit` datom with before /
  after op-hash + step-id" — same content-address-based identity end
  to end.
- Aligns with § 3.7 byte-identity replay: walking the audit log
  reconstructs the edit history because `step_id` + `before_op_hash`
  pairs are deterministic across replays.

**Caveat (documented in module docstring + propagated to user-facing
docstring on each public fn):** if a Plan AST contains two
content-identical subtrees (two `(:llm-call :prompt "hi")` siblings,
say), they share a `Node.id`. `edit_step` / `insert_step_after` /
`insert_step_before` / `delete_step` then target the **first
occurrence in pre-order DFS walk order**. The identical-subtree case is
rare in practice (homoiconic Plans tend to differentiate via attrs),
but the contract is explicit. A separate `find_step_by_path` helper
for path-based addressing is out of scope for #140 — log as
follow-up if Phase 2 dogfood surfaces a need.

## Decision 2 — downstream-execution check posture for `delete_step`

**Posture:** TODO (partial / documented gap).

**Rationale:** the design § 4.1 line 285 says `delete_step` is "only
allowed if no downstream step has executed". Execution state lives in
the Transaction object passed to the dosync body — but the Transaction
exposes `read_set` / `write_set` / `effect_intent_log`, NOT a "completed
step ids" set. Plan execution (`persistence.plan._execute`) returns an
`ExecutionResult` after the dosync exits; mid-flight execution state
during a dosync is not currently tracked at the step level.

Implementing the check honestly would require adding a
`completed_step_ids: set[str]` field on `Transaction` and threading it
through the executor, which crosses module boundaries (`txn` ↔ `plan`)
in a way that should land as its own substrate change with its own
ADR. That is **scope shift** for #140.

**Therefore:** `delete_step` ships with the simpler "always allowed
inside dosync" semantics in v0.8.5a1 / Phase 2.0a; the downstream-
execution check is gated behind a substrate-backlog item:

> **Substrate-backlog #200 (proposed):** thread `Transaction.completed_step_ids`
> through `persistence.plan._execute` so `Plan.delete_step` can falsify
> the "no downstream step has executed" precondition. Until then,
> callers MUST validate downstream state at the call-site if they care.

A `# TODO #140 follow-up: downstream-execution check (substrate-backlog
#200)` comment marks the spot in `_edit.py::delete_step`. The §13
changelog entry calls it out explicitly.

## Decision 3 — audit-emission integration point

`:plan/edit` rides on the existing `tx.effect()` intent queue. Each
`edit_step` / `insert_step_*` / `delete_step` call requires an explicit
`tx: Transaction` argument (mirrors `tx.assoc` ergonomics; no global
context-fetching). Inside the function body:

1. Validate we are in a dosync (via `is_in_dosync()` ContextVar). If
   not, raise `PlanEditOutsideDosync`.
2. Locate the step by `Node.id` walk; raise `StepIdNotFound` if absent.
3. Build the new plan tree functionally (no mutation; relies on `Node`'s
   `frozen=True`).
4. Call `tx.effect(":plan/edit", plan_id=..., step_id=..., before_op_hash=..., after_op_hash=...)`
   — no explicit `txn_id` kwarg because the existing intent-replay
   path (`_replay_effect_intents`) injects `_txn_commit` (the commit_id)
   on the way to the effect runtime, so the audit handler links the
   `:plan/edit` request datom into the same Merkle chain as every other
   `:effect/*` datom on this commit (matches ADR-6 + § 3.7 row).
5. Return the new plan `Node`.

**No new Merkle-chain code is needed** — the audit handler at
`src/persistence/effect/handlers/audit.py` already chains every effect
intent into the `:audit/entry` chain via `prev_hash` (PG-W1 confirmed
this still works under cross-process Merkle ordering). `:plan/edit`
inherits that chain by being a regular effect intent.
