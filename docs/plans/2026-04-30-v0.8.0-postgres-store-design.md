## 1. Summary

Persistence OS today has two production-style fact stores: `InMemoryStore` (tests + CLI demo, no durability) and `SQLiteStore` (single-process durable; `BEGIN IMMEDIATE` writer-lock semantics). Module 5 Txn (`v0.5.2`) ships Clojure-parity STM (Refs / `dosync` / Atoms / `tx.commute` / `tx.ensure`) but the v0.5.0a1 design doc § "Design constraints" explicitly defers **multi-process** STM to v0.6.x: *"single-process only — multi-process via Postgres SERIALIZABLE deferred."* Today the Txn module's `_commit_attempt` (`src/persistence/txn/transaction.py:572-630`) holds `tx.db.store._lock` across the conflict-check + transact-batch core, which is correct for one OS process but is a no-op across processes. Multi-process correctness needs a **shared isolation barrier** that lives in the storage layer, not in the Python lock.

This design ships `PostgresStore` — a third backend that implements the same `Store` Protocol, runs under Postgres SERIALIZABLE isolation, and preserves byte-identical replay across processes. The store is opted into via the URI scheme `postgres://...` per Adapter SDK ADR-9; existing in-process consumers using `Substrate.open(store="memory")` or `Substrate.open(store="sqlite:///...")` are not affected.

This design ships three artifacts:

1. **`persistence.fact.postgres_store.PostgresStore`** — a new backend in the `[postgres]` optional-dependency extra (psycopg 3 + psycopg-pool); implements the full `Store` Protocol; runs all writes under `ISOLATION LEVEL SERIALIZABLE`; surfaces 40001 retry semantics through the existing `Transaction._run` retry loop without changing public API.
2. **`src/persistence/fact/migrations/0001_datom_log.postgres.sql`** — the Postgres-flavoured schema (TIMESTAMPTZ, JSONB, BIGINT/BIGSERIAL, partial+expression indexes, a single `audit_chain_lock` row for `prev_audit_hash` ordering). Sibling to the existing `0001_datom_log.sql` which becomes the SQLite-flavoured copy. Migration runner picks the right file by URI scheme.
3. **`tests/persistence/fact/test_postgres_store_*` + cross-process Hypothesis property** — the new byte-identity test mirrors v0.5.2 N6/N7 at `max_examples=200` and adds a cross-process variant: spawn N child processes via `multiprocessing.Process`, each writes a strategy-drawn op-sequence concurrently, parent reads and projects to the same filtered tuple list as N6/N7 — must equal the single-process baseline. Replay byte-identity is the falsifiable invariant.

**Not** in this stream: Citus sharding, multi-region read replicas, pgvector-backed `recall`, audit-tail logical-replication streaming. All deferred — see § 9.

**Ships:** ~700 LOC + ~250 tests + Postgres migration SQL. **5 days** (parallel to SDK impl after SDK1).

## 2. Strategic frame

### Why this stream ships as Phase 1 stream #2 (not later)

Three reasons, in order of weight:

1. **It is the production deployment surface.** Phase 2 ships `persistence-coder` as a commercial CLI agent. Any commercial customer who cares about durability + multi-process scaling will run it on Postgres, not SQLite. Without `PostgresStore` we ship a Phase 2 product that cannot be deployed past one operator's laptop. The Adapter SDK already commits to URI dispatch for `postgres://...` (ADR-9); the missing piece is the actual implementation.
2. **It is the load-bearing closure for the v0.5.0a1 deferral.** v0.5.0a1 carved Module 5 Txn deliberately narrow: single-process STM with the multi-process map onto Postgres SERIALIZABLE explicitly punted to v0.6.x. v0.5.1 + v0.5.2 then absorbed the entire window for Clojure-parity surface (Refs / `dosync` / Atoms / `tx.commute` / `tx.ensure`) without revisiting the multi-process deferral. v0.8 is where it gets paid down — and the Clojure-parity primitives need to stay correct across processes, not just within one.
3. **It de-risks every later stream's storage assumptions.** Plan Edit API (#140), `:code` sandbox (#141), and `fold()` executor (#145) all write datoms that go through the Store. If we discover only at v1.0 that one of those streams breaks under SERIALIZABLE (e.g. an interceptor that writes mid-transaction in a way that triggers 40001), the fix lands deep in product work. Doing PostgresStore in Phase 1 surfaces those bugs while the substrate-side ARIS gates are still active.

## 4. Schema design

### 4.1 Datom log table

The Postgres schema mirrors the SQLite schema's column shape so the codec (`_encode` / `_decode`) stays cross-backend identical. Type promotions are mechanical: `INTEGER` → `BIGINT`, `TEXT` (ISO-8601 timestamp) → `TIMESTAMPTZ`, `TEXT` (JSON) → `JSONB`. The 8-tuple semantics + provenance + `invalidated_by` pointer are unchanged.

```sql
CREATE TABLE datom_log (
    seq             BIGSERIAL    PRIMARY KEY,
    e               TEXT         NOT NULL,
    a               TEXT         NOT NULL,
    v               JSONB        NOT NULL,
    tx              BIGINT       NOT NULL,
    tx_time         TIMESTAMPTZ  NOT NULL,
    valid_from      TIMESTAMPTZ  NOT NULL,
    valid_to        TIMESTAMPTZ,
    op              TEXT         NOT NULL CHECK (op IN ('assert', 'retract')),
    provenance      JSONB        NOT NULL,
    invalidated_by  BIGINT
);

-- Five primary indexes (mirror SQLite); JSONB v + provenance use BTREE on the
-- canonical-string projection (NOT GIN) so equality lookups stay byte-identical
-- across backends. GIN indexes are added in § 4.4 as opt-in for value-set queries.
CREATE INDEX idx_datom_eavt ON datom_log (e, a, valid_from, tx);
CREATE INDEX idx_datom_aevt ON datom_log (a, e, valid_from, tx);
CREATE INDEX idx_datom_avet ON datom_log (a, (v::text), e, tx);  -- canonical-string
CREATE INDEX idx_datom_vaet ON datom_log ((v::text), a, e, tx);  -- canonical-string
CREATE INDEX idx_datom_vte  ON datom_log (valid_from, valid_to, e);

-- LOG — tx_time ordered, drives since(t) and replication; (tx_time, tx) is
-- the strict-monotonic projection for the MVCC primitive in § 6.
CREATE INDEX idx_datom_log_txtime ON datom_log (tx_time, tx);
```

**Single table vs split (datoms / audit / commits)?** Single. ADR-1 below pins this. Three reasons. (a) The audit chain is encoded as datoms with `a IN ('audit/op-...', 'persistence.txn/commit-id', 'mcts/op-...', 'repl/op-...', 'mcp/op-...')` — splitting them out would force every replay path to do a UNION ALL across tables, with a hand-maintained ordering invariant on top; staying single-table makes the existing `db.log()` projection identical across backends and keeps the EAVT/AEVT indexes useful for audit reads. (b) The commit-datom shape (`a = 'persistence.txn/commit-id'`) is the same shape as a user-write datom; v0.5.2 N6/N7 explicitly filter commit datoms via `not d.a.startswith("persistence.txn/")` at the comparator boundary (`tests/persistence/txn/test_replay_byte_identity_property.py:43-49`), and that filter is byte-identical across backends only if the rows live in the same table and order. (c) Splitting tables makes RLS policies (§ 7) harder to author — a single `datom_log` lets the tenant-id policy live on one row-shape.

### 4.2 Audit chain ordering primitive

Multi-process audit chain continuity requires that `prev_audit_hash` always points to the most-recent committed audit datom *as observed by the writer that just committed*. A single `audit_chain_lock` table (with exactly one row) plus `SELECT ... FOR UPDATE` is the ordering primitive.

```sql
CREATE TABLE audit_chain_lock (
    id          INTEGER PRIMARY KEY CHECK (id = 1),  -- exactly one row
    head_hash   TEXT,                                 -- last committed audit hash
    head_tx     BIGINT,                               -- tx of the last audit datom
    head_seq    BIGINT                                -- seq of the last audit datom
);
INSERT INTO audit_chain_lock (id, head_hash, head_tx, head_seq)
    VALUES (1, NULL, 0, 0);
```

The Txn-side commit attempt that contains audit datoms takes `SELECT * FROM audit_chain_lock WHERE id = 1 FOR UPDATE` BEFORE building any audit datom that needs `prev_audit_hash`. The lock row is held until the surrounding SERIALIZABLE transaction commits, so two concurrent writers hash-chain in commit-order, not write-order. ADR-3 pins this.

### 4.3 Tx-id allocation

Postgres has two natural primitives: `BIGSERIAL` (a sequence), or `MAX(tx) + 1` under the SERIALIZABLE barrier. v0.8 uses **the latter**, mirroring SQLite. ADR-4 pins this. Rationale: (a) replay byte-identity demands deterministic tx-ids per op-sequence; sequences are not deterministic across crash-recovery (a sequence advance can outlive a rolled-back transaction, leaving gaps that a single-process backend would not produce); (b) SERIALIZABLE already gives us the read-committed-write barrier `MAX(tx) + 1` needs; (c) `BIGSERIAL` adds nothing the SERIALIZABLE barrier doesn't already give. The cost is one `SELECT COALESCE(MAX(tx), 0) + 1 FROM datom_log` per `allocate_and_append`, which is microseconds with the `(tx_time, tx)` index pinned.

The `seq BIGSERIAL` column does keep insertion-order for `since(tx_time)` and `all_datoms()` reads — it is the *physical* row order, not the *logical* tx-id. Gaps in `seq` after a rollback are fine because all reads order by `seq` and skip nothing.

### 4.4 Indexes — canonical-string vs GIN

Two index strategies for JSONB `v` / `provenance`:

- **BTREE on `(v::text)`** — matches SQLite's `(v)` BTREE, equality lookups are byte-identical across backends. Used for AVET/VAET/AEVT.
- **GIN on `provenance`** — added in v0.8 as an *opt-in* index for `:audit/parent-provenance-hash` graph traversal queries. NOT used by core Store reads.

ADR-5 pins the canonical-string-BTREE choice for cross-backend identity. GIN indexes are opt-in (a separate `0002_optional_gin_indexes.postgres.sql` migration the operator runs only if their workload needs JSONB containment queries).

## 5. SERIALIZABLE isolation — Txn integration

### 5.1 Connection isolation

Every connection in the pool is opened with `SET default_transaction_isolation = 'serializable'`. Every `_commit_attempt` runs inside a `BEGIN ISOLATION LEVEL SERIALIZABLE` block. Reads-only paths (`all_datoms`, `since`, `next_tx`) run in the same SERIALIZABLE isolation but at `READ ONLY` mode — Postgres's predicate-locking gives us the read consistency we need for `any_datoms_since(t)` to be sound across processes, and `READ ONLY` allows the planner to skip SSI bookkeeping for reads that cannot conflict.

### 5.2 dosync / tx.alter / tx.commute / tx.ensure → SQL

The v0.5.2 conflict-detection union (`transaction.py:603-609`) is:

```python
touched = (
    {r.eid for r in tx.read_set}
    | {r.eid for r in tx.write_set}
    | {r.eid for r in tx.ensure_set}
)
if any_datoms_since(tx.db, tx.t_start, touched):
    return False
```

Mapping this onto Postgres SERIALIZABLE has two layers:

**Layer 1 (correctness)** — Postgres SSI (Serializable Snapshot Isolation) detects rw-dependency conflicts automatically. As soon as the transaction issues a `SELECT ... FROM datom_log WHERE e = ANY($touched)` inside the SERIALIZABLE transaction, Postgres places SIReadLock predicate locks on the matching rows + index entries; any concurrent writer that would have invalidated the read will cause one of the two transactions to fail at commit with SQLSTATE `40001` (`SerializationFailure`). The existing v0.5.2 conflict-detection union becomes a single Postgres SELECT — Postgres does the conflict detection.

**Layer 2 (preserve our explicit semantics)** — we keep the explicit `any_datoms_since(t, touched)` check too. It is redundant with SSI for the common case but it is *cheaper* (no predicate locks) and it gives us our existing deterministic conflict-rejection path. The explicit check runs first; if it says "conflict", we return `False` and let the Txn-side retry loop spin. If it says "no conflict", SSI may still raise 40001 at commit — we catch `psycopg.errors.SerializationFailure` and treat it as a conflict (return `False`) so the same retry loop handles both sources of conflict.

```python
# transaction.py — minimal hook (~8 lines added)
def _commit_attempt(tx: "Transaction") -> bool:
    ...
    try:
        with tx.db.store._lock:  # PostgresStore._lock holds the connection out
            ...                  # explicit any_datoms_since + transact_batch
        return True
    except SerializationFailure:  # NEW: psycopg 40001
        return False              # retry loop in _run picks this up unchanged
```

This is the **only** Txn-side change. The retry loop, the `read_set | ensure_set | write_set` union, the conflict-rejection contract — all unchanged.

### 5.3 What `tx.alter` / `tx.commute` / `tx.ensure` look like at the SQL layer

| Txn primitive | What runs at the Postgres layer |
|---|---|
| `tx.deref(ref)` | `SELECT v FROM datom_log WHERE e = $eid AND a = $spec_attr ORDER BY tx DESC LIMIT 1` (snapshot read inside SERIALIZABLE; SIReadLock placed) |
| `tx.assoc(ref, v)` | adds entry to `tx.write_set`; the actual INSERT lands at commit |
| `tx.alter(ref, fn)` | reads via `tx.deref` (SIReadLock placed); fn runs in Python; writes deferred to commit |
| `tx.ensure(ref)` | `SELECT v FROM datom_log WHERE e = $eid AND a = $spec_attr ORDER BY tx DESC LIMIT 1` (same shape as deref — places the same SIReadLock; the txn-side `ensure_set` is the local mirror) |
| `tx.commute(ref, fn_id, args)` | adds entry to `tx.commute_log`; **does NOT place a predicate lock** — that is the entire point of commute (v0.5.2 § F3). Reapply at commit reads under `store._lock` after the SERIALIZABLE conflict check passes |
| commit | `BEGIN ISOLATION LEVEL SERIALIZABLE; SELECT 1 FROM datom_log WHERE tx > $t_start AND e = ANY($touched_eids); INSERT INTO datom_log VALUES ...; COMMIT;` |

**`tx.commute` requires NOT placing a predicate lock on the commute target.** This is achieved by *not* issuing a `SELECT` against the commute-target eid inside the `tx.commute` call — the body-eager value comes from `tx.commute_log` state, the reapply at commit issues the SELECT under `store._lock` post-conflict-check. ADR-6 pins this.

### 5.4 SSI false-positive rate

SERIALIZABLE retries on rw-dependency cycles even when the application logic would have committed correctly. Empirically the false-positive rate is workload-dependent; for our workload (low-thousands of datoms / hr / writer at the coding-agent MVP scale) the expected retry rate is < 1% based on community benchmarks. ARIS R2 measures this against the cross-process Hypothesis property and budgets `max_retries=256` (mirrors v0.5.2's `dosync` retry budget) so retry exhaustion is a real bug, not a hot-loop.

Statement-level retry (Postgres's natural error-then-retry) does NOT preserve user-level transaction semantics for our case — the *whole* `dosync` body must replay including any Python-side fn calls. This matches Postgres's own guidance (the docs explicitly say automatic retry cannot be done with correctness — § 13.5 PG18 docs). The retry lives in `Transaction._run`, not in psycopg.

## 6. Audit-chain Merkle continuity

### 6.1 The invariant

Every audit datom carries `provenance["parent_provenance_hash"]` pointing to the immediately-prior audit datom's hash, forming a Merkle chain. Across multi-process inserts, this means: if two processes both commit audit datoms at "the same instant," exactly one chain order must be observable. Replay byte-identity then re-derives that exact same order from the persisted log.

### 6.2 The mechanism

`audit_chain_lock` (§ 4.2) is a single-row table with `id = 1` invariant. Every commit-attempt that emits an audit datom takes `SELECT * FROM audit_chain_lock WHERE id = 1 FOR UPDATE` BEFORE building the audit datom. This blocks any concurrent audit-emitting transaction at the row lock until the holder commits or rolls back. The holder reads `head_hash` from the row, builds the audit datom with `parent_provenance_hash = head_hash`, INSERTs the audit datom, and `UPDATE audit_chain_lock SET head_hash = $new_hash, head_tx = $new_tx, head_seq = $new_seq WHERE id = 1`. The COMMIT releases the row lock.

ADR-3 pins this choice over four alternatives:

| Alternative | Why rejected |
|---|---|
| `pg_advisory_xact_lock(<chain_id>)` | Works for single-key serialisation but does not give a typed `head_hash` to read; we'd still need a row to hold the hash |
| Postgres LSN ordering | LSN orders writes globally but is not exposed as a transaction-rendezvous primitive; would need a trigger to emit LSN into the row, adding a write per audit datom |
| Table-level `LOCK TABLE datom_log IN ACCESS EXCLUSIVE` | Blocks all reads + writes; catastrophic for throughput |
| Per-chain advisory lock + auxiliary lookup | Same shape as ADR-3 but split across two primitives — more moving parts, no benefit |

The single-row `FOR UPDATE` is the simplest correct shape and gives us exactly the semantics we need. The cost is one `SELECT FOR UPDATE` + one `UPDATE` per audit-emitting transaction — microseconds.

### 6.3 Replay byte-identity

`replay_byte_identity` (existing in `persistence.replay`) walks `db.log()` in `seq` order and re-derives the audit chain. The cross-process invariant: if process A and process B both committed audit datoms at "the same instant," the persisted `seq` order is the order the `audit_chain_lock` was acquired — process A's audit datom has `parent_provenance_hash = OLD_head_hash`, process B's audit datom has `parent_provenance_hash = A's_audit_hash`, and replay re-derives both correctly. ADR-3 makes this falsifiable: the cross-process Hypothesis property (§ 8) asserts byte-identical replay across N concurrent writers.

### 6.4 What is NOT load-bearing on the lock row

- The `head_hash` column on `audit_chain_lock` is a **performance optimisation** (avoids a `SELECT ... ORDER BY seq DESC LIMIT 1` against `datom_log` for each audit-emit). It is **not** the source of truth — the source of truth is the `datom_log` itself. If the lock row's `head_hash` ever drifts from the actual log tip (would require a bug), `verify_chain()` (existing audit handler invariant) catches it.
- `head_tx` and `head_seq` are stored for the same performance reason, plus they make the chain's tip findable in O(1) for the SDK MCP `audit_tail` resource.

## 7. `any_datoms_since(t)` MVCC primitive

### 7.1 Today's shape

`InMemoryStore.since(t)` and `SQLiteStore.since(t)` both yield datoms with `tx_time > t`, ordered by insertion (`seq` in SQLite, list-position in InMemory). The Txn-side `any_datoms_since(t, touched)` reads `db.store.since(t_start)` filtered by `eid in touched`.

### 7.2 Postgres mapping — ADR-8

Three candidate primitives:

| Candidate | Why considered | Why rejected (or chosen) |
|---|---|---|
| `xmin` / `xmax` system columns | Postgres-native MVCC bookkeeping | NOT serialisable across `pg_dump`/`pg_restore` cycles (txid wraparound, vacuum) — replay against a restored DB would diverge |
| `pg_logical_emit_message` + LSN | Cross-backend ordering primitive | Requires logical-replication slot for any reader; adds infra dependency |
| `(tx_time, tx)` strict-monotonic projection | Already in the table; matches SQLite | **Chosen.** No new infra. Replay byte-identity preserved trivially because the same tuple lives in both backends. |

ADR-8 pins `(tx_time, tx)` ordered by `idx_datom_log_txtime` as the MVCC primitive. The implementation is one query:

```sql
SELECT 1 FROM datom_log
 WHERE tx_time > $t_start AND e = ANY($touched_eids)
 LIMIT 1;
```

Inside the SERIALIZABLE transaction, this places SIReadLocks on the matching rows; any concurrent writer that would invalidate the read causes the *other* transaction to fail at commit with 40001.

### 7.3 Why not also rely on `xmin`?

`xmin` is the Postgres-native answer to "did this row exist as of my snapshot," but it is a per-row, per-snapshot, per-vacuum-state value. Two issues: (a) replay against a restored DB needs the same `xmin` ordering — `xmin` doesn't survive `pg_dump`/`pg_restore`; (b) `xmin` is a 32-bit txid that wraps around, requiring `pg_xact` lookups for ordering across freezing — fragile. `(tx_time, tx)` is a substrate-owned monotonic projection and is the safe choice.

## 8. `allocate_and_append` semantics

### 8.1 Single-process

`InMemoryStore` holds `_lock`; `SQLiteStore` holds `BEGIN IMMEDIATE`. Both are correct within one process.

### 8.2 Multi-process via PostgresStore — ADR-9

The atomic primitive is the SERIALIZABLE transaction itself, plus an `INSERT` whose `tx` value is `SELECT COALESCE(MAX(tx), 0) + 1 FROM datom_log`:

```python
def allocate_and_append(self, datoms: Iterable[Datom]) -> list[Datom]:
    materialised = list(datoms)
    if not materialised:
        return []
    with self._lock:  # in-process serialisation, NOT cross-process
        with self._pool.connection() as conn:
            with conn.transaction():  # SERIALIZABLE
                cur = conn.execute(
                    "SELECT COALESCE(MAX(tx), 0) + 1 FROM datom_log"
                )
                tx = int(cur.fetchone()[0])
                out = [_with_tx(d, tx) for d in materialised]
                conn.executemany(
                    "INSERT INTO datom_log (e, a, v, tx, ...) VALUES (...)",
                    [_encode(d) for d in out],
                )
    return out
```

The SERIALIZABLE level is what gives us multi-process atomicity. Two writers that race the `MAX(tx)` read both see the same value; both insert datoms with the same `tx`; one of the two commits successfully, the other fails at COMMIT with 40001 (because the other writer's INSERT invalidated its read). The 40001 surfaces as a `SerializationFailure` exception, the in-process retry catches it, re-runs `allocate_and_append`, the second writer reads the now-bumped `MAX(tx)`, gets the next tx-id, and commits. No same-tx collision is ever observable.

ADR-9 pins this over alternatives:

| Alternative | Why rejected |
|---|---|
| `pg_advisory_xact_lock` on a global "tx-allocator" key | Works but serialises ALL writers — defeats SSI's point; same-row contention shape as a single mutex |
| `BIGSERIAL tx` column with `RETURNING tx` | Tx-ids drift after rollback (sequence advance survives rollback); not deterministic; replay-byte-identity-incompatible |
| `INSERT ... SELECT COALESCE(MAX(tx), 0) + 1 ... RETURNING tx` in one statement | Same shape; less flexible if we later want to validate tx-id pre-insert |
| `uuid7` for `tx` | Drops monotonicity; breaks `since(t)` ordering invariant |

### 8.3 In-process `_lock` retained

`PostgresStore._lock` (a `threading.RLock`) stays — it serialises in-process writers so they don't all race the `MAX(tx)` read and force same-process retries. The lock is purely a performance optimisation; correctness is on the SERIALIZABLE barrier.

## 10. ADRs

### ADR-1: Single `datom_log` table (no split into datoms / audit / commits)

**Decision.** Postgres backend uses one `datom_log` table for all datom kinds — user writes, txn commit datoms, audit-handler datoms, MCTS datoms, REPL datoms, MCP datoms. No table split.

**Rationale.** Audit datoms ARE datoms — they have the same 8-tuple shape, the same provenance dict, the same `invalidated_by` semantics. Splitting them out forces every replay path to UNION ALL across N tables with a hand-maintained ordering invariant on top, breaks `db.log()`'s cross-backend identity, and makes RLS hard to author. Single-table keeps the byte-identity invariant trivial.

**Consequence.** All indexes are over `datom_log`; the audit-chain Merkle continuity uses a separate `audit_chain_lock` table for the head pointer (one row), not a separate audit table.

### ADR-2: Backend-flavoured migration files; `load_migrations(backend)` dispatches

**Decision.** Existing `0001_datom_log.sql` is renamed `0001_datom_log.sqlite.sql`. NEW `0001_datom_log.postgres.sql` ships alongside. `load_migrations(backend)` filters by suffix.

**Rationale.** SQLite uses `INTEGER PRIMARY KEY AUTOINCREMENT` + ISO-8601 TEXT timestamps; Postgres uses `BIGSERIAL` + TIMESTAMPTZ + JSONB. The two cannot share one SQL file; a single file with `IF` conditionals would make schema review brittle. Per-backend files are explicit and reviewable.

**Consequence.** `_apply_migrations(backend)` becomes `for name, sql in load_migrations(backend): ...`; each store passes its own backend flavour. The InMemoryStore needs no migrations. Future backends (Cloudflare D1, DuckDB, etc.) drop in their own `0001_*.<flavour>.sql` without touching the existing two.

### ADR-3: Audit-chain ordering uses a single-row `audit_chain_lock` + `SELECT FOR UPDATE`

**Decision.** Multi-process audit-chain Merkle continuity is enforced by a one-row `audit_chain_lock(id=1, head_hash, head_tx, head_seq)` table, locked via `SELECT * FROM audit_chain_lock WHERE id = 1 FOR UPDATE` BEFORE building any audit datom needing `prev_audit_hash`.

**Rationale.** § 6.2 enumerates four alternatives (`pg_advisory_xact_lock`, LSN ordering, table-level lock, advisory + auxiliary lookup). The single-row `FOR UPDATE` is the simplest correct shape: typed `head_hash` for O(1) tip lookup, automatic release at COMMIT/ROLLBACK, predicate-lock-free (no SSI bookkeeping).

**Consequence.** Every audit-emitting transaction carries one extra `SELECT FOR UPDATE` + one `UPDATE audit_chain_lock`. Microseconds. The `head_hash` column is a performance optimisation; truth lives in `datom_log` and `verify_chain()` re-derives it.

### ADR-4: `tx` allocation uses `MAX(tx) + 1` under SERIALIZABLE, not `BIGSERIAL`

**Decision.** The `tx` column is `BIGINT` not `BIGSERIAL`. Allocation is `SELECT COALESCE(MAX(tx), 0) + 1 FROM datom_log` inside the SERIALIZABLE transaction. The `seq BIGSERIAL` column does keep physical insertion-order separately.

**Rationale.** `BIGSERIAL` advances on rollback (sequence side-effect); replay byte-identity demands deterministic tx-ids per op-sequence. `MAX(tx) + 1` under SERIALIZABLE has the deterministic shape SQLite already gives us. The cost of one extra read is microseconds with the `(tx_time, tx)` index pinned.

**Consequence.** Two concurrent writers may both compute the same `MAX(tx) + 1`; one commits, the other gets 40001 and retries. Bounded by `max_retries=256` in the v0.5.2 retry loop. False-positive rate measured at ARIS R2.

### ADR-5: BTREE on `(v::text)` for AVET / VAET / AEVT (not GIN)

**Decision.** AVET / VAET / AEVT use BTREE indexes over the canonical-string projection `(v::text)`, NOT GIN over JSONB. GIN indexes are opt-in (separate migration) for operator workloads that need JSONB containment queries.

**Rationale.** Equality lookups must be byte-identical across SQLite and Postgres. SQLite stores `v` as JSON-encoded TEXT with `sort_keys=True` (`store.py:_encode`); Postgres's `JSONB::text` is *not* the same canonical form (different whitespace, different key ordering). The `_encode` codec already produces canonical-string JSON before the INSERT; storing it as JSONB and querying as `(v::text)` against the canonical string keeps cross-backend identity. The trade-off: GIN-style "find datoms whose value contains key X" queries are not free; if a workload needs them, the operator runs the opt-in migration.

**Consequence.** `_encode` (existing) outputs canonical JSON. The Postgres column is JSONB (for storage compactness + potential future GIN), but queries that need byte-identity go through `(v::text)`. The opt-in `0002_optional_gin_indexes.postgres.sql` migration lives in v0.8 but is not auto-applied.

### ADR-6: `tx.commute` does NOT place predicate locks at body call site

**Decision.** `tx.commute(ref, fn_id, args)` adds an entry to `tx.commute_log` and does NOT issue a `SELECT` against the commute target. The reapply at commit reads under `store._lock` AFTER the SERIALIZABLE conflict check passes.

**Rationale.** Placing a predicate lock on the commute target would force SSI to flag two parallel commutes on the same ref as a rw-conflict — but commutativity is the entire point of `tx.commute` (v0.5.2 § F3: "two parallel transactions calling `tx.commute(counter, "inc-by", 1)` both succeed with no retry"). SSI without a SELECT on the target stays silent; reapply-under-lock at commit serialises the actual write.

**Consequence.** Replay byte-identity holds because the commute log is in the commit-datom provenance (`:persistence.txn/commute-log`); replay reads the commute log + applies fn-id against the snapshot value at the *replay's* commit time, getting the same result as the original commit. v0.5.2's commute-replay test extends to PostgresStore unchanged.

### ADR-7: Audit hashing stays Python-side; pgcrypto is NOT used for v0.8 chain integrity

**Decision.** `prev_audit_hash` is computed in Python via `hashlib.sha256` (existing `make_audit_handler` codepath). Postgres-native pgcrypto is NOT used in v0.8.

**Rationale.** (a) Replay byte-identity demands one canonical hash function across backends; SQLite has no pgcrypto, so a Postgres-only hash diverges. (b) Chain integrity is verified by `verify_chain()` which walks `db.log()` in `seq` order — a Python-side hash gives that walker authority over Python-side replay. (c) Server-side hashing forces a server-side trust boundary we don't need yet.

**Consequence.** v0.9 may add a Postgres `BEFORE INSERT` trigger that re-hashes server-side and rejects mismatches as defence-in-depth; that's additive, not a replacement. v0.8 doesn't ship it.

### ADR-8: `(tx_time, tx)` is the MVCC primitive for `since(t)` and `any_datoms_since(t)`

**Decision.** `Store.since(t)` queries `WHERE tx_time > $t ORDER BY seq ASC`; `any_datoms_since(t, touched)` adds `AND e = ANY($touched)`. The strict-monotonic projection is `(tx_time, tx)`; `seq` is the physical insertion-order tiebreaker.

**Rationale.** § 7.2 enumerates `xmin/xmax` and `pg_logical_emit_message`. Both fail the cross-backend determinism + restore-survival tests. `(tx_time, tx)` is substrate-owned, survives `pg_dump`/`pg_restore`, matches SQLite trivially, and is indexed.

**Consequence.** No new infra. Cross-process `any_datoms_since` is a one-query operation. The SERIALIZABLE barrier + SIReadLocks give us the consistency guarantee — no additional Python-side coordination needed.

### ADR-9: `allocate_and_append` runs inside SERIALIZABLE; in-process `_lock` retained for performance

**Decision.** `PostgresStore.allocate_and_append` opens a SERIALIZABLE transaction, reads `MAX(tx)+1`, executemany-INSERTs the datoms, COMMITs. An in-process `threading.RLock` serialises *within-process* writers as a performance optimisation; correctness is on the SERIALIZABLE barrier across processes.

**Rationale.** § 8.2 enumerates `pg_advisory_xact_lock`, `BIGSERIAL RETURNING`, single-statement INSERT-with-MAX, uuid7. SERIALIZABLE + `MAX(tx)+1` is the simplest correct shape that preserves replay determinism.

**Consequence.** SerializationFailure (40001) on the COMMIT bubbles up as `psycopg.errors.SerializationFailure`; the Txn-side `_commit_attempt` (8-line hook in `transaction.py`) catches it and returns `False`, the existing retry loop retries.

### ADR-10: psycopg 3 + psycopg-pool; opt-in `[postgres]` extra; lazy import

**Decision.** PostgresStore uses psycopg 3 (`>= 3.2`) with `psycopg_pool.ConnectionPool`. The dependency is in a `[postgres]` optional-dependency extra (`pip install "persistence[postgres]"`). The module imports psycopg lazily inside `__init__` so the substrate is importable without psycopg.

**Rationale.** (a) psycopg 3 is the modern Postgres driver for Python; native asyncio support (psycopg-pool 3.x) lets us extend to async paths in v0.9 without driver swap. (b) Optional-dependency keeps the substrate small for users who only run InMemory or SQLite. (c) Lazy import lets the SDK URI dispatcher (Adapter SDK ADR-9) raise a clean `BackendNotInstalled` error rather than an obscure ImportError.

**Consequence.** `pip install persistence` works without Postgres; `pip install "persistence[postgres]"` enables it. CI matrix gains a `[postgres]` extras job that runs against a real Postgres via `testcontainers`.

### ADR-11: Migration runner uses raw SQL files; NO Alembic

**Decision.** Migrations are a sorted list of `.sql` files loaded by `load_migrations(backend)` and applied via `executescript` (SQLite) / `cursor.execute` (Postgres). No Alembic, no SQLAlchemy.

**Rationale.** (a) Substrate already has zero ORM; adding Alembic for one backend introduces a new dependency tree (SQLAlchemy + Mako) for a runner that only needs `apply N forward-only SQL files`. (b) Forward-only is sufficient for v0.8; multi-step rollbacks are a v1.0+ concern when schema evolves with breaking changes. (c) Raw SQL files are what operators read for SOC2 audit; no DSL between them and the schema.

**Consequence.** Migrations are append-only. Any breaking change to the schema is a new `0002_*.sql` with an explicit upgrade path documented. Operator-driven rollbacks ship a documented `*_rollback.sql` (per existing project pattern: see InfraFlow `008_rollback.sql`); not auto-applied.

### ADR-12: RLS-ready, RLS-not-authored

**Decision.** `datom_log` rows carry an implicit tenant via `provenance->>'tenant_id'`. v0.8 adds NO RLS policy. The schema is *RLS-ready*: an operator can `ALTER TABLE datom_log ENABLE ROW LEVEL SECURITY` and author a policy on `provenance->>'tenant_id'`. The substrate does not enforce one.

**Rationale.** v0.8 ships single-tenant dev/MVP deploys. Two distinct timelines bear on the multi-tenant story and must NOT be conflated: (a) **substrate-level tenant model** — a `provenance->>'tenant_id'` convention promoted to a first-class `Substrate.open(tenant=...)` parameter with cap-token tenant-scoping — lands in **v0.9** as additive surface; (b) **Juba OS multi-tenant deploy** that authors RLS policies against (a)'s tenant model — lands in **Phase 4** (post-incorporation, product-side) once a tenant model exists to author against. Authoring policies now would invent a tenant model the substrate doesn't know how to use; staying RLS-ready means adopting tenants later is additive in both layers.

**Consequence.** Multi-tenant Juba OS (Phase 4 timeline) ships its own `0003_tenant_rls.postgres.sql` migration when needed, against the v0.9 substrate-level tenant model. The substrate documentation (`postgres-deployment-v0.8.md`) flags this as a layered v0.9+/Phase-4 concern with a worked example showing both the substrate-level `tenant=` pattern and the RLS policy authored on top.

ARIS R1 (this design doc) target: mean ≥ 8.5 / min ≥ 7.5 per the auto-review-loop hard-mode pattern (matches v0.5.2 + Adapter SDK).

ARIS R2 (post-implementation) target: mean ≥ 8.5 / min ≥ 7.5 on the merged branch tip.

### G1 — Smoke: PostgresStore satisfies the `Store` Protocol

`isinstance(PostgresStore(...), Store)` is True via duck-typing; calling each Protocol method (`append`, `allocate_and_append`, `all_datoms`, `since`, `mark_invalidated`, `next_tx`) against a real Postgres returns shapes equivalent to InMemoryStore.

### G2 — Single-process replay byte-identity (v0.5.2 N6/N7 extended)

The existing v0.5.2 Hypothesis property at `max_examples=200` (`tests/persistence/txn/test_replay_byte_identity_property.py`) parametrises over `[InMemoryStore, SQLiteStore, PostgresStore]`. All three pass at 200 examples each on the same draw seeds. Filter scope unchanged: `not d.a.startswith("persistence.txn/")` + per-run UUIDs excluded + clock frozen.

### G3 — Cross-process replay byte-identity (the load-bearing test)

A new test `tests/persistence/fact/test_postgres_store_concurrency.py` spawns N=4 child processes via `multiprocessing.Process`. Each child writes a strategy-drawn op-sequence concurrently against the same PostgresStore (shared connection pool). After all children join, the parent reads the entire log, projects to the same filtered tuple list, and asserts: (a) the projection is byte-identical to a single-process baseline that processed the same op-sequences serially in the same order; (b) `verify_chain()` returns True for the audit-chain projection; (c) total tx-id count = sum of per-process op counts (no same-tx collision survived). Hypothesis property at `max_examples=50` over op-sequence shape (lower than single-process 200 because of multiprocessing fork overhead).

### G4 — SerializationFailure is the only retryable cross-process exception

A targeted test forces a 40001 by colluding two writers' `MAX(tx)` reads with a barrier. Asserts: `_commit_attempt` returns `False` (not raises); the retry loop catches it and re-runs; second attempt succeeds. Asserts: NO other psycopg exception is silently caught.

### G5 — Audit-chain Merkle continuity across processes

Two child processes write 100 audit datoms each in parallel. Parent reads `audit_chain_lock` row + walks `datom_log` filtered by `a LIKE 'audit/%'`. Asserts: every audit datom's `parent_provenance_hash` equals the immediately-prior audit datom's hash (in `seq` order); `audit_chain_lock.head_hash` equals the last audit datom's hash; `verify_chain()` returns True.

### G6 — `since(t)` cross-backend identity

Same op-sequence applied to InMemoryStore + SQLiteStore + PostgresStore. `since(t_mid)` returns the same projected tuples in the same order. (Implementation: `seq` ordering on Postgres = `seq` ordering on SQLite = list ordering on InMemory.)

### G7 — Migration apply + rollback on a real Postgres

`testcontainers` Postgres 17 fresh image. Apply `0001_datom_log.postgres.sql` cleanly. Apply `0001_datom_log.postgres_rollback.sql` cleanly. Re-apply the forward migration cleanly. Suite passes.

### G8 — `BackendNotInstalled` error path

`pip install persistence` (without `[postgres]`). `Substrate.open(store="postgres://...")` raises `BackendNotInstalled` with a message including the install hint `pip install "persistence[postgres]"`.

### G9 — Pool sizing + statement timeout configuration

`PostgresStore(uri="postgres://...?pool_min=1&pool_max=10&statement_timeout=30000")` parses the query string and applies the pool size + statement timeout. Out-of-band test asserts a long-running `pg_sleep(60)` is interrupted at 30s.

### G10 — Cross-process Atom CAS

A targeted test spawns 4 processes, each calls `atom.swap(lambda v: v + 1)` 100 times against a shared PostgresStore. Final atom value = 400. Total writes = 400 + retries (logged but bounded). No `AtomCASExhausted` raised at `max_retries=256`.

### G11 — Cross-process `tx.commute` correctness

Spawn 4 processes, each calls `tx.commute(counter_ref, "inc-by", 1)` 100 times. Final counter = 400. Zero retry-rejected commits (since commute does not place predicate locks per ADR-6).

### G12 — Cross-process `tx.ensure` write-skew prevention

Classical bank-transfer write-skew test (v0.5.2 § F2) extended to multi-process: two processes, each transfers from the joint Alice+Bob balance to a third party. With `tx.ensure` on the partner's account, exactly one commits; the other retries and overdraws is prevented.

### G13 — Coverage gate

`pytest --cov=persistence.fact.postgres_store` shows ≥ 90% line coverage on the new module. Branches uncovered are documented.

## 13. Open questions (defer to ARIS R1 review or impl phase)

1. **Statement timeout default in URI?** v0.8 default: `statement_timeout=30000` (30s) baked into the URI parser. Alternative: 60s. Recommend 30s — coding-agent workload is interactive, runaway queries are a real failure mode, 30s is generous for our access patterns.
2. **Pool sizing default?** `pool_min=1, pool_max=10`. Alternative: `pool_max=20`. Recommend 10 — testcontainers + dev-laptop runs cleanly at 10, scales linearly; bigger defaults invite "why are we eating Postgres connections" questions in deploy.
3. **`testcontainers` vs ephemeral CI Postgres for tests?** `testcontainers` (lazy-pulled Docker image) is the simpler shape. Alternative: GitHub-Actions service-container. Recommend `testcontainers` for local-dev-loop fidelity; CI can use GH service-container as a parallel path.
4. **Cross-process Hypothesis `max_examples`?** 50 is the proposed budget (vs single-process 200) because multiprocessing fork overhead is ~50× slower than in-process. Alternative: 100 with parallel pytest-xdist workers. Recommend 50 for v0.8; revisit at v0.9 if false-pos rate findings warrant more.
5. **Optional GIN indexes — ship in v0.8 as an opt-in migration, or fully defer?** Recommend ship `0002_optional_gin_indexes.postgres.sql` in v0.8 but DO NOT auto-apply. Operators run it explicitly if their workload needs it.
6. **`pgaudit` settings in `postgres-deployment-v0.8.md`?** Recommend `pgaudit.log = 'write,ddl,role'` as the documented default; lighter (`'ddl'` only) and heavier (`'all'`) variants documented as alternatives. v0.8 ships no code-level pgaudit dependency.
7. **Should `PostgresStore` expose an `async_append` / `async_all_datoms` for psycopg-async?** v0.8: NO — Module 5 Txn is sync; introducing async-Store paths forces an async-Txn rewrite that's out of scope. v0.9 may grow async paths once we need them.

## 14. Risks + mitigations

| # | Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|---|
| R1 | SSI false-positive rate higher than benchmarks suggest, retry exhaustion at `max_retries=256` | medium | high | G4 measures the rate at ARIS R2; Hypothesis property at `max_examples=50` cross-process covers the tail. If rate is > 5%, we lift the budget to 1024 and re-measure; if structural, revisit ADR-9 in v0.8.1. |
| R2 | Audit-chain `audit_chain_lock` becomes a write-throughput bottleneck | medium | medium | Lock is held only for the single SELECT FOR UPDATE + UPDATE — microseconds. Quantitative gate at ARIS R2: throughput regression vs SQLite ≤ 2×. If exceeded, ADR-3 alternative (advisory lock + LSN ordering) is the v0.8.1 backout. |
| R3 | psycopg 3 + connection pool flakiness in CI (testcontainers + connection refusal during PG warmup) | medium | low | `testcontainers` `wait_for(strategies.LogMessageWaitStrategy("ready to accept connections"))`; retry on `OperationalError` for first 5s of test setup. Documented in `tests/persistence/fact/conftest.py`. |
| R4 | Replay byte-identity divergence under JSONB ↔ TEXT codec drift | low | high | ADR-5 pins `(v::text)` BTREE for cross-backend identity. G6 falsifiable test asserts `since(t)` projection identity across all three backends on the same op-sequence. |
| R5 | Migration runner conflict with future Alembic adoption | low | low | ADR-11 pins forward-only raw SQL; future Alembic adoption (if needed) lands as a separate migration system for product-layer code, NOT for substrate. Substrate stays raw-SQL forever. |
| R6 | Operator runs Postgres < 17 and a feature regresses | low | medium | Deployment guide pins minimum to PG 14 (matches pg_partman + pgmq + pgaudit support floor). PG 17 features (partition-wise aggregate pushdown, logical-failover) are documented as enhancements, NOT required. |
| R7 | RLS-not-authored stance bites at multi-tenant deploy | low | medium | ADR-12 explicit and layered: v0.9 ships substrate-level tenant model (additive; `Substrate.open(tenant=...)` + `provenance->>'tenant_id'` first-class), Phase 4 (post-incorporation) ships Juba OS multi-tenant RLS authoring on top. The two timelines are distinct deliberately: substrate has a tenant model before any deploy needs one. |
| R8 | Cross-process Hypothesis property is flaky under multiprocessing on macOS forkserver vs spawn | medium | medium | Pin `multiprocessing.set_start_method("spawn")` at test module level. Spawn is slower but deterministic across macOS / Linux. |
| R9 | Postgres + SQLite cross-backend codec drift on TIMESTAMPTZ | low | high | `_encode` / `_decode` already round-trip ISO-8601; psycopg 3 returns `datetime` with tzinfo; the codec asserts `tzinfo is not None` (matches `datom.py:148-159` invariant). G6 catches drift. |
