## 1. Summary

Persistence OS today has two production-style fact stores: `InMemoryStore` (tests + CLI demo, no durability) and `SQLiteStore` (single-process durable; `BEGIN IMMEDIATE` writer-lock semantics). Module 5 Txn (`v0.5.2`) ships Clojure-parity STM (Refs / `dosync` / Atoms / `tx.commute` / `tx.ensure`) but the v0.5.0a1 design doc § "Design constraints" explicitly defers **multi-process** STM to v0.6.x: *"single-process only — multi-process via Postgres SERIALIZABLE deferred."* Today the Txn module's `_commit_attempt` (`src/persistence/txn/transaction.py:572-630`) holds `tx.db.store._lock` across the conflict-check + transact-batch core, which is correct for one OS process but is a no-op across processes. Multi-process correctness needs a **shared isolation barrier** that lives in the storage layer, not in the Python lock.

This design ships `PostgresStore` — a third backend that implements the same `Store` Protocol, runs under Postgres SERIALIZABLE isolation, and preserves byte-identical replay across processes. The store is opted into via the URI scheme `postgres://...` per Adapter SDK ADR-9; existing in-process consumers using `Substrate.open(store="memory")` or `Substrate.open(store="sqlite:///...")` are not affected.

This design ships three artifacts:

1. **`persistence.fact.postgres_store.PostgresStore`** — a new backend in the `[postgres]` optional-dependency extra (psycopg 3 + psycopg-pool); implements the full `Store` Protocol; runs all writes under `ISOLATION LEVEL SERIALIZABLE`; surfaces 40001 retry semantics through the existing `Transaction._run` retry loop without changing public API.
2. **`src/persistence/fact/migrations/0001_datom_log.postgres.sql`** — the Postgres-flavoured schema (TIMESTAMPTZ, JSONB, BIGINT/BIGSERIAL, partial+expression indexes, a single `audit_chain_lock` row for `prev_audit_hash` ordering). Sibling to the existing `0001_datom_log.sql` which becomes the SQLite-flavoured copy. Migration runner picks the right file by URI scheme.
3. **`tests/persistence/fact/test_postgres_store_*` + cross-process Hypothesis property** — single-process replay byte-identity (the v0.5.2 N6/N7 property) is parametrised over `[InMemoryStore, SQLiteStore, PostgresStore]` at `max_examples=200` per backend (G2). The cross-process Hypothesis variant runs at `max_examples=50` (lower because of multiprocessing spawn overhead) and asserts a 5-sub-invariant gate (G3a-G3e per ADR-14 W2-clarified, W3-rewritten): replay-determinism, no-same-tx-collision, audit-chain-continuity, sum-of-ops, some-serial-order-exists (rw-cycle check from persisted commit provenance). The persisted log is its own canonical baseline; we no longer compare to a hypothetical single-process serial-order baseline (which SERIALIZABLE doesn't guarantee anyway). Replay byte-identity is the falsifiable invariant.

**Not** in this stream: Citus sharding, multi-region read replicas, pgvector-backed `recall`, audit-tail logical-replication streaming. All deferred — see § 9.

**Ships:** ~770 LOC + ~270 tests + Postgres migration SQL (~150 lines including `tx_allocator` + `audit_chain_lock` + opt-in JSONB-columns sibling). **5 days** (parallel to SDK impl after SDK1). LOC bump from ~700 to ~770 in W1-revised reflects the `transact_serializable` Protocol extension + `persist_repl_audit` migration scope expansion (per ARIS R1 W1 ADR-13).

## 2. Strategic frame

### Why this stream ships as Phase 1 stream #2 (not later)

Three reasons, in order of weight:

1. **It is the production deployment surface.** Phase 2 ships `persistence-coder` as a commercial CLI agent. Any commercial customer who cares about durability + multi-process scaling will run it on Postgres, not SQLite. Without `PostgresStore` we ship a Phase 2 product that cannot be deployed past one operator's laptop. The Adapter SDK already commits to URI dispatch for `postgres://...` (ADR-9); the missing piece is the actual implementation.
2. **It is the load-bearing closure for the v0.5.0a1 deferral.** v0.5.0a1 carved Module 5 Txn deliberately narrow: single-process STM with the multi-process map onto Postgres SERIALIZABLE explicitly punted to v0.6.x. v0.5.1 + v0.5.2 then absorbed the entire window for Clojure-parity surface (Refs / `dosync` / Atoms / `tx.commute` / `tx.ensure`) without revisiting the multi-process deferral. v0.8 is where it gets paid down — and the Clojure-parity primitives need to stay correct across processes, not just within one.
3. **It de-risks every later stream's storage assumptions.** Plan Edit API (#140), `:code` sandbox (#141), and `fold()` executor (#145) all write datoms that go through the Store. If we discover only at v1.0 that one of those streams breaks under SERIALIZABLE (e.g. an interceptor that writes mid-transaction in a way that triggers 40001), the fix lands deep in product work. Doing PostgresStore in Phase 1 surfaces those bugs while the substrate-side ARIS gates are still active.

### Where this stream sits in the v1.0 product roadmap

```
Phase 0 ──→ Phase 1 Substrate completion ──┬──→ Phase 2 Coding agent MVP
                │                          │
                ├─ Adapter SDK [stream 1]   ├── persistence-coder MVP runs against PostgresStore in dev
                ├─ POSTGRES STORE [this]    │

```

This stream is the load-bearing **second** Phase 1 item because every later Phase 1 item assumes a shared multi-process isolation surface for its own correctness tests.

### What this stream preserves (zero canonical-form change)

- `Store` Protocol (`src/persistence/fact/store.py:62-127`) — the **6 existing methods** (`append`, `allocate_and_append`, `all_datoms`, `since`, `mark_invalidated`, `next_tx`) keep their exact shape. The Protocol gains **TWO additive methods** per ADR-15 W2: `_txn()` (store-level transaction context-manager) and `transact_serializable(facts, provenance, tx_time?) -> int` (atomic primitive for cross-process correctness). Both have default implementations on InMemoryStore + SQLiteStore that preserve existing semantics — no behavioural change for those backends; PostgresStore overrides both with the row-lock + SERIALIZABLE-txn shape. All adapters bound to the existing 6 methods continue to work; the two new methods are opt-in surface for adapters that want multi-process atomicity. (W2-revised + W3-revised: the original "Store Protocol shape unchanged — adds no new methods" framing was retracted; W3 corrects "ONE method" → "TWO methods" to match ADR-15.)
- `Datom` 8-tuple shape + frozen-dataclass invariants (`datom.py:108-194`) unchanged.
- `_encode` / `_decode` JSON-canonicalisation (`store.py:356-385`) unchanged for cross-backend wire consistency.
- Audit chain + `prev_audit_hash` linkage exactly preserved (§ 5).
- v0.5.2 conflict-detection union `read_set | ensure_set | write_set` (`transaction.py:609`) unchanged — Postgres is the new substrate the union maps onto, not a redesign of the union itself.
- Replay byte-identity Hypothesis property (v0.5.2 N6/N7 at `max_examples=200`) parametrised over PostgresStore (G2) — the existing single-process invariant stays a passing gate. A cross-process Hypothesis variant runs at `max_examples=50` per ADR-14 with a 5-sub-invariant gate (G3a-e); it does NOT compare to a single-process baseline — see § 11 G3 + ADR-14 W2-clarified.
- AGPL-3.0-or-later licensing.

### What this stream changes

- New `persistence.fact.postgres_store.PostgresStore` class. New optional extras `[postgres]` (psycopg 3 + psycopg-pool).
- New migration file `0001_datom_log.postgres.sql` (TIMESTAMPTZ + JSONB + BIGSERIAL + GIN indexes on JSONB columns + a single-row `audit_chain_lock` table).
- Migration loader (`load_migrations`) gains a backend-flavour discriminator — see ADR-2.
- `Substrate.open(store="postgres://...")` (Adapter SDK ADR-9) wires through to this backend.
- Adds a 40001 retry hook on the Txn-side commit attempt: when `_commit_attempt` raises `psycopg.errors.SerializationFailure`, it returns `False` (same shape as the v0.5.2 conflict-rejection path), and the existing retry loop in `_run` retries — no public API change.
- New CHANGELOG-fact.md section for v0.8.

PG6 deploy guide includes a "minting tokens against a Postgres-backed substrate" walkthrough.

## 4. Schema design

### 4.1 Datom log table

The Postgres schema mirrors the SQLite schema's column shape so the codec (`_encode` / `_decode`) stays cross-backend identical. Type promotions are mechanical: `INTEGER` → `BIGINT`, `TEXT` (ISO-8601 timestamp) → `TIMESTAMPTZ`. **`v` and `provenance` stay as `TEXT` (canonical JSON)** in Postgres v0.8 — see ADR-5 (W1-revised) for why JSONB was rejected as the storage column type for byte-identity reasons. An optional sibling JSONB column for GIN-indexable containment queries is shipped as the opt-in `0002_optional_jsonb_columns.postgres.sql` migration. The 8-tuple semantics + provenance + `invalidated_by` pointer are unchanged.

```sql
CREATE TABLE datom_log (
    seq             BIGSERIAL    PRIMARY KEY,
    e               TEXT         NOT NULL,
    a               TEXT         NOT NULL,
    v               TEXT         NOT NULL,     -- canonical JSON (sort_keys=True), byte-identical to SQLite
    tx              BIGINT       NOT NULL,
    tx_time         TIMESTAMPTZ  NOT NULL,
    valid_from      TIMESTAMPTZ  NOT NULL,
    valid_to        TIMESTAMPTZ,
    op              TEXT         NOT NULL CHECK (op IN ('assert', 'retract')),
    provenance      TEXT         NOT NULL,     -- canonical JSON (sort_keys=True)
    invalidated_by  BIGINT,
    -- Hard schema-level uniqueness primitive for tx allocation safety net.
    -- Per ADR-4 (W1-revised): the SERIALIZABLE allocator (tx_allocator
    -- in § 4.2) is the *correctness* primitive; this UNIQUE constraint is
    -- the *safety net* that converts any allocator regression into a
    -- visible unique_violation rather than silent corruption. Two writers
    -- that ever produce the same (tx, e, a) pair → one INSERT raises
    -- 23505 (unique_violation), which the Txn-side retry loop treats as
    -- equivalent to 40001 (SerializationFailure) per ADR-4.
    UNIQUE (tx, e, a)
);

-- Five primary indexes (mirror SQLite); v + provenance are stored as TEXT
-- so equality lookups are byte-identical to SQLite without any cast.
CREATE INDEX idx_datom_eavt ON datom_log (e, a, valid_from, tx);
CREATE INDEX idx_datom_aevt ON datom_log (a, e, valid_from, tx);
CREATE INDEX idx_datom_avet ON datom_log (a, v, e, tx);
CREATE INDEX idx_datom_vaet ON datom_log (v, a, e, tx);
CREATE INDEX idx_datom_vte  ON datom_log (valid_from, valid_to, e);

-- LOG — tx_time ordered, drives since(t) and replication; (tx_time, tx) is
-- the strict-monotonic projection for the MVCC primitive in § 6.
CREATE INDEX idx_datom_log_txtime ON datom_log (tx_time, tx);
```

**On the existing SQLite migration's `(tx, a, e) UNIQUE` comment.** The current `0001_datom_log.sql:24-26` claims a `(tx, a, e)` uniqueness constraint that is NOT actually present in the SQLite `CREATE TABLE`. PG1 fixes this in two ways: (a) renames the SQLite migration to `0001_datom_log.sqlite.sql` and corrects the comment to match the actual schema (the auto-retraction invariant is enforced at the Python `db.transact` layer, not at the schema level); (b) the new Postgres migration ships the `UNIQUE (tx, e, a)` constraint *for real* as a hard safety net behind the SERIALIZABLE allocator (per ADR-4 W1-revised). The two backends now have aligned schema-level guarantees on the Postgres side strictly stronger than SQLite's (which is fine because SQLite is single-process by `BEGIN IMMEDIATE`).

**Single table vs split (datoms / audit / commits)?** Single. ADR-1 below pins this. Three reasons. (a) The audit chain is encoded as datoms with `a IN ('audit/op-...', 'persistence.txn/commit-id', 'mcts/op-...', 'repl/op-...', 'mcp/op-...')` — splitting them out would force every replay path to do a UNION ALL across tables, with a hand-maintained ordering invariant on top; staying single-table makes the existing `db.log()` projection identical across backends and keeps the EAVT/AEVT indexes useful for audit reads. (b) The commit-datom shape (`a = 'persistence.txn/commit-id'`) is the same shape as a user-write datom; v0.5.2 N6/N7 explicitly filter commit datoms via `not d.a.startswith("persistence.txn/")` at the comparator boundary (`tests/persistence/txn/test_replay_byte_identity_property.py:43-49`), and that filter is byte-identical across backends only if the rows live in the same table and order. (c) Splitting tables makes RLS policies (§ 7) harder to author — a single `datom_log` lets the tenant-id policy live on one row-shape.

### 4.2 Audit chain ordering primitive

Multi-process audit chain continuity requires that `prev_audit_hash` always points to the most-recent committed audit datom *as observed by the writer that just committed*. A single `audit_chain_lock` table (with exactly one row) plus `SELECT ... FOR UPDATE` is the ordering primitive.

```sql
CREATE TABLE audit_chain_lock (
    id          INTEGER PRIMARY KEY CHECK (id = 1),  -- exactly one row
    head_hash   TEXT,                                 -- last committed audit hash
    head_tx     BIGINT,                               -- tx of the last audit datom
    head_seq    BIGINT                                -- seq of the last audit datom
);
INSERT INTO audit_chain_lock (id, head_hash, head_tx, head_seq)
    VALUES (1, NULL, 0, 0);
```

The Txn-side commit attempt that contains audit datoms takes `SELECT * FROM audit_chain_lock WHERE id = 1 FOR UPDATE` BEFORE building any audit datom that needs `prev_audit_hash`. The lock row is held until the surrounding SERIALIZABLE transaction commits, so two concurrent writers hash-chain in commit-order, not write-order. ADR-3 pins this.

### 4.2a Tx-allocator primitive (W1-revised — hard correctness primitive)

**Why a separate primitive (not just `MAX(tx)+1` under SERIALIZABLE).** The original draft pinned `tx` allocation as `SELECT COALESCE(MAX(tx), 0) + 1 FROM datom_log` under the SERIALIZABLE barrier and claimed SSI's predicate-locking would force one of two racing writers to abort with 40001. ARIS R1 W1 flagged this as too implicit: SSI's predicate-lock placement on a `MAX()` aggregate depends on planner choices (sequential scan vs index scan vs index-only scan) and PG version; relying on it as the *only* correctness primitive is brittle. The W1-revised design layers a **hard primitive** alongside SSI:

```sql
CREATE TABLE tx_allocator (
    id          INTEGER PRIMARY KEY CHECK (id = 1),  -- exactly one row
    next_tx     BIGINT  NOT NULL DEFAULT 1
);
INSERT INTO tx_allocator (id, next_tx) VALUES (1, 1);
```

**Allocation protocol** (every `allocate_and_append` and every audit-emitting transaction — see ADR-13 for the audit migration):

```sql
-- Inside the SERIALIZABLE transaction:
SELECT next_tx FROM tx_allocator WHERE id = 1 FOR UPDATE;
-- holds the row lock until COMMIT/ROLLBACK; serialises ALL tx-id allocators
UPDATE tx_allocator SET next_tx = next_tx + N WHERE id = 1;
-- caller now owns tx-ids [next_tx .. next_tx + N - 1]
INSERT INTO datom_log (e, a, v, tx, ...) VALUES ...;
-- every row's tx is in the allocated half-open range
COMMIT;
```

**Why a row lock + `UPDATE`, not `BIGSERIAL`.** The allocator row lock gives us three properties `BIGSERIAL` does not: (a) **Determinism on rollback** — `BIGSERIAL` advances on rollback (the sequence side-effect outlives the transaction); the allocator row's `next_tx` only advances on COMMIT, so a rolled-back batch returns its tx-ids to the pool. (b) **Cross-batch atomicity** — one allocator-row lock covers all N rows of a batch insert, no `RETURNING tx` per-row plumbing. (c) **Plan independence** — correctness no longer depends on which plan PG picks for `MAX(tx)`. The cost is one extra `SELECT FOR UPDATE` + one `UPDATE` per allocate, microseconds.

**Why also `UNIQUE (tx, e, a)` on `datom_log` (§ 4.1)?** Defence in depth. If a future code-path bypasses `allocate_and_append` and calls `INSERT INTO datom_log` directly without first locking `tx_allocator`, the unique constraint converts the bug into a visible `psycopg.errors.UniqueViolation` rather than silent same-tx duplication. The Txn-side retry loop treats `UniqueViolation` (SQLSTATE 23505) on `datom_log_tx_e_a_key` the same as `SerializationFailure` (40001) — re-allocate and retry. ADR-4 (W1-revised) pins both layers.

**Lock ordering.** When a transaction emits BOTH an audit datom AND user-write datoms, the lock acquisition order is fixed: `tx_allocator` (FOR UPDATE on `id=1`) FIRST, `audit_chain_lock` (FOR UPDATE on `id=1`) SECOND, then INSERT into `datom_log`. The fixed order eliminates deadlock between two concurrent audit-emitting transactions (PG's default `deadlock_timeout=1s` would catch any accidental inversion, but we prevent it by convention). ADR-13 pins the audit-side discipline.

### 4.3 Tx-id allocation (W1-revised)

Tx-id allocation is the `tx_allocator` row + `SELECT FOR UPDATE` + `UPDATE next_tx` primitive defined in § 4.2a, with `UNIQUE (tx, e, a)` on `datom_log` as defence-in-depth. ADR-4 (W1-revised) pins this.

Three primitives were considered: `BIGSERIAL`, `MAX(tx)+1` under SERIALIZABLE, and the allocator row.

| Primitive | Determinism | Plan-independent correctness | Verdict |
|---|---|---|---|
| `BIGSERIAL` | NO — sequence advance survives rollback | YES | Rejected (replay byte-identity demands no rollback gaps) |
| `MAX(tx)+1` under SSI | YES | NO — relies on SSI predicate-lock placement on aggregate plan | Rejected (W1; not a hard primitive) |
| `tx_allocator` row + `FOR UPDATE` + `UPDATE` | YES | YES — explicit row lock, plan-independent | **Chosen.** |

The `seq BIGSERIAL` column does keep insertion-order for `since(tx_time)` and `all_datoms()` reads — it is the *physical* row order, not the *logical* tx-id. Gaps in `seq` after a rollback are fine because all reads order by `seq` and skip nothing.

### 4.4 v / provenance storage column type — TEXT, not JSONB (W1-revised)

ADR-5 (W1-revised) pins `v` and `provenance` as `TEXT` (canonical JSON `json.dumps(..., sort_keys=True)`) on Postgres, NOT JSONB. The original draft proposed JSONB with `BTREE on (v::text)` for cross-backend identity; ARIS R1 W1 flagged this as unsafe: `jsonb::text` is not a stable byte stream (whitespace + key-order normalisation differ across PG versions and `jsonb_set` rewrites can change the representation between INSERT and SELECT). Storing as TEXT removes the canonicalisation question entirely — the bytes that go in are the bytes that come out, and they match SQLite's TEXT column shape exactly.

**Trade-off accepted.** TEXT storage means no native JSONB containment queries (`v @> '{...}'`). For v0.8 workloads (coding-agent MVP, recall is substring + tag per Adapter SDK ADR-8) this is fine. Operators who need containment queries run the opt-in `0002_optional_jsonb_columns.postgres.sql` migration which adds parallel `v_jsonb` / `provenance_jsonb` columns + GIN indexes, populated via a one-time `UPDATE datom_log SET v_jsonb = v::jsonb`. The parallel columns are NEVER read by the Store Protocol (which sees only `v`/`provenance` TEXT); they are operator-side analytics surface.

Index strategy: simple BTREE on `v` (TEXT). Identical to SQLite. No `(v::text)` cast; no GIN-on-JSONB in the core schema. R4 risk (codec drift) drops from "high" to "low" — the TEXT column eliminates the JSONB-rendering surface that drove the original concern.

## 5. SERIALIZABLE isolation — Txn integration

### 5.1 Connection isolation

Every connection in the pool is opened with `SET default_transaction_isolation = 'serializable'`. Every `_commit_attempt` runs inside a `BEGIN ISOLATION LEVEL SERIALIZABLE` block. Reads-only paths (`all_datoms`, `since`, `next_tx`) run in the same SERIALIZABLE isolation but at `READ ONLY` mode — Postgres's predicate-locking gives us the read consistency we need for `any_datoms_since(t)` to be sound across processes, and `READ ONLY` allows the planner to skip SSI bookkeeping for reads that cannot conflict.

### 5.2 dosync / tx.alter / tx.commute / tx.ensure → SQL

The v0.5.2 conflict-detection union (`transaction.py:603-609`) is:

```python
touched = (
    {r.eid for r in tx.read_set}
    | {r.eid for r in tx.write_set}
    | {r.eid for r in tx.ensure_set}
)
if any_datoms_since(tx.db, tx.t_start, touched):
    return False
```

Mapping this onto Postgres SERIALIZABLE has two layers:

**Layer 1 (correctness)** — Postgres SSI (Serializable Snapshot Isolation) detects rw-dependency conflicts automatically. As soon as the transaction issues a `SELECT ... FROM datom_log WHERE e = ANY($touched)` inside the SERIALIZABLE transaction, Postgres places SIReadLock predicate locks on the matching rows + index entries; any concurrent writer that would have invalidated the read will cause one of the two transactions to fail at commit with SQLSTATE `40001` (`SerializationFailure`). The existing v0.5.2 conflict-detection union becomes a single Postgres SELECT — Postgres does the conflict detection.

**Layer 2 (preserve our explicit semantics)** — we keep the explicit `any_datoms_since(t, touched)` check too. It is redundant with SSI for the common case but it is *cheaper as a query shape* than reading full datoms (it returns at most one row per touched eid because of `LIMIT 1` in the implementation; it does NOT skip predicate-lock placement — under SERIALIZABLE every read places SIReadLocks, including this one) (W2-revised: the original "no predicate locks" phrasing was wrong — the cheapness is wall-clock + bandwidth, not lock-set; SIReadLocks ARE placed and that is in fact the load-bearing SSI mechanism). The explicit check runs first; if it says "conflict", we return `False` and let the Txn-side retry loop spin. If it says "no conflict", SSI may still raise 40001 at commit — we catch `psycopg.errors.SerializationFailure` and treat it as a conflict (return `False`) so the same retry loop handles both sources of conflict.

#### 5.2.1 Cross-process atomicity boundary (W1-revised, W2-clarified — `_lock` and `_txn` are distinct)

ARIS R1 W1 flagged that the original "8-line hook" framing under-specified where the multi-process atomicity boundary lives. ARIS R1 W2 (codex round 2) flagged that conflating `_lock` (in-process mutex) with the SERIALIZABLE-transaction context-manager confuses implementers. The W2-clarified contract introduces TWO distinct concepts:

| Concept | InMemoryStore | SQLiteStore | PostgresStore |
|---|---|---|---|
| **`store._lock`** — in-process mutex (existing surface) | `threading.Lock` | `threading.RLock` | `threading.RLock` (NEW — performance-only, see § 8.3) |
| **`store._txn(...)`** — store-level transaction context-manager (NEW additive Protocol method per ADR-15 W2) | yields the in-process datom list (no real transaction) | wraps `BEGIN IMMEDIATE` ... `COMMIT` and yields the cursor | wraps `BEGIN ISOLATION LEVEL SERIALIZABLE` ... `COMMIT` and yields the active `psycopg.Connection`; sets `_current_conn: ContextVar` on entry / clears on exit |

**Authoritative shape:**

- **`store._lock` stays a pure mutex** on every backend. Under PostgresStore it is `threading.RLock` and serialises in-process writers AHEAD of opening the SERIALIZABLE conn (so two threads in one Python process don't both queue on `tx_allocator` `FOR UPDATE` against each other — they queue in-process first, which is cheaper than queueing on a row lock through the network round-trip). § 8.3 covers the in-process serialisation rationale.
- **`store._txn()` is the store-level transaction context-manager** (additive Protocol method per ADR-15 W2). On Postgres it opens `BEGIN ISOLATION LEVEL SERIALIZABLE` and yields the connection; on SQLite it opens `BEGIN IMMEDIATE` and yields the cursor; on InMemory it is effectively a no-op that yields the in-process list. The Txn-side `_commit_attempt` (`transaction.py:604`) is rewritten in PG3 from `with tx.db.store._lock:` to `with tx.db.store._lock, tx.db.store._txn() as _ctx:` — first acquire in-process mutex, then enter the store-level transaction.
- **`store.transact_serializable(facts, provenance, tx_time?) -> int`** is the store-level helper that wraps the full multi-step sequence: enter `_txn()` → `SELECT FOR UPDATE tx_allocator` + `UPDATE next_tx` → optional `SELECT FOR UPDATE audit_chain_lock` (if any datom in `facts` is audit-shaped) → INSERT all datoms → COMMIT (via `_txn.__exit__`). This is the actual atomic primitive Txn binds to. InMemory + SQLite carry a default impl that calls existing `allocate_and_append` + `append` paths inside `_txn()`; behaviour is preserved for those backends.
- **`_current_conn: ContextVar[Connection | None]`** is set by PostgresStore's `_txn()` `__enter__` and cleared by `__exit__`. Any Store method called inside an active `_txn()` (read or write) routes through `_current_conn.get()` to use the active connection. Any Store method called OUTSIDE an active `_txn()` opens its own short-lived autocommit connection (read-only paths only — write paths require an active txn).

**Store Protocol delta (additive, per ADR-15 W2).** Two new methods land in v0.8: `_txn()` (context manager) and `transact_serializable(facts, provenance, tx_time?)`. Both have default implementations on InMemoryStore + SQLiteStore that preserve existing semantics — adapters bound to the original 6 methods continue to work without changes; new code that wants the multi-process atomicity guarantee uses `transact_serializable` (or feature-detects via `hasattr(store, "transact_serializable")`).

**LOC delta on `_commit_attempt`** (revised from W1's "~30 LOC"): the realistic shape is ~50 LOC including the `with store._lock, store._txn() as ctx:` rewrite, the dual-exception catch (`SerializationFailure` 40001 + `UniqueViolation` 23505), and the route-through-`transact_serializable`-when-available branch. Task PG3 estimate (§ 12) holds at ~80 LOC across `transaction.py` + `_audit.py` + `store.py` Protocol extension + `postgres_store.py` override.

```python
# transaction.py — W2-clarified hook (~50 lines)
def _commit_attempt(tx: "Transaction") -> bool:
    ...
    try:
        # Two distinct primitives:
        #   store._lock — in-process mutex (threading.RLock under Postgres)
        #   store._txn() — store-level transaction context manager
        with tx.db.store._lock, tx.db.store._txn() as _ctx:
            # Inside _txn(): on Postgres, _ctx is the active SERIALIZABLE conn;
            # on SQLite, _ctx is a cursor inside BEGIN IMMEDIATE; on InMemory,
            # _ctx is the bound write-list. Store reads/writes from this point
            # use the active context.
            #
            # explicit any_datoms_since check — cheap query shape (LIMIT 1
            # per touched eid); SIReadLocks ARE placed under SERIALIZABLE
            # and that's the load-bearing SSI mechanism for layer 1
            ...
            # transact_batch routes through store.transact_serializable
            # under PostgresStore:
            #   1. SELECT next_tx FROM tx_allocator WHERE id=1 FOR UPDATE
            #   2. UPDATE tx_allocator SET next_tx = next_tx + N
            #   3. (if audit datoms in batch) SELECT FROM audit_chain_lock FOR UPDATE
            #   4. INSERT all datoms (with allocated tx-ids)
            #   5. (if audit datoms) UPDATE audit_chain_lock SET head_*
            #   6. COMMIT — _txn.__exit__ — releases both row locks
            tx.db.transact_batch(facts, provenance=provenance)
        return True
    except SerializationFailure:    # psycopg 40001 — SSI rw-conflict
        return False                # retry loop in _run picks this up unchanged
    except UniqueViolation:         # psycopg 23505 on (tx, e, a) — defence-in-depth
        return False                # treat same as 40001; allocator regression caught
```

The `read_set | ensure_set | write_set` union, the conflict-rejection contract, and the retry loop are all unchanged. What changes is (a) `_lock` and `_txn` are now distinct concepts (W2 rename split), (b) `tx_allocator` becomes the tx-id source, (c) we now catch two retryable exceptions (40001 + 23505).

### 5.3 What `tx.alter` / `tx.commute` / `tx.ensure` look like at the SQL layer

| Txn primitive | What runs at the Postgres layer |
|---|---|
| `tx.deref(ref)` | `SELECT v FROM datom_log WHERE e = $eid AND a = $spec_attr ORDER BY tx DESC LIMIT 1` (snapshot read inside SERIALIZABLE; SIReadLock placed) |
| `tx.assoc(ref, v)` | adds entry to `tx.write_set`; the actual INSERT lands at commit |
| `tx.alter(ref, fn)` | reads via `tx.deref` (SIReadLock placed); fn runs in Python; writes deferred to commit |
| `tx.ensure(ref)` | `SELECT v FROM datom_log WHERE e = $eid AND a = $spec_attr ORDER BY tx DESC LIMIT 1` (same shape as deref — places the same SIReadLock; the txn-side `ensure_set` is the local mirror) |
| `tx.commute(ref, fn_id, args)` | adds entry to `tx.commute_log`; **does NOT place a predicate lock on the commute target** — that is the entire point of commute (v0.5.2 § F3). Reapply at commit reads happen inside the active `store._txn()` (after the in-process `_lock` is held and after the explicit `any_datoms_since` conflict check passes) |
| commit | `BEGIN ISOLATION LEVEL SERIALIZABLE; SELECT next_tx FROM tx_allocator WHERE id=1 FOR UPDATE; UPDATE tx_allocator SET next_tx = next_tx + N; SELECT 1 FROM datom_log WHERE tx_time > $t_start AND e = ANY($touched_eids); INSERT INTO datom_log VALUES ...; COMMIT;` (W1-revised — `tx_time` not `tx`; `tx_allocator` lock acquired before the conflict check) |

**`tx.commute` requires NOT placing a predicate lock on the commute target.** This is achieved by *not* issuing a `SELECT` against the commute-target eid inside the `tx.commute` call — the body-eager value comes from `tx.commute_log` state, the reapply at commit issues the SELECT inside the active `store._txn()` (post-`_lock` + post-conflict-check). ADR-6 pins this.

### 5.4 SSI false-positive rate

SERIALIZABLE retries on rw-dependency cycles even when the application logic would have committed correctly. Empirically the false-positive rate is workload-dependent; for our workload (low-thousands of datoms / hr / writer at the coding-agent MVP scale) the expected retry rate is < 1% based on community benchmarks. ARIS R2 measures this against the cross-process Hypothesis property and budgets `max_retries=256` (mirrors v0.5.2's `dosync` retry budget) so retry exhaustion is a real bug, not a hot-loop.

Statement-level retry (Postgres's natural error-then-retry) does NOT preserve user-level transaction semantics for our case — the *whole* `dosync` body must replay including any Python-side fn calls. This matches Postgres's own guidance (the docs explicitly say automatic retry cannot be done with correctness — § 13.5 PG18 docs). The retry lives in `Transaction._run`, not in psycopg.

## 6. Audit-chain Merkle continuity

### 6.1 The invariant

Every audit datom carries `provenance["parent_provenance_hash"]` pointing to the immediately-prior audit datom's hash, forming a Merkle chain. Across multi-process inserts, this means: if two processes both commit audit datoms at "the same instant," exactly one chain order must be observable. Replay byte-identity then re-derives that exact same order from the persisted log.

### 6.2 The mechanism

`audit_chain_lock` (§ 4.2) is a single-row table with `id = 1` invariant. Every commit-attempt that emits an audit datom takes `SELECT * FROM audit_chain_lock WHERE id = 1 FOR UPDATE` BEFORE building the audit datom. This blocks any concurrent audit-emitting transaction at the row lock until the holder commits or rolls back. The holder reads `head_hash` from the row, builds the audit datom with `parent_provenance_hash = head_hash`, INSERTs the audit datom, and `UPDATE audit_chain_lock SET head_hash = $new_hash, head_tx = $new_tx, head_seq = $new_seq WHERE id = 1`. The COMMIT releases the row lock.

ADR-3 pins this choice over four alternatives:

| Alternative | Why rejected |
|---|---|
| `pg_advisory_xact_lock(<chain_id>)` | Works for single-key serialisation but does not give a typed `head_hash` to read; we'd still need a row to hold the hash |
| Postgres LSN ordering | LSN orders writes globally but is not exposed as a transaction-rendezvous primitive; would need a trigger to emit LSN into the row, adding a write per audit datom |
| Table-level `LOCK TABLE datom_log IN ACCESS EXCLUSIVE` | Blocks all reads + writes; catastrophic for throughput |
| Per-chain advisory lock + auxiliary lookup | Same shape as ADR-3 but split across two primitives — more moving parts, no benefit |

The single-row `FOR UPDATE` is the simplest correct shape and gives us exactly the semantics we need. The cost is one `SELECT FOR UPDATE` + one `UPDATE` per audit-emitting transaction — microseconds.

### 6.3 Replay byte-identity

`replay_byte_identity` (existing in `persistence.replay`) walks `db.log()` in `seq` order and re-derives the audit chain. The cross-process invariant: if process A and process B both committed audit datoms at "the same instant," the persisted `seq` order is the order the `audit_chain_lock` was acquired — process A's audit datom has `parent_provenance_hash = OLD_head_hash`, process B's audit datom has `parent_provenance_hash = A's_audit_hash`, and replay re-derives both correctly. ADR-3 makes this falsifiable: the cross-process Hypothesis property (§ 8) asserts byte-identical replay across N concurrent writers.

### 6.4 What is NOT load-bearing on the lock row

- The `head_hash` column on `audit_chain_lock` is a **performance optimisation** (avoids a `SELECT ... ORDER BY seq DESC LIMIT 1` against `datom_log` for each audit-emit). It is **not** the source of truth — the source of truth is the `datom_log` itself. If the lock row's `head_hash` ever drifts from the actual log tip (would require a bug), `verify_chain()` (existing audit handler invariant) catches it.
- `head_tx` and `head_seq` are stored for the same performance reason, plus they make the chain's tip findable in O(1) for the SDK MCP `audit_tail` resource.

## 7. `any_datoms_since(t)` MVCC primitive

### 7.1 Today's shape

`InMemoryStore.since(t)` and `SQLiteStore.since(t)` both yield datoms with `tx_time > t`, ordered by insertion (`seq` in SQLite, list-position in InMemory). The Txn-side `any_datoms_since(t, touched)` reads `db.store.since(t_start)` filtered by `eid in touched`.

### 7.2 Postgres mapping — ADR-8

Three candidate primitives:

| Candidate | Why considered | Why rejected (or chosen) |
|---|---|---|
| `xmin` / `xmax` system columns | Postgres-native MVCC bookkeeping | NOT serialisable across `pg_dump`/`pg_restore` cycles (txid wraparound, vacuum) — replay against a restored DB would diverge |
| `pg_logical_emit_message` + LSN | Cross-backend ordering primitive | Requires logical-replication slot for any reader; adds infra dependency |
| `(tx_time, tx)` strict-monotonic projection | Already in the table; matches SQLite | **Chosen.** No new infra. Replay byte-identity preserved trivially because the same tuple lives in both backends. |

ADR-8 pins `(tx_time, tx)` ordered by `idx_datom_log_txtime` as the MVCC primitive. The implementation is one query:

```sql
SELECT 1 FROM datom_log
 WHERE tx_time > $t_start AND e = ANY($touched_eids)
 LIMIT 1;
```

Inside the SERIALIZABLE transaction, this places SIReadLocks on the matching rows; any concurrent writer that would invalidate the read causes the *other* transaction to fail at commit with 40001.

### 7.3 Why not also rely on `xmin`?

`xmin` is the Postgres-native answer to "did this row exist as of my snapshot," but it is a per-row, per-snapshot, per-vacuum-state value. Two issues: (a) replay against a restored DB needs the same `xmin` ordering — `xmin` doesn't survive `pg_dump`/`pg_restore`; (b) `xmin` is a 32-bit txid that wraps around, requiring `pg_xact` lookups for ordering across freezing — fragile. `(tx_time, tx)` is a substrate-owned monotonic projection and is the safe choice.

## 8. `allocate_and_append` semantics

### 8.1 Single-process

`InMemoryStore` holds `_lock` (in-process mutex); `SQLiteStore` holds `_lock` + `BEGIN IMMEDIATE` writer-lock inside `_txn()`. Both are correct within one process.

### 8.2 Multi-process via PostgresStore — ADR-9 (W1-revised, W2-clarified)

The atomic primitive is the SERIALIZABLE transaction (`store._txn()`) PLUS the `tx_allocator` row lock (§ 4.2a) for the tx-id source. ARIS R1 W1 retracted the original `MAX(tx)+1` design — see ADR-4 (W1-revised) for the upgrade rationale. ARIS R1 W2 (round 2) clarified that `_lock` and `_txn()` are distinct concepts (§ 5.2.1 W2-clarified). The new shape:

```python
def allocate_and_append(self, datoms: Iterable[Datom]) -> list[Datom]:
    materialised = list(datoms)
    if not materialised:
        return []
    n = len(materialised)
    # Two distinct primitives: in-process mutex + store-level transaction.
    with self._lock:                          # threading.RLock — in-process serialisation (§ 8.3)
        with self._txn() as conn:             # BEGIN ISOLATION LEVEL SERIALIZABLE; yields psycopg.Connection
            # Hard tx allocation primitive — row lock on tx_allocator
            cur = conn.execute(
                "SELECT next_tx FROM tx_allocator WHERE id = 1 FOR UPDATE"
            )
            next_tx = int(cur.fetchone()[0])
            conn.execute(
                "UPDATE tx_allocator SET next_tx = next_tx + %s WHERE id = 1",
                (n,),
            )
            # Caller now owns tx-ids [next_tx .. next_tx + n - 1]
            out = [_with_tx(d, next_tx) for d in materialised]  # batch shares one tx
            conn.executemany(
                "INSERT INTO datom_log (e, a, v, tx, ...) VALUES (...)",
                [_encode(d) for d in out],
            )
            # COMMIT happens at _txn().__exit__; releases tx_allocator row lock.
    return out
```

**Why this is now a hard correctness primitive.** Two writers that race the allocator both queue on the `FOR UPDATE` row lock; the first acquires it, advances `next_tx`, INSERTs its datoms with `tx=N`, and COMMITs (releasing the row lock). The second writer's `SELECT FOR UPDATE` was queued; it now sees `next_tx=N+1` (the committed value), advances it, INSERTs with `tx=N+1`, COMMITs. The two writers' tx-ids are guaranteed disjoint by Postgres row-lock semantics — independent of SSI predicate-lock placement and independent of the planner's choice for `MAX()`.

**Belt-and-suspenders: `UNIQUE (tx, e, a)`.** If a future code-path bypasses `allocate_and_append` and INSERTs into `datom_log` without first locking `tx_allocator`, the unique constraint converts the bug into a visible `psycopg.errors.UniqueViolation` (SQLSTATE 23505). The `_commit_attempt` retry loop treats this identically to 40001 (per § 5.2.1).

ADR-9 (W1-revised) pins this over alternatives:

| Alternative | Why rejected |
|---|---|
| `pg_advisory_xact_lock` on a global "tx-allocator" key | Works but serialises ALL writers — defeats SSI's point; same-row contention shape as a single mutex |
| `BIGSERIAL tx` column with `RETURNING tx` | Tx-ids drift after rollback (sequence advance survives rollback); not deterministic; replay-byte-identity-incompatible |
| `INSERT ... SELECT COALESCE(MAX(tx), 0) + 1 ... RETURNING tx` in one statement | Same shape; less flexible if we later want to validate tx-id pre-insert |
| `uuid7` for `tx` | Drops monotonicity; breaks `since(t)` ordering invariant |

### 8.3 In-process `_lock` retained

`PostgresStore._lock` (a `threading.RLock`) stays — it serialises in-process writers so they don't all queue on the network round-trip to `tx_allocator FOR UPDATE` against each other (which would force same-process retries via SSI predicate-lock contention on the allocator row). The in-process mutex collapses N in-process writers down to a single allocator-row contender, after which the cross-process row lock handles the actual cross-process serialisation. The lock is purely a performance optimisation; correctness is on the `tx_allocator` row lock + SERIALIZABLE barrier (W2-clarified — `_lock` is distinct from `_txn()`; see § 5.2.1).

LISTEN/NOTIFY is the Postgres-internal transport; `notifications/resources/updated` is the wire-protocol the LLM client actually sees — the layering is explicit so a Postgres operator inspecting `pg_stat_activity` sees the LISTEN session, while an MCP client SDK sees its conformant subscription. Single-consumer in v0.8; multi-consumer ack semantics (e.g. multiple parallel MCP clients each subscribed to `audit_tail`) is the v0.9 pgmq evaluation venue. The trigger ships in the Postgres migration (PG1); the LISTEN consumer + MCP relay live in the Adapter SDK SDK3, not this stream. |

### Summary of extension decisions

- **Use (in v0.8 deploy guide):** `pg_stat_statements`, native LISTEN/NOTIFY (for SDK MCP audit_tail).
- **Compose (opt-in, no code dep):** `pgaudit` (recommended for SOC2-style operator audit; documented in deployment guide).
- **Defer to v0.9:** `pgvector` (recall semantics expansion), `pglogical` / native-logical-replication (multi-region), `timescaledb` (retention/compression at scale), `pg_partman` (declarative partitioning), `pgmq` (multi-consumer audit-tail), server-side `pgcrypto` validation trigger.
- **Reject:** `temporal_tables`, `periods` (bitemporal already ours at datom layer), `pg_cron` (orchestration above substrate), `Citus` (sharding violates replay invariant).

## 10. ADRs

### ADR-1: Single `datom_log` table (no split into datoms / audit / commits)

**Decision.** Postgres backend uses one `datom_log` table for all datom kinds — user writes, txn commit datoms, audit-handler datoms, MCTS datoms, REPL datoms, MCP datoms. No table split.

**Rationale.** Audit datoms ARE datoms — they have the same 8-tuple shape, the same provenance dict, the same `invalidated_by` semantics. Splitting them out forces every replay path to UNION ALL across N tables with a hand-maintained ordering invariant on top, breaks `db.log()`'s cross-backend identity, and makes RLS hard to author. Single-table keeps the byte-identity invariant trivial.

**Consequence.** All indexes are over `datom_log`; the audit-chain Merkle continuity uses a separate `audit_chain_lock` table for the head pointer (one row), not a separate audit table.

### ADR-2: Backend-flavoured migration files; `load_migrations(backend)` dispatches

**Decision.** Existing `0001_datom_log.sql` is renamed `0001_datom_log.sqlite.sql`. NEW `0001_datom_log.postgres.sql` ships alongside. `load_migrations(backend)` filters by suffix.

**Rationale.** SQLite uses `INTEGER PRIMARY KEY AUTOINCREMENT` + ISO-8601 TEXT timestamps; Postgres uses `BIGSERIAL` + TIMESTAMPTZ + JSONB. The two cannot share one SQL file; a single file with `IF` conditionals would make schema review brittle. Per-backend files are explicit and reviewable.

**Consequence.** `_apply_migrations(backend)` becomes `for name, sql in load_migrations(backend): ...`; each store passes its own backend flavour. The InMemoryStore needs no migrations. Future backends (Cloudflare D1, DuckDB, etc.) drop in their own `0001_*.<flavour>.sql` without touching the existing two.

### ADR-3: Audit-chain ordering uses a single-row `audit_chain_lock` + `SELECT FOR UPDATE`

**Decision.** Multi-process audit-chain Merkle continuity is enforced by a one-row `audit_chain_lock(id=1, head_hash, head_tx, head_seq)` table, locked via `SELECT * FROM audit_chain_lock WHERE id = 1 FOR UPDATE` BEFORE building any audit datom needing `prev_audit_hash`.

**Rationale.** § 6.2 enumerates four alternatives (`pg_advisory_xact_lock`, LSN ordering, table-level lock, advisory + auxiliary lookup). The single-row `FOR UPDATE` is the simplest correct shape: typed `head_hash` for O(1) tip lookup, automatic release at COMMIT/ROLLBACK, predicate-lock-free (no SSI bookkeeping).

**Consequence.** Every audit-emitting transaction carries one extra `SELECT FOR UPDATE` + one `UPDATE audit_chain_lock`. Microseconds. The `head_hash` column is a performance optimisation; truth lives in `datom_log` and `verify_chain()` re-derives it.

### ADR-4: `tx` allocation uses a `tx_allocator` row + `SELECT FOR UPDATE`, with `UNIQUE (tx, e, a)` defence-in-depth (W1-revised)

**Decision.** The `tx` column is `BIGINT` not `BIGSERIAL`. Allocation is **`SELECT next_tx FROM tx_allocator WHERE id=1 FOR UPDATE` followed by `UPDATE tx_allocator SET next_tx = next_tx + N`** inside the SERIALIZABLE transaction (§ 4.2a). The `datom_log` table carries `UNIQUE (tx, e, a)` as a defence-in-depth safety net (§ 4.1). The `seq BIGSERIAL` column keeps physical insertion-order separately.

**Rationale.** ARIS R1 W1 (codex gpt-5.4 xhigh) flagged the original `MAX(tx)+1` design as too implicit: SSI's predicate-lock placement on a `MAX()` aggregate depends on planner choices and PG version; relying on it as the *only* correctness primitive is brittle. The W1-revised design layers (a) a hard row-lock primitive on a single allocator row that gives plan-independent serialisation, (b) a hard schema-level unique constraint that converts any allocator regression into a visible `UniqueViolation` (23505) rather than silent same-tx duplication. `BIGSERIAL` was rejected because it advances on rollback (sequence side-effect outlives the transaction); replay byte-identity demands deterministic tx-ids per op-sequence.

**Consequence.** Two concurrent writers queue on the `tx_allocator` `FOR UPDATE` row lock; tx-ids are guaranteed disjoint by Postgres row-lock semantics, independent of planner choices. The `_commit_attempt` retry loop catches both `SerializationFailure` (40001 — SSI rw-conflict on the touched-eid layer) AND `UniqueViolation` (23505 — defence-in-depth catch on `(tx, e, a)`); both return `False` and trigger the existing v0.5.2 `max_retries=256` retry loop. False-positive rate measured at ARIS R2 (R1 risk).

### ADR-5: `v` and `provenance` stored as canonical TEXT, not JSONB (W1-revised)

**Decision.** The `v` and `provenance` columns are `TEXT` on Postgres (canonical JSON via `json.dumps(..., sort_keys=True)` — exactly what SQLite already stores). NOT `JSONB`. AVET / VAET / AEVT indexes are plain BTREE over the TEXT column. Operator-side JSONB containment queries are served by an opt-in `0002_optional_jsonb_columns.postgres.sql` migration that adds parallel `v_jsonb` / `provenance_jsonb` columns (populated via `UPDATE`) plus GIN indexes; the Store Protocol never reads these.

**Rationale.** ARIS R1 W1 (codex gpt-5.4 xhigh) flagged the original "JSONB + BTREE on `(v::text)`" design as unsafe: `jsonb::text` is NOT a stable byte stream (whitespace + key-order normalisation differ across PG versions; `jsonb_set` / `||` operations can rewrite representation between INSERT and SELECT). Storing `v` as TEXT removes the canonicalisation question entirely — the bytes that go in are the bytes that come out, byte-identical to SQLite. The `_encode` codec runs the same on both backends; no `(v::text)` cast in any index expression.

**Consequence.** v0.8 native containment queries (`v @> '{...}'`) are not supported on the core column. The opt-in migration is the operator-side path. R4 risk (codec drift) drops from "high" to "low" — the TEXT column eliminates the JSONB-rendering surface that drove the original concern. Cross-backend `since(t)` byte-identity (G6) is now trivially preserved.

### ADR-6: `tx.commute` does NOT place predicate locks at body call site

**Decision.** `tx.commute(ref, fn_id, args)` adds an entry to `tx.commute_log` and does NOT issue a `SELECT` against the commute target. The reapply at commit reads under `store._lock` AFTER the SERIALIZABLE conflict check passes.

**Rationale.** Placing a predicate lock on the commute target would force SSI to flag two parallel commutes on the same ref as a rw-conflict — but commutativity is the entire point of `tx.commute` (v0.5.2 § F3: "two parallel transactions calling `tx.commute(counter, "inc-by", 1)` both succeed with no retry"). SSI without a SELECT on the target stays silent; reapply-under-lock at commit serialises the actual write.

**Consequence.** Replay byte-identity holds because the commute log is in the commit-datom provenance (`:persistence.txn/commute-log`); replay reads the commute log + applies fn-id against the snapshot value at the *replay's* commit time, getting the same result as the original commit. v0.5.2's commute-replay test extends to PostgresStore unchanged.

### ADR-7: Audit hashing stays Python-side; pgcrypto is NOT used for v0.8 chain integrity

**Decision.** `prev_audit_hash` is computed in Python via `hashlib.sha256` (existing `make_audit_handler` codepath). Postgres-native pgcrypto is NOT used in v0.8.

**Rationale.** (a) Replay byte-identity demands one canonical hash function across backends; SQLite has no pgcrypto, so a Postgres-only hash diverges. (b) Chain integrity is verified by `verify_chain()` which walks `db.log()` in `seq` order — a Python-side hash gives that walker authority over Python-side replay. (c) Server-side hashing forces a server-side trust boundary we don't need yet.

**Consequence.** v0.9 may add a Postgres `BEFORE INSERT` trigger that re-hashes server-side and rejects mismatches as defence-in-depth; that's additive, not a replacement. v0.8 doesn't ship it.

### ADR-8: `(tx_time, tx)` is the MVCC primitive for `since(t)` and `any_datoms_since(t)`

**Decision.** `Store.since(t)` queries `WHERE tx_time > $t ORDER BY seq ASC`; `any_datoms_since(t, touched)` adds `AND e = ANY($touched)`. The strict-monotonic projection is `(tx_time, tx)`; `seq` is the physical insertion-order tiebreaker.

**Rationale.** § 7.2 enumerates `xmin/xmax` and `pg_logical_emit_message`. Both fail the cross-backend determinism + restore-survival tests. `(tx_time, tx)` is substrate-owned, survives `pg_dump`/`pg_restore`, matches SQLite trivially, and is indexed.

**Consequence.** No new infra. Cross-process `any_datoms_since` is a one-query operation. The SERIALIZABLE barrier + SIReadLocks give us the consistency guarantee — no additional Python-side coordination needed.

### ADR-9: `allocate_and_append` runs inside `_txn()` SERIALIZABLE with the `tx_allocator` row-lock primitive (W1-revised, W2-clarified)

**Decision.** `PostgresStore.allocate_and_append` enters two distinct primitives in order: (1) `with self._lock` — `threading.RLock` in-process mutex; (2) `with self._txn() as conn` — store-level SERIALIZABLE transaction context-manager (`BEGIN ISOLATION LEVEL SERIALIZABLE`); then takes `SELECT next_tx FROM tx_allocator WHERE id=1 FOR UPDATE`, advances `next_tx` by N, executemany-INSERTs the datoms with the allocated tx-ids, exits `_txn()` (which COMMITs and releases the row lock). The `_lock` mutex is the in-process serialiser; `_txn()` is the SERIALIZABLE-transaction-opener — these are **distinct concepts** per ADR-15 W2 (W2-clarified: the W1 framing of "`_lock` IS the SERIALIZABLE-transaction-opener" was retracted in W2; W3 propagates the rename to ADR-9).

**Rationale.** ARIS R1 W1 retracted the original `MAX(tx)+1` story (see ADR-4 W1-revised). The row-lock primitive on `tx_allocator` is plan-independent and gives a hard correctness primitive that does not depend on SSI predicate-lock placement on aggregates. ARIS R1 W2 (round 2) flagged that conflating `_lock` with the SERIALIZABLE-transaction context-manager confuses implementers; ADR-15 W2 introduces `_txn()` as a separate Protocol method. § 8.2 enumerates `pg_advisory_xact_lock`, `BIGSERIAL RETURNING`, `MAX(tx)+1`, uuid7 alternatives — all rejected for determinism / plan-dependence reasons.

**Consequence.** Two retryable exceptions surface from `_txn()` `__exit__` (the SERIALIZABLE COMMIT): `SerializationFailure` (40001, SSI rw-conflict on the touched-eids predicate-lock layer) and `UniqueViolation` (23505, defence-in-depth on `UNIQUE (tx, e, a)` if any future code-path bypasses the allocator). The Txn-side `_commit_attempt` (~50 LOC hook per § 5.2.1 W2-clarified — NOT 8 LOC, NOT 30 LOC, the W3 honest estimate is ~50 LOC including the `with _lock, _txn() as ctx:` rewrite + dual-exception catch + `transact_serializable` route-through) catches both and returns `False`; the existing v0.5.2 retry loop retries up to `max_retries=256`.

### ADR-10: psycopg 3 + psycopg-pool; opt-in `[postgres]` extra; lazy import

**Decision.** PostgresStore uses psycopg 3 (`>= 3.2`) with `psycopg_pool.ConnectionPool`. The dependency is in a `[postgres]` optional-dependency extra (`pip install "persistence[postgres]"`). The module imports psycopg lazily inside `__init__` so the substrate is importable without psycopg.

**Rationale.** (a) psycopg 3 is the modern Postgres driver for Python; native asyncio support (psycopg-pool 3.x) lets us extend to async paths in v0.9 without driver swap. (b) Optional-dependency keeps the substrate small for users who only run InMemory or SQLite. (c) Lazy import lets the SDK URI dispatcher (Adapter SDK ADR-9) raise a clean `BackendNotInstalled` error rather than an obscure ImportError.

**Consequence.** `pip install persistence` works without Postgres; `pip install "persistence[postgres]"` enables it. CI matrix gains a `[postgres]` extras job that runs against a real Postgres via `testcontainers`.

### ADR-11: Migration runner uses raw SQL files; NO Alembic

**Decision.** Migrations are a sorted list of `.sql` files loaded by `load_migrations(backend)` and applied via `executescript` (SQLite) / `cursor.execute` (Postgres). No Alembic, no SQLAlchemy.

**Rationale.** (a) Substrate already has zero ORM; adding Alembic for one backend introduces a new dependency tree (SQLAlchemy + Mako) for a runner that only needs `apply N forward-only SQL files`. (b) Forward-only is sufficient for v0.8; multi-step rollbacks are a v1.0+ concern when schema evolves with breaking changes. (c) Raw SQL files are what operators read for SOC2 audit; no DSL between them and the schema.

**Consequence.** Migrations are append-only. Any breaking change to the schema is a new `0002_*.sql` with an explicit upgrade path documented. Operator-driven rollbacks ship a documented `*_rollback.sql` (per existing project pattern: see InfraFlow `008_rollback.sql`); not auto-applied.

### ADR-12: RLS-ready, RLS-not-authored (W1-revised cast note)

**Decision.** `datom_log` rows carry an implicit tenant via `(provenance::jsonb)->>'tenant_id'` — note the explicit JSONB cast since `provenance` is stored as TEXT (ADR-5 W1-revised). v0.8 adds NO RLS policy. The schema is *RLS-ready*: an operator can `ALTER TABLE datom_log ENABLE ROW LEVEL SECURITY` and author a policy on the cast expression. The substrate does not enforce one. Operators on the v0.9 substrate-tenant timeline who want sub-microsecond RLS-policy-evaluation cost may run the opt-in `0002_optional_jsonb_columns.postgres.sql` migration which adds a parallel `provenance_jsonb` column (with GIN index) and pivot RLS authoring to that column.

**Rationale.** v0.8 ships single-tenant dev/MVP deploys. Two distinct timelines bear on the multi-tenant story and must NOT be conflated: (a) **substrate-level tenant model** — a `provenance->>'tenant_id'` convention promoted to a first-class `Substrate.open(tenant=...)` parameter with cap-token tenant-scoping — lands in **v0.9** as additive surface; (b) **Juba OS multi-tenant deploy** that authors RLS policies against (a)'s tenant model — lands in **Phase 4** (post-incorporation, product-side) once a tenant model exists to author against. Authoring policies now would invent a tenant model the substrate doesn't know how to use; staying RLS-ready means adopting tenants later is additive in both layers.

**Consequence.** Multi-tenant Juba OS (Phase 4 timeline) ships its own `0003_tenant_rls.postgres.sql` migration when needed, against the v0.9 substrate-level tenant model. The substrate documentation (`postgres-deployment-v0.8.md`) flags this as a layered v0.9+/Phase-4 concern with a worked example showing both the substrate-level `tenant=` pattern and the RLS policy authored on top, plus the cast-vs-parallel-column trade-off for hot-path RLS evaluation.

### ADR-13: Audit datom persistence routes through `transact_serializable`, NOT through `db.store.append` (W1 NEW)

**Decision.** `persist_repl_audit` (`src/persistence/repl/_audit.py:131-195`) is rewritten in PG3 (Task PG3 scope expanded — see § 12) to route audit-datom INSERTs through `db.store.transact_serializable([audit_datom], provenance=...)` instead of `db.store.append([audit_datom])` after a private `_allocate_next_tx` log scan. The full sequence under PostgresStore: open SERIALIZABLE conn → `SELECT FOR UPDATE tx_allocator` → `UPDATE next_tx` → `SELECT FOR UPDATE audit_chain_lock` → build audit datom with `prev_audit_hash=head_hash` → INSERT → `UPDATE audit_chain_lock SET head_hash, head_tx, head_seq` → COMMIT.

**Rationale.** ARIS R1 W4 (codex gpt-5.4 xhigh) flagged that the existing `persist_repl_audit` allocates `tx` via `_allocate_next_tx` (full `db.log()` scan + `max+1`) and routes through `db.store.append`, NOT `allocate_and_append`. This is the SAME pattern (`MAX(tx)+1` without an atomic allocator) that ADR-4 W1-revised retracts as unsafe under multi-process. Without an explicit migration, the audit chain breaks on day one of multi-process deploy: two writers both compute `max_tx+1=N`, both append audit datoms with `tx=N`, the `UNIQUE (tx, e, a)` constraint fires for one (defence-in-depth catch), but the in-memory `_audit_chain_state` (Python-side) has already advanced for both — the chain head pointer in `audit_chain_lock` and the in-Python state diverge.

**Why preserve the audit handler's `recorded_at`.** The current rationale for bypassing `allocate_and_append` was: `db.transact` would re-stamp `tx_time` from the DB clock, losing the audit handler's own `recorded_at` instant. PG3 preserves this by giving `transact_serializable` an explicit per-datom `tx_time` parameter that bypasses the DB-clock stamping; the audit handler passes its `recorded_at` through. The Stream A G2 gate's seeder pattern (`tests/plan/test_promotion_g1_g2.py:329-342`) — which also wanted `tx_time` preservation — is migrated in the same PG3 task to use `transact_serializable` with `tx_time=` rather than direct `store.append`.

**What "audit-emitting transaction" means precisely.** A datom whose `a` (after lstripping the leading `:`) starts with `audit/` (covers `audit/repl.op`, `audit/mcts.op`, `audit/skill.op`, etc.). The `transact_serializable` helper inspects the batch and takes the `audit_chain_lock` `FOR UPDATE` only when at least one datom in the batch is audit-shaped. Non-audit batches skip the audit-chain lock acquisition (no contention).

**Consequence.** Task PG3 LOC estimate revised from `~20` to `~80` LOC (§ 12). The migration of `persist_repl_audit` is a real refactor, not a one-line catch. The existing in-Python `_audit_chain_state` becomes a *cache* of the `audit_chain_lock.head_hash` row, refreshed on every `persist_repl_audit` call (cheap — the SELECT FOR UPDATE returns the same row that holds the cached value).

### ADR-14: G3 cross-process replay byte-identity is recast as a multi-invariant gate (W1 NEW, W2-clarified G3e)

**Decision.** G3 is recast (§ 11 below) from "byte-identical to a single-process baseline that processed the same op-sequences serially in the same order" — which is not falsifiable since SERIALIZABLE doesn't guarantee a *specific* serial order — to a multi-invariant gate that captures what SERIALIZABLE actually promises:

- **G3a** (replay-determinism): replay the persisted log against an empty store; the resulting projection (filtered per N6/N7) is byte-identical when replayed twice. The log replays to itself deterministically.
- **G3b** (no-same-tx): zero rows in `datom_log` share `(tx, e, a)` after N concurrent writers complete (UNIQUE constraint trivially enforces this).
- **G3c** (audit-chain-continuity): `verify_chain()` returns True for the audit-shaped projection of the persisted log walked in `seq` order.
- **G3d** (sum-of-ops): non-audit, non-commit datom count equals the sum of per-process op counts.
- **G3e** (some-serial-order-exists, W4-rewritten — fixes vacuous-cycle bug): the rw-conflict graph reconstructed from persisted commit provenance (`:persistence.txn/read-set` and `:persistence.txn/ensure-set` from `_build_commit_provenance` `transaction.py:541-542`, `:persistence.txn/started-at` and `:persistence.txn/committed-at` `transaction.py:535-536`, plus the **reconstructed** write-set computed as `{d.e for d in datom_log if d.tx == C.tx and not d.a.startswith('persistence.txn/') and not d.a.startswith('audit/')}`) contains no cycle over concurrent-commit pairs. SSI promises an acyclic rw-graph; the gate verifies the persisted log actually has the property. The W4-corrected algorithm restricts edges to concurrent-commit pairs (overlapping `[t_start, committed_at]` intervals) and removes the `C_b.committed_at > C_a.committed_at` constraint that W3 had — that constraint forced edges to follow commit-time order and made cycles structurally impossible (round-4 codex caught this). With the constraint removed, two concurrent commits CAN have rw-edges in both directions — the load-bearing 2-cycle that SSI is supposed to prevent.

**Rationale.** ARIS R1 W3 (round 1) flagged that comparing concurrent-execution output to a "single-process baseline" is fundamentally not falsifiable. ARIS R1 W10 (round 2) flagged G3e's first cut was underspecified. ARIS R1 round 3 caught the field-mismatch error (the W2 framing referenced non-persisted `write-set` + `t-start`). ARIS R1 round 4 caught the algorithmic error (the W3 cycle-edge condition forced commit-time ordering and made cycles structurally impossible). The W4-rewritten G3e: (a) uses actually-persisted fields, (b) reconstructs write-set from same-`tx`-group filtered datoms, (c) restricts edges to concurrent-commit pairs only, (d) removes the spurious `committed_at` ordering constraint, (e) ships an explicit READ-COMMITTED-fallback proof-of-falsifiability test in PG4 to demonstrate the gate fails when SSI is bypassed. See § 11 G3e for the corrected algorithm.

**Consequence.** G3 ships as 5 sub-gates, each Hypothesis-property-falsifiable at `max_examples=50`. Task PG4 asserts each sub-gate independently and ships `test_g3e_falsifiability.py` (the proof-of-falsifiability harness — explicitly demonstrates G3e fails under READ COMMITTED and passes under SERIALIZABLE). The single "byte-identical to a single-process baseline" claim is removed; the rw-cycle check uses only persisted fields that exist in code today.

### ADR-15: `Store` Protocol grows two additive methods in v0.8 — `_txn()` and `transact_serializable` (W2 NEW)

**Decision.** The `Store` Protocol (`src/persistence/fact/store.py:62-127`) gains two additive methods in v0.8: `_txn()` (store-level transaction context-manager) and `transact_serializable(facts, provenance, tx_time?) -> int` (atomic primitive for cross-process correctness). Both methods carry default implementations on InMemoryStore and SQLiteStore that preserve their existing transactional semantics — no behavioural change for those backends. PostgresStore overrides both with the SERIALIZABLE-transaction + `tx_allocator`-row-lock + `audit_chain_lock`-row-lock shape.

**Rationale.** ARIS R1 W2 (round 2) flagged that the W1 fix-pass simultaneously claimed "Store Protocol unchanged" (§ 1) and "PG3 adds `transact_serializable` to the Protocol" (§ 12). Two contradictory framings cannot both be correct; W2 picks "additive Protocol extension" because:
- (a) it makes the contract explicit for adapters (anyone binding to the Protocol now knows the new methods exist and can choose to use them);
- (b) feature-detection via `hasattr(store, "transact_serializable")` would let backends silently skip multi-process correctness, which is the wrong default;
- (c) cross-backend default impls preserve existing semantics, so the additivity is non-breaking.

Naming: `_lock` (in-process mutex) and `_txn()` (store-level transaction context-manager) are now distinct concepts (W2 rename split per § 5.2.1). The original draft conflated the two; the W2-clarified contract is implementable without ambiguity.

**Consequence.** § 1 "What this stream preserves" is W2-revised to call out the additive Protocol extension explicitly (the original "shape unchanged — adds no new methods" line is retracted). Adapters bound to the original 6 methods continue to work; new code that wants multi-process atomicity opts into `transact_serializable`. The Adapter SDK's `@stable("v0.8")` decorator (Adapter SDK ADR-5 + ADR-16) covers the original 6 methods unchanged; the new methods are decorated `@stable("v0.8")` as well so adapters can pin against them.

### ADR-16: Codec consolidates to a `DatomCodec` ABC; SQLite + Postgres share the canonical implementation but keep distinct datetime wire shapes (PG2 NEW)

**Decision.** A new module `src/persistence/store/_codec.py` ships a `DatomCodec` ABC plus two concrete implementations: `TextDatomCodec` (datetimes serialised as ISO-8601 TEXT — used by SQLiteStore + the InMemoryStore optional round-trip) and `NativeDatomCodec` (datetimes pass through to psycopg 3's TIMESTAMPTZ adapter unchanged — used by PostgresStore). Both codecs share canonical-JSON encoding for `v` / `provenance` (`json.dumps(..., sort_keys=True, default=str)`) and the `with_tx` placeholder-rewrite primitive. `persistence.fact.store._encode` / `_decode` / `_with_tx` and `persistence.store.postgres._encode` / `_decode_tuple` / `_with_tx` are reduced to thin module-level shims that delegate into the canonical codec — preserving every existing import surface unchanged.

**Rationale.** PG1 (commit `d911270`) shipped a local copy of the codec inside `postgres.py` so the module would stay importable on machines without `sqlite3` (the previous codec's home was `fact/store.py`, which imports `sqlite3` unconditionally at module import). That decision was correct for PG1's "ship the SERIALIZABLE backbone" scope but left two structurally-different copies of the codec that were drifting — the `_with_tx` placeholder rewrite, in particular, is a load-bearing primitive (`{"superseded_by_tx": TX_PLACEHOLDER}` rewrite at allocation time) and having two implementations is a bug-magnet.

The W2-deferred consolidation question was: *should the consolidation also unify datetime serialisation?* Two options:

| Option | Wire shape on Postgres | Implication |
|---|---|---|
| (A) ISO-string everywhere | `tx_time TEXT` (drop TIMESTAMPTZ) OR `.isoformat()` round-trip outside psycopg's adapter | breaks ADR-8's `idx_datom_log_txtime` efficiency on `(tx_time, tx)` (every `since(t)` query casts string→timestamp); microseconds-around-DST risk if we round-trip outside psycopg |
| (B) Per-backend datetime shape | SQLite ISO-string, Postgres native TIMESTAMPTZ via psycopg adapter | preserves ADR-8 index efficiency; codec becomes ABC + 2 concrete impls; byte-identity invariant moves from raw-row-tuple to Datom-post-decode |

Option (B) is chosen. The per-backend wire-shape divergence at the *encoded* layer is real and acceptable — what matters for replay byte-identity is the *decoded* layer, and the PG2 parametrised property at `max_examples=200` (per backend) pins that exact invariant: same generated Datom batches go in, decoded sequences come out equal across backends. The codec ABC makes the divergence explicit in the code (no silent "is the round-trip safe?" question for a future contributor) and keeps the index-efficiency property that ADR-8 depends on.

**Trade-off accepted.** Datom-level equality, NOT raw-tuple-level equality, is what the byte-identity gate asserts. This is consistent with how every downstream consumer uses the store: `db.log()` returns `Datom` objects, replay walks `Datom` objects, `verify_chain()` reads `Datom` objects. Nothing in the substrate inspects raw row tuples — they are an internal implementation detail of each backend. Pinning the wrong invariant (raw tuples) would force option (A) and pay a real cost (ADR-8 index efficiency); pinning the right invariant (decoded Datoms) costs nothing and matches the actual consumer contract.

**Consequence.** §13 "Codec helpers locally redefined in `postgres.py`" is RESOLVED in PG2: both backends now import their codec implementation from `persistence.store._codec`. The `[postgres]` extra is still optional (the codec module imports `Datom` from `persistence.fact.datom` only — neither `sqlite3` nor `psycopg` is in the codec's dependency closure). PG2 ships the parametrised Hypothesis property at `max_examples=200` (InMemory + SQLite, fast in-process) and `max_examples=50` (Postgres, gated on `PERSISTENCE_PG_DSN`) — see `tests/store/test_replay_byte_identity_cross_store.py`. Future backends (Cloudflare D1, DuckDB, native MotherDuck, etc.) drop in their own `DatomCodec` subclass and inherit the canonical-JSON + `with_tx` plumbing for free.

## 11. Acceptance gates (ARIS R1 + R2)

ARIS R1 (this design doc) target: mean ≥ 8.5 / min ≥ 7.5 per the auto-review-loop hard-mode pattern (matches v0.5.2 + Adapter SDK).

ARIS R2 (post-implementation) target: mean ≥ 8.5 / min ≥ 7.5 on the merged branch tip.

### G1 — Smoke: PostgresStore satisfies the `Store` Protocol

`isinstance(PostgresStore(...), Store)` is True via duck-typing; calling each Protocol method (`append`, `allocate_and_append`, `all_datoms`, `since`, `mark_invalidated`, `next_tx`) against a real Postgres returns shapes equivalent to InMemoryStore.

### G2 — Single-process replay byte-identity (v0.5.2 N6/N7 extended)

The existing v0.5.2 Hypothesis property at `max_examples=200` (`tests/persistence/txn/test_replay_byte_identity_property.py`) parametrises over `[InMemoryStore, SQLiteStore, PostgresStore]`. All three pass at 200 examples each on the same draw seeds. Filter scope unchanged: `not d.a.startswith("persistence.txn/")` + per-run UUIDs excluded + clock frozen.

### G3 — Cross-process replay byte-identity (the load-bearing test, W1-recast per ADR-14)

A new test `tests/persistence/fact/test_postgres_store_concurrency.py` spawns N=4 child processes via `multiprocessing.Process` (`spawn` start method per R8). Each child opens its own `PostgresStore` (per-process pool — psycopg connections are NOT fork-safe; spawn isolates them) against the same database URI, then writes a strategy-drawn op-sequence concurrently. After all children join, the parent reads the entire log and asserts five sub-invariants (per ADR-14 W1):

- **G3a** (replay-determinism): replay the persisted log against an empty store; the resulting projection (filtered per N6/N7 — commit datoms excluded, per-run UUIDs excluded, clock frozen) is byte-identical when replayed twice. The log replays to itself deterministically.
- **G3b** (no-same-tx-collision): `SELECT (tx, e, a), COUNT(*) FROM datom_log GROUP BY 1 HAVING COUNT(*) > 1` returns zero rows. The `UNIQUE (tx, e, a)` constraint guarantees this; the test asserts the constraint was never violated mid-run (no rows would exist if it had).
- **G3c** (audit-chain-continuity): `verify_chain()` returns True for the audit-shaped projection of the persisted log walked in `seq` order. The `audit_chain_lock.head_hash` row equals the last audit datom's hash.
- **G3d** (sum-of-ops): `SELECT COUNT(*) FROM datom_log WHERE NOT a LIKE 'audit/%' AND a != 'persistence.txn/commit-id'` equals the sum of per-process op counts (no lost writes, no duplicated writes from spurious retries).
- **G3e** (some-serial-order-exists, W4-rewritten — fixes vacuous-cycle bug from W3): every commit datom in the persisted log carries the provenance fields actually emitted by `_build_commit_provenance` (`src/persistence/txn/transaction.py:534-544`): `:persistence.txn/commit-id` (UUID), `:persistence.txn/started-at` (ISO datetime string), `:persistence.txn/committed-at` (ISO datetime string), `:persistence.txn/read-set` (sorted list of eids), `:persistence.txn/ensure-set` (sorted list of eids), `:persistence.txn/intent-log`, `:persistence.txn/commute-log`. NOT in the persisted provenance: `write-set` and `t-start` (W3-corrected — earlier framings of G3e referenced fields that don't exist in code). The reconstruction:

    - **Write-set reconstruction:** for each commit identified by `(tx, commit-id)`, the write-set is the set of `e` values across ALL non-commit, non-audit datoms in the same `tx` group: `{d.e for d in datom_log where d.tx == C.tx and not d.a.startswith('persistence.txn/') and not d.a.startswith('audit/')}`. This is the v0.5.2 N6/N7 filtered projection, restricted to one commit.
    - **t_start parse:** `t_start_C := datetime.fromisoformat(C.provenance[':persistence.txn/started-at'])`.
    - **Commit-time:** `committed_at_C := datetime.fromisoformat(C.provenance[':persistence.txn/committed-at'])`.
    - **Concurrency relation:** two commits `C_a` and `C_b` are *concurrent* iff their `[t_start, committed_at]` intervals overlap, i.e., `C_a.t_start < C_b.committed_at AND C_b.t_start < C_a.committed_at`. Non-concurrent commits cannot form a cycle (the later one saw the earlier one's writes; SSI is trivially satisfied).
    - **rw-anti-dependency edge `C_a → C_b`** ("C_b reads a stale view from before C_a's writes; logical order is C_a precedes C_b"): exists iff (1) C_a and C_b are concurrent, AND (2) `C_a.committed_at > C_b.t_start` (C_a's writes were not yet visible at C_b's snapshot — so C_b read a pre-`C_a` view), AND (3) `C_a.write_set ∩ (C_b.read_set ∪ C_b.ensure_set) != ∅` (touched eids overlap on the rw direction). **The W3 condition `C_b.committed_at > C_a.committed_at` is REMOVED** — it was wrong. With concurrent commits, both `C_a → C_b` AND `C_b → C_a` rw-edges can simultaneously exist (each saw a stale view of the other's writes), forming the load-bearing 2-cycle that SSI is supposed to prevent.
    - **Cycle check:** build the directed graph over concurrent-commit pairs only, run DFS with white/gray/black colouring — if any cycle exists (length 2 or longer), G3e fails. The standard "dangerous structure" detection from Cahill, Röhm, Fekete (2008, "Serializable Isolation for Snapshot Databases") — two consecutive rw-edges `T1 → T2 → T3` with T3 committing first — is the more sensitive test; the simpler full-cycle check is sufficient for our gate (any actual anomaly produces a cycle, regardless of dangerous-structure shape).

    Falsifiable from persisted state alone: a true SSI escape produces a 2-cycle between two concurrent commits in the reconstructed graph and fails the gate. **Concrete falsification scenario** (proof-of-falsifiability, exercised by an explicit test in PG4): set up an environment override that opens `BEGIN ISOLATION LEVEL READ COMMITTED` instead of SERIALIZABLE in `_txn()` (test-only); run a write-skew workload (two transactions each read both Alice + Bob's joint balance, each writes a different one). Under READ COMMITTED both commit; the persisted log shows the 2-cycle on the rw-edges; G3e fails. Restore SERIALIZABLE: G3e passes. This explicit falsification harness lives in `tests/persistence/fact/test_g3e_falsifiability.py` (PG4 task scope).

    **Why this catches real anomalies.** SSI's promise is "no rw-cycle escapes commit." If our PostgresStore implementation accidentally runs below SERIALIZABLE (e.g., a future regression where `_txn()` defaults `isolation_level` to autocommit), the persisted log can contain rw-cycles, and G3e detects them. If our implementation ignores `audit_chain_lock` and two audit writes interleave in a way that violates the chain ordering, the audit-datom rw-graph also reflects it. The gate is not "pass iff PG SSI is correct" — PG SSI is a black box; the gate is "pass iff the persisted log is consistent with SOME serial order," which is the property our user-facing replay byte-identity claim depends on.

Hypothesis property at `max_examples=50` over op-sequence shape (lower than single-process 200 because of multiprocessing spawn overhead). Each sub-invariant is its own assertion in the property body — any single failure surfaces a single-invariant Hypothesis shrink.

### G4 — Both retryable cross-process exceptions are caught (W2-rewritten)

The original G4 narrative ("force 40001 via colluding two writers' MAX(tx) reads") is W2-retracted because the W1-revised allocator no longer relies on `MAX(tx)+1` (per ADR-4 W1-revised). The W2-rewritten gate asserts both retryable exceptions are caught and recover correctly:

- **G4a** (SerializationFailure 40001 path): a targeted test forces a true rw-dependency conflict on the touched-eids path. Two writers issue `tx.alter(ref, fn)` against the same Ref in parallel; both pass the explicit `any_datoms_since` check (because they read at the same `t_start`); both INSERT; one COMMITs, the other gets `SerializationFailure` at COMMIT (SSI rw-conflict). Asserts: `_commit_attempt` returns `False` (does not raise); the retry loop catches it and re-runs; second attempt succeeds.
- **G4b** (UniqueViolation 23505 defence-in-depth path): a targeted test directly INSERTs into `datom_log` with a `tx` that bypasses the `tx_allocator` (test-fixture only — uses a `_dangerous_direct_insert` helper exposed under a sentinel env var). Asserts: the second writer's `transact_serializable` call surfaces `UniqueViolation` on `(tx, e, a)`; `_commit_attempt` returns `False`; retry loop catches it and re-runs against the now-advanced `tx_allocator`; second attempt succeeds with a fresh tx-id.
- **G4c** (no other psycopg exception silently caught): asserts that a synthetic non-retryable error (e.g., `OperationalError` from a closed connection) propagates out of `_commit_attempt` and is NOT swallowed by the retry loop.

### G5 — Audit-chain Merkle continuity across processes

Two child processes write 100 audit datoms each in parallel. Parent reads `audit_chain_lock` row + walks `datom_log` filtered by `a LIKE 'audit/%'`. Asserts: every audit datom's `parent_provenance_hash` equals the immediately-prior audit datom's hash (in `seq` order); `audit_chain_lock.head_hash` equals the last audit datom's hash; `verify_chain()` returns True.

### G6 — `since(t)` cross-backend identity

Same op-sequence applied to InMemoryStore + SQLiteStore + PostgresStore. `since(t_mid)` returns the same projected tuples in the same order. (Implementation: `seq` ordering on Postgres = `seq` ordering on SQLite = list ordering on InMemory.)

### G7 — Migration apply + rollback on a real Postgres

`testcontainers` Postgres 17 fresh image. Apply `0001_datom_log.postgres.sql` cleanly. Apply `0001_datom_log.postgres_rollback.sql` cleanly. Re-apply the forward migration cleanly. Suite passes.

### G8 — `BackendNotInstalled` error path

`pip install persistence` (without `[postgres]`). `Substrate.open(store="postgres://...")` raises `BackendNotInstalled` with a message including the install hint `pip install "persistence[postgres]"`.

### G9 — Pool sizing + statement timeout configuration

`PostgresStore(uri="postgres://...?pool_min=1&pool_max=10&statement_timeout=30000")` parses the query string and applies the pool size + statement timeout. Out-of-band test asserts a long-running `pg_sleep(60)` is interrupted at 30s.

### G10 — Cross-process Atom CAS

A targeted test spawns 4 processes, each calls `atom.swap(lambda v: v + 1)` 100 times against a shared PostgresStore. Final atom value = 400. Total writes = 400 + retries (logged but bounded). No `AtomCASExhausted` raised at `max_retries=256`.

### G11 — Cross-process `tx.commute` correctness (W1-recast)

Spawn 4 processes, each calls `tx.commute(counter_ref, "inc-by", 1)` 100 times. **G11a** (correctness): final counter = 400. **G11b** (bounded retries): total retries across all 4 processes ≤ 4× `max_retries` budget (≤ 1024); ARIS R2 measures the actual rate. The original "zero retry-rejected commits" framing was W1-retracted: SSI may still flag rw-conflicts on the audit-chain or the tx_allocator-row paths even when the commute target itself sees no predicate lock; what we actually need is "commutativity preserved + retries bounded." The replay-byte-identity test for commute (v0.5.2 § F3 acceptance) is preserved unchanged — the commute log replays deterministically regardless of retry count.

**Why the original "zero retries" claim was wrong.** ADR-6 says `tx.commute` does NOT place a predicate lock on the commute *target*. But every audit-emitting commit takes `audit_chain_lock` `FOR UPDATE`, and every commit takes `tx_allocator` `FOR UPDATE`. Two parallel commute commits both contend on these row locks; one queues, runs, commits; the other queues, runs, commits — no SSI 40001, no retry. But SSI may still flag a rw-conflict if the two transactions touched any of the same eids in their explicit read-set / ensure-set / write-set, which a real-world `tx.commute` body usually does (e.g., the same `dosync` reads other refs to compose the commute argument). The "zero retries" framing only holds for a pathological body that does NOTHING but `tx.commute(counter, "inc-by", 1)` — and even then `audit_chain_lock` is contended.

### G12 — Cross-process `tx.ensure` write-skew prevention

Classical bank-transfer write-skew test (v0.5.2 § F2) extended to multi-process: two processes, each transfers from the joint Alice+Bob balance to a third party. With `tx.ensure` on the partner's account, exactly one commits; the other retries and overdraws is prevented.

### G13 — Coverage gate

`pytest --cov=persistence.fact.postgres_store` shows ≥ 90% line coverage on the new module. Branches uncovered are documented.

## 13. Open questions (defer to ARIS R1 review or impl phase)

The OQs below are split into two tiers per ARIS R1 W8: structural (the design's actual blockers; resolved as ADRs in W1-revised) and parametric (defaults / tuning that can be revisited at impl time).

### Structural (resolved in W1-revised; listed here for traceability)

- **OQ-S1 (W1-revised → ADR-4):** Tx-id allocator primitive — `MAX(tx)+1` vs allocator row vs `BIGSERIAL`. **Resolved:** `tx_allocator` row + `SELECT FOR UPDATE` + `UPDATE next_tx`, with `UNIQUE (tx, e, a)` defence-in-depth.
- **OQ-S2 (W1-revised → ADR-5):** `v` / `provenance` storage column type — TEXT vs JSONB. **Resolved:** TEXT (canonical JSON), with opt-in parallel JSONB column migration for operator-side containment queries.
- **OQ-S3 (W1-revised → ADR-13):** Audit datom persistence under multi-process — bypass allocator (current shape) vs route through `transact_serializable`. **Resolved:** route through `transact_serializable`; preserve `recorded_at` via explicit `tx_time` parameter.
- **OQ-S4 (W1-revised → ADR-14):** G3 baseline definition — "byte-identical to a single-process serial baseline" vs multi-invariant. **Resolved:** multi-invariant (G3a-e); the persisted log is its own canonical baseline.
- **OQ-S5 (W1-revised + W2-clarified → § 5.2.1 + ADR-15):** Cross-process atomicity boundary — "8-line `_lock` swap" vs explicit `transact_serializable` helper. **Resolved (W2-clarified, W3-propagated):** PostgresStore exposes TWO distinct Store-Protocol-additive primitives: `_txn()` (store-level transaction context-manager, opens `BEGIN ISOLATION LEVEL SERIALIZABLE` under Postgres) and `transact_serializable(facts, provenance, tx_time?)` (the atomic helper that uses `_txn()` + `tx_allocator` + optional `audit_chain_lock`). The `_lock` field stays as a pure `threading.RLock` in-process mutex on every backend (NOT polymorphic, NOT a context-manager that opens transactions — that was W1's incorrect framing, retracted in W2). PG3 task LOC: ~80 across all touched files.

### Parametric (defaults / tuning — defer to impl time)

1. **Statement timeout default in URI?** v0.8 default: `statement_timeout=30000` (30s) baked into the URI parser. Alternative: 60s. Recommend 30s — coding-agent workload is interactive, runaway queries are a real failure mode, 30s is generous for our access patterns.
2. **Pool sizing default?** `pool_min=1, pool_max=10`. Alternative: `pool_max=20`. Recommend 10 — testcontainers + dev-laptop runs cleanly at 10, scales linearly; bigger defaults invite "why are we eating Postgres connections" questions in deploy. **Per-process pool init under `multiprocessing.spawn` is mandatory** (psycopg connections are NOT fork-safe — see R10 W1-NEW); each child opens its own pool against the shared DB URI.
3. **`testcontainers` vs ephemeral CI Postgres for tests?** `testcontainers` (lazy-pulled Docker image) is the simpler shape. Alternative: GitHub-Actions service-container. Recommend `testcontainers` for local-dev-loop fidelity; CI can use GH service-container as a parallel path.
4. **Cross-process Hypothesis `max_examples`?** 50 is the proposed budget (vs single-process 200) because multiprocessing spawn overhead is ~50× slower than in-process. Alternative: 100 with parallel pytest-xdist workers. Recommend 50 for v0.8; revisit at v0.9 if false-pos rate findings warrant more.
5. **Optional JSONB columns + GIN — ship in v0.8 as an opt-in migration, or fully defer?** Recommend ship `0002_optional_jsonb_columns.postgres.sql` in v0.8 but DO NOT auto-apply. Operators run it explicitly if their workload needs containment queries (e.g., RLS hot-path tenant-id evaluation per ADR-12 W1-revised).
6. **`pgaudit` settings in `postgres-deployment-v0.8.md`?** Recommend `pgaudit.log = 'write,ddl,role'` as the documented default; lighter (`'ddl'` only) and heavier (`'all'`) variants documented as alternatives. v0.8 ships no code-level pgaudit dependency.
7. **Should `PostgresStore` expose an `async_append` / `async_all_datoms` for psycopg-async?** v0.8: NO — Module 5 Txn is sync; introducing async-Store paths forces an async-Txn rewrite that's out of scope. v0.9 may grow async paths once we need them.

**Operational gotchas worth carrying forward.** (1) `multiprocessing.get_context('spawn')` is mandatory — the macOS default has shifted toward spawn but explicit is better than implicit, and CI on Linux still defaults to fork. (3) Across-process exception propagation is uneven; the harness ships exceptions back as `repr(exc)` + SQLSTATE strings rather than the live psycopg objects.
