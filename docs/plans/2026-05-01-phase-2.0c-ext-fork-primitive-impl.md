## Why this exists

Phase 2.0c shipped `s.txn.fold_into` as a Path-A foldl-with-`:fold/chosen`
marker. It works but does not match § 3.7 + § 4.3: non-chosen branches'
facts persist in the substrate, and only one audit datom emits (the
chosen marker) instead of the canonical 4-datom shape (`:fold/probe`,
`:fold/branch`xN, `:fold/score`xN, `:fold/chosen`).

The "speculate / score / commit only the winner / rollback the rest"
shape is the wedge story for persistence-coder (rewind/branch/replay).
Without per-branch rollback the wedge is just transactional foldl.

This task ships the substrate-true primitive `DB.fork` and rewires
`s.txn.fold_into` on top of it. `DB.fold` (the foldl/reduce primitive)
stays untouched. Two primitives now live side-by-side:

- `DB.fold` — foldl/reduce; commits all facts; emits `:fold/chosen` marker
  (kept for users who want "audit the chosen one of a foldl"). 2.0c
  semantics intact.
- `DB.fork` — speculate-rollback-pick; commits only chosen branch's facts;
  emits `:fork/probe` + `:fork/branch`xN + `:fork/score`xN + `:fork/chosen`.

The two-namespace split (`:fold/*` vs `:fork/*`) is intentional and is
called out in design doc § 3.7 + § 4.3 + ADR-7 reconciliation in C6.

## Public surface (target)

### `DB.fork` (new)

```python
def fork(
    self,
    items: Sequence[Any],
    fn: Callable[[Any, Any], Any],          # (branch_state, item) -> branch_state
    choose: Callable[[list[ForkBranchResult]], int],
    *,
    seed: Any = None,
    on_error: Literal["stop", "continue"] = "stop",
    provenance: dict | None = None,
) -> ForkResult: ...
```

- `seed` — initial branch state (optional; default None).
- `fn(branch_state, item)` — called once per branch (one item -> one branch);
  returns the branch's terminal state.
- `choose(branches)` — picks winning index (must be `int`, not `bool`).
- `on_error="stop"` — first branch failure aborts whole fork.
- `on_error="continue"` — failed branches marked score=None, excluded from
  `choose`.
- `provenance` — merged into the chosen branch's commit provenance only.

### `ForkBranchResult` / `ForkResult`

Both `@dataclass(frozen=True)`:

```python
@dataclass(frozen=True)
class ForkBranchResult:
    branch_index: int
    branch_id: str             # 16-hex content-hash of (item, fn-result)
    item: Any
    branch_state: Any          # fn(seed, item) — terminal state
    score: float | None        # None on failure under continue
    error: str | None          # type-name + message on failure

@dataclass(frozen=True)
class ForkResult:
    chosen_index: int
    chosen_state: Any
    all_branches: tuple[ForkBranchResult, ...]
    txn_commit_uuid: str
```

### Audit datom shape (canonical 4-datom emission)

Per fork call, in this order under outer txn (so all share `txn_commit`
and Merkle prev-hash of N+2 entries):

1. **`:fork/probe`** — `{seed_hash, items_hash, fn_hash, choose_hash, branch_count}`
2. **`:fork/branch`** xN — `{branch_index, branch_id, item_hash, branch_state_hash}`
3. **`:fork/score`** xN — `{branch_index, score_value, score_hash}`
4. **`:fork/chosen`** — `{chosen_index, chosen_branch_id, chosen_state_hash, txn_commit_uuid}`

Hashes are sha256 over canonical-JSON, first 16 hex chars (matches
existing `_hash_fact` in `db.py`). `fn_hash` and `choose_hash` are over
`(callable.__qualname__, callable.__module__)` tuples — best-effort,
documented as such (we cannot byte-hash callables).

### `s.txn.fork` (new SDK surface)

Thin pass-through to `DB.fork`. `@experimental("v0.8.5a1")` per Phase
2.0c precedent.

### `s.txn.fold_into` (rewired)

Stays at the same public signature. Internally now calls `DB.fork` with
a wrapper that (a) bridges the 3-tuple `fn(acc, item, db) -> (new_acc,
facts, score)` shape into the 2-arg `fn(branch_state, item)` shape (b)
threads `facts` through into branch-local datom emission so chosen
branch commits them as part of the outer txn (c) bridges
`FoldBranchScore` <- `ForkBranchResult` for the `choose` callback.

The Path-A `:fold/chosen` marker is REMOVED from `fold_into`. The new
`:fork/*` shape IS what `fold_into` emits internally. Tests that
asserted `:fold/chosen` shape for `fold_into` are updated to assert the
4-datom `:fork/*` shape — be honest about which were testing the wrong
thing.

`DB.fold` itself (the foldl primitive) keeps `:fold/chosen` for users
who still need that pattern.

## Design decisions

### D1 — Use `DB.branch()` for child snapshots, not new infra
`DB.branch(t, assertions)` already implements "fresh InMemoryStore seeded
from as_of(t)". Per branch, we call `db.branch(t_now, [])` to get a
counterfactual sub-DB; run `fn(seed, item)` against it (or against an
in-memory state, since `fn` operates on opaque `branch_state` not the
DB); the branch state is independent of the parent store by construction.

### D2 — Branch state is opaque to substrate
`fn(branch_state, item)` returns a Python value; the substrate does NOT
mandate it be facts. Facts emission is opt-in: `fold_into`'s adapter
will collect `facts` from its 3-tuple shape and stash them in a
per-branch list to commit only on the winner. `DB.fork` itself keeps the
shape pure: branches are just states.

This means rollback is trivial — non-chosen branches' state is just
discarded Python objects; nothing was ever committed. The "facts of the
non-chosen branch" question is moot at this level.

For `fold_into`'s adapter layer: facts ARE collected per-branch into a
side-list, and only the chosen branch's facts get applied via
`tx.db.transact_batch` inside the outer txn at the choose step.

### D3 — Outer txn ownership
`DB.fork` MUST run inside an enclosing `dosync` (mirrors `:plan/edit`).
The dosync gate is enforced via `is_in_dosync()` ContextVar and an
explicit `tx` kwarg (mirrors `fold_into`'s contract).

The 4-datom audit emission queues 4 effect intents on `tx.effect_intent_log`,
which `_replay_effect_intents` flushes at commit time, automatically
threading `_txn_commit` into each.

### D4 — Why `:fork/*` not `:fold/*`
- `:fold/chosen` is already shipped with foldl-marker semantics (chosen
  branch within an all-committed foldl). Reusing the keyword for
  rollback semantics would silently change meaning across replays of the
  same trajectory before/after upgrade — unsafe.
- The two primitives are semantically distinct; the namespaces should
  reflect that. `fold` = reduce / accumulate; `fork` = speculate / pick.
- Updates land in design doc § 3.7 (table row split), § 4.3 (#145
  description rename), and ADR-7 (surface naming).

### D5 — `branch_id` content-addressed
`branch_id` = sha256(canonical_json({item: ..., branch_state: ...}))[:16].
Lets the `:fork/chosen` datom point at the winner unambiguously across
replays, even if items happen to collide.

### D6 — `score` derived from terminal `branch_state`
The score isn't a separate `fn` return; it IS what `fn` returns
(`branch_state`). `choose` reads `branch.branch_state` and decides.
Optionally, `choose` can call `branch.score` if the adapter populated
it (e.g. `fold_into`'s 3-tuple `fn` extracts a numeric `score` and
populates `ForkBranchResult.score`).

For the bare `s.txn.fork` API, `score=None` by default; users who want
`:fork/score` datoms with non-None values must populate it themselves
via the wrapper layer (or `fold_into`).

Wait — the spec says emit `:fork/score` per branch. Re-reading spec D2:
"`:fork/score` × N — one per candidate: `{branch_index, score_value,
score_hash}` (where `score` is whatever `fn` returned as the branch's
terminal state, since `choose` scores them)."

Resolution: `score_value` in the audit datom is the `branch_state` (the
fn's return), serialized via canonical JSON. `score_hash` is the 16-hex
hash thereof. This works for both numeric-score callers (`fold_into`)
and structural-state callers (bare `fork`). The naming `:fork/score`
is slightly aspirational for the non-numeric case; the kwarg name
`score_value` makes it unambiguous in the datom payload.

### D7 — Test-restructuring honesty
The existing 2.0c tests assert "all branches' facts persist". The new
contract is "only chosen branch's facts persist". This is a behavior
change, not a test bug — the prior tests were correctly verifying
Path-A's shipped semantics, which we are now superseding.

Tests that were testing the wrong (path-A) thing get updated, with a
git commit message making the swap explicit. Tests like the
choose-validation, fn-contract, dosync-gate, byte-identity property are
behavior-invariant under the rewire and stay green.

## Honest scope-reduction triggers

If `DB.branch()` does not cleanly support the per-branch isolation we
need, OR if `Transaction.effect_intent_log` cannot accept 2N+2 entries
in a single dosync without spec-validation choking on the new op names,
STOP and report. Do NOT hack around. Phase 2.0a `delete_step`
permissive-shipping precedent: honest scope reduction beats
overconfident impl.

## Out of scope (deferred)

- `:fork/*` re-execution-replay (re-run `fn` on replay). § 3.7 says
  audit-replay is the default and re-exec is opt-in only for
  `:code/exec`. The new datoms have audit-replay as the load-bearing
  contract; re-exec lands in v0.10 alongside FS-snapshot work.
- Multi-item-per-branch (each branch consumes a sub-list of items).
  Current shape: 1 item -> 1 branch. Multi-item folding is what
  `DB.fold` already does; users who want both compose them.
- Cross-process fork (fork branches in `multiprocessing.Pool`). Future
  perf work; in-process is sufficient for the agent's MCTS-N=4-8 case.
