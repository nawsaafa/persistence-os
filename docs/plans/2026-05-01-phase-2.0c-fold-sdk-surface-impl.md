## Architecture

`fold_into` is a **convenience method** that runs `fold` (which already commits facts as it iterates), then calls a user-supplied `choose` callback over the per-item scores returned by `fn`, and emits a single `:fold/chosen` audit datom marking the winning branch.

This matches the design doc's §4.3 framing of `fold_into` as "a `fold_into(seed, items, fn, choose)` convenience method on the same namespace that runs `fold` and commits the chosen branch in one txn (the current surface returns scores; the agent has to thread the commit step itself)."

**Key design decision:** the existing `DB.fold` is a foldl/reduce primitive (commits facts as it goes), NOT a speculate-rollback-pick primitive. `fold_into` therefore does NOT speculatively run N branches and discard losers — it runs the fold once (which commits all `fn`-emitted facts), then records WHICH branch was chosen via the audit datom. The non-chosen branches' facts remain in the substrate as "considered but not chosen" datoms, which matches Persistence OS' append-only ethos and the wider audit-replay invariant from § 3.7.

**The `choose` callback shape:** `Callable[[list[FoldBranchScore]], int]` returning the index of the chosen branch. Each `FoldBranchScore` is a frozen dataclass `(item, score, accumulator_after)` so the callback has access to the per-item score AND the accumulator state at that point.

**`fn` contract extension:** `fn(acc, item, db) -> (new_acc, facts, score)` — the third tuple element is the score for THIS branch. `fold_into` validates the 3-tuple and routes through `DB.fold` via a wrapper `fn` that strips the score back to the 2-tuple `DB.fold` expects, while collecting `(item, score, new_acc)` triples into a side list for `choose`. **Backwards compatible**: `s.txn.fold` still takes the 2-tuple `(new_acc, facts)` shape.

**Atomicity:** `fold_into` REQUIRES the caller to be inside a `db.dosync(...)` body — it raises `FoldIntoOutsideDosync` otherwise. This mirrors the Phase 2.0a `PlanEditOutsideDosync` pattern. Inside dosync, the `:fold/chosen` audit datom is queued via `tx.effect(":fold/chosen", ...)` and rides the existing Merkle chain at `effect/handlers/audit.py` — no new chain code.

**Determinism contract:** for fixed `(seed, items, fn, choose)` inputs (where `fn` and `choose` are pure / deterministic), the `:fold/chosen` audit datom is byte-identical across replays. This is the §4.3 gate; verified via Hypothesis property test at `@max_examples=200`.

## Decisions

### Decision 1 — `fn` returns 3-tuple `(new_acc, facts, score)`

`DB.fold` takes `fn(acc, item, db) -> (new_acc, facts)`. `fold_into` extends this to `(new_acc, facts, score)` — the third element is the score for THIS branch. A wrapper at the `fold_into` layer strips the score back to the 2-tuple shape `DB.fold` expects, recording `(item, score, new_acc)` triples for the `choose` callback.

**Rationale:** keeps `DB.fold` signature unchanged (no breaking change to the underlying surface). The score is part of the agent's MCTS scoring contract, not a substrate concern.

**Validation:** the wrapper validates the return shape; if `fn` returns a 2-tuple (forgets the score), raises `TypeError` wrapped in `FoldIntoChooseError("fn must return 3-tuple (acc, facts, score)")`. Score must be `int | float | bool`; non-finite (NaN/+/-Inf) scores raise the same error. Empty `items` raises `ValueError` (no branches to choose between).

### Decision 2 — `choose` callback signature

`choose: Callable[[list[FoldBranchScore]], int]` where `FoldBranchScore = (item, score, accumulator_after)` is a frozen dataclass. Returns the 0-based index of the chosen branch in `items` order.

**Rationale:** the `choose` callback owns the policy (argmax score, sample by softmax temperature, latest-wins, etc.). Keeping it pure means the audit datom is reproducible across replays as long as `choose` is deterministic. The frozen dataclass enforces immutability so callers can't mutate the score list.

**Errors:**
- `choose` returns non-int -> `TypeError` wrapped in `FoldIntoChooseError`
- `choose` returns int outside `[0, len(items))` -> `ValueError` wrapped in `FoldIntoChooseError`
- `choose` raises arbitrarily -> wrapped in `FoldIntoChooseError` with `__cause__` = original

### Decision 3 — `:fold/chosen` audit datom shape

```python
tx.effect(
    ":fold/chosen",
    chosen_index=int,         # 0-based index in items
    chosen_score=float,       # score of the winning branch
    all_scores=tuple[float],  # per-branch scores in items order
    branch_count=int,         # len(items)
)
```

`_txn_commit` is auto-injected by `persistence.txn.transaction._replay_effect_intents` at commit time (same path as `:plan/edit` and `:code/exec`). No new chain code; `:fold/chosen` rides the existing Merkle chain at `effect/handlers/audit.py`.

**`all_scores` is a tuple** (not a list) so it's hashable and frozen — the audit handler canonicalizes args via JSON, but tuples and lists serialize the same; the tuple here documents intent (caller must NOT mutate after emit).

**Why not include `accumulator_after`?** It's caller-domain state (could be anything — a model object, a numpy array, an arbitrary tree); not generally JSON-serializable. The audit datom captures the deterministic decision identity (which index, score), not the full state. Replay determinism only needs the index.

### Decision 4 — `FoldIntoResult` return type

```python
@dataclass(frozen=True)
class FoldIntoResult:
    chosen_index: int
    chosen_score: float
    all_scores: tuple[float, ...]
    chosen_accumulator: Any           # accumulator after the chosen branch
    final_accumulator: Any            # accumulator after fold completes
    total_datoms_committed: int       # forwarded from DB.fold's return tuple
```

`chosen_accumulator` and `final_accumulator` may be the same value if `choose` picks the last branch; they differ when `choose` picks an earlier branch (e.g. argmax).

### Decision 5 — Outside-dosync gate

`fold_into` MUST run inside `db.dosync(...)` so the `:fold/chosen` datom rides the same Merkle chain as the rest of the trajectory. Outside dosync, raises `FoldIntoOutsideDosync` (mirrors `PlanEditOutsideDosync`).

**Why gate this differently from `s.txn.fold`?** `s.txn.fold` does NOT require dosync today (each item's facts go through `transact_batch` in their own implicit txn). `fold_into` adds the `:fold/chosen` audit step which ONLY makes sense inside an enclosing dosync (otherwise `tx.effect()` has no transaction to attach to). The dosync gate is therefore a `fold_into`-specific contract; `s.txn.fold` keeps its current "callable anywhere" shape.

### Decision 6 — Score type coercion

Scores from `fn` are coerced to `float` for the audit datom + `FoldIntoResult.all_scores` field, even if `fn` returned `int` or `bool`. This keeps the audit-datom JSON canonicalization stable (Python's `json.dumps(1)` vs `json.dumps(1.0)` differ at byte level — `1` vs `1.0`). The `chosen_score` field is `float` for the same reason.

**Rejected alternative:** preserve original numeric type. Would mean two replays of the same `fn` returning `int 5` produce a different audit datom than one replay returning `float 5.0` even though the comparison is semantically identical. Coercing to `float` at the boundary closes that footgun.

### Decision 7 — `on_error` and `checkpoint_every` propagation

`fold_into` accepts `on_error` and `checkpoint_every` and passes them through to `fold`. Implications:
- `on_error="abort"` (default): if any branch's `fn` raises, `fold_into` raises `FoldError` (wrapped, from `DB.fold`). `choose` is NEVER called. No `:fold/chosen` datom emitted.
- `on_error="skip"`: skipped branches are NOT included in the score list passed to `choose`; the index returned by `choose` is into the SUCCESSFUL-branches list, not the original `items` list. The `:fold/chosen` datom records `chosen_index` as the index into the SUCCESSFUL list (not `items`) for replay determinism. `branch_count` reflects successful count.
- `on_error="checkpoint"`: same as `"abort"` (fold raises, choose not called).

**Rationale:** `choose` operating over a possibly-shorter successful list is the natural shape — agents picking among MCTS rollouts shouldn't have to pre-filter failures. The audit datom records the exact list `choose` saw.

**Documented in module docstring + CHANGELOG.**

## Hypothesis strategies

```python
@st.composite
def fold_input(draw):
    seed = draw(st.integers(min_value=-1000, max_value=1000))
    n = draw(st.integers(min_value=1, max_value=8))
    items = draw(st.lists(st.integers(min_value=-100, max_value=100), min_size=n, max_size=n))
    return seed, items
```

`fn` is a fixed pure deterministic op like `lambda acc, item, db: (acc + item, [{...item-fact...}], item * item)` — **no closures over draw'd state** (closures break Hypothesis shrinking determinism and we want byte-identity across replays, which closures over per-test state would violate).
