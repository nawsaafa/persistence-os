# Phase 2.1c — Context Substrate Endpoints (Design)

**Status:** DRAFT — awaiting ARIS R1
**Date:** 2026-05-04 (amended same day: claim-namespace separation; caller identity rescoped to **2.1c.5** with falsifiable acceptance signal)
**Phase placement:** 2.1c (between 2.1b LLM provider [merged at `dac29ed`] and 2.1d+ persistence-coder critical-path work)
**Branch base:** `feat/v0.9-persistence-coder` @ `dac29ed`
**Effort:** ~1 day implementation + tests (~700-1050 LOC, +100-150 vs pre-amendment for `persistence.claim/` module skeleton)
**Critical-path posture:** orthogonal — does NOT block persistence-coder MVP (Phase 2.1d+)
**Trust posture:** HTTP surface emits **claims** (caller-asserted records), not facts. Substrate accountability covers what was *claimed and stamped*, not what was *executed and witnessed*. See §5.0 and §16.

---

## 1. Goal (one sentence)

Add a thin HTTP surface to persistence-os so external processes — Claude Code hooks, MCP tool wrappers, and (eventually) Mimir's CLI — can emit **claims** (records of what the caller asserts happened), store/retrieve content-addressed blobs, and run bounded session-scoped queries against the claims log.

## 2. Why this, why now

Phase 2.1b shipped the first `:llm/decision` datoms via `_decide`. Substrate has all the persistence primitives it needs — `txn.commit`, datom log, audit chain (hash-chained today, Ed25519-signed under Mimir Phase B), plan AST, `DB.fork`. What it lacks is **a remote-process emit path**: today, only in-process Python code can write datoms.

Two near-term consumers need this surface:

1. **context-mimir** (separate repo `~/Projects/context-mimir/`, not yet scaffolded): a Claude Code plugin + MCP server that intercepts tool output, stores raw blobs, replaces in-context payload with `<<sandboxed:hash=…size=…hint=…>>` stubs, and retrieves relevant past **`:claim/tool-exec` records** on PreCompact. Cuts daily Claude Code session token-burn by an estimated 20-25× on top of `opus-4-7[1m]` usage. Blocked on persistence-os exposing claim emit + blob HTTP endpoints.

2. **Mimir** (track `mimir-os-product_20260503`): the `mimir` CLI is specified to dumb-wrap `POST box.tail89def3.ts.net:8005/v1/dispatch` in MEMORY.md. The dispatch endpoint there is Mimir-specific, but the underlying datom-emit / blob / query primitives are substrate-shaped and belong in persistence-os, not in Mimir. Mimir's `/v1/dispatch` will internally call into these substrate endpoints once 2.1c lands.

Building this once unblocks both consumers and gives persistence-os its first real production tenant before Mimir D-phase. No reason to wait.

### 2.1 Epistemic separation (load-bearing for trust posture)

Pre-2.1c, every datom in the substrate trace back to in-process Python that called the curated SDK — code the substrate witnessed executing. Accountability is enforceable because *the substrate observed the action*.

A network emit endpoint breaks that. Any localhost process can `POST /v1/claim/emit` asserting "I ran `Bash` `command='rg foo'` exit_code=0 in 247 ms" — and the substrate has no ground-truth verification of any of those fields. It can record the assertion; it cannot validate it.

The 2.1c design treats this as a real architectural distinction, not a documentation footnote:

| Category | Provenance | Substrate witnessed | Datom kind prefix | Owned by |
|---|---|---|---|---|
| **Fact** | in-process curated SDK | Yes — execution observed here | `:llm/*`, `:txn/*`, `:plan/*`, etc. | existing modules |
| **Claim** | external process via HTTP | No — caller asserts X happened | `:claim/*` | NEW `persistence.claim/` module |

Both flow through the same `txn.commit` and the same audit chain (hash-linear, Ed25519-signed at Mimir Phase B). What differs is **what the chain attests to**: substrate-execution for facts, caller-stamping for claims. Consumers and replay can opt into "facts only," "claims only," or "both."

Caller identity attestation (cryptographic pinning of *which caller* made a claim) is the second half of the trust story and is rescoped to **2.1c.5** — see §16. 2.1c lands the namespace and the module skeleton so the trust delta is **visible in the data from day one**, not retrofitted later.

### 3.1 Why a separate `persistence.claim/` module instead of folding into `http/`

If the trust boundary lives only in HTTP route handlers, then anyone who later adds a different transport (Unix socket, gRPC, MCP server) has to re-implement it — and the first one to forget gets to write malformed claims into the substrate. Putting namespace ownership and validation in a transport-agnostic module means the trust boundary travels with the substrate, not with the wire.

## 4. HTTP surface — 4 endpoints

**Kind restriction (load-bearing, enforced server-side):** every `kind` field accepted by `/v1/claim/emit` and returned by `/v1/claim/query` MUST be in `persistence.claim.CLAIM_KINDS`. Any non-claim kind (e.g. `:llm/decision`, `:plan/step`) is rejected `400 not_a_claim_kind`. The HTTP surface cannot mint or expose facts.

### 4.1 `POST /v1/claim/emit`

Emit one or many claims in a single transaction. All claims in a single request commit atomically (one `txn.commit` call) or all fail. Each claim's `kind` is validated against `CLAIM_KINDS` and its `attrs` against the per-kind schema in `persistence.claim._registry`.

**Request**
```json
{
  "claims": [
    {
      "kind": ":claim/tool-exec",
      "attrs": {
        "tool": "Bash",
        "args": {"command": "rg --json 'foo'"},
        "body_hash": "sha256:ab12cd34...",
        "body_summary": "47 matches across 12 files in src/",
        "started_at": 1714839028000,
        "duration_ms": 247,
        "exit_code": 0,
        "session_id": "session-2026-05-04-abc",
        "parent_correlation_id": null
      }
    }
  ]
}
```

**Response 200**
```json
{
  "tx": 4961,
  "claim_ids": ["datom-2026-05-04-xyz-1"],
  "audit_chain_head": "sha256:7eb7...",
  "caller_identity": null
}
```

(Internal substrate identifier is the standard datom-id; the HTTP shape calls them `claim_ids` for API-side consistency with the rest of the surface. Both refer to the same thing.)

`caller_identity` is `null` in 2.1c (no attestation). In 2.1c.5 it becomes the verified caller pubkey-hash.

**Response 4xx**
- `400` malformed body, schema violation, `kind` not in `CLAIM_KINDS` (`not_a_claim_kind`), unknown claim kind
- `401` missing/invalid bearer token
- `413` request body exceeds 1 MiB (blobs go via `/v1/blob/put`, not emit)
- `422` `attrs` shape mismatch with curated schema for `kind`

### 4.2 `POST /v1/blob/put`

Content-addressed put. Same content → same hash → single underlying write (idempotent). Emits a `:claim/blob-put` datom for every put-attempt (including duplicates).

**Request**
- Header `Content-Type: application/octet-stream` (or any — stored verbatim)
- Header `X-Session-Id: session-…` (required, for `:claim/blob-put` datom emission)
- Body: raw bytes (max 16 MiB initially; tunable via env)

**Response 200**
```json
{
  "hash": "sha256:ab12cd34...",
  "size_bytes": 56432,
  "duplicate": false
}
```

`duplicate: true` when content already existed. The corresponding `:claim/blob-put` datom is still emitted (audit chain records every put-attempt for traceability).

### 4.3 `GET /v1/blob/get/{hash}`

Fetch raw blob by sha256.

**Response 200** — raw bytes, `Content-Length` set
**Response 404** — blob absent (no datom emitted; reads are not audit-chained)

### 4.4 `GET /v1/claim/query`

Bounded session-scoped query over the **claim subset** of the datom log. Facts are not exposed via HTTP — see §4 kind restriction.

**Query params**
- `session_id` (required) — scope to one session
- `since` (optional, default 0) — `tx` lower bound, exclusive
- `kind` (optional, repeatable) — filter to one or more claim kinds. Any kind not in `CLAIM_KINDS` is rejected `400 not_a_claim_kind`. Unspecified = all claim kinds.
- `limit` (optional, default 50, max 500)

**Response 200**
```json
{
  "claims": [
    {
      "tx": 4961,
      "kind": ":claim/tool-exec",
      "attrs": {...},
      "ts": 1714839028000,
      "caller_identity": null
    }
  ],
  "next_since": 4961
}
```

`next_since` returns the `tx` of the last datom in the page; pass it back as `since` to paginate. `caller_identity` is `null` until 2.1c.5 (see §16).

**Out of scope for v0.1**: Datalog query expressions, joins, aggregations. Structured filters only — `session_id`, `since`, `kind`, `limit`. FTS over `body_summary` deferred to Phase 2.x of context-mimir. Cross-namespace queries (claim + fact in one response) are explicitly out of scope and not on the 2.1c.x roadmap — facts stay in-process-only by design.

## 5. Claims layer — datom kind additions (curated, `@experimental("v0.9.x")`)

### 5.0 The claims layer (epistemic posture)

`:claim/*` is a distinct datom-kind namespace owned by `persistence.claim`. Semantically:

- **Record-of-action, not action.** Substrate does not execute a `:claim/tool-exec` — it stores the caller's assertion that a tool ran. This is a quiet but meaningful break from the "every action is an effect" framing on facts; the substrate's accountability invariant for claims is "we recorded what the caller said, hashed into the chain."
- **No audit-stack middleware.** Unlike `:llm/call` (a `_WRAPPED_OP` from 2.1b which the audit stack wraps because the substrate *invoked* it), claim emissions are direct `txn.commit` calls. There is no effect handler to wrap; there is no execution to retry. Claims are a write-only ledger entry.
- **No bypass paths.** Claims still go through `txn.commit` and are still hashed into the audit chain identically to fact datoms. The trust delta is entirely in the *kind namespace*, not in the persistence machinery.
- **Replay-safe.** Replay of a claim = re-applying the recorded datom. The substrate doesn't try to re-execute the underlying tool; it never executed it in the first place.

### 5.1 `:claim/tool-exec`

| field | type | notes |
|---|---|---|
| `tool` | `str` | E.g. `"Bash"`, `"Read"`, `"Grep"`, `"WebFetch"`, `"ctx_bash"` |
| `args` | `dict` | Tool-specific args. JSON-serializable. |
| `body_hash` | `Optional[str]` | sha256 of full output if blobbed; `None` if output was inline-summary-only. |
| `body_summary` | `str` | Bounded ≤512 chars. Either rule-based for cheap tools (file count, exit code) or LLM-extracted hint downstream of context-mimir. Substrate stores verbatim. |
| `body_disposition` | `Literal["inline","blobbed","discarded"]` | What the caller did with the full output. `inline` = body fits in summary, `blobbed` = body was sent to `/v1/blob/put` and `body_hash` is set, `discarded` = caller chose not to retain. Distinguishes "summary IS the result" from "summary is a hint, body lives elsewhere." |
| `started_at` | `int` | Unix epoch milliseconds. |
| `duration_ms` | `int` | Wall-clock duration. |
| `exit_code` | `Optional[int]` | Process exit code if applicable, else `None`. |
| `session_id` | `str` | Identifier of the session (Claude Code session, Mimir agent run, etc.). Substrate is opaque to its meaning. |
| `parent_correlation_id` | `Optional[str]` | For traceability across spawn boundaries (Mimir → child agent). |

### 5.2 `:claim/blob-put`

| field | type | notes |
|---|---|---|
| `hash` | `str` | sha256:… |
| `size_bytes` | `int` | |
| `content_type` | `str` | Verbatim from request header, or `"application/octet-stream"` if absent. |
| `session_id` | `str` | |
| `duplicate` | `bool` | `True` if content was already present; useful for retroactive analytics on dedup ratio. |

Both kinds flow through existing `txn.commit` → audit chain hashes them in identically to existing datom kinds. No bypass paths, no new persistence machinery.

## 6. Blob store

- **Backend**: filesystem
- **Root**: `<persistence_root>/blobs/` where `persistence_root` is the existing persistence data directory
- **Sharding**: git-style two-char prefix → `<root>/<sha256[:2]>/<sha256[2:]>`
- **Idempotency**: write-if-absent. `os.link()` from a temp file written under the same directory to make the put atomic on POSIX.
- **Read path**: direct file open, no caching layer in v0.1
- **Concurrency**: filesystem semantics handle concurrent puts (same content → same target inode via link). Concurrent puts of the same hash are safe; first-writer-wins on the actual byte write, others see existing file.

**Capacity / GC**: explicit non-goal in v0.1. The tenant is single-host, single-user, and a typical Claude Code session blob volume is on the order of MB-to-low-GB per week. Quota / GC / retention design lives in a separate Phase 2.x ticket once we have utilization data.

## 7. Trust posture and auth

The HTTP surface has **two trust layers** with distinct responsibilities. 2.1c lands the first; 2.1c.5 lands the second (§16).

### 7.1 Layer 1 — transport access control (2.1c)

- **Mechanism**: bearer token in `Authorization: Bearer <token>` header.
- **Token source**: env var `PERSISTENCE_API_KEY` (read at server start, hot-reload not in scope).
- **Loopback bypass**: requests originating from `127.0.0.1` or `::1` skip the bearer check. Justification: on Mac the Claude Code hooks emit from the same host as the substrate; plumbing API keys through hook environment is friction with negligible security gain (any process on localhost already has shell-level reach into the substrate process anyway). Tailscale-mode remote access still requires the token.
- **HTTPS**: out of scope. Local dev uses loopback; remote access uses Tailscale, which provides transport encryption. Reverse-proxy / public TLS is a deployment concern, not a substrate concern.

This layer answers the question "is the caller allowed to talk to the server at all." It does NOT answer "who is the caller and is their claim authentic."

### 7.2 Layer 2 — caller identity attestation (2.1c.5, see §16)

Bearer tokens are blanket access — a leaked or shared token gives an attacker the substrate's claim namespace. The 2.1c design accepts this for v0.1 because:
- Single-host, single-user pre-alpha
- Loopback bypass is the dominant code path during development
- Caller-identity work is real (~1-2 days, Ed25519 keypair registry + signed payload verification + revocation list)

Caller identity is rescoped to **2.1c.5** with a **falsifiable acceptance signal** (xfail-strict marker):

> Tailscale-mode emit (i.e. non-loopback request) MUST be rejected when no caller signature is present, even if the bearer token is valid.

The marker is added in 2.1c (xfail-strict) and flips to passing when 2.1c.5 lands. This is the W3 honest-rescope pattern from `CLAUDE.md`. See §16.

### 7.3 Trust posture summary

| Question | 2.1c | 2.1c.5 |
|---|---|---|
| Can the caller reach the server? | bearer + loopback | bearer + loopback |
| Is the caller who they say they are? | **NO — accepted for v0.1** | Ed25519 verified |
| Is what the caller claims happened actually true? | **Never — substrate cannot verify** | Never — out of substrate's epistemic reach |
| Is the claim cryptographically pinned to a specific caller? | No | Yes (`:caller/identity` attribute) |

## 8. Module layout

Two new top-level modules, sibling to existing `fact/`, `effect/`, `plan/`, `txn/`, `replay/`, `spec/`, `repl/`, `sdk/`.

```
src/persistence/claim/                 # NEW — trust boundary, transport-agnostic
├── __init__.py                        # public API: CLAIM_KINDS, is_claim_kind,
│                                      #   validate_attrs, CallerIdentity (stub)
├── _registry.py                       # CLAIM_KINDS frozenset, per-kind schemas
├── _validate.py                       # validate_attrs(kind, attrs) → attrs | raises
└── _identity.py                       # CallerIdentity dataclass + attest() stub
                                       #   (filled in 2.1c.5 with Ed25519 verify)

src/persistence/http/                  # NEW — transport, depends on persistence.claim
├── __init__.py
├── __main__.py                        # uvicorn entry: `python -m persistence.http`
├── server.py                          # FastAPI app, lifespan, DI
├── auth.py                            # Bearer + loopback dependency (Layer 1)
├── schemas.py                         # Pydantic models (ClaimEmitRequest, etc.)
├── blob_store.py                      # Filesystem CAS, sha256 sharded
└── routes/
    ├── __init__.py
    ├── claim.py                       # /v1/claim/{emit,query}
    └── blob.py                        # /v1/blob/{put,get}
```

**Dependency direction:** `http/` imports `claim/`. `claim/` imports only `persistence.txn` and `persistence.fact`. Neither imports the other in reverse. The existing `sdk/mcp/_server.py` is left untouched.

**Curated SDK surface (additive to `s.fact`, `s.txn`, `s.plan`, etc.):**
- `s.claim.kinds` — frozenset of valid claim kinds
- `s.claim.is_kind(kind: str) -> bool` — discriminator for query/replay filters
- `s.claim.validate(kind, attrs) -> attrs` — total-function validator (raises `ClaimValidationError`)
- `s.claim.identity` — `CallerIdentity` namespace (stub in 2.1c, real in 2.1c.5)

All `s.claim.*` are `@experimental("v0.9.x")`.

## 10. Tests

Target test count: **~45-55 new tests** (raised from pre-amendment ~30-40 to cover the `persistence.claim/` module + kind-restriction enforcement). Final pyright posture: 0/0/0 on touched files.

### 10.1 Claim module tests (`tests/claim/test_registry.py`, `test_validate.py`)
- `CLAIM_KINDS` is a frozenset, exact membership: `{":claim/tool-exec", ":claim/blob-put"}`
- `is_claim_kind(":claim/tool-exec")` → `True`; `is_claim_kind(":llm/decision")` → `False`; `is_claim_kind("garbage")` → `False`
- `validate_attrs(":claim/tool-exec", valid_attrs)` returns canonical attrs
- `validate_attrs(":claim/tool-exec", missing_required_field)` raises `ClaimValidationError`
- `validate_attrs(":claim/tool-exec", body_summary_>512_chars)` raises
- `validate_attrs(":claim/tool-exec", body_disposition="bogus")` raises (Literal enforcement)
- `validate_attrs(":claim/blob-put", valid_attrs)` returns canonical attrs
- `validate_attrs(":not/a-claim", anything)` raises `UnknownClaimKindError`
- `CallerIdentity.attest(...)` stub returns `None` in 2.1c (xfail-strict marker for 2.1c.5 wires real verify — see §16)

### 10.2 HTTP contract tests (`tests/http/test_routes_claim.py`, `test_routes_blob.py`)
- `POST /v1/claim/emit` — happy path single, happy path multi, atomic-failure (one bad claim → no commit), schema-violation rejected, **kind-restriction enforcement: emit `:llm/decision` rejected `400 not_a_claim_kind`**, unknown claim kind rejected, oversize body rejected
- `POST /v1/blob/put` — happy path, idempotent (same content twice → `duplicate: true`, single file on disk, two `:claim/blob-put` datoms in audit chain), oversize rejected, missing `X-Session-Id` rejected
- `GET /v1/blob/get/:hash` — happy path, 404 on absent, raw bytes round-trip exact equality
- `GET /v1/claim/query` — by `session_id` only, with `since`, with `kind` filter, **kind-restriction: `kind=:llm/decision` rejected `400 not_a_claim_kind`**, pagination via `next_since`, `limit` enforced, `session_id` missing rejected, **fact datom (e.g. `:llm/decision` previously emitted in-process) NOT returned even when no `kind` filter set**

### 10.3 Auth tests (`tests/http/test_auth.py`)
- Loopback request without token → 200
- Non-loopback request without token → 401
- Non-loopback request with valid token → 200
- Non-loopback request with malformed `Authorization` header → 401
- **(xfail-strict for 2.1c.5)** Non-loopback request with valid token but no caller signature → 401. Marker flips to PASS when 2.1c.5 lands. See §16.

### 10.4 Roundtrip / integration tests (`tests/http/test_roundtrip.py`)
- Emit `:claim/tool-exec` → query by `session_id` returns it byte-for-byte
- Put blob → claim emit referencing `body_hash` with `body_disposition="blobbed"` → query → fetch blob → contents match
- 100-emit fuzz: emit 100 random claims, query all back, audit chain head advances monotonically and is reachable from genesis
- **Mixed-namespace audit chain integrity:** emit a `:llm/decision` fact in-process AND a `:claim/tool-exec` via HTTP, verify both hash into the same chain in tx-order, and `/v1/claim/query` returns only the claim

### 10.5 Audit chain tests (`tests/http/test_audit_chain.py`)
- Audit chain head before emit + emit + audit chain head after = expected hash relation
- Concurrent emit from 2 simulated processes (`asyncio.gather`) — datom IDs distinct, audit chain remains linear, no missing tx
- `:claim/blob-put` datoms appear in audit chain in put order

### 10.6 Hypothesis property tests (`tests/http/test_props.py`, `tests/claim/test_props.py`, `@max_examples=200`)
- Round-trip property: any valid `ClaimEmitRequest` → emit → query returns equivalent claim
- `validate_attrs` totality: any string `kind` × any JSON-shaped `attrs` → returns valid attrs OR raises a documented exception (never returns invalid, never silently drops)
- Idempotency property: any `bytes` blob put twice yields same hash, single on-disk file, `duplicate: true` on second
- Pagination property: emit N claims, paginate with various `limit`s, union of pages = full set, no duplicates

### 10.7 Smoke / hermetic
- Add `tests/http/test_main_smoke.py` with hermetic `subprocess.run(env=…, cwd=…)` per the F1 lesson from Phase 2.1a (no reliance on inherited `PYTHONPATH=src`)

## 11. ARIS pre-mortem (self-eval before R1)

Surfacing risks now so R1 either confirms them addressed or flags new ones.

| risk | severity | mitigation |
|---|---|---|
| **Substrate accountability invariant erodes — claims-as-stored ≠ claims-as-verified** | HIGH | Addressed by namespace separation (`:claim/*` distinct kind prefix, `persistence.claim/` module owns the trust boundary, HTTP emits cannot mint fact-kinds). Trust posture explicit in §5.0, §7. Cryptographic pinning of *which caller* made the claim is the second half — rescoped to **2.1c.5** with falsifiable acceptance signal (§16). |
| **Audit chain race under concurrent emit** | HIGH | Single `txn.commit` per request; rely on existing txn serialization. Concurrent-emit test in §10.5 verifies. |
| **Blob store fills disk** | MEDIUM | `PERSISTENCE_BLOB_MAX_BYTES` per-put cap; total-disk monitoring is operational concern, not 2.1c scope. Document in CHANGELOG. |
| **Loopback-bypass is too permissive** | MEDIUM | Trade-off accepted for v0.1: any process on localhost owns the substrate already. Documented justification §7.1. Tightened in 2.1c.5 — non-loopback requests will require caller signature on top of bearer (§16). |
| **Bearer token in env var (cleartext at rest)** | LOW | Standard for v0.1. Secret-manager integration deferred. |
| **`session_id` is opaque to substrate** | LOW (intentional) | Substrate doesn't validate session_ids — that's the consumer's contract. Documented. Reduces coupling. With caller identity (2.1c.5), session_id forgery is bounded to the caller's blast radius. |
| **`body_summary` storage cost** | LOW | Capped at 512 chars in schema; enforced at validation layer. |
| **HTTP server adds startup time / memory footprint** | LOW | Optional component — `python -m persistence.http` is opt-in; in-process SDK usage unaffected. New deps (FastAPI, uvicorn) gated under optional install group `persistence[http]`. |
| **Pydantic version drift** | LOW | Pin to existing pyproject.toml version; no new dep. |
| **FastAPI is a new dep** | LOW-MED | New top-level dep, optional install group. Justified: standard, well-maintained, async-native. Alternative (raw `aiohttp`) considered and rejected as more boilerplate for the same outcome. |
| **`persistence.claim/` module is dead weight if HTTP is the only emitter forever** | LOW | Accepted: even if HTTP stays the sole transport, having the trust boundary in a transport-agnostic module simplifies the 2.1c.5 caller-identity work and any future Unix-socket/gRPC/MCP-write transports. |

## 12. Out of scope (YAGNI — explicit non-goals)

- ❌ **Caller identity attestation (Ed25519-signed claims)** — rescoped to **2.1c.5** with falsifiable acceptance signal (§16). 2.1c lands the namespace and the module skeleton so consumers don't have to migrate later.
- ❌ **HTTP exposure of fact-namespace datoms** (e.g. `:llm/decision`, `:plan/step`). Permanently out of scope for 2.1c.x. If a use case emerges, it gets its own design pass with its own ARIS — facts crossing the HTTP boundary is a strictly distinct trust posture from claims doing so.
- ❌ **Cross-namespace query** (claim + fact in one HTTP response) — same rationale.
- ❌ FTS / BM25 over `body_summary` (downstream context-mimir Phase 2)
- ❌ Datalog query expressions at the HTTP layer — structured filters only
- ❌ Blob GC / retention policy / quota enforcement
- ❌ Multi-tenant auth, scoped tokens, RBAC
- ❌ HTTPS termination — Tailscale or reverse-proxy concern
- ❌ Streaming endpoints / WebSocket / SSE for live tailing
- ❌ Compatibility with Mimir's full `/v1/dispatch` schema — only the substrate primitives Mimir's dispatch will internally wrap
- ❌ Cross-session query (queries always require `session_id`) — global queries deferred until we know the use case
- ❌ Claim retraction / amendment via HTTP (substrate-internal concern, not exposed even after 2.1c.5)
- ❌ Audit chain verification endpoint (separate diagnostics surface, not 2.1c)
- ❌ Hot-reload of `PERSISTENCE_API_KEY` (restart required)

## 15. Downstream consumer (informational, NOT in scope of 2.1c)

For context only — context-mimir (separate repo) will consume this surface as follows. Listed here so the substrate-side reviewer can sanity-check the endpoint shapes against the actual usage pattern.

**Capture path (Claude Code PostToolUse hook → persistence-os):**
1. Hook receives tool result via stdin (Claude Code hook protocol).
2. If `len(result.body) > 2 KB`: hook does `POST /v1/blob/put` with raw body, captures returned `hash`. (Substrate emits `:claim/blob-put`.)
3. Hook generates `body_summary` (rule-based for cheap tools, LLM-extracted for `Read`/`WebFetch`).
4. Hook does `POST /v1/claim/emit` with `:claim/tool-exec` carrying `body_hash` (or `None`) + summary + `body_disposition`.
5. Hook returns rewritten tool result to Claude Code: `<<sandboxed:tool=Bash hash=ab12cd size=56KB hint="3 fr matches"/>>`. The original 56KB is now out of context.

**Retrieval path (Claude Code PreCompact hook → persistence-os):**
1. Hook fires when Claude Code is about to compact.
2. Hook does `GET /v1/claim/query?session_id=<this>&kind=:claim/tool-exec&limit=200`.
3. Hook ranks by recency × `body_summary` BM25 against current task description (BM25 done client-side in v0.1, server-side FTS deferred).
4. Hook returns top-N digest as `additionalContext` to inject before compaction.

**Model-explicit retrieval (MCP tool wrapper):**
- `ctx_search(query)` MCP tool → `GET /v1/claim/query` + client-side ranking → returns digest
- `ctx_fetch(hash)` MCP tool → `GET /v1/blob/get/:hash` → returns blob inline (model has paid the token cost knowingly)

These flows are NOT implemented as part of 2.1c. They live in the context-mimir repo and depend on this design landing. Note that context-mimir running on Mac (loopback) is unblocked by 2.1c alone; running over Tailscale needs 2.1c.5.

## 16. v0.9.x rescope — Phase 2.1c.5: caller identity attestation

**Pattern:** W3 honest-rescope per `CLAUDE.md` ("a finding that can't be fixed in scope gets queued as a separate v0.9.x track with a falsifiable acceptance signal").

**Why rescoped, not folded:** Caller identity is genuinely 1-2 days of design + implementation (Ed25519 keypair registry, signed-payload canonicalization, verification pipeline, revocation list, key-rotation protocol, install-time keygen UX). Folding into 2.1c would double the scope and dilute ARIS focus. Splitting lets 2.1c land its core architectural decision (claims layer + namespace separation) while 2.1c.5 lands the cryptographic completeness as a focused follow-up.

**Sized:** ~300-500 LOC across `persistence/claim/_identity.py` + `persistence/http/auth.py` + key-registry CLI command. ~1-2 days of work.

**Target landing:** before context-mimir's first non-loopback emit, AND before any Tailscale-mode external consumer comes online. Calendar target: within 1 week of 2.1c merge.

**Falsifiable acceptance signal (xfail-strict marker, added in 2.1c, flips to PASS when 2.1c.5 lands):**

```python
# tests/http/test_auth.py
@pytest.mark.xfail(strict=True, reason="2.1c.5 — caller identity attestation")
def test_non_loopback_without_caller_signature_rejected_even_with_valid_bearer():
    """
    A non-loopback request carrying a valid bearer token but NO caller signature
    must be rejected 401. This is the load-bearing trust property that bearer-only
    auth (2.1c) cannot enforce — it requires Ed25519 caller identity attestation
    (2.1c.5). When this test starts passing, 2.1c.5 has landed.
    """
    response = client_with_valid_bearer_no_signature.post(
        "/v1/claim/emit",
        json=valid_claim,
        headers={"X-Forwarded-For": "100.64.0.5"},  # simulate Tailscale CGNAT
    )
    assert response.status_code == 401
    assert response.json()["error"] == "caller_signature_required"
```

**Companion stub design doc:** `docs/plans/2026-05-04-phase-2.1c.5-caller-identity-design.md` (one-pager — full design lands when 2.1c.5 is dispatched).

**Out of scope for 2.1c.5 in turn (queued further):** key-rotation automation, multi-key per caller, hardware-key (YubiKey) integration, attestation chains for delegated callers (parent agent → child agent), revocation-list distribution. These are 2.1c.6+ if they ever come up.

## Appendix B — comparison to mksglu/context-mode

`mksglu/context-mode` (HN #1, ELv2 license) ships the same problem space — context-window optimization for AI coding agents — but stores everything in a sidecar SQLite + FTS5 with no audit-chain, no replay, no `DB.fork`, and no decoupling between log (datoms) and bulk data (blobs). Our design uses `mksglu/context-mode` as the reference for the surface behavior consumers will need (sandbox tool output, session-scoped retrieval, think-in-code via subprocess), but shapes the storage layer to fit persistence-os' substrate primitives. ELv2 prevents direct vendoring; reimplementation is required regardless.
