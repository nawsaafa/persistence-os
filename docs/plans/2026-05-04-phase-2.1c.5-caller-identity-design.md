## 1. Why this exists as a separate phase

Phase 2.1c lands the HTTP claim surface with bearer-token + loopback-bypass auth. That answers **"is the caller allowed to talk to the server"** but not **"is the caller authentically who they say they are."** A leaked bearer token = blanket access to the claim namespace. For Mac-localhost development this is acceptable (loopback bypass dominates); for anything over Tailscale or any future remote consumer it is not.

Folding this into 2.1c would have:
- Doubled the change surface (~700-1050 → ~1100-1500 LOC)
- Pulled in keypair UX, registry persistence, revocation, key-rotation protocol — all genuinely 1-2 days of focused work
- Diluted ARIS focus on the load-bearing 2.1c decision (claims-layer separation)

Splitting follows the W3 honest-rescope pattern from `CLAUDE.md`: the finding (substrate accountability invariant erodes under HTTP emit) gets queued as a separate v0.9.x track with a **falsifiable acceptance signal** that flips when the fix lands.

## 3. Sketch — what 2.1c.5 ships (full design TBD)

The full design lands when 2.1c.5 is dispatched. Sketch only:

### 3.1 Caller registry
- File-backed registry under `<persistence_root>/claim/callers/<caller_id>.pub`
- One Ed25519 public key per registered caller
- CLI: `python -m persistence.claim.callers register --caller-id=<id> --pubkey-file=...`
- CLI: `python -m persistence.claim.callers revoke --caller-id=<id> --reason=...` (records revocation tx; future emits from that caller fail; past datoms remain valid)

### 3.2 Signed payload protocol
- Caller signs canonical-JSON of the claim payload (`canonical_dumps` from `persistence.effect.canonical`, already in use for 2.1b audit chain)
- Header `X-Caller-Id: <caller_id>` + `X-Caller-Signature: <base64-ed25519-sig>`
- Server verifies signature against registered pubkey for `<caller_id>` before forwarding to `persistence.claim.validate_attrs`
- Verified `caller_identity` (caller_id + pubkey-hash + signature-tx) is added as a substrate-stamped field on every emitted datom

### 3.3 Key UX
- Install-time keygen for context-mimir, Mimir CLI: `<consumer> init` generates keypair under `~/.config/<consumer>/keypair`, registers pubkey via the substrate CLI
- Loopback bypass keeps working — local dev unchanged
- `caller_identity` field is `null` for loopback emits, populated for verified non-loopback emits

### 3.4 Replay safety
- Replay re-applies the recorded datom including its `caller_identity` field
- No re-verification at replay time (sig was verified at emit time and pinned in the chain)
- Revocation does NOT invalidate past datoms — only blocks future emits from that caller_id

## 4. Out of scope for 2.1c.5 in turn

- ❌ Key rotation automation (manual rotate via revoke + re-register for v0.1)
- ❌ Multi-key per caller
- ❌ Hardware-key (YubiKey, Secure Enclave) integration
- ❌ Delegation chains (parent agent → child agent attestation)
- ❌ Revocation list distribution / propagation across multiple substrate instances

These are 2.1c.6+ if/when use cases emerge.

## 5. ARIS posture

2.1c.5 will get its own ARIS hard-mode pass when dispatched. Pre-mortem topics:

- Canonical-JSON drift between caller and server (must be byte-identical for sig to verify)
- Replay clock-skew / timestamp validity windows (or none — pin sig at tx-time, no window)
- Revocation tx-ordering vs concurrent emit-from-revoked-caller race
- Pubkey registry corruption / tampering (file-system permissions, optional checksum)
- Bootstrap problem: who registers the first caller? (Substrate admin = whoever has filesystem access to `<persistence_root>/claim/callers/`)
