## 1. Why this exists as a separate phase

Phase 2.1c lands claim emit/query HTTP routes backed by `substrate.fact.transact(datoms)`.
During ARIS R1.1, the controller verified interactively that:

- `Substrate.open()` defaults `audit=True` — that is NOT the issue.
- `s.fact.transact(datoms)` does **NOT** populate `substrate._canonical_audit_entries`.
- `s.txn.dosync() + tx.add_facts(...)` also does not populate audit entries.
- The canonical audit chain is only populated by `tx.effect(...)` calls (effect-level writes), not fact-level writes.

The design (§§ 19, 42, 275) claimed "claims flow through the same audit chain as facts" — this was incorrect. The substrate-side gap was pre-existing: Phase 2.1b's `Coder._decide` also uses `s.fact.transact` for `:llm/messages` and `:llm/decision` datoms and has the same uncovered gap.

The original implementation silently returned `"sha256:placeholder"` from `_extract_audit_chain_head`, masking the gap entirely. Phase 2.1c R1.1 removes the placeholder and returns `None` — honest and correct per the W3 honest-rescope pattern from `CLAUDE.md`.

Folding the fix into 2.1c would have required:
- Understanding and modifying the substrate's `fact.transact` internals (non-trivial)
- Or wrapping every claim emit in a `tx.effect(":claim/emit", ...)` call (requires understanding effect-audit coupling)
- Either path is a distinct design exercise requiring its own ARIS pass

Splitting follows the W3 honest-rescope pattern: the finding gets queued as a separate v0.9.x track with a **falsifiable acceptance signal** that flips when the fix lands.

## 3. Sketch — what 2.1c.6 ships (full design TBD)

Two candidate approaches — full design TBD when 2.1c.6 is dispatched.

### 3.1 Approach A: `tx.effect(":claim/emit", ...)` wrapper in the emit handler

Wrap the `substrate.fact.transact(datoms)` call in the emit handler with an effect call that both:
1. Writes the claim datoms via the substrate
2. Records the emit as an effect in the canonical audit chain

```python
# sketch — contract TBD
substrate.txn.dosync(lambda tx: (
    tx.add_facts(datoms) or
    tx.effect(":claim/emit", {"claim_ids": claim_ids, "session_id": ...})
))
# _canonical_audit_entries[-1].id is now the real head
```

This is the lighter-weight approach — adds an effect wrapper without modifying `fact.transact` internals.

### 3.2 Approach B: Extend `s.fact.transact` to write canonical audit entries

Modify `fact.transact` at the substrate level to optionally record a canonical audit entry alongside each fact batch. This is load-bearing for correctness — it would close the gap for all `s.fact.transact` callers (including `Coder._decide` which has the same gap for `:llm/messages` / `:llm/decision` datoms).

This is the broader fix but has more surface area and needs careful design around:
- Whether every `s.fact.transact` call should produce an audit entry, or only certain ones
- How the audit entry content is determined for arbitrary datom batches
- Backward compatibility with existing callers

### 3.3 Approach C: Dedicated claim-audit event via substrate effect at server startup

Register a substrate effect handler for `:claim/emit` events at `build_app` time, wired to the canonical audit stack. The HTTP handler calls `s.effect.perform(":claim/emit", {...})` instead of `s.fact.transact` directly.

This is closer to the Phase 2.1b pattern (Coder uses `s.effect.perform(":llm/call", ...)` for LLM calls). Adds clean separation — claims go through the effect audit stack, not the fact stack.

**Recommendation (tentative):** Approach C is most consistent with Phase 2.1b patterns and keeps the audit contract explicit. But this is a sketch — the full design should evaluate all three.

## 5. ARIS posture

2.1c.6 will get its own ARIS hard-mode pass when dispatched. Pre-mortem topics:

- **Substrate coupling risk**: modifying `fact.transact` internals (Approach B) is high surface area — must not break existing tests
- **Effect-audit stack coupling**: Approach C pulls the emit handler into the effect lifecycle — what happens if the effect handler fails? Is the claim transaction rolled back?
- **Replay correctness**: does the audit chain head at replay time match the original? Needs G3 replay byte-identity test equivalent for the HTTP layer
- **Concurrent emit serialization**: two concurrent emits must produce monotonically increasing audit heads — verify under `asyncio.gather` (deferred test from T16)
- **`_canonical_audit_entries` list semantics**: is it a global singleton per Substrate instance, or per-transaction? Must verify append semantics under concurrent fact writes
