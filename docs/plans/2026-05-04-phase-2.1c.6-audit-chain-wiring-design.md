## 1. Why this exists as a separate phase

Phase 2.1c lands claim emit/query HTTP routes backed by `substrate.fact.transact(datoms)`. During ARIS R1.1, the controller verified interactively that:

- `Substrate.open()` defaults `audit=True` â€” that is NOT the issue.
- `s.fact.transact(datoms)` does **NOT** populate `substrate._canonical_audit_entries`.
- `s.txn.dosync() + tx.add_facts(...)` also does not populate audit entries.
- The canonical audit chain is only populated by ops in `CANONICAL_AUDIT_WRAPPED_OPS` going through `runtime.perform()`. See `src/persistence/effect/_audit_stack.py:71-87`.

The stub-doc claim that the substrate gap "also affects 2.1b coder retroactively" turns out to be **partially incorrect on closer reading**. The Phase 2.1b coder DOES advance the audit chain â€” on each `s.effect.perform(":llm/call", ...)`. The `:llm/call` op IS in `CANONICAL_AUDIT_WRAPPED_OPS` (added in Phase 2.1b at `_audit_stack.py:83-86`). The `:llm/messages` and `:llm/decision` datoms transacted via `s.fact.transact` ARE supporting facts that bracket the LLM call, NOT independent auditable events. The audit anchor is `:llm/call`. This is correct-by-design, not a gap.

**That reframes 2.1c.6:** it narrows to wiring `:claim/emit` and `:blob/put` as audit-wrapped ops, matching the established `:llm/call` pattern.

## 3. Approach (locked after design pass)

**Approach C, narrowed.** HTTP claim/blob routes write facts first (provenance survives), then perform an audit-anchor effect via `s.effect.perform`. Two new audit-wrapped ops join the canonical set; both are audit-only (no functional side effect beyond producing the audit entry).

`_extract_audit_chain_head(substrate)` in the same file returns the real head:

```python
def _extract_audit_chain_head(substrate) -> str:
    """Return the substrate's canonical audit chain head id.

    Phase 2.1c.6: audit chain advances on every :claim/emit perform.
    Returns the most recent AuditEntry.id from _canonical_audit_entries.
    """
    entries = substrate._canonical_audit_entries
    if entries is None or len(entries) == 0:
        # Substrate opened with audit=False, or no perform yet â€”
        # neither should occur in production HTTP serving (build_app
        # always opens with audit=True default and the perform happens
        # before this helper is called). Defensive: return a sentinel
        # the caller can detect.
        raise RuntimeError(
            "audit chain head requested but canonical chain is empty; "
            "indicates a missing :claim/emit perform or audit=False mode"
        )
    return entries[-1].id  # str of shape "sha256:<hex>"
```

### 3.4 Op argument shapes (canonical, append-only contract)

These shapes are stable v0.9.x. Future fields ADD â€” never modify or remove.

```python
# :claim/emit  (R1 IMPORTANT 2 fold: dropped `session_id` â€”
# session is a per-claim attribute, not a batch attribute, and the
# 2.1c contract permits mixed sessions per emit batch.)
{
    "claim_ids": list[str],         # per-claim entity ids
    "tx": int,                      # the substrate tx id of the fact transact
    "kind_counts": dict[str, int],  # e.g. {":claim/tool-exec": 3, ":claim/blob-put": 1}
}

# :blob/put  (session_id retained â€” POST /v1/blob/put has a top-level
# X-Session-Id header so the route boundary owns a single session for
# the put, unlike emit batches.)
{
    "hash": str,                    # "sha256:<hex>"
    "size_bytes": int,
    "session_id": str,
    "duplicate": bool,              # True iff the put was a no-op (CAS hit)
}
```

**What gets stored on the AuditEntry (R1 BLOCKING 1 fold):** the audit middleware computes `args_hash = canonical_hash(args)` after popping the `_txn_commit` sentinel; ONLY `args_hash` is persisted on the `AuditEntry` (verified at `effect/handlers/audit.py:52` field def + line 495 hash computation). The raw `args` dict is NOT round-tripped through `to_edn`/`from_edn`. This is by design â€” the audit entry attests "an emit with these arguments happened at time T," not "here are the arguments themselves." Test assertions on the entry must use `args_hash == canonical_hash(expected_args_dict)`, not `entry.args == expected_args_dict`.

This means:
- **Replay determinism**: the same args produce the same entry hash given the same prev-hash + clock + signing key.
- **Privacy**: claim payloads are NOT in the audit args â€” only ids, counts, and tx. The full claim attrs live in the fact log; the audit entry attests "an emit happened at time T with these ids."
- **Test surface**: G2 tests must hash the expected args dict via the same `canonical_hash` function the middleware uses (importable from `persistence.effect.canonical`).

- **No changes to the claim module public API** (`CLAIM_KINDS`, `validate_attrs`, `is_claim_kind`, `CallerIdentity`, `s.claim.*` namespace).
- **No changes to `/v1/claim/query`** response shape or pagination semantics.
- **No changes to CallerIdentity** â€” the 2.1c.5 track owns Ed25519 caller-attestation. When 2.1c.5 lands, the audit entry payload may extend with `caller_id` â€” additive, not breaking.
- **No changes to the Phase 2.1b coder** `_decide` flow. The `:llm/call` op is already audit-wrapped; `:llm/messages` and `:llm/decision` are correctly designed as supporting provenance facts that bracket the auditable call. If a future phase wants to audit-anchor the decision-parsing step independently, that's a separate design.
- **No changes to `s.fact.transact`** internals. The substrate's most load-bearing function stays untouched; the audit gap is closed at the call-site (HTTP route) by adding an effect perform, not at the substrate level.
- **No new database backend, no new wire format, no new dependencies.**

## 5. ARIS posture â€” pre-mortem topics

These are the foreseeable failure modes. Each will be exercised by a test gate (see Â§ 6).

### 5.1 Substrate coupling risk

**Risk:** modifying `_audit_stack.py` to add ops can drift from existing wraps-set assumptions. The `make_audit_handler(entries, wraps=...)` middleware iterates `wraps` to install per-op clauses; the raw terminator clauses must mirror exactly.

**Mitigation:** The drift-pin test pins `frozenset(CANONICAL_AUDIT_WRAPPED_OPS)` and `frozenset(CANONICAL_AUDIT_RAW_OPS)` against expected values. Any addition to one without the other (or against the documented exclusion `:llm/call âˆ‰ RAW_OPS`) breaks the test loud.

### 5.2 Effect-audit failure isolation

**Risk:** if `:claim/emit` perform raises (e.g., signing key invariant, audit entries list is None due to `audit=False`), does the claim transaction roll back?

**Decision:** No. Fact-write happens BEFORE effect-perform. Per Â§ 3 design choice locked with user, "provenance survives audit failure" â€” same shape as 2.1b coder where `:llm/messages` survives even if `:llm/call` raises. The HTTP route returns 500 on audit-perform failure (FastAPI default), but the fact log is consistent. A retry by the client would write duplicate facts; idempotency is NOT a 2.1c.6 concern (it isn't in 2.1c either; deferred).

**Test:** `tests/http/test_audit_chain.py::test_audit_perform_failure_does_not_roll_back_facts` â€” install a handler that raises on `:claim/emit`, POST a valid emit, verify (a) HTTP returns 500, (b) the claim datoms ARE in the fact log, (c) the audit chain did NOT advance. **NEW falsifiability gate**.

### 5.3 Replay correctness (G3 byte-identity)

**Risk:** the audit entry's `args_hash` includes a clock-ish field (the audit middleware reads `:clock/now` for `recorded_at`). Replay with a pinned clock should produce byte-identical audit entries.

**Mitigation:** the existing G3 pattern from Phase 2.1b coder applies. The audit middleware reads clock via `make_system_clock_handler()`; replay swaps it for a fixed-clock handler. The new `tests/http/test_audit_replay_byte_identity.py` runs two HTTP emits on fresh substrates with pinned clock + pinned signing key + pinned audit-args (deterministic claim ids and session_id) and asserts the resulting `_canonical_audit_entries[-1].id` is byte-identical across runs.

**Subtlety:** the existing 2.1c routes use wall-clock `datetime.now()` and `uuid.uuid4()` (with `# noqa: wall-clock` per design Â§ 6). For the replay test to work, the test must inject a deterministic substrate state (no real wall-clock drift in the test path). This matches 2.1b's `_FixedClock` precedent.

### 5.4 Concurrent emit serialization

**Risk:** two concurrent `:claim/emit` performs against the same substrate produce two AuditEntry records; the prev-hash chain links must be monotonic and consistent.

**Mitigation:** the audit middleware's `entries.append(...)` is the linearization point. The `_MirroringEntryList` (or its underlying list) is appended-to under the GIL â€” Python list append is atomic. Two concurrent `runtime.perform` calls produce two entries; both reference some prev hash, but only one will be the "real prev" of the other. Under `asyncio.gather`, the order is determined by the asyncio task scheduler.

**Existing test coverage:** Phase 2.1c T16 already exercises concurrent emit (`tests/http/test_audit_chain.py::test_concurrent_emit_serializes`). It is NOT currently strict-xfail â€” it asserts the post-condition that the chain is internally consistent (each entry's prev-hash references a valid prior entry). Verify this test STILL passes after 2.1c.6.

### 5.5 `_canonical_audit_entries` list semantics

**Risk:** is the entries list a global singleton per Substrate instance, or per-transaction?

**Answer (verified in code):** per Substrate instance. `Substrate.open` allocates exactly one `_MirroringEntryList(mirror=audit_entries)` and binds it to `_canonical_audit_entries` for the substrate's lifetime. All performs against that substrate's runtime append to the same list. This is the intended design â€” the audit chain is session-scoped (one chain per substrate-open), not per-transaction.

This means: **HTTP server uptime = one canonical audit chain.** Server restart = new chain (with no prev-link to the prior session, by design â€” replay rebuilds chains, not extends across server restarts).

### 5.6 `audit=False` mode (sandbox tests only)

**Risk:** if a test opens `Substrate.open(audit=False)` and emits a claim, `_canonical_audit_entries is None` and the new `:claim/emit` perform raises `Unhandled` (no handler in the bare runtime).

**Mitigation:** the `:claim/emit` op gets a raw terminator from the canonical stack â€” present iff `audit=True`. Under `audit=False`, `Substrate.open` constructs a bare empty `Runtime()` with no handlers. The HTTP server always opens with `audit=True` default (verified at `http/server.py:build_app`); `audit=False` is sandbox-test-only. If a test wants to exercise HTTP routes under `audit=False`, the test must install its own raw `:claim/emit` handler â€” that's expected for sandbox tests, NOT a 2.1c.6 concern.

**Test gate:** `tests/http/test_emit_under_audit_false_raises_unhandled` (NEW, asserts the `Unhandled` raises cleanly with a useful message). Documents the expectation rather than silently passing.

### 5.7 ContextVar coupling â€” `runtime._active` (R1 NICE-TO-HAVE fold)

**Risk:** the audit middleware's `make_audit_handler` calls module-level `perform(":clock/now")` to read `recorded_at`/`latency_ms`. That module-level perform reads `persistence.effect.runtime._active` â€” the ContextVar set by `Substrate.open(audit=True)` and released in `close()`. If a test (or production multi-substrate path) opens substrate A, opens substrate B, then performs against A's runtime while `_active` points at B, the clock read could land on the wrong runtime stack.

**Mitigation in 2.1c.6:** no new exposure beyond the existing canonical stack. The audit handler invocation path is unchanged â€” we only add new wrapped ops. The ContextVar lifecycle invariant (one substrate per opened scope; close releases the token) holds for HTTP serving (single substrate per `build_app` instance, lifetime = server uptime) and for tests with the `app_client` fixture's `app.state.substrate.close()` finally-block (T14.1 lock-in).

**Test fixture discipline:** any 2.1c.6 test opening a fresh substrate explicitly (NOT via `app_client`) MUST close-then-reopen sequentially, OR explicitly activate the right runtime via `with persistence.effect.with_runtime(rt): ...`. The new G3 replay-byte-identity test will use sequential substrates (close one, open next) to avoid the trap. Document this in the test module docstring.

**Out of scope for 2.1c.6:** redesigning the ContextVar lifecycle to support concurrent multi-substrate runtimes is a Phase 2.4a / 2.4b concern. 2.1c.6 inherits the existing invariant.
