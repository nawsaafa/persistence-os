## 1. Overview

Phase 2.2b adds the next batch of audit-anchored substrate ops the persistence-coder ReAct loop needs to be useful: `:git/{diff, status, log, commit}` + `:code/run`. It also closes a load-bearing gap in 2.2a's `_observe` rendering that left the LLM blind on its own past tool outputs.

This phase is the LLM-meets-substrate moment. After 2.2b ships, the coder can read code (`:fs/read` from 2.2a), shell out (`:shell/exec` from 2.2a), execute Python in the soft-isolation sandbox (`:code/run`, new), and inspect/commit git state (`:git/*`, new) — and **see its own outputs** in the next iteration's prompt.

Phase 2.2b explicitly does NOT widen the substrate beyond these ops. Plan AST escalation (2.3a), MCTS branch (2.3b), and REPL pause (2.3d) remain queued.

## 2. Locked design decisions (LD1–LD3)

Locked via 4 codex high adversarial passes during brainstorming (2026-05-05). Process: receipts via Serena scan → controller proposal → adversarial codex (argue against) → fold or defend → iterate → lock.

### LD1 — `:code/run` is a new op, not a re-wiring of `:code/exec`

**Decision:** Add `:code/run` to `CANONICAL_AUDIT_WRAPPED_OPS` only (NOT `RAW_OPS`). New handler `make_code_run_dispatch_handler(*, project_root: Path, scratch_dir: Path) -> Handler` installed at `position="bottom"` by `__main__.py`, mirroring 2.2a's `:fs/*` install pattern. Single-mode substantive-return: clause runs subprocess inline + computes hashes + returns `{stdout, stderr, exit_code, wall_clock_ms, output_hash}` dict.

**Legacy `:code/exec` + `exec_code()` + 12+ tests in `tests/effect/test_code_exec.py` + the F4 strict-xfail are completely UNCHANGED.** This is the load-bearing decider — the F4 strict-xfail is the v0.9.x real-OS-sandbox acceptance signal; preserving it dominates aesthetic concerns about "two ops for one logical effect."

**Why a new op name (rejected: shared `:code/exec` op):** A shared-name dispatch handler would need dual-mode args-shape switching (`source` vs `source_hash` keys) — a footgun where correctness depends on a negative guarantee (never call `k(args)` for one branch). Accepted tradeoff: future audit-replay queries for "all code execution events" must union `:code/exec OR :code/run`.

**Code reuse:** Extract `_run_subprocess` and `_output_hash` from `code.py` as shared helpers (rename `_run_subprocess` → `run_subprocess`; `_output_hash` → `output_hash`; both stay in `code.py`). Both `exec_code` (legacy) and the new `:code/run` handler call them. Keeps timeout/env/subprocess-spawn semantics identical across both ops by SHARED IMPLEMENTATION, not by convention.

**Op shape:**
```
:code/run args:
  source: str            (required; Python source to execute)
  stdin: str = ""        (optional; piped to subprocess stdin)
  timeout_seconds: float = 5.0
  memory_mb: int = 128

:code/run result:
  stdout: str
  stderr: str
  exit_code: int
  wall_clock_ms: int
  output_hash: str       ("sha256:<hex>" of canonical-JSON of {stdout, stderr, exit_code})
```

**Audit chain:** `:code/run` AuditEntry has `args_hash = canonical_hash({"source", "stdin", "timeout_seconds", "memory_mb"})` (source-shape; differs from legacy `:code/exec` audit which is hash-shape). `result_hash = canonical_hash(<result dict>)`.

### LD2 — `:git/*` is a thin wrapper over `:shell/exec`, not a parallel native handler

**Decision:** Add four `:git/*` ops to `CANONICAL_AUDIT_WRAPPED_OPS` only: `:git/diff`, `:git/status`, `:git/log`, `:git/commit`. New handler `make_git_handler(*, project_root: Path, audit_handler_name: str = "audit") -> Handler` with quartet `wraps={":git/diff", ":git/status", ":git/log", ":git/commit"}` mirroring `make_fs_handler` factory pattern. Each clause:

1. validates the op-specific args dict against the schema below (raises `GitArgValidation` on bad input — validation error, NOT a git error string)
2. enforces cwd confinement via `_safe_resolve(args["cwd"], project_root)` (helper from `fs.py`; already accepts `str | Path`; raises `FsCapabilityDenied` on path outside project_root). The op shapes spec `cwd: str` (wire form); `_safe_resolve` returns the resolved `Path` for the subprocess call.
3. constructs the deterministic `argv` per the per-op rules below (B4 fold — pinned `-c color.ui=false -c core.pager=cat`, sorted `paths`, explicit `--` separator, allowed `format` values, etc.)
4. delegates execution via `runtime.perform(":shell/exec", argv=<argv>, cwd=<resolved>, allowlist_version=shell.ALLOWLIST_VERSION, env_allowlist_subset=["PATH", "HOME", "LANG", "LC_ALL", "LC_CTYPE"], timeout_s=<op_default>)` — using the **module-level `runtime.perform`** from `persistence.effect.runtime` (NOT `s.effect.perform` — handler clauses don't have `s` in scope; they import `from persistence.effect.runtime import perform as runtime_perform, mask` and call `runtime_perform(...)` directly, mirroring the audit handler precedent at `audit.py:497-498` / `:516-517`)
5. wraps step 4 in `with mask(audit_handler_name):` — the inner `:shell/exec` perform is hidden from the audit middleware, so only the OUTER `:git/<sub>` AuditEntry is emitted (B3 fold)
6. passes through the result shape from `:shell/exec` (`{exit, stdout, stderr, wall_clock_ms}`) — no per-op extras

**Excluded ops** (out of 2.2b scope): `:git/{push, pull, fetch, checkout, branch, merge, reset, rebase, tag, stash, cherry-pick, amend}`. `:git/push` is forbidden by privacy mandate. The other ref-state mutations are out of 2.2b — branching for MCTS lands in 2.3b via `s.txn.fork`, not git refs.

**Why thin wrapper (rejected: native handler with own subprocess):** A native handler would duplicate ~60 LOC of timeout/env/subprocess/error-shaping that already lives in `shell.py:81-103` (including the macOS bytes-branch for `TimeoutExpired` discovered in 2.2a). Divergence risk is real and unbounded. The thin wrapper inherits `:shell/exec`'s tested execution path. Defense-in-depth (cwd confinement, narrow op shapes, deterministic argv) lives at the wrapper layer, where the LLM's intent is shaped before it hits the shared shell substrate.

**Why narrow op shapes (rejected: arbitrary git flag passthrough):** Bans on `--amend` / `-S` / `--no-verify` / `--allow-empty` are arbitrary policy in code that creates a compatibility burden if ever relaxed. Narrow op shapes enforce safety BY CONSTRUCTION — the schema simply doesn't model those degrees of freedom, so there's nothing to relax. If a future workflow legitimately needs `--amend`, it's a separate v0.9.x op (e.g. `:git/amend`), not a flag toggle on `:git/commit`.

**Op shapes (with B4 determinism pins):**

All argvs start with `["git", "-c", "color.ui=false", "-c", "core.pager=cat"]` (suppress ANSI color + pager regardless of host TERM/git config) before the subcommand. `paths` are always **sorted lexicographically** before insertion (so same input args produce same argv). `--` separator is always emitted before pathspecs.

```
:git/diff args:
  ref: str = "HEAD"          (optional; tree-ish to diff against; rejected if it contains shell metacharacters)
  paths: list[str] = []      (optional; sorted before argv construction)
  cached: bool = False       (optional; emits --cached when True)
  cwd: str = <project_root>  (validated via _safe_resolve)

:git/diff argv (deterministic):
  ["git", "-c", "color.ui=false", "-c", "core.pager=cat", "diff", "--no-color"]
  + (["--cached"] if cached else [])
  + [ref]
  + ["--"] + sorted(paths)

:git/diff result:
  exit: int
  stdout: str       (the diff)
  stderr: str
  wall_clock_ms: int

:git/status args:
  paths: list[str] = []      (optional; sorted before argv construction; default = repo-wide)
  cwd: str = <project_root>

:git/status argv (deterministic):
  ["git", "-c", "color.ui=false", "-c", "core.pager=cat", "status", "--porcelain=v1", "--no-renames"]
  + (["--"] + sorted(paths) if paths else [])

:git/status result:
  exit: int
  stdout: str       (porcelain v1 output, deterministic format)
  stderr: str
  wall_clock_ms: int

:git/log args:
  n: int = 10                 (--max-count; rejected if < 1 or > 1000)
  paths: list[str] = []       (optional; sorted before argv construction)
  format: str = "oneline"     (allowed enum: "oneline" | "short" | "medium"; other values raise GitArgValidation)
  cwd: str = <project_root>

:git/log argv (deterministic):
  ["git", "-c", "color.ui=false", "-c", "core.pager=cat", "log", "--no-color", f"--max-count={n}", f"--pretty={format}"]
  + (["--"] + sorted(paths) if paths else [])

:git/log result:
  exit: int
  stdout: str
  stderr: str
  wall_clock_ms: int

:git/commit args:
  message: str               (required; non-empty after strip; rejected if empty)
  paths: list[str]           (required; non-empty; sorted before argv construction; each path is_relative_to project_root after _safe_resolve)
  cwd: str = <project_root>

:git/commit argv (deterministic):
  ["git", "-c", "color.ui=false", "-c", "core.pager=cat", "commit", "-m", message, "--"]
  + sorted(paths)

:git/commit result:
  exit: int          (0 on success; 1 on "nothing to commit" — wrapper passes through, does NOT raise)
  stdout: str        (porcelain commit output; e.g. "[branch hash] msg\n N files changed...")
  stderr: str
  wall_clock_ms: int
```

**Paths semantics (I2 fold):** `:git/commit` requires `paths` non-empty. Paths-unchanged (no diff) returns `exit=1` with stderr `"nothing to commit, working tree clean"` — the wrapper passes through unchanged, NOT an exception. The LLM can read `exit` and stderr to handle the case. `sha` is NOT returned (was fragile to parse and YAGNI — the LLM can call `:git/log -n 1 --pretty=oneline` to read the new commit if needed).

**Audit chain (ONE entry per `:git/*` op — B3 fold):**

The clause uses `with mask(audit_handler_name): runtime.perform(":shell/exec", ...)` to suppress the inner `:shell/exec` audit. Only the OUTER `:git/<sub>` AuditEntry is emitted by the audit middleware, with:
- `args_hash = canonical_hash({"_op_args": <op-args>, "_argv": <constructed-argv>, "_allowlist_version": shell.ALLOWLIST_VERSION})` — the audited args fold both the LLM's intent AND the actual constructed argv, so replay can verify both layers from a single entry.
- `result_hash = canonical_hash(<result-dict>)` — note that `result_hash` is over the FULL result including `wall_clock_ms`; see B1 fold below for what this means for replay tests.

**Constraint on dispatch context (I1 fold — corrected from R0):** The `:git/*` clause uses module-level `runtime.perform(":shell/exec", ...)` which DOES check `is_in_dosync()` and raises `EffectInIoBlock` if true. The semantics:

- Direct `s.effect.perform(":git/diff", args)` from outside dosync (the coder's `_act` path) → inner `runtime.perform(":shell/exec", ...)` runs OUTSIDE dosync (no dosync context) → succeeds.
- `tx.effect(":git/diff", args)` queued inside a dosync body → queue happens fine; perform fires at COMMIT-REPLAY time via `_replay_effect_intents` AFTER dosync body has exited → inner `runtime.perform(":shell/exec", ...)` ALSO outside dosync at that point → succeeds.
- Direct `s.effect.perform(":git/diff", args)` called from INSIDE a dosync body (which the substrate forbids by contract — direct perform inside dosync is a contract violation regardless of op) → inner call would correctly raise `EffectInIoBlock`. This is the substrate's intended behavior, NOT a `:git/*`-specific limitation.

The real rule the substrate enforces: **no side-effectful direct performs inside dosync; use `tx.effect(op, **kwargs)` to queue.** `:git/*` participates in this rule like every other op. 2.3a Plan AST execution will queue via `tx.effect`, which works correctly.

### LD3 — `build_messages` adds a "Latest action output" section

**Decision:** Modify `src/persistence/coder/_prompt.py:build_messages` to add a NEW section between the task header and the "Recent loop history" block. The section is omitted on iter 0 (when `obs.recent_actions` is empty); otherwise it pulls `obs.recent_actions[-1]` and renders it with explicit field formatting:

```
Latest action output (iter <N>):
  op: <op-string>
  error: <error-string or "none">
  exit_code: <int (if present in result_summary)>
  stdout: |
    <full _summarize_result-truncated stdout, ~600 chars max>
  stderr: |
    <full _summarize_result-truncated stderr, ~600 chars max>
  matches: count=<N>, first_3=[<first 3 entries>]   (only for list-shaped result fields like :fs/glob)
  <other-keys>: <repr-truncated-to-200>
```

Older history entries (iters before the latest) keep the existing `[:200]` cap on `json.dumps(d, sort_keys=True)`. This preserves prompt-cost control while guaranteeing the LLM sees its most recent action's full output.

**Why CHANGE (rejected: NO_CHANGE):** Receipts step (Serena scan) revealed a SECOND truncation in `build_messages` (line 54-58) that the controller initially missed. Math: a `:act/result` JSON-encoded with `sort_keys=True` has alphabetical keys `args_hash, error, latency_ms, op, result_summary` — the first 200 chars never reach `result_summary.stdout`. Concrete failure case: a `:code/run` with a 10KB Python traceback in `result_summary.stderr` is INVISIBLE to the LLM in iter+1's prompt. The LLM cannot fix errors it cannot see. This is "blind on its own outputs" — a structural failure of the ReAct loop, not an edge case.

**Why explicit-field rendering for the latest action (rejected: just bump `[:200]` to `[:1500]` universally):** A universal cap would (a) inflate prompt cost on every history entry regardless of relevance, (b) still not solve list-shaped results like `:fs/glob`'s `matches: list[str]` which can be hundreds of items. Explicit-field rendering with shape-aware collapsing (count + first 3 for lists) gives bounded prompt cost AND full visibility on the last action's stdout/stderr.

**Acceptance signals (load-bearing, three falsifiable tests — I3 fold):**

1. **stdout sentinel** — `tests/coder/test_observe_latest_action.py::test_latest_action_stdout_visible_in_prompt` constructs an `Observation` with one action whose `result_summary.stdout` contains `"STDOUT_SENTINEL_2_2_B"`. Asserts `build_messages(...)[0]["content"]` contains the sentinel verbatim.

2. **stderr sentinel** (motivating-failure-mode coverage) — symmetric test asserting `result_summary.stderr` containing `"STDERR_SENTINEL_2_2_B"` reaches the LLM verbatim. Most tool-error paths land on stderr (Python tracebacks from `:code/run`, git error messages from `:git/*`, command-not-found from `:shell/exec`); without this test the coder is blind to the modal failure case.

3. **exception path with `result_summary=None`** — `_act` records `result_summary=None` when the underlying perform raises (`_session.py:209-220`'s `_record(None, error)` branch). Test constructs an Observation with one action where `result_summary is None` and `error="<exception class>: <message>"`. Asserts the rendered "Latest action output" section shows `op:`, `error:`, and explicitly `(no result_summary — exception path)` (or equivalent placeholder) so the LLM sees the failure mode without crashing the renderer on a missing dict.

If ANY of the three fails, we picked wrong — the LLM is still partially blind on its own actions.

## 6. Forced spec deviations to expect

Calibration: 2.1b had 5 deviations, 2.1c had 9, 2.1c.6 had 3, 2.2a had 4. Phase 2.2b expected ~3-5 because the design surface is smaller (one new module + one widening + one extracted helper).

Likely hotspots — leave room in each task to verify substrate APIs against actual behavior before committing to spec'd usage:

1. **`runtime.perform` + `mask` semantics in the `:git/*` clause** — LD2 specifies `from persistence.effect.runtime import perform as runtime_perform, mask` then `with mask(audit_handler_name): runtime_perform(":shell/exec", argv=..., cwd=..., ...)`. The pattern matches the audit handler precedent (`audit.py:497-498` / `:516-517`), but verify the `mask` actually suppresses the audit handler's `wraps` membership check. If the audit middleware uses a different name registry, the `audit_handler_name` factory parameter may need adjustment at install time.

## 7. Known limitations & W3 honest-rescope deferrals

These are NOT closed by 2.2b; they remain queued artifacts:

- **2.1c.5** — caller identity Ed25519 attestation (~1-2 days when dispatched, blocking for non-loopback HTTP consumers).
- **F4 strict-xfail in `tests/effect/test_code_exec.py`** — v0.9.x real-OS-sandbox confidentiality acceptance signal. UNCHANGED by 2.2b. `:code/run` shares the same soft-isolation guarantees as `:code/exec` because they share `run_subprocess`. - **2.4a `:loop/exhausted` sentinel + `:sys/now`/`:sys/random` clock/random** — telemetry-driven, deferred.
- **`tx` extraction race** under concurrent HTTP serving (pre-existing from 2.1c).
- **`:shell/exec` cwd confinement at the `:shell/exec` layer** — LD2 puts confinement at the `:git/*` wrapper layer (the right altitude for git semantics), but `:shell/exec` itself remains unconstrained. If/when a future op set needs cwd confinement on shell directly, that's a separate `:shell/exec` v2 design (allowlist-version-bumping change).
- **Audit-replay queries for "all code execution"** — must union `:code/exec OR :code/run`. Accepted cost of LD1's namespace separation.
- **AuditEntry full byte-identity for `:git/*` and `:code/run`** — NOT achievable across runs because `result_hash = canonical_hash(result)` includes `wall_clock_ms`. Replay tests assert deterministic invariants only: `args_hash` stability + `output_hash` byte-identity (for `:code/run`, which excludes timing) + argv-construction determinism. If a future regulatory-replay tool needs full audit-entry byte-identity, that's a v0.9.x audit-stack redesign (timing fields would move to a sidecar, or `result_hash` would be computed over a deterministic subset).
- **Direct perform inside dosync raises `EffectInIoBlock`** — substrate-wide rule, not `:git/*`-specific. `tx.effect(op, **kwargs)` is the correct queueing mechanism inside dosync. The `:git/*` clause uses module-level `runtime.perform(":shell/exec", ...)` internally; the only path where this would surface `EffectInIoBlock` is if a caller does `s.effect.perform(":git/*", ...)` from inside their own dosync — which is already forbidden for ALL ops. 2.3a Plan AST execution queues via `tx.effect`, which works correctly.
