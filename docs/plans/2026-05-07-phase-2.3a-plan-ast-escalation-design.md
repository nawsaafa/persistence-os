## 1. Overview

Phase 2.3a fills the `_escalate_plan` stub at `src/persistence/coder/_session.py:237-240` (`CoderStubNotImplemented("Phase 2.3a — Plan AST builder + s.plan.execute")`). The escalation gate is already wired at `_session.py:234-235` by Phase 2.1a (`_should_escalate_plan: return decision.kind == "plan"`); 2.3a is the BRIDGE phase that connects the LLM-driven decision loop to the existing 5,695-LOC Plan AST substrate at `src/persistence/plan/`.

The phase has zero new substrate primitives. It exposes one curated SDK addition (`s.plan.new_dispatcher()` factory), introduces one new coder module (`src/persistence/coder/_planner.py`), and adds rendering/validation logic for the `:plan/*` slice of `EMIT_DECISION_TOOL_SCHEMA`. Plan execution is a TERMINAL mode-switch — when the LLM emits `kind="plan"`, the coder builds a `Node`, runs `s.plan.execute(plan, dispatcher)`, datomizes per-leaf results into the existing `:act/result` shape, and exits cleanly (or re-raises on failure mirroring `_act`'s contract).

## 2. Locked design decisions (LD0–LD7)

### LD0 — Approach C (LLM-hinted opt-in plan escalation)

**Decision:** Plan execution is a terminal mode-switch triggered when the LLM emits `decision.kind == "plan"`. `_escalate_plan` builds a Plan AST, runs it through `s.plan.execute`, and `Coder.run()` returns immediately afterward. The flat `_observe → _decide → _act` loop is preserved unchanged for non-plan decisions.

**Rationale (codex R1 ACCEPT, mean 8/10 across 5 dims):** the `_should_escalate_plan` gate is ALREADY locked at `_session.py:234-235`; the stub at line 237-240 is named exactly `"Phase 2.3a — Plan AST builder + s.plan.execute"`; Phase 2 design § 3.4 loop diagram (lines 194-197) explicitly diagrams the C-style conditional. Approach A (top-down, every-task-is-a-plan) is structurally incompatible with the `for i in range(max_iters)` loop shape that the skeleton header marks as ARIS-reviewable; Approach B (retroactive datomization) is rejected because `Node` is a content-addressed Merkle DAG with structurally future-pointing semantics — there is no shape for "this happened" inside the AST.

**Falsifiable acceptance signal:** scripted scenario, iter-0 emits one `:llm/decision` with `kind=="plan"`, `Coder.run()` terminates immediately after `_escalate_plan` returns, executes leaves via the registered dispatcher (NOT via flat `_act`), and constructs a plan with no `:branch`/`:code` leaves so `UnimplementedNodeKindError` cannot escape (`_execute.py:166-170`).

### LD1 — Gate contract: `kind=="plan"` wins; design doc prose updated

**Decision:** the escalation gate is `decision.kind == "plan"` per the shipped code at `_session.py:234-235`. The design doc prose at line 211 (mentioning a `:strategy/plan` payload field) is OUTDATED and gets updated as a 2.3a deliverable. There is no `:strategy/plan` field in `EMIT_DECISION_TOOL_SCHEMA` — the `kind` enum at `_prompt.py:34` is the contract.

**Rationale:** mechanical receipt-driven choice — the implemented gate is shipped, tested, and gated by `_should_escalate_plan`'s one-liner. The design prose is a documentation drift, not a design question.

### LD2 — Terminality: `_escalate_plan` returns; `:plan/done` audit datom marks the handoff

**Decision:** on `ExecutionResult.status == "ok"`, `_escalate_plan` emits a single `:plan/done` audit datom carrying `{plan_id, leaf_count, walk_order_node_ids}`, then returns. `Coder.run()`'s post-escalation early return at `_session.py:79` exits the loop. No "feed back into next iter" — codex R1's terminal-mode-switch refinement.

**Rationale (codex R1 finding):** `Coder.run()` returns immediately after `_escalate_plan` (`_session.py:79`) — there is NO next iter inside the same `run()` to consume any "translated-back" results. The bridge owns its provenance emission directly. The `:plan/done` datom gives downstream readers (replay, telemetry, the next session's `_observe` if the user re-invokes the coder) a structured handoff marker.

### LD3 — REUSE `:act/result` for plan leaves; encode plan context inside the value envelope

**Decision (CONTROLLER FOLDED to codex R1 REJECT-FOR-A):** plan leaves emit the same `:act/result` datom shape used by flat `_act` (`{op, args_hash, result_summary, error, latency_ms}`). For plan leaves, `op` is set to the leaf node's `tag` (e.g. `:fs/read`); `args_hash` is `canonical_hash(node.attrs)`; `result_summary` is the existing `_summarize_result(LeafResult.result)` projection EXTENDED to carry plan context as additional dict keys: `{plan_id, node_id, tag, handler_id, ...rest_of_summary}`. The `_render_latest_action` LD3 widening from 2.2b continues to work without branching.

**Rationale (codex R1 finding the controller missed):** (1) `CANONICAL_AUDIT_WRAPPED_OPS` is for `s.effect.perform` ops, NOT for `s.fact.transact` provenance datoms — `:act/result` BYPASSES the canonical audit chain by design (see `CHANGELOG-coder.md:203-209`); registering a NEW `:plan/leaf-result` op there would be category-wrong. (2) `_render_latest_action` (`_prompt.py:42-103`) assumes a stable `{op, error, result_summary}` shape — the very fix that solved 2.2b's "blind on its own outputs" failure mode. Forking the prompt-history shape would either weaken that latest-action visibility guarantee OR force a branching renderer (losing the single-shape benefit motivating reuse). (3) Schema-drift risk: `CHANGELOG-coder.md` already has `summary` vs `result_summary` divergence; adding a new datom kind widens the drift surface.

### LD4 — RE-RAISE on `ExecutionResult.status == "failed"`; mirror `_act`'s contract

**Decision (CONTROLLER FOLDED to codex R1 REJECT-FOR-A):** on plan execution failure, `_escalate_plan` (a) emits one `:act/result` per leaf in `result.leaf_results` (the partial-trace prefix preserved by `execute()`'s capture-then-record contract at `_execute.py:178-205`), (b) emits one final summary `:act/result` with `op=":plan/execute"`, `args_hash=canonical_hash(plan.id)`, `error=f"{result.failure.error_class}: {result.failure.error_repr}"`, (c) RE-RAISES a synthesized `PlanExecutionFailed` exception carrying `result.failure`. `Coder.run()` exits via raise, mirroring `_act`'s established control-plane signal (`_session.py:226-228`).

**Rationale (codex R1):** "failed but returned successfully" is a footgun for CLIs/tests/invokers that rely on exception semantics to detect failure (which is the entire 2.2a `_act` contract). Silent return on plan failure forces every caller to scrape the fact log — a reliability hazard. Re-raising preserves caller-visible failure while datomizing the partial trace fully.

### LD5 — `Dispatcher` per-escalation (fresh registry, no cross-call leakage)

**Decision:** `_escalate_plan` constructs a fresh `Dispatcher` instance via `s.plan.new_dispatcher()` at the start of each invocation, registers the 5 substrate ops needed for plan leaf dispatch (`:fs/read`, `:fs/write`, `:fs/glob`, `:fs/grep`, `:shell/exec`, `:code/run`, `:git/diff`, `:git/status`, `:git/log`, `:git/commit`, `:llm/call` — adapter shape: `(node, env) -> Any` wrapping `runtime.perform(node.tag, **node.attrs)`), and discards the registry when `_escalate_plan` returns. No cross-call state.

**Rationale:** `Dispatcher.register` raises on duplicate registration without `replace=True` (`_dispatch.py:52-57`) — fresh-per-call means we never need the duplicate-tolerance mode. Lifetime is exactly the `_escalate_plan` body. Memory cost is negligible (one dict + 11 callables per escalation; escalations are infrequent by design — most decisions stay flat).

### LD6 — Plan AST builder: EDN string in `payload["plan_edn"]` + post-parse semantic validator

**Decision (CONTROLLER FOLDED to codex R1 REJECT-FOR-B):** the LLM emits its plan as an EDN string in a single tool-use payload field: `payload["plan_edn"]: "<canonical-EDN-text>"`. The bridge code calls `s.plan.parse(payload["plan_edn"], strict=True)` to parse + spec-validate in one step (canonical defense-in-depth: keyword/symbol normalization, `:id` stripping for hash poisoning per `_parse.py:121-137`, shorthand handling, spec validation). After successful parse, a new `_planner.validate_plan_for_2_3a(node)` semantic validator enforces three additional 2.3a-specific constraints:
1. Root tag must be `:seq` (no other root kinds at 2.3a)
2. Leaves must NOT be `:branch` or `:code` (would otherwise crash `_walk` per `_execute.py:166-170`)
3. Every leaf tag MUST be in the registered handler set (otherwise `Dispatcher.dispatch` SILENTLY skips it per `_dispatch.py:73-80` — the controller-missed silent-no-op hazard codex flagged)

Validation failures raise `PlanPayloadValidation(field, reason)` BEFORE any call to `s.plan.execute`.

**Rationale (codex R1 finding the controller missed):** (1) `s.plan.parse` already concentrates defense-in-depth: hash-poisoning prevention ("DEFENSE IN DEPTH — DO NOT REMOVE" at `_parse.py:121-137`), keyword/symbol normalization, spec validation. A parallel JSON-subset validator would mirror 60% of those behaviors — exactly where security/identity bugs hide. (2) `Dispatcher.dispatch(...)` silently SKIPS nodes without a registered handler (`_dispatch.py:73-80`) — so "schema-valid" ≠ "effects occur"; without a leaf-tag-must-be-registered check, a syntactically valid plan could silently no-op. The post-parse validator is the right place to enforce that constraint.

**LLM ergonomics note:** `payload` is currently `type: object` (loose) per `_prompt.py:36` — the schema permits `payload["plan_edn"]: <string>` without a tool-schema change. The system prompt gains 2-3 few-shot examples showing canonical EDN syntax for `:seq` plans with `:fs/read` / `:code/run` leaves, mitigating EDN syntax-error rates that codex flagged as a real concern.

### LD7 — `s.plan.new_dispatcher()` curated factory (substrate-prereq T0)

**Decision (codex R1 ACCEPT-C):** add a new method `new_dispatcher() -> Dispatcher` to `_PlanNamespace` in `src/persistence/sdk/_facade.py`, decorated with `@experimental("v0.8.5a1")` matching the rest of `s.plan.*`. Thin pass-through: `from persistence.plan import Dispatcher; return Dispatcher()`. Lands as task T0 BEFORE any 2.3a coder-side work.

**Rationale:** `_PlanNamespace` is explicitly "functions only" per `_facade.py:383-390` — `Dispatcher` and `Handler` types stay in `persistence.plan` and are NOT re-exported. Direct import in coder code (`from persistence.plan import Dispatcher`) violates the agent's SDK-only consumption rule (Phase 2 design § 3.2). The `s.escape.plan.Dispatcher` escape-hatch path is also rejected: it adds a tracked TODO with sunset issue ID per the v0.9.0 escape-hatch sunset rule (more debt to close before GA). The `new_dispatcher()` factory follows the `s.txn.fork` / `s.txn.fold_into` precedent — curated method that returns an instance without exposing the type name, ~30min substrate-prereq with one new method on the SDK surface.

## 6. Forced spec deviations to expect

Per § 6 of 2.2b, expect 4-9 deviations documented in commit bodies. Five hotspots flagged in advance:

1. **`Node.attrs` keyword-vs-string keys.** `_ast.py:99` rejects colon-prefixed keys at construction; EDN parser strips colons (`_parse.py:121-137`). Plan-leaf `tag` is a colon-prefixed string in the EDN source but stored bare on `Node.tag`. The handler-id wiring needs to use the BARE form for handler registration but emit `:foo` form in `:act/result.op`.
2. **`Dispatcher.dispatch` silent-skip vs `execute()` capture-on-raise.** Two different "no handler" semantics: `dispatch()` skips silently; `execute()` calls `dispatcher.get_handler` returns None, also skips silently. The leaf-tags-must-be-registered validator (G2) is the only enforcement layer.
3. **`runtime.perform(node.tag, **node.attrs)` adapter shape.** Whether `**node.attrs` splat is the right shape for every op (e.g. `:fs/read` takes `path: str`, but `node.attrs` is a `Mapping[str, Any]` after EDN parsing) needs verification. May require per-tag adapter functions if the splat doesn't work universally.
4. **`canonical_hash(node.attrs)` for `args_hash` in plan-leaf `:act/result`.** Verify `Mapping` shape acceptance in `canonical_hash` (`effect/canonical.py`) — should match the existing `_act` `canonical_hash(args)` discipline.
5. **`:plan/done` op routing — `s.effect.perform` vs `s.fact.transact`.** Pinned at T2. The `:act/result` precedent uses `s.fact.transact` (bypassing canonical audit chain); `:plan/done` SHOULD follow that convention for consistency, but the LD2 framing said "audit datom" — may need to choose. Default: `s.fact.transact`-only matching `:act/result`, no drift-pin update. If empirically the canonical audit chain needs the marker, escalate at T2.
