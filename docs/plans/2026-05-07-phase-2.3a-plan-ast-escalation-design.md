## 1. Overview

Phase 2.3a fills the `_escalate_plan` stub at `src/persistence/coder/_session.py:237-240` (`CoderStubNotImplemented("Phase 2.3a ŌĆö Plan AST builder + s.plan.execute")`). The escalation gate is already wired at `_session.py:234-235` by Phase 2.1a (`_should_escalate_plan: return decision.kind == "plan"`); 2.3a is the BRIDGE phase that connects the LLM-driven decision loop to the existing 5,695-LOC Plan AST substrate at `src/persistence/plan/`.

The phase has zero new substrate primitives. It exposes one curated SDK addition (`s.plan.new_dispatcher()` factory), introduces one new coder module (`src/persistence/coder/_planner.py`), and adds rendering/validation logic for the `:plan/*` slice of `EMIT_DECISION_TOOL_SCHEMA`. Plan execution is a TERMINAL mode-switch ŌĆö when the LLM emits `kind="plan"`, the coder builds a `Node`, runs `s.plan.execute(plan, dispatcher)`, datomizes per-leaf results into the existing `:act/result` shape, and exits cleanly (or re-raises on failure mirroring `_act`'s contract).

### LD0 ŌĆö Approach C (LLM-hinted opt-in plan escalation)

**Decision:** Plan execution is a terminal mode-switch triggered when the LLM emits `decision.kind == "plan"`. `_escalate_plan` builds a Plan AST, runs it through `s.plan.execute`, and `Coder.run()` returns immediately afterward. The flat `_observe ŌåÆ _decide ŌåÆ _act` loop is preserved unchanged for non-plan decisions.

**Rationale (codex R1 ACCEPT, mean 8/10 across 5 dims):** the `_should_escalate_plan` gate is ALREADY locked at `_session.py:234-235`; the stub at line 237-240 is named exactly `"Phase 2.3a ŌĆö Plan AST builder + s.plan.execute"`; Phase 2 design ┬¦ 3.4 loop diagram (lines 194-197) explicitly diagrams the C-style conditional. Approach A (top-down, every-task-is-a-plan) is structurally incompatible with the `for i in range(max_iters)` loop shape that the skeleton header marks as ARIS-reviewable; Approach B (retroactive datomization) is rejected because `Node` is a content-addressed Merkle DAG with structurally future-pointing semantics ŌĆö there is no shape for "this happened" inside the AST.

**Falsifiable acceptance signal:** scripted scenario, iter-0 emits one `:llm/decision` with `kind=="plan"`, `Coder.run()` terminates immediately after `_escalate_plan` returns, executes leaves via the registered dispatcher (NOT via flat `_act`), and constructs a plan with no `:branch`/`:code` leaves so `UnimplementedNodeKindError` cannot escape (`_execute.py:166-170`).

### LD1 ŌĆö Gate contract: `kind=="plan"` wins; design doc prose updated

**Decision:** the escalation gate is `decision.kind == "plan"` per the shipped code at `_session.py:234-235`. There is no `:strategy/plan` field in `EMIT_DECISION_TOOL_SCHEMA` ŌĆö the `kind` enum at `_prompt.py:34` is the contract.

**Rationale:** mechanical receipt-driven choice ŌĆö the implemented gate is shipped, tested, and gated by `_should_escalate_plan`'s one-liner. The design prose is a documentation drift, not a design question.

### LD2 ŌĆö Terminality: `_escalate_plan` returns; `:plan/done` PROVENANCE datom (fact-only) marks the handoff

**Decision (R0-fold I1 ŌĆö locked at design):** on `ExecutionResult.status == "ok"`, `_escalate_plan` emits a single `:plan/done` PROVENANCE datom via `s.fact.transact` (NOT `s.effect.perform`) carrying `{plan_id, leaf_count, walk_order_node_ids}`, then returns. `Coder.run()`'s post-escalation early return at `_session.py:79` exits the loop. No "feed back into next iter" ŌĆö codex's terminal-mode-switch refinement.

**Rationale (R0-fold I1 closure):** `:plan/done` follows the `:act/result` precedent ŌĆö `s.fact.transact`-only, BYPASSES the canonical audit chain (per `CANONICAL_AUDIT_WRAPPED_OPS` being for `s.effect.perform` ops only). Calling it a "provenance datom" not an "audit datom" matches the actual routing. Drift-pin in `tests/effect/test_canonical_audit_stack.py` is therefore UNAFFECTED ŌĆö G7 collapses (originally proposed as conditional drift-pin update; now confirmed not needed). Downstream readers (replay, telemetry, next-session `_observe` on re-invocation) consume the marker via `s.fact.history` / `s.fact.since` queries, same access pattern as `:act/result`.

**Rationale (LD2 base ŌĆö codex R1 LD0):** `Coder.run()` returns immediately after `_escalate_plan` (`_session.py:79`) ŌĆö there is NO next iter inside the same `run()` to consume any "translated-back" results. The bridge owns its provenance emission directly.

### LD3 ŌĆö REUSE `:act/result` for plan leaves; encode plan context inside the value envelope

**Decision (CONTROLLER FOLDED to codex R1 REJECT-FOR-A):** plan leaves emit the same `:act/result` datom shape used by flat `_act` (`{op, args_hash, result_summary, error, latency_ms}`). For plan leaves, `op` is set to the leaf node's `tag` (e.g. `:fs/read`); `args_hash` is `canonical_hash(node.attrs)`; `result_summary` is the existing `_summarize_result(LeafResult.result)` projection EXTENDED to carry plan context as additional dict keys: `{plan_id, node_id, tag, handler_id, ...rest_of_summary}`. The `_render_latest_action` LD3 widening from 2.2b continues to work without branching.

**Rationale (codex R1 finding the controller missed):** (1) `CANONICAL_AUDIT_WRAPPED_OPS` is for `s.effect.perform` ops, NOT for `s.fact.transact` provenance datoms ŌĆö `:act/result` BYPASSES the canonical audit chain by design (see `CHANGELOG-coder.md:203-209`); registering a NEW `:plan/leaf-result` op there would be category-wrong. (2) `_render_latest_action` (`_prompt.py:42-103`) assumes a stable `{op, error, result_summary}` shape ŌĆö the very fix that solved 2.2b's "blind on its own outputs" failure mode. Forking the prompt-history shape would either weaken that latest-action visibility guarantee OR force a branching renderer (losing the single-shape benefit motivating reuse). (3) Schema-drift risk: `CHANGELOG-coder.md` already has `summary` vs `result_summary` divergence; adding a new datom kind widens the drift surface.

### LD4 ŌĆö RE-RAISE on `ExecutionResult.status == "failed"`; mirror `_act`'s caller-visible-failure-via-raise contract (NOT traceback)

**Decision (CONTROLLER FOLDED to codex R1 REJECT-FOR-A; R0-fold B1+I3):** on plan execution failure, `_escalate_plan`:

1. Pre-walks the plan into an `id ŌåÆ Node` map BEFORE calling `s.plan.execute` (so the bridge can recover the failing node's `attrs` and `handler-id` after `execute()` returns).
2. Emits one `:act/result` per `LeafResult` in `result.leaf_results` (the SUCCESSFUL prefix only ŌĆö `execute()` does NOT append a `LeafResult` for the failing node; it sets `failure` and bails per `_execute.py:193-205`).
3. Emits one ADDITIONAL failure-shaped `:act/result` for the failing leaf (R0-fold B1): `op=<failed.tag>`, `args_hash=canonical_hash(failed_node.attrs)` (recovered from the id-map), `result_summary={plan_id, node_id, tag, handler_id}`, `error=failure.error_repr`.
4. Emits one final summary `:act/result` with `op=":plan/execute"`, `args_hash=canonical_hash(plan.id)`, `error=failure.error_repr`. (R0-fold N2 + R1-NICE: error-string format is `failure.error_repr` ALONE ŌĆö already class-prefixed, e.g. `"RuntimeError('SENTINEL_2_3_A')"` ŌĆö NOT `f"{error_class}: {error_repr}"` which would duplicate the class name. Asymmetry with `_act`'s `f"{type(e).__name__}: {e}"` shape is accepted: `_act` uses `str(e)` which has no class prefix, while plan-failure summarizes from `FailureInfo.error_repr` which already does. Tests pin the exact `repr()`-derived format so future drift surfaces immediately.)
5. RE-RAISES a synthesized `PlanExecutionFailed` exception carrying `result.failure` so `Coder.run()` exits via raise, mirroring `_act`'s established CALLER-VISIBLE-FAILURE-VIA-RAISE control-plane signal (`_session.py:226-228`).

**R0-fold I3 ŌĆö diagnostic surface boundaries:** "Mirror `_act`'s contract" applies to the caller-visible-failure-via-raise property ONLY, not to traceback fidelity. `_act` re-raises the ORIGINAL exception with original traceback (`raise` at `_session.py:228`). Plan-leaf handler exceptions cannot be re-raised originally ŌĆö `execute()` captures them into `FailureInfo` (a string-only dataclass at `_execute.py:96-114`); the original `__traceback__` is gone by the time `_escalate_plan` sees the result. `PlanExecutionFailed` carries the structured `FailureInfo` (failed_node_id, failed_tag, error_class, error_repr) for caller introspection, but **native traceback is unavailable in 2.3a ŌĆö documented limitation, queued to v0.9.x for re-evaluation if dogfood feedback warrants extending `execute()`'s capture surface**.

**Rationale (codex R1):** "failed but returned successfully" is a footgun for CLIs/tests/invokers that rely on exception semantics to detect failure (the entire 2.2a `_act` contract). Silent return on plan failure forces every caller to scrape the fact log. Re-raising preserves caller-visible failure while datomizing the partial trace + the failure leaf + a final summary.

### LD5 ŌĆö `Dispatcher` per-escalation (fresh registry, no cross-call leakage); 10 substrate ops

**Decision (R0-fold N1 ŌĆö count corrected):** `_escalate_plan` constructs a fresh `Dispatcher` instance via `s.plan.new_dispatcher()` at the start of each invocation, registers the **10 substrate ops** that compose the 2.3a plan-leaf alphabet, and discards the registry when `_escalate_plan` returns. No cross-call state.

The 10 ops are exactly the 2.2a + 2.2b shipped audit-anchored effect ops:
1. `:fs/read` 2. `:fs/write` 3. `:fs/glob` 4. `:fs/grep` (Phase 2.2a)
5. `:shell/exec` (Phase 2.2a)
6. `:code/run` (Phase 2.2b)
7. `:git/diff` 8. `:git/status` 9. `:git/log` 10. `:git/commit` (Phase 2.2b)

`:llm/call` is INTENTIONALLY EXCLUDED from plan leaves: LLM calls are the OUTER decision-loop primitive, not a plan leaf. Nested LLM emission inside plans is queued to 2.3c (skill library) where it composes via `:compose-with-skill`. (Original design draft listed 11 ops; corrected to 10 in R0-fold.)

Adapter shape: `(node, env) -> Any` wrapping `substrate.effect.perform(node.tag, dict(node.attrs))`. Adapter coerces frozen Mapping to plain dict for the substrate.effect.perform args contract; T3 forced spec deviation (splat `**node.attrs` was original sketch ŌĆö impl confirmed Mapping form is required). Per-tag adapter functions not needed ŌĆö all 10 ops use the same generic adapter factory `_make_adapter(substrate, tag)`.

**Rationale:** `Dispatcher.register` raises on duplicate registration without `replace=True` (`_dispatch.py:52-57`) ŌĆö fresh-per-call means we never need the duplicate-tolerance mode. Lifetime is exactly the `_escalate_plan` body. Memory cost is negligible (one dict + 10 callables per escalation; escalations are infrequent by design ŌĆö most decisions stay flat).

### LD6 ŌĆö Plan AST builder: EDN string in `payload["plan_edn"]` + bounded resource limits + post-parse semantic validator

**Decision (CONTROLLER FOLDED to codex R1 REJECT-FOR-B; R0-fold I2 ŌĆö bounds added):** the LLM emits its plan as an EDN string in a single tool-use payload field: `payload["plan_edn"]: "<canonical-EDN-text>"`. The bridge code applies a bounds-check + parse + semantic-validate pipeline in this exact order:

**Stage 1 ŌĆö Pre-parse byte budget (R0-fold I2):**
- `len(payload["plan_edn"].encode("utf-8")) Ōēż MAX_PLAN_EDN_BYTES` (default 8192). Reject with `PlanPayloadValidation("plan_edn", "exceeds 8192-byte budget")` BEFORE invoking the parser.

**Stage 2 ŌĆö `s.plan.parse(plan_edn, strict=False)`: plain parse + Node `__post_init__` invariants. Strict mode is unusable here because the plan-spec enum at `_parse.py:211` rejects every valid coder leaf (`:fs/*`, `:code/run`, `:git/*`, `:shell/exec` are absent from the enum). Stage 3 (`validate_plan_for_2_3a`) is the SOLE validation layer for the 2.3a coder subset and is strictly tighter than what strict-mode would have caught for that subset (explicit 10-op allowlist + `:seq`-only interior tag + banned-leaf list + node-count/depth bounds).**
- Defense-in-depth properties RETAINED even in strict=False: keyword/symbol normalization, `:id` stripping for hash poisoning per `_parse.py:121-137` ("DEFENSE IN DEPTH ŌĆö DO NOT REMOVE"), shorthand handling.
- (R0-fold I2 honesty correction): `s.plan.parse` is **schema validation, NOT sandboxing**. The parser raises `ParseError` on malformed EDN; pathological inputs (deep recursion, large strings) reach the parser before being rejected. The byte-budget at Stage 1 is the upstream sandbox.
- Wrap any raised `ParseError` in `PlanPayloadValidation` (consistent error class) per G1.

**Stage 3 ŌĆö Post-parse semantic validator** `_planner.validate_plan_for_2_3a(node)`:
1. Root tag must be `:seq` (no other root kinds at 2.3a)
2. **Plan-node-count budget**: `len(list(walk(node))) Ōēż MAX_PLAN_NODES` (default 64). Rejection rationale: prevents deeply-nested plans from exhausting handler-dispatch CPU budget; bound is empirically tunable in 2.4a dogfood.
3. **Plan-depth budget**: max depth from root to deepest leaf Ōēż MAX_PLAN_DEPTH (default 4). Reuses `MAX_PLAN_DEPTH` from `persistence.plan` if available; else defines locally.
4. Leaves must NOT be `:branch` or `:code` (would otherwise crash `_walk` per `_execute.py:166-170`)
5. Every leaf tag MUST be in the registered handler set (otherwise `Dispatcher.dispatch` SILENTLY skips it per `_dispatch.py:73-80` ŌĆö the controller-missed silent-no-op hazard codex flagged)

Validation failures at any stage raise `PlanPayloadValidation(field, reason)` BEFORE any call to `s.plan.execute`.

**Rationale (codex R1 finding the controller missed):** (1) `s.plan.parse` already concentrates defense-in-depth: hash-poisoning prevention ("DEFENSE IN DEPTH ŌĆö DO NOT REMOVE" at `_parse.py:121-137`), keyword/symbol normalization, spec validation. A parallel JSON-subset validator would mirror 60% of those behaviors ŌĆö exactly where security/identity bugs hide. (2) `Dispatcher.dispatch(...)` silently SKIPS nodes without a registered handler (`_dispatch.py:73-80`) ŌĆö so "schema-valid" ŌēĀ "effects occur"; without a leaf-tag-must-be-registered check, a syntactically valid plan could silently no-op. The post-parse validator is the right place to enforce that constraint.

**LLM ergonomics note:** `payload` is currently `type: object` (loose) per `_prompt.py:36` ŌĆö the schema permits `payload["plan_edn"]: <string>` without a tool-schema change. The system prompt gains 2-3 few-shot examples showing canonical EDN syntax for `:seq` plans with `:fs/read` / `:code/run` leaves, mitigating EDN syntax-error rates that codex flagged as a real concern.

### LD7 ŌĆö `s.plan.new_dispatcher()` curated factory (substrate-prereq T0)

**Decision (codex R1 ACCEPT-C):** add a new method `new_dispatcher() -> Dispatcher` to `_PlanNamespace` in `src/persistence/sdk/_facade.py`, decorated with `@experimental("v0.8.5a1")` matching the rest of `s.plan.*`. Thin pass-through: `from persistence.plan import Dispatcher; return Dispatcher()`. Lands as task T0 BEFORE any 2.3a coder-side work.

**Rationale:** `_PlanNamespace` is explicitly "functions only" per `_facade.py:383-390` ŌĆö `Dispatcher` and `Handler` types stay in `persistence.plan` and are NOT re-exported. Direct import in coder code (`from persistence.plan import Dispatcher`) violates the agent's SDK-only consumption rule (Phase 2 design ┬¦ 3.2). The `s.escape.plan.Dispatcher` escape-hatch path is also rejected: it adds a tracked TODO with sunset issue ID per the v0.9.0 escape-hatch sunset rule (more debt to close before GA). The `new_dispatcher()` factory follows the `s.txn.fork` / `s.txn.fold_into` precedent ŌĆö curated method that returns an instance without exposing the type name, ~30min substrate-prereq with one new method on the SDK surface.

## 6. Forced spec deviations to expect

Per ┬¦ 6 of 2.2b, expect 4-9 deviations documented in commit bodies. Five hotspots flagged in advance:

1. **`Node.attrs` keyword-vs-string keys.** `_ast.py:99` rejects colon-prefixed keys at construction; EDN parser strips colons (`_parse.py:121-137`). Plan-leaf `tag` is a colon-prefixed string in the EDN source but stored bare on `Node.tag`. The handler-id wiring needs to use the BARE form for handler registration but emit `:foo` form in `:act/result.op`.
2. **`Dispatcher.dispatch` silent-skip vs `execute()` capture-on-raise.** Two different "no handler" semantics: `dispatch()` skips silently; `execute()` calls `dispatcher.get_handler` returns None, also skips silently. The leaf-tags-must-be-registered validator (G2) is the only enforcement layer.
3. **`runtime.perform(node.tag, **node.attrs)` adapter shape.** Whether `**node.attrs` splat is the right shape for every op (e.g. `:fs/read` takes `path: str`, but `node.attrs` is a `Mapping[str, Any]` after EDN parsing) needs verification. May require per-tag adapter functions if the splat doesn't work universally.
4. **`canonical_hash(node.attrs)` for `args_hash` in plan-leaf `:act/result`.** Verify `Mapping` shape acceptance in `canonical_hash` (`effect/canonical.py`) ŌĆö should match the existing `_act` `canonical_hash(args)` discipline.
5. **`:plan/done` op routing ŌĆö LOCKED at design (R0-fold I1):** `s.fact.transact`-only, mirroring `:act/result` precedent. BYPASSES the canonical audit chain. NOT in `CANONICAL_AUDIT_WRAPPED_OPS`. Drift-pin (G7) is therefore unaffected. (Originally pinned to T2 impl decision; closed at design after R0 found that calling it an "audit datom" while routing through `s.fact.transact` was internally inconsistent.)
