## 1. Overview

Phase 2.3b fills the `_escalate_branch` stub at `src/persistence/coder/_session.py:254-257` (`CoderStubNotImplemented("Phase 2.3b — s.plan.mcts_search + s.txn.fork + s.plan.judge")`). The escalation gate is already wired at `_session.py:251-252`; 2.3b is the BRIDGE phase that connects the LLM-driven decision loop to the existing 45-KB v0.6.5 MCTS engine at `src/persistence/plan/_mcts.py` (PUCT search + `:mcts/iteration` Merkle-chained provenance + `Action` ADT + `Expander`/`Evaluator` Protocols).

The phase has zero new substrate primitives — `s.plan.mcts_search` (`_facade.py:709`), `s.plan.judge` (`_facade.py:728`), and the `LLMExpander` + `LLMJudgeEvaluator` constructors (`_mcts.py:388, 469`) all ship today on the curated SDK surface. The phase introduces one new coder module (`src/persistence/coder/_searcher.py`) modelled on 2.3a's `_planner.py`. MCTS exploration is a TERMINAL mode-switch — when the LLM emits `kind="branch"`, the coder builds a seed `Node`, constructs an LLM-provider-backed expander+evaluator, runs `s.plan.mcts_search`, and then **executes the winner exactly ONCE** via 2.3a's `_escalate_plan_body` bridge. Losing branches are STRUCTURAL — they never run their leaf handlers, so no real-world side effects (`:fs/write`, `:shell/exec`, `:git/commit`) leak.

**R0-fold B1 correction:** `s.txn.fork` is NOT used by `mcts_search` — the engine achieves transposition via an in-memory `dict[plan_id, MCTSNode]` and never calls `s.txn.fork` (verified at `_mcts.py:881-882`). Earlier draft framing that listed `s.txn.fork` as a 2.3b composition primitive was wrong; the locked SCOPE LD0 stands (single-execution invariant), but the audit claims at LD5 are corrected to drop `:fork/*` and keep only `:mcts/iteration` + `:mcts/search` + `:act/result` (winner) + `:plan/done` (winner). The pre-existing `_escalate_branch` stub string at `_session.py:256` mentions `s.txn.fork`; it gets corrected to `"Phase 2.3b — s.plan.mcts_search + s.plan.judge"` as a T8 deliverable.

## 2. Locked design decisions (LD0–LD7)

8 LDs, all closed via the `consensus-driven-architecture-decision` skill (2 codex `--reasoning high` passes converged). Acceptance signal for the skill: zero user-weighing during the brainstorm. ✅

### LD0 — SCOPE: MCTS-search drives; winner executes once via 2.3a's bridge; NO execution-time handler dispatch during search

**Decision (CONTROLLER FOLDED to codex R0 REJECT-FOR-A; R0-fold B1 amended):** the 2 substrate primitives compose as `s.plan.mcts_search` (drives the search over plan STRUCTURE via `Action`-based plan-tree mutations on opaque Python `Node` objects) → `s.plan.judge` (scores via `LLMJudgeEvaluator` evaluator). After `MCTSResult.winner` is selected, the bridge invokes `_escalate_plan_body(coder, synthesized_decision)` (2.3a's existing happy/failure paths) ONCE on the winner's plan EDN. Losing branches are never executed — their `:fs/write` / `:shell/exec` / `:git/commit` leaves are explored only as Plan-AST structures during search, not as real handler dispatches. **No `s.txn.fork`** in the 2.3b composition (R0-fold B1: codex verified `mcts_search` uses in-memory dict transposition at `_mcts.py:881-882`, no `fork` usage).

**Rationale (codex R0 finding the controller missed):** the controller's original Option B proposed forking each candidate through `_escalate_plan_body` and letting `s.txn.fork` "roll back losers". Codex's decider: *"`s.txn.fork` only guarantees rollback/isolation for facts, not external effects. 2.3a plan execution is wired to call `substrate.effect.perform(...)` for `:fs/write`, `:shell/exec`, `:git/commit` — losers' real-world mutations leak regardless of whether the substrate rolls back facts."* The right safety invariant is "only ONE plan ever executes per `_escalate_branch` call", not "losers' facts don't persist". MCTS over plan STRUCTURE (substitute leaf / add step / compose with skill `Action`s — `_mcts.py:60-78`) explores plan-tree variations as content-addressed Merkle DAG operations on opaque Python `Node` objects; no `Dispatcher` is invoked during search.

**R0-fold B1 gold-star verification (codex):** *"LD0's single-execution invariant is backed by the engine: `_mcts.py` does structural `apply_action(...)` + `db.transact(...)` only (no dispatcher / no effect handlers), so the 'losers never execute' story is mechanically enforceable if the branch bridge never calls `_escalate_plan_body` except on the winner."*

**Falsifiable acceptance signal:** scripted scenario, K=3 candidate plans (the seed plan plus 2 expander-generated variants) where each candidate would `:fs/write` a distinct marker on disk if executed (`marker_A`, `marker_B`, `marker_C`); after `_escalate_branch` returns, assert `len([f for f in os.listdir(scratch) if f.startswith("marker_")]) == 1` AND only the winner's `:act/result` datoms appear in the parent fact-store. Two+ markers → losers' handlers ran → fail.

### LD1 — TRIGGER: `decision.kind == "branch"` ONLY; drop the confidence-based escalation half of the gate

**Decision (codex R0 ACCEPT):** the escalation gate is `decision.kind == "branch"` per the FROZEN intent of `_session.py:251-252`. The current gate ALSO checks `decision.confidence < self.confidence_threshold` (default `0.65`); 2.3b ships the EXPLICIT half ONLY and removes the confidence-based half. Confidence-based escalation defers to Phase 2.4a once dogfood calibration data exists.

**Rationale (codex R0 sharper than controller's):** *"a confidence-based trigger can route non-`branch` payloads (e.g. `kind=="act"` with `{op,args}`) into a branch escalator that necessarily expects a branch-specific payload contract, creating a type/shape mismatch that either forces implicit conversions (unsafe) or silent failures."* This is a real correctness risk, not just a "tunable threshold" issue. Mirroring 2.3a's LD1 (`kind=="plan"` only) keeps the gate single-cause, testable, and payload-shape aligned. The confidence threshold default at `_session.py:39` is "starting value, will be tuned during dogfood (Phase 2.4a) on calibration data" per the design doc — meaning 2.3b shipping it would force an arbitrary threshold to be load-bearing for tests.

### LD2 — PAYLOAD: `payload["seed_plan_edn"]: <EDN-string>`; mirror 2.3a's plan_edn discipline

**Decision (codex R0 ACCEPT-A; R0-fold N1 simplified):** the `:llm/decision` payload when `kind=="branch"` carries `{"seed_plan_edn": "<canonical-EDN-text>"}`. The bridge applies the SAME 3-stage pipeline 2.3a's `_planner` uses: byte-budget (≤8192) → `s.plan.parse(strict=False)` → **2.3a's `validate_plan_for_2_3a` STRICT** (no looser sibling). Any extra structural constraints introduced by MCTS-applied actions (e.g. interior tags beyond `:seq`) are enforced at the EXPANDER layer — the wrapper rejects expander proposals whose `apply_action` result would fail `validate_plan_for_2_3a`. This way the post-search winner is guaranteed to satisfy 2.3a's strict validator at execution time.

**R0-fold N1 simplification:** earlier draft contemplated "tightened sibling `validate_seed_plan_for_2_3b`" — codex flagged this as over-complicated. Since the winner is executed via `_escalate_plan_body` which already enforces `validate_plan_for_2_3a`, any looser seed validator only wastes search budget on plans that can never execute. Use the strict 2.3a validator at the seed layer AND filter expander proposals against it.

The parsed `Node` becomes `mcts_search`'s `initial_plan` argument.

**Rationale (codex R0 decider):** *"the current MCTS API requires a concrete `initial_plan: Node` and is written as 'search over a Plan AST,' not an 'intent-to-plan synthesis' engine, so an EDN seed is the only payload that feeds the existing ingestion path without inventing new expander responsibilities."* (`_mcts.py:844-853`.) Rejected: `payload["intent"]` (NL string) — would force the expander to synthesize plans from scratch, which `LLMExpander` does NOT do today (it proposes Actions to apply to an existing Node). Rejected: `payload["candidates"]` (pre-baked K plans) — collapses MCTS to a no-op judge-only layer, contradicting LD0.

**LLM ergonomics note:** `_PLAN_EDN_GUIDANCE` in `_prompt.py` (added at 2.3a T7) gets a sibling `_BRANCH_EDN_GUIDANCE` constant explaining the seed-plan shape: "Provide a starting Plan AST as the seed; MCTS will explore structural variations (substitute leaves, insert steps) around it. Use `kind="branch"` when uncertain which leaf composition will best achieve the goal — NOT when you have a definite linear plan (use `kind="plan"` instead)."

### LD3 — EXPANDER + EVALUATOR: providers route through `substrate.effect.perform(":llm/call", ...)`; `LLMExpander` constrained to non-skill Actions; `ComposeWithSkillAction` queued for 2.3c

**Decision (R0-fold I3 amended):** the bridge constructs both `LLMExpander` and `LLMJudgeEvaluator` per-escalation. The provider closures wired into both **MUST route their LLM dispatch through `substrate.effect.perform(":llm/call", request)`** (matching the `_decide` path at `_session.py:108`). This preserves replay/audit semantics — every expander proposal and every evaluator score becomes a `:llm/call` audit datom on the canonical chain (request_hash + response_hash + model + tokens), wrapped under the `mcts_search` iteration's `:mcts/iteration` provenance. NO new audit shapes for expander/evaluator-specific calls; `:llm/call` is the substrate boundary. The expander provider is constrained to propose ONLY `SubstituteLeafAction` and `AddStepAction` — `ComposeWithSkillAction` requires a populated `SkillLibrary` which is queued for Phase 2.3c. A defensive validator wraps the provider closure: if the LLM returns a `ComposeWithSkillAction`, reject the proposal (treat as if the LLM proposed nothing for that slot) rather than letting `apply_action` raise downstream.

**Minimal request shape for both providers (R1-fold I2 corrected):** the request payload to `:llm/call` matches `_decide`'s actual shape at `_session.py:134-139` — `{model, messages, tools}`. NO `system` / `response_format` keys (those are NOT in the existing `:llm/call` dispatch). The system message is embedded as the first message in `messages` (role=`"system"`); JSON-mode response shape is enforced via `tools` (a single tool-use schema for action proposals or score emission). Bridge-side post-processing parses tool-use response and softmax-normalizes priors (for expander). If `system`/`response_format` capabilities are needed for replay-byte-identity at 2.4a, they get added as a `:llm/call` substrate-API extension (queued v0.9.x; non-blocking for 2.3b). Differences vs `_decide`'s tool: `EMIT_DECISION_TOOL_SCHEMA` is replaced per-call with either `EMIT_BRANCH_PROPOSAL_TOOL_SCHEMA` (expander) or `EMIT_BRANCH_SCORE_TOOL_SCHEMA` (evaluator) — both new constants in `_prompt.py`.

**Rationale (R0-fold I3 codex finding):** the controller's draft said "thread the coder's existing LLM provider closure" but `_decide` does NOT use a reusable closure — it calls `substrate.effect.perform(":llm/call", ...)` directly. Pinning the provider routing to `:llm/call` is what preserves replay byte-identity AND matches the Phase 2 design § 3.4 LLM-handler contract. `LLMExpander` (`_mcts.py:388-411`) is pure delegation — its `provider` closure is caller-owned, so 2.3b's bridge owns the prompt template + JSON parsing + softmax normalization layer ABOVE the `:llm/call` boundary.

**Forward-pointer:** `_searcher.py:_make_branch_llm_provider(coder)` is the seam where 2.3c will widen the action space (skill-aware proposals) AND where 2.4a will tune calibration (max-iters, temperature, prompt iteration).

### LD4 — FAILURE: mirror 2.3a LD4 — `PlanExecutionFailed` re-raise on the winner's execution; native traceback unavailable

**Decision:** if the winner's execution (the post-search `_escalate_plan_body(coder, synthesized_decision)` call) raises `PlanExecutionFailed`, the failure propagates UNCHANGED through `_escalate_branch` — `Coder.run()` exits via raise. No additional summary datom emission at the branch layer beyond what 2.3a already emits (partial-trace `:act/result` per leaf + failure-shaped `:act/result` for the failing leaf + final `:plan/execute` summary). The branch layer is a search-and-pick wrapper; once the winner is chosen, failure semantics are 2.3a's exclusively.

**Rationale:** there is exactly ONE plan execution per `_escalate_branch` call (LD0). Failure is therefore single-source-of-truth. Adding a redundant `:branch/failed` summary datom would duplicate `:plan/execute` (2.3a) and risk drift. Native traceback availability inherits 2.3a's W3 rescope to v0.9.x (queued).

**MCTS-search-failure separation (R0-fold I2 + R1-fold I1 corrected):** the bridge handles two distinct search-layer-failure paths, BOTH of which produce `BranchSearchFailed`:

1. **Bubble-out exceptions (assuming validated inputs — bridge already checks `started_at_ms` is positive int and `mcts_config` field shapes before invoking `mcts_search`):** `mcts_search` itself raises. The engine-level catches mean only a SUBSET of exceptions actually propagate: (a) `ExpanderContractError` for prior-sum-≠-1 violations (`_mcts.py:_PRIOR_TOL`); (b) any expander provider exception (i.e. the `Callable[[Node, int], Sequence[(Action, float)]]` itself raises) — these bubble through `mcts_search` because the engine does NOT wrap expander invocations in try/except. **NOT bubble-out** (R1-fold I1 codex finding): `PlanDepthExceeded` from `apply_action` is caught inside `_populate_children` and recorded as `phase="reject"`; `evaluator.evaluate(...)` exceptions are similarly absorbed. The bridge wraps `mcts_search` with `try/except`, emits ONE `:act/result` with `op=":mcts/search"` + `error=<error_repr>`, and re-raises `BranchSearchFailed`.
2. **Post-search detection** — `mcts_search` returns successfully but `MCTSResult.terminated_by == "all_evaluations_failed"` (per `_mcts.py` evaluator-exception handling: evaluator failures are absorbed into `phase="reject"` rows; if all evaluations fail, the engine terminates with that signal rather than raising). The bridge inspects `MCTSResult.terminated_by` post-search and converts this terminal state into the same `BranchSearchFailed` path: emit `:act/result` with `op=":mcts/search"` + `error="all_evaluations_failed"` + summary fields from `MCTSResult` (iter_count, search_id) + raise `BranchSearchFailed`.

Both paths funnel into the same exception class so `Coder.run()` exits uniformly. Tests G5a covers BOTH paths (one test per branch-of-failure).

**`BranchSearchFailed` payload shape:** dataclass mirroring `PlanExecutionFailed`: `BranchSearchFailed(error_repr: str, search_id: str | None, iter_count: int | None, terminated_by: TerminatedBy | None)`. Path-1 carries `error_repr` only (search_id/iter_count/terminated_by None — search never completed); Path-2 carries all four fields from the returned `MCTSResult`.

### LD5 — AUDIT: `:mcts/iteration` Merkle-chained provenance + `:mcts/search` summary + `:llm/call` (per expander/evaluator query) + `:plan/done` (winner) from 2.3a; NO new audit shapes

**Decision (R0-fold B1 amended — `:fork/*` removed):** the bridge passes `db=substrate._db` to `mcts_search` so the existing `:mcts/iteration` provenance fires (one `db.transact` per search iteration with `phase="expand"` / `phase="evaluate"` / `phase="reject"` rows + `mcts/prev-hash` chain + `mcts/search` summary on iter 0 + winner+iter-count+terminated-by+finished-at on search end — `_mcts.py:862-869`). Each expander proposal and evaluator score routes through `substrate.effect.perform(":llm/call", ...)` per LD3, emitting standard `:llm/call` audit datoms wrapped under the iteration's `:mcts/iteration` group. After `mcts_search` returns, the bridge invokes `_escalate_plan_body(coder, synthesized_decision)` which fires 2.3a's existing `:act/result` + `:plan/done` PROVENANCE datoms on the winner's leaves.

**No fork-namespace audit datoms in 2.3b** (R0-fold B1 codex finding): `mcts_search` does NOT use `s.txn.fork`. The engine maintains transposition via `dict[plan_id, MCTSNode]` in-memory at `_mcts.py:881-882` and emits no fork-namespace audit datoms. Earlier draft framing was wrong; corrected here.

**Rationale:** the audit shapes that DO emit are: (a) `:mcts/iteration` (design §13 + ADR-7, already shipped); (b) `:mcts/search` start/end summary (already shipped via `mcts_search`'s `db` argument); (c) `:llm/call` per expander/evaluator dispatch (LD3); (d) `:act/result` + `:plan/done` PROVENANCE on winner leaves (2.3a LD2 + LD3). Adding a new `:branch/*` shape would duplicate provenance for zero added information — the iter-count + winner_plan_id + tree_dump fields on `MCTSResult` (`_mcts.py:772-793`) are the wire format already.

### LD6 — MCTS CONFIG: branch-bridge default `MCTSConfig(max_iter=50, expander_k=4)`; tuning hooks via `payload["mcts_config"]` (optional, capped)

**Decision (R0-fold I1 amended — explicit two-layer policy):** there are TWO config defaults, distinct on purpose:
- **Engine default** (`MCTSConfig()` at `_mcts.py:270-277`): `max_iter=200`, `max_unique_plans=64`, `c_puct=1.4`, `expander_k=4`. Unchanged; this is the v0.6.5 substrate-level default.
- **Branch-bridge default** (`_searcher.py::_BRANCH_BRIDGE_DEFAULT_CONFIG`): `MCTSConfig(max_iter=50, expander_k=4)`. ALWAYS used by 2.3b unless `decision.payload["mcts_config"]` overrides. The 50-iter ceiling matches the wall-clock posture (50 iters × 4 expander_k ≈ 200 LLM calls ≈ ~3 min wall-clock per branch escalation at ~1s per `:llm/call`).

When `payload["mcts_config"]` is present, the bridge merges the supplied dict over the BRIDGE default via `dataclasses.replace`. Bounds-check on LLM-supplied values: `max_iter ≤ MAX_BRANCH_MAX_ITER` (default 50); `expander_k ≤ MAX_BRANCH_EXPANDER_K` (default 4). Out-of-bound values rejected with `BranchPayloadValidation("mcts_config.<field>", "exceeds 2.3b cap")` BEFORE search starts.

**Rationale (R0-fold I1 codex finding):** earlier draft said "construct `MCTSConfig()` default" which would inherit `max_iter=200` (not 50), defeating the wall-clock posture. The two-layer split makes intent explicit: engine is generic-substrate-level, branch-bridge is demo-safe-by-default. 2.4a dogfood may tune branch-bridge defaults; engine defaults remain the substrate's reference values.

### LD7 — SUBSTRATE PREREQS: NONE — all primitives shipped on `_facade.py`

**Decision (R0-fold B1 amended — `s.txn.fork` removed):** unlike 2.3a (which added `s.plan.new_dispatcher()` as substrate-prereq T0), 2.3b adds NO new SDK methods. The 4 SDK touchpoints (`s.plan.mcts_search`, `s.plan.judge`, `s.fact.transact`, `s.effect.perform`) all ship on `_facade.py` today. The 5 `persistence.plan.*` types consumed (`Node`, `MCTSConfig`, `MCTSResult`, `LLMExpander`, `LLMJudgeEvaluator`) are SDK-private; the bridge imports them via `from persistence.plan import ...` matching 2.3a's `from persistence.plan import Dispatcher` pattern after T0 introduced the curated factory. (2.3b does NOT introduce a similar factory for `LLMExpander` / `LLMJudgeEvaluator` — they are constructed inline; if dogfood reveals a need, factory-curation is queued for 2.3c.)

**Rationale:** zero substrate-prereq tasks means 2.3b is pure coder-side wiring. Calendar projection: 2-3 days vs 2.3a's 1 day — the extra day is the `LLMExpander` + `LLMJudgeEvaluator` provider-closure construction (new prompt templates, new JSON-mode response parsing, softmax normalization at the boundary), which 2.3a did not need.

## 6. Forced spec deviations to expect

Per 2.3a's calibration (12 forced deviations from 1 primitive), 2.3b composes 3 primitives — expect 6-12 deviations on substrate-API surface verification:

1. **`mcts_search.started_at_ms`** is a REQUIRED positive int — not `None`-able, not `float`. The bridge synthesizes `time.time_ns() // 1_000_000` at call site. (Wall-clock injection is queued for 2.4a `:sys/now`; 2.3b uses raw `time` module at the boundary as a documented forced deviation.) **Determinism note (R1-fold I3):** `_derive_search_id(initial_plan_id, config, started_at_ms)` is deterministic GIVEN `started_at_ms` — but two fresh runs of the same scenario will NOT produce byte-identical `search_id` / audit chain because `started_at_ms` advances. This is acceptable because audit-replay records `started_at_ms` and replays it. Cross-run byte-identity is NOT a 2.3b acceptance criterion; replay byte-identity (record + replay produces same chain) IS, and that holds.
2. **`MCTSConfig.__post_init__`** rejects `bool` values for numeric fields via `isinstance(v, bool)` check FIRST. Payload-merge from `decision.payload["mcts_config"]` must coerce JSON `true`/`false` → reject before `MCTSConfig(...)` construction.
3. **`LLMExpander.provider`** signature is `Callable[[Node, int], Sequence[tuple[Action, float]]]` — NOT a Protocol with `propose` method. The bridge wires a closure, not a class.
4. **`LLMJudgeEvaluator.provider`** signature is `Callable[[Node], float]` (per `_mcts.py:469`) — single-plan-arg, returns score in [0.0, 1.0]. Bridge wraps the score via `:llm/call` dispatch + tool-use response parsing.
5. **`MCTSResult.tree_dump`** is a frozen tuple of `(parent_plan_id, child_plan_id, action_hash, visits, q)` per edge, lex-sorted by first three. The bridge's "winner" extraction reads `MCTSResult.winner` (a `Node`) directly — `tree_dump` is for replay verification only.
6. **`mcts_search` with `db=None`** still advances the chain symmetrically (unobservable). Tests for `db is not None` must verify the chain is non-empty, not just non-None.
7. **`Action` ADT** is closed-isinstance-strict (`_mcts.py:80`). The expander wrapper's `ComposeWithSkillAction` rejection must use `isinstance` check, not duck-typing.
8. **2.3a's `validate_plan_for_2_3a` is the seed-plan validator** (R0-fold N1). Expander proposals that would produce a winner failing this validator are rejected at the expander wrapper layer (defensive `apply_action`-then-validate dry-run). Avoids the "looser seed validator wastes search budget" trap codex flagged.

9. **`mcts_search` evaluator-failure absorption** (R0-fold I2). The engine catches evaluator exceptions as `phase="reject"` rows internally — they do NOT bubble out as exceptions. `BranchSearchFailed` for evaluator-failure is detected via `MCTSResult.terminated_by == "all_evaluations_failed"` post-search, not via `try/except` around `mcts_search`.

10. **LLM provider routing through `:llm/call`** (R0-fold I3). The expander/evaluator providers are NOT reusable closures — they construct request payloads and dispatch via `substrate.effect.perform(":llm/call", request)`. This preserves replay byte-identity AND matches `_decide`'s pattern at `_session.py:108`.

11. **`MCTSResult.terminated_by` enum coverage** (R1-fold N2). With `max_iter=50` and `expander_k=4`, the search can introduce up to 200 unique plans across iterations — well above `max_unique_plans=64` engine default. Realistic terminal modes for 2.3b: `max_iter`, `max_unique_plans`, `simple_regret_converged`, `wall_clock_exhausted`, `all_evaluations_failed`. Tests G4 should NOT assume termination is `max_iter`-driven; they parametrize over `max_unique_plans`-driven termination as a normal happy-path subcase.

Forced deviations beyond #1-#8 will be captured in commit bodies + CHANGELOG-coder.md per 2.3a precedent.
