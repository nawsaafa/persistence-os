## 1. Overview

Phase 2.3b fills the `_escalate_branch` stub at `src/persistence/coder/_session.py:254-257` (`CoderStubNotImplemented("Phase 2.3b — s.plan.mcts_search + s.txn.fork + s.plan.judge")`). The escalation gate is already wired at `_session.py:251-252`; 2.3b is the BRIDGE phase that connects the LLM-driven decision loop to the existing 45-KB v0.6.5 MCTS engine at `src/persistence/plan/_mcts.py` (PUCT search + `:mcts/iteration` Merkle-chained provenance + `Action` ADT + `Expander`/`Evaluator` Protocols).

The phase has zero new substrate primitives — `s.plan.mcts_search` (`_facade.py:709`), `s.plan.judge` (`_facade.py:728`), `s.txn.fork` (`_facade.py:329`), and the `LLMExpander` + `LLMJudgeEvaluator` constructors (`_mcts.py:388, 469`) all ship today on the curated SDK surface. The phase introduces one new coder module (`src/persistence/coder/_searcher.py`) modelled on 2.3a's `_planner.py`. MCTS exploration is a TERMINAL mode-switch — when the LLM emits `kind="branch"`, the coder builds a seed `Node`, constructs an LLM-provider-backed expander+evaluator, runs `s.plan.mcts_search`, and then **executes the winner exactly ONCE** via 2.3a's `_escalate_plan_body` bridge. Losing branches are STRUCTURAL — they never run their leaf handlers, so no real-world side effects (`:fs/write`, `:shell/exec`, `:git/commit`) leak.

## 2. Locked design decisions (LD0–LD7)

8 LDs, all closed via the `consensus-driven-architecture-decision` skill (2 codex `--reasoning high` passes converged). Acceptance signal for the skill: zero user-weighing during the brainstorm. ✅

### LD0 — SCOPE: MCTS-search drives; winner executes once via 2.3a's bridge; NO execution-time fork over real-world effects

**Decision (CONTROLLER FOLDED to codex R0 REJECT-FOR-A):** the 3 substrate primitives compose as `s.plan.mcts_search` (drives the search over plan STRUCTURE) → `s.plan.judge` (scores via `LLMJudgeEvaluator` evaluator) → `s.txn.fork` (used INTERNALLY by the search loop for transposition-table speculation over opaque Plan-AST objects, NOT at execution-time over substrate side effects). After `MCTSResult.winner` is selected, the bridge invokes `_escalate_plan_body(coder, synthesized_decision)` (2.3a's existing happy/failure paths) ONCE on the winner's plan EDN. Losing branches are never executed — their `:fs/write` / `:shell/exec` / `:git/commit` leaves are explored only as Plan-AST structures during search, not as real handler dispatches.

**Rationale (codex R0 finding the controller missed):** the controller's original Option B proposed forking each candidate through `_escalate_plan_body` and letting `s.txn.fork` "roll back losers". Codex's decider: *"`s.txn.fork` only guarantees rollback/isolation for facts, not external effects. 2.3a plan execution is wired to call `substrate.effect.perform(...)` for `:fs/write`, `:shell/exec`, `:git/commit` — losers' real-world mutations leak regardless of whether the substrate rolls back facts."* The right safety invariant is "only ONE plan ever executes per `_escalate_branch` call", not "losers' facts don't persist". MCTS over plan STRUCTURE (substitute leaf / add step / compose with skill `Action`s — `_mcts.py:60-78`) explores plan-tree variations as content-addressed Merkle DAG operations on opaque Python `Node` objects; no `Dispatcher` is invoked during search.

**Falsifiable acceptance signal:** scripted scenario, K=3 candidate plans (the seed plan plus 2 expander-generated variants) where each candidate would `:fs/write` a distinct marker on disk if executed (`marker_A`, `marker_B`, `marker_C`); after `_escalate_branch` returns, assert `len([f for f in os.listdir(scratch) if f.startswith("marker_")]) == 1` AND only the winner's `:act/result` datoms appear in the parent fact-store. Two+ markers → losers' handlers ran → fail.

### LD1 — TRIGGER: `decision.kind == "branch"` ONLY; drop the confidence-based escalation half of the gate

**Decision (codex R0 ACCEPT):** the escalation gate is `decision.kind == "branch"` per the FROZEN intent of `_session.py:251-252`. The current gate ALSO checks `decision.confidence < self.confidence_threshold` (default `0.65`); 2.3b ships the EXPLICIT half ONLY and removes the confidence-based half. Confidence-based escalation defers to Phase 2.4a once dogfood calibration data exists.

**Rationale (codex R0 sharper than controller's):** *"a confidence-based trigger can route non-`branch` payloads (e.g. `kind=="act"` with `{op,args}`) into a branch escalator that necessarily expects a branch-specific payload contract, creating a type/shape mismatch that either forces implicit conversions (unsafe) or silent failures."* This is a real correctness risk, not just a "tunable threshold" issue. Mirroring 2.3a's LD1 (`kind=="plan"` only) keeps the gate single-cause, testable, and payload-shape aligned. The confidence threshold default at `_session.py:39` is "starting value, will be tuned during dogfood (Phase 2.4a) on calibration data" per the design doc — meaning 2.3b shipping it would force an arbitrary threshold to be load-bearing for tests.

### LD2 — PAYLOAD: `payload["seed_plan_edn"]: <EDN-string>`; mirror 2.3a's plan_edn discipline

**Decision (codex R0 ACCEPT-A):** the `:llm/decision` payload when `kind=="branch"` carries `{"seed_plan_edn": "<canonical-EDN-text>"}`. The bridge applies the SAME 3-stage pipeline 2.3a's `_planner` uses: byte-budget (≤8192) → `s.plan.parse(strict=False)` → 2.3a's `validate_plan_for_2_3a` (or a tightened sibling `validate_seed_plan_for_2_3b` if MCTS allows additional structural shapes). The parsed `Node` becomes `mcts_search`'s `initial_plan` argument.

**Rationale (codex R0 decider):** *"the current MCTS API requires a concrete `initial_plan: Node` and is written as 'search over a Plan AST,' not an 'intent-to-plan synthesis' engine, so an EDN seed is the only payload that feeds the existing ingestion path without inventing new expander responsibilities."* (`_mcts.py:844-853`.) Rejected: `payload["intent"]` (NL string) — would force the expander to synthesize plans from scratch, which `LLMExpander` does NOT do today (it proposes Actions to apply to an existing Node). Rejected: `payload["candidates"]` (pre-baked K plans) — collapses MCTS to a no-op judge-only layer, contradicting LD0.

**LLM ergonomics note:** `_PLAN_EDN_GUIDANCE` in `_prompt.py` (added at 2.3a T7) gets a sibling `_BRANCH_EDN_GUIDANCE` constant explaining the seed-plan shape: "Provide a starting Plan AST as the seed; MCTS will explore structural variations (substitute leaves, insert steps) around it. Use `kind="branch"` when uncertain which leaf composition will best achieve the goal — NOT when you have a definite linear plan (use `kind="plan"` instead)."

### LD3 — EXPANDER + EVALUATOR: `LLMExpander` constrained to non-skill Actions; `LLMJudgeEvaluator` via `s.plan.judge`; `ComposeWithSkillAction` queued for 2.3c

**Decision:** the bridge constructs both `LLMExpander` and `LLMJudgeEvaluator` per-escalation, threading the coder's existing LLM provider closure (the same callable installed at `Coder.__init__` for `_decide`). The expander provider is constrained to propose ONLY `SubstituteLeafAction` and `AddStepAction` — `ComposeWithSkillAction` requires a populated `SkillLibrary` which is queued for Phase 2.3c. A defensive validator wraps the provider closure: if the LLM returns a `ComposeWithSkillAction`, reject the proposal (treat as if the LLM proposed nothing for that slot) rather than letting `apply_action` raise downstream.

**Rationale:** `LLMExpander` (`_mcts.py:388-411`) is pure delegation — its `provider` closure is caller-owned, so 2.3b's bridge owns the prompt template + JSON parsing + softmax normalization. The provider closure for the expander generates "what's the next thing to try?" prompts; the evaluator closure generates "score this plan structurally" prompts. Both run during search and emit `:mcts/iteration` audit datoms via `mcts_search`'s built-in `db.transact` per iteration (LD5). 2.3c's skill-library wiring will REPLACE the constraint — the same `_searcher.py` will inject `ComposeWithSkillAction` proposals once skills are content-addressed.

**Forward-pointer:** `_searcher.py:_make_branch_provider(coder, prompt_template)` is the seam where 2.3c will widen the action space.

### LD4 — FAILURE: mirror 2.3a LD4 — `PlanExecutionFailed` re-raise on the winner's execution; native traceback unavailable

**Decision:** if the winner's execution (the post-search `_escalate_plan_body(coder, synthesized_decision)` call) raises `PlanExecutionFailed`, the failure propagates UNCHANGED through `_escalate_branch` — `Coder.run()` exits via raise. No additional summary datom emission at the branch layer beyond what 2.3a already emits (partial-trace `:act/result` per leaf + failure-shaped `:act/result` for the failing leaf + final `:plan/execute` summary). The branch layer is a search-and-pick wrapper; once the winner is chosen, failure semantics are 2.3a's exclusively.

**Rationale:** there is exactly ONE plan execution per `_escalate_branch` call (LD0). Failure is therefore single-source-of-truth. Adding a redundant `:branch/failed` summary datom would duplicate `:plan/execute` (2.3a) and risk drift. Native traceback availability inherits 2.3a's W3 rescope to v0.9.x (queued).

**MCTS-search-failure separation:** if `mcts_search` itself raises (e.g. `ExpanderContractError` per `_mcts.py:_PRIOR_TOL`, or LLM-provider exception during expand/evaluate), the bridge catches it, emits ONE `:act/result` with `op=":mcts/search"` + `error=<error_repr>`, and re-raises a new `BranchSearchFailed` exception so `Coder.run()` exits. This mirrors the 2.3a pattern of "search-layer failure ≠ execution-layer failure"; tests for both paths (G5a search-failure, G5b execution-failure) keep them disambiguated.

### LD5 — AUDIT: `:mcts/iteration` Merkle-chained provenance + `:mcts/search` summary + `:plan/done` from 2.3a; NO new audit shapes

**Decision:** the bridge passes `db=substrate._db` to `mcts_search` so the existing `:mcts/iteration` provenance fires (one `db.transact` per search iteration with `phase="expand"` / `phase="evaluate"` / `phase="reject"` rows + `mcts/prev-hash` chain + `mcts/search` summary on iter 0 + winner+iter-count+terminated-by+finished-at on search end — `_mcts.py:862-869`). After `mcts_search` returns, the bridge then invokes `_escalate_plan_body(coder, synthesized_decision)` which fires 2.3a's existing `:act/result` + `:plan/done` PROVENANCE datoms on the winner's leaves. Internal `s.txn.fork` calls inside `mcts_search`'s transposition table emit the canonical 4-datom `:fork/probe` + `:fork/branch`×N + `:fork/score`×N + `:fork/chosen` shape per the v0.6.5 implementation — 2.3b inherits this audit trail without adding new shapes.

**Rationale:** all three audit shapes are already shipped and battle-tested. `:mcts/iteration` is design §13 + ADR-7; `:fork/*` is Phase 2.0c-extended #145ext; `:plan/done` is 2.3a LD2. Adding a new `:branch/*` shape would duplicate provenance for zero added information — the iter-count + winner_plan_id + tree_dump fields on `MCTSResult` (`_mcts.py:772-793`) are the wire format already.

### LD6 — MCTS CONFIG: ship sane defaults; expose tuning hooks via `payload["mcts_config"]` (optional)

**Decision:** `_searcher.py` constructs `MCTSConfig()` with default values (`max_iter=200`, `max_unique_plans=64`, `c_puct=1.4`, `expander_k=4`) UNLESS `decision.payload["mcts_config"]` is present, in which case it merges the supplied dict over the defaults via `dataclasses.replace`. Bounds-check on the LLM-supplied values: `max_iter ≤ MAX_BRANCH_MAX_ITER` (default 50 for 2.3b — tighter than the 200 default to keep search wall-clock bounded for the demo); same for `expander_k ≤ MAX_BRANCH_EXPANDER_K` (default 4). Out-of-bound values rejected with `BranchPayloadValidation("mcts_config.<field>", "exceeds 2.3b cap")` BEFORE search starts.

**Rationale:** `mcts_search` is byte-deterministic in terminating-state given fixed inputs (`_mcts.py:856-861`), but wall-clock is not — `max_iter=200` × LLM latency per iteration ≈ minutes per branch escalation. The 2.3b cap (50 iters × 4 expander_k) is empirically tunable in 2.4a dogfood. The `payload["mcts_config"]` hook lets the LLM signal "deeper search needed" for hard problems, but the substrate-side cap prevents runaway costs.

### LD7 — SUBSTRATE PREREQS: NONE — all primitives shipped on `_facade.py`

**Decision:** unlike 2.3a (which added `s.plan.new_dispatcher()` as substrate-prereq T0), 2.3b adds NO new SDK methods. The 5 SDK touchpoints (`s.plan.mcts_search`, `s.plan.judge`, `s.txn.fork`, `s.fact.transact`, `s.effect.perform`) all ship on `_facade.py` today. The 4 `persistence.plan.*` types consumed (`Node`, `MCTSConfig`, `MCTSResult`, `LLMExpander`, `LLMJudgeEvaluator`) are SDK-private; the bridge imports them via `from persistence.plan import ...` matching 2.3a's `from persistence.plan import Dispatcher` pattern after T0 introduced the curated factory. (2.3b does NOT introduce a similar factory for `LLMExpander` / `LLMJudgeEvaluator` — they are constructed inline; if dogfood reveals a need, factory-curation is queued for 2.3c.)

**Rationale:** zero substrate-prereq tasks means 2.3b is pure coder-side wiring. Calendar projection: 2-3 days vs 2.3a's 1 day — the extra day is the `LLMExpander` + `LLMJudgeEvaluator` provider-closure construction (new prompt templates, new JSON-mode response parsing, softmax normalization at the boundary), which 2.3a did not need.

## 6. Forced spec deviations to expect

Per 2.3a's calibration (12 forced deviations from 1 primitive), 2.3b composes 3 primitives — expect 6-12 deviations on substrate-API surface verification:

1. **`mcts_search.started_at_ms`** is a REQUIRED positive int — not `None`-able, not `float`. The bridge synthesizes `time.time_ns() // 1_000_000` at call site. (Wall-clock injection is queued for 2.4a `:sys/now`; 2.3b uses raw `time` module at the boundary as a documented forced deviation.)
2. **`MCTSConfig.__post_init__`** rejects `bool` values for numeric fields via `isinstance(v, bool)` check FIRST. Payload-merge from `decision.payload["mcts_config"]` must coerce JSON `true`/`false` → reject before `MCTSConfig(...)` construction.
3. **`LLMExpander.provider`** signature is `Callable[[Node, int], Sequence[tuple[Action, float]]]` — NOT a Protocol with `propose` method. The bridge wires a closure, not a class.
4. **`LLMJudgeEvaluator.provider`** signature TBD on first inspection — verify before T3 begins.
5. **`MCTSResult.tree_dump`** is a frozen tuple of `(parent_plan_id, child_plan_id, action_hash, visits, q)` per edge, lex-sorted by first three. The bridge's "winner" extraction reads `MCTSResult.winner` (a `Node`) directly — `tree_dump` is for replay verification only.
6. **`mcts_search` with `db=None`** still advances the chain symmetrically (unobservable). Tests for `db is not None` must verify the chain is non-empty, not just non-None.
7. **`Action` ADT** is closed-isinstance-strict (`_mcts.py:80`). The expander wrapper's `ComposeWithSkillAction` rejection must use `isinstance` check, not duck-typing.
8. **2.3a's `validate_plan_for_2_3a`** may be too tight for seed plans (e.g. it requires `:seq` interior; if MCTS produces `:par` or other interior tags via `AddStepAction`, the post-search winner could fail re-validation). Decide at T2 whether `validate_seed_plan_for_2_3b` is a tightened sibling, a loosened sibling, or whether the post-search winner is re-validated against `validate_plan_for_2_3a` (executed via 2.3a's bridge which already enforces it).

Forced deviations beyond #1-#8 will be captured in commit bodies + CHANGELOG-coder.md per 2.3a precedent.
