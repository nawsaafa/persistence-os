## 1. Overview

Phase 2.3c.1 is the FIRST half of the 2.3c skill-system rollout. The substrate-side `SkillLibrary` (`src/persistence/plan/_skill_library.py`, 357 LOC, content-addressed registry with cross-instance fact-store idempotency, 3 datoms per registration: `skill/plan`, `skill/promotion-record`, `skill/registered-at`) and the SDK factory `s.plan.skill_library(db)` are ALREADY shipped (Phase 2.0c-prime #147). The MCTS-side `ComposeWithSkillAction` + `_apply_compose_with_skill` + `_SkillNotRegistered` + `_SkillCircularComposition` are ALSO shipped (`src/persistence/plan/_mcts.py`).

What's missing for the coder demo:

1. **Audit-chain integration.** Skill registration today bypasses `CANONICAL_AUDIT_WRAPPED_OPS` ‚Äî `SkillLibrary.register` writes 3 fact-store datoms but emits no AuditEntry on the canonical chain. The coder cannot register skills under audit accountability. Same gap 2.1c.6 closed for `:claim/emit` + `:blob/put`.
2. **Coder-side use.** No path for the coder to (a) emit a "register this Plan AST as a skill" operation under the existing `kind="plan"` decision flow, or (b) recall a registered skill's Plan EDN and incorporate it into a subsequent plan. Without this, the registry exists but never gets exercised by the persistence-coder demo.
3. **End-to-end usage proof.** The 2.3c kickoff codex consensus pass identified the load-bearing risk: shipping the registry without an exercised define‚Üíresolve‚Üíexpand‚Üíexecute path lets wrong invariants (identity model, audit anchoring granularity, expansion determinism) ship undetected until 2.3c.2 piles on `:llm/call` recursion + `ComposeWithSkillAction` proposals. Codex's verbatim flag: *"acceptance without use is weak: you can pass define/recall/audit-chain tests while getting expansion semantics, identity constraints, and determinism subtly wrong ‚Äî then 2.3c.2 forces a rewrite."*

2.3c.1 ships TWO new substrate-effect ops (`:skill/define`, `:skill/lookup`) + their inclusion in `s.plan.new_dispatcher()`'s registered leaf set + a load-bearing G4 end-to-end test that exercises define‚Üílookup‚Üíinline-into-plan‚Üíexecute under the canonical audit chain.

**Scope discipline (deferred to 2.3c.2):** `:llm/call` recursion (skills with embedded LLM dispatches), `ComposeWithSkillAction` proposal acceptance in the MCTS expander (lifting 2.3b's FD7 rejection), and any RUNTIME skill resolution at the dispatcher level. The 2.3c.1 model is **define + procedural recall** ‚Äî the coder retrieves a registered skill's Plan EDN as data and inlines it into a subsequent plan; no runtime resolution, no recursion.

## 2. Locked design decisions (LD0‚ÄìLD7)

7 LDs.

### LD0 ‚Äî SCOPE: define + procedural recall, NO runtime resolution, NO recursion, NO MCTS interaction

**Decision:** 2.3c.1 ships TWO new audit-wrapped substrate effect ops (`:skill/define`, `:skill/lookup`) and registers them as plan-leaf handlers via `_register_substrate_handlers` (12 ops total post-2.3c.1). The coder uses them via standard Plan AST leaves emitted in `kind="plan"` decisions. Skill RESOLUTION is procedural ‚Äî `:skill/lookup` returns the skill's Plan EDN as a substantive-return result; the coder inlines the EDN content into a subsequent `plan_edn` payload. NO dispatcher-level magic substitution, NO `:skill/compose` Plan AST primitive.

**Rationale:** keeps the substrate Plan AST unchanged; preserves 2.3a's strict `validate_plan_for_2_3a` semantics; defers dispatch-time substitution semantics to 2.3c.2 where `:llm/call` recursion + `ComposeWithSkillAction` proposals will already need to address it. 2.3c.1 is a "data-flow demonstration" that the registered skill's content survives a round-trip through fact-store + canonical audit chain + plan execution under coder control.

**Falsifiable acceptance signal (G4 ‚Äî LOAD-BEARING per codex consensus):** scripted 3-iter coder scenario:
- Iter 1: emit `kind="plan"` with body `[:seq {} [:skill/define {":plan-edn" "[:seq {} [:fs/write {...}]]" ":promotion-id" "p-1" ":registered-at-ms" 1700000000000}]]`. Plan executes; `:skill/define` audit datom emitted; SkillLibrary now contains `skill/abc...`.
- Iter 2: emit `kind="plan"` with body `[:seq {} [:skill/lookup {":skill-id" "skill/abc..."}]]`. Plan executes; `:skill/lookup` audit datom emitted; LeafResult carries `{":plan-edn": "...", ":promotion-id": "p-1", ":plan-id": "..."}`. Coder observes the result via standard `_observe` ‚Üí next iter's prompt context.
- Iter 3: emit `kind="plan"` with the looked-up plan-edn inlined as plan body ‚Äî `[:seq {} [:fs/write {...}]]` (the original skill's body). Plan executes; `:fs/write` happens; produces `marker.txt` on disk.

Assertions: `:skill/define` + `:skill/lookup` audit entries linked via prev-hash chain; `:plan/done` PROVENANCE datom for each iter; the inlined skill's `:fs/write` produces the marker; ZERO `:skill/compose` audit datoms (proves no dispatcher-level substitution); SkillLibrary state is consistent across the 3 iters (idempotent re-define = no extra datoms).

### LD1 ‚Äî `:skill/define` op shape

**Decision:** new audit-wrapped substantive-return op. Follows the 2.2a pattern (`:fs/read` etc.): wrapped at audit middleware, bottom-of-stack handler installed via `s.effect.install_handler(handler, position="bottom")`. Handler signature (sketch):

```python
def _skill_define_clause(args: Mapping[str, Any], kwargs: Mapping[str, Any], ctx) -> Mapping[str, Any]:
    plan_edn = _required_str(args, ":plan-edn")
    promotion_id = _required_str(args, ":promotion-id")
    registered_at_ms = _required_int(args, ":registered-at-ms")
    plan = parse(plan_edn, strict=False)
    skill_lib = ctx.substrate.plan.skill_library(ctx.substrate._db)
    skill_id = skill_lib.register(
        plan,
        _PromotionRecordStub(promotion_id=promotion_id),
        registered_at_ms=registered_at_ms,
    )
    return {":skill-id": skill_id, ":plan-id": plan.id}
```

The 3 fact datoms (`skill/plan`, `skill/promotion-record`, `skill/registered-at`) are written by `SkillLibrary.register` via `db.transact` BEFORE the audit middleware emits the AuditEntry ‚Äî matches 2.1c.6 `:claim/emit`/`:blob/put` "fact-write first, audit second" pattern (provenance-survives-audit-failure). Idempotency on re-define: handled by `SkillLibrary.register` (returns same skill_id, ZERO additional fact datoms; the audit middleware DOES emit a fresh AuditEntry for the redundant call ‚Äî mirrors 2.2a `:fs/read` semantics).

**Rationale:** the substantive-return shape `{":skill-id", ":plan-id"}` lets the coder (a) capture the skill_id from `:act/result.result_summary` for use in subsequent `:skill/lookup` calls, and (b) verify content-addressing by comparing the returned `:plan-id` against the input plan's expected hash.

### LD2 ‚Äî `:skill/lookup` op shape

**Decision:** new audit-wrapped substantive-return op. Same pattern as LD1. Handler signature:

```python
def _skill_lookup_clause(args, kwargs, ctx) -> Mapping[str, Any]:
    skill_id = _required_str(args, ":skill-id")
    skill_lib = ctx.substrate.plan.skill_library(ctx.substrate._db)
    result = skill_lib.lookup(skill_id)
    if result is None:
        raise SkillNotFound(skill_id=skill_id)
    plan, record = result
    return {
        ":plan-edn": unparse(plan),
        ":promotion-id": record.promotion_id,
        ":plan-id": plan.id,
    }
```

`SkillNotFound(ValueError)` is a new error class in `handlers/skill.py`. Raises propagate through `_execute.py` per the standard plan-execution failure pathway (handler returns dict on success; raises on failure ‚Üí `LeafResult.error_repr` populated ‚Üí `PlanExecutionFailed` if leaf is final).

**Rationale:** unparsing the Plan AST back to canonical EDN means the coder receives a STRING it can splice into subsequent `plan_edn` payloads. The Plan AST `Node` itself is NOT returned (would be a Python object leaked across the substrate boundary).

### LD3 ‚Äî `_PromotionRecordStub`: minimal in-coder fabrication

**Decision:** the coder has NO access to A7's `PromotionRecord` dataclass (Phase 3 NeSy 2027 scope per `_skill_library.py` docstring lines 40-45). For 2.3c.1's `:skill/define`, fabricate a `_PromotionRecordStub` that satisfies `_PromotionRecordLike`:

```python
@dataclass(frozen=True, slots=True)
class _PromotionRecordStub:
    promotion_id: str
```

Same minimal-stub pattern `tests/plan/test_skill_library.py` uses today. It does NOT perform G1/G2/G3/G4 promotion validation (those gates live in A7's PromotionRecord). 2.3c.1 ships the stub explicitly and the design doc names it as a deferred-validation deferral: **skills registered via the coder demo skip A7 promotion gates**; A7 integration is queued for v0.9.x.

**Rationale:** A7 isn't shipped, isn't blocking 2.3c.1's acceptance signal (audit-chain anchoring + define‚Üílookup‚Üíinline-execute round-trip), and forcing A7 to ship inside 2.3c.1 would expand scope to include the `PromotionRecord` data-class definition + its G1-G4 validation gates (~600+ LOC alone).

### LD4 ‚Äî Failure modes mirror 2.3a LD4

`:skill/define` / `:skill/lookup` failures during plan execution propagate as `PlanExecutionFailed` per the 2.3a path. Three new error classes in `handlers/skill.py`:

- `SkillNotFound(ValueError)` ‚Äî `:skill/lookup` on unregistered `skill_id`
- `SkillDefineValidation(ValueError)` ‚Äî `:skill/define` with invalid args (missing/wrong-type fields, unparseable `:plan-edn`)
- `SkillLookupValidation(ValueError)` ‚Äî `:skill/lookup` with invalid args (missing/wrong-type `:skill-id`)

Native traceback inheritance from 2.3a LD4's W3 rescope (queued v0.9.x, `__cause__ is None` G5 acceptance signal flips when fix lands).

## 5. Anticipated forced spec deviations

(Confirmed during impl; speculative until then)

- **FD1 (anticipated):** `:skill/define` arg keys use keyword-form (`":plan-edn"`, `":promotion-id"`, `":registered-at-ms"`) per EDN convention; verify against actual ingested arg-key shape during T1 (the dispatcher adapter may strip leading colons per `Datom.a` convention ‚Äî checked at impl).
- **FD2 (anticipated):** `_PromotionRecordStub` is a stub (LD3); A7 PromotionRecord integration deferred to v0.9.x. Document explicitly in CHANGELOG.
- **FD3 (anticipated):** `parse(plan_edn, strict=False)` is required for `:skill/define` because skills can have arbitrary leaf tags (not just 2.3a's 10-op closed set). Strict mode would reject most real skill bodies. Match 2.3a's parse pattern.
- **FD4 (anticipated):** `unparse(node)` for `:skill/lookup` returns canonical EDN form (bare-keyword); coder downstream MUST splice this verbatim into next plan's `plan_edn` payload (not re-format).
- **FD5 (anticipated):** `:skill/lookup` does NOT use `s.fact.transact` (no fact write); audit-only side-effect is the AuditEntry. Differs from `:skill/define` at the fact-store boundary. Document.
