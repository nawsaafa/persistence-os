## 1. Overview

Phase 2.3c.2 is the SECOND half of the 2.3c skill-system rollout, completing the trio: registry (2.3c.1) + recursion + composition. Two intertwined lifts:

1. **`:llm/call` recursion.** The substrate-side `:llm/call` op is currently a LEAF in the dispatcher (`CANONICAL_AUDIT_WRAPPED_OPS`); when a registered skill's Plan AST contains a `:llm/call` leaf, it WORKS today (2.3c.1 procedural-recall demo proves this), but the audit-chain semantics were never specified for the case where a skill body's `:llm/call` triggers another `:llm/call` deeper in the recursive expansion. 2.3c.2 specifies and audits the recursion explicitly: outer call's prev-hash chain becomes the parent of inner-call audit entries; depth limit + cycle detection + global resource budget enforce safety.

2. **`ComposeWithSkillAction` proposal acceptance.** 2.3b's MCTS expander wrapper (`_searcher.py:361, 369`) REJECTS `ComposeWithSkillAction` at TWO layers (kind-string + isinstance — FD7 belt-and-braces) because no SkillLibrary was populated. 2.3c.2 lifts both rejections. Expander now legitimately proposes `ComposeWithSkillAction`; MCTS engine's `_apply_compose_with_skill` (`_mcts.py:187-209`) grafts the looked-up skill into the Plan AST during search; winner unparses through 2.3a's `_escalate_plan_body` unchanged. The expander wrapper threads the active SkillLibrary into search context.

**Design discipline lock from kickoff codex consensus pass (2026-05-08):**
- (a) Recursion safety: HARD `MAX_LLM_CALL_DEPTH=3` floor + GLOBAL token-budget soft cap across the full nested call tree (codex NUANCED-HYBRID — depth-only is bypass vector for big-context calls)
- (b) Composition timing: search-time RESOLUTION + content-hash PINNING in `:mcts/iteration` provenance; eager grafting OK, but pinned hash makes execution-time replay byte-identical (codex NUANCED-OTHER)
- (c) Recursion × composition: UNIFIED budget at dispatcher boundary — both feed same depth + token counter (codex DISAGREE-INDEPENDENT — composition introduces skills with recursive `:llm/call`, independent limits are bypass loopholes)
- (d) Confidence-based escalation: DEFER to 2.4a (codex AGREE — out of 2.3c.2 scope)
- (NEW) Cycle detection keyed by **content hash** (not `skill_id` alone) — codex's missed-flag callout
- (NEW) Audit chain shape decision: linear append + parent pointer (best of both — see LD5)

## 2. Locked design decisions (LD0-LD7)

### LD0 — SCOPE: lift FD7 + audit-anchor recursion + global budget; NO confidence-based escalation; NO new substrate primitives

**Decision:** 2.3c.2 makes 4 cross-cutting changes to the coder + bridge:
1. Remove `_searcher.py:361` `kind == "ComposeWithSkillAction"` rejection AND remove `_searcher.py:369` belt-and-braces isinstance rejection (FD7 from 2.3b)
2. 3. Audit-anchor `:llm/call` recursion via a per-dispatch DispatcherContext that carries depth + token-counter + cycle set (content-hash-keyed); audit middleware emits AuditEntries with `parent_audit_entry_id` field linking nested calls to the outer call's id
4. Wire global resource budget at dispatcher boundary — both `:llm/call` recursion AND `ComposeWithSkillAction`-grafted skill bodies feed the same DispatcherContext counter; budget exhaustion raises `LLMRecursionBudgetExceeded`

Out of scope (deferred):
- Confidence-based escalation (2.4a per kickoff codex (d))
- Production CLI wiring of recursion-aware dispatcher (2.4a, mirrors 2.2b deferral pattern)
- Multi-process skill registration races (already-queued v0.9.x from 2.3c.1)
- `Action` ADT extension beyond `ComposeWithSkillAction` (no new Action variants)

**Falsifiable acceptance signal (G4 LOAD-BEARING per consensus pattern):**
End-to-end scenario: register skill A whose body contains `[:llm/call ...]`. Then in a separate iter, MCTS expander proposes `ComposeWithSkillAction(skill_id=A)`. Winner Plan AST is the grafted form including A's `:llm/call`. Execute the winner; the inner `:llm/call` triggers RECURSION (depth=1); audit chain shows OUTER `:llm/call` AuditEntry → INNER `:llm/call` AuditEntry with `parent_audit_entry_id` pointing to outer. Assertions: (a) chain integrity prev-hash linkage; (b) parent_audit_entry_id correct; (c) DispatcherContext depth counter reaches 1 max (within MAX_LLM_CALL_DEPTH=3); (d) cycle detection rejects skill A re-invoking itself; (e) content-hash pinning in `:mcts/iteration` provenance survives expansion.

### LD1 — `:llm/call` recursion enabled at dispatcher; per-call DispatcherContext

**Decision:** `:llm/call` is no longer a leaf — handlers can themselves dispatch nested `:llm/call`s. Implementation:
- New `DispatcherContext` dataclass (`@dataclass(frozen=False)` — mutates depth counter; alternative: `@dataclass(frozen=True)` with `replace`-style copies — pick whichever minimizes lock contention, documented in impl)
- Fields: `depth: int = 0`, `token_count: int = 0`, `request_count: int = 0`, `cycle_set: set[str]` (content-hash-keyed), `parent_audit_entry_id: str | None = None`, `budget: RecursionBudget`
- ContextVar binding so nested calls inherit the outer context (similar to `_audit_stack.py` ContextVar pattern)
- Outermost `:llm/call` initializes a fresh DispatcherContext
- Each nested `:llm/call` increments depth + (post-call) token_count from response.usage + request_count
- On entry, check depth ≤ MAX_LLM_CALL_DEPTH (3) and token_count + estimated-tokens ≤ MAX_RECURSIVE_TOKENS (default 20_000) and request_count ≤ MAX_RECURSIVE_REQUESTS (default 10)
- Budget exhaustion raises `LLMRecursionBudgetExceeded(field="depth"|"tokens"|"requests", limit=..., observed=...)`

**Resource limits:**
```python
MAX_LLM_CALL_DEPTH: int = 3        # hard floor (codex consensus a)
MAX_RECURSIVE_TOKENS: int = 20_000  # soft cap, total across nested tree
MAX_RECURSIVE_REQUESTS: int = 10   # soft cap, total across nested tree
```

Defaults are demo-grade per codex (a); 2.4a dogfood may tune. Bounds-check pattern mirrors 2.3b `MAX_BRANCH_MAX_ITER=50` / `MAX_BRANCH_EXPANDER_K=4`.

### LD2 — Content-hash-keyed cycle detection (codex missed-flag callout)

**Decision:** the DispatcherContext.cycle_set tracks **content hashes**, not skill_ids. When a `ComposeWithSkillAction(skill_id=X)` is grafted at search-time (LD3), its plan body's content hash is computed (`plan.id` per `_skill_library.py:_derive_skill_id`); the FULL hash (not the 16-hex-char skill_id slice) is added to the cycle_set. Re-grafting any skill whose content hash is already in the set raises `SkillCircularComposition` (already exists in `_mcts.py:165`; reuse).

**Rationale (codex callout):** content-addressed registry can have semantically-identical skill aliases (different `skill_id` prefixes pointing to byte-identical Plan ASTs via re-registration with different `promotion_id`). skill_id-only cycle detection would (a) reject legitimate updates of the same logical skill across versions, OR (b) allow recursion via aliases that should be detected as cycles. Content-hash keying prevents both failure modes.

**Falsifiable acceptance signal (G6):** register skill A and skill B where both have IDENTICAL Plan AST bodies (different promotion_ids → different skill_ids → SAME content hash). Compose A inside B's body. Cycle detection MUST reject (assertion: `SkillCircularComposition` raised) even though `skill_id_A != skill_id_B`.

### LD3 — Composition expansion: search-time RESOLUTION + content-hash PINNING (codex NUANCED-OTHER on b)

**Decision:** when MCTS engine applies `ComposeWithSkillAction(skill_id=X)` via `_apply_compose_with_skill` (`_mcts.py:187-209`):
1. Look up the skill from SkillLibrary at SEARCH time (via the `skill_library: SkillLibrary` parameter threaded through MCTS engine — already shipped, just populated by 2.3c.2)
2. Compute the looked-up skill's content hash (`plan.id`)
3. **Pin the content hash in `:mcts/iteration` provenance** — emit a new datom field `composed_skill_content_hash: str` alongside the existing `composed_skill_id` field at `_mcts_datoms.py:209`
4. Graft the skill body into the Plan AST eagerly (existing engine behavior)
5. Winner Plan AST is FULLY-RESOLVED (no remaining `[:compose-with-skill ...]` references) — runs through 2.3a's `_escalate_plan_body` unchanged

**Replayability invariant:** at execution time, the dispatcher uses the PINNED content hash from search-time provenance, NOT a fresh SkillLibrary lookup. This prevents skill drift (where SkillLibrary gets mutated between MCTS selection and winner execution) from breaking replay byte-identity.

**Rationale (codex b NUANCED-OTHER):** search-time resolution + content-hash pinning is the strictly-better alternative to "search-time graft only" (loses replay determinism if library mutates) AND "execution-time graft only" (loses provenance falsifiability). Content-hash pinning is the load-bearing addition.

### LD4 — UNIFIED dispatcher budget (codex DISAGREE on c)

**Decision:** `:llm/call` recursion AND `ComposeWithSkillAction`-grafted skill execution feed the SAME DispatcherContext budget counter. Specifically:
- When MCTS grafts a skill into the Plan AST, the skill body's `:llm/call` leaves count toward the SAME depth + token + request counter as direct recursive calls
- Cycle detection (LD2) is enforced at the DispatcherContext level — applies to BOTH composition-driven and recursion-driven skill activation

**Rationale (codex c DISAGREE-INDEPENDENT):** independent limits create bypass loopholes — N composed skills each with depth-1 recursion would all be locally legal but globally exceed depth=N. Unified budget closes the loophole.

**Test gate (G3 falsifiability):** scenario where MCTS proposes ComposeWithSkillAction whose grafted body contains a recursive `:llm/call` — verify the recursive call's audit entry inherits the outer DispatcherContext's depth counter (continues at 1, not resets to 0).

### LD5 — Audit chain: linear append + parent pointer (NEW)

**Decision:** AuditEntries are appended LINEARLY to the canonical audit chain (existing semantics — prev_hash links each entry to its predecessor regardless of nesting). NEW field `parent_audit_entry_id: str | None` records the OUTER call's audit entry id when the current entry is a nested dispatch.

```python
# Example chain for outer-then-nested call:
# entry_outer:  prev_hash=PRIOR_HASH, parent_audit_entry_id=None,         op=":llm/call"
# entry_inner:  prev_hash=entry_outer.id, parent_audit_entry_id=entry_outer.id, op=":llm/call"
# entry_after:  prev_hash=entry_inner.id, parent_audit_entry_id=None,        op=":fs/write"
```

**Rationale (codex missed-flag callout):** linear append alone obscures causality (you can't tell which `:llm/call` was a child of which other from the chain shape). Adding parent pointer is cheap (one optional string field) and recovers the parent/child structure for replay/debug. Doesn't break replay byte-identity because parent_audit_entry_id is deterministic given the call structure.

**Migration: AuditEntry shape change.** Add `parent_audit_entry_id: str | None = None` field to `audit.py:AuditEntry` (currently has `id`, `prev_hash`, `op`, `args_hash`, `verdict`, `result_hash`, `latency_ms`, `recorded_at`). Default `None` preserves backward compatibility with all 2.0a-2.3c.1 audit entries. The chain-hash computation includes parent_audit_entry_id so prev-hash chain integrity covers it.

### LD6 — FD7 lift: 2 lines removed from `_searcher.py`

**Decision:** delete the two rejection sites at `_searcher.py:361` (kind-string drop) and `_searcher.py:369` (isinstance drop). Add: `_make_branch_expander(coder, skill_library: SkillLibrary)` signature change — threads SkillLibrary through to the search-context for `_apply_compose_with_skill` to use.

`_dry_run_apply_action_safely` at `_searcher.py:371` already calls `apply_action(seed_plan, action, skill_library=...)` per the engine contract; just needs the `skill_library` argument populated (currently passes None per the FD7 rejection regime).

**Falsifiability:** the existing 2.3b G3 test asserting `ComposeWithSkillAction` rejection MUST be either (a) removed and replaced by an acceptance test, OR (b) kept as a NEGATIVE test for cases where SkillLibrary is genuinely None or skill_id is unregistered (different rejection condition than 2.3b's blanket FD7 ban).

### LD7 — SUBSTRATE PREREQS: NONE

`SkillLibrary` + `_apply_compose_with_skill` + `Action` ADT (incl. `ComposeWithSkillAction`) all already shipped. Audit middleware extension for `parent_audit_entry_id` is INSIDE `audit.py` — not a new substrate primitive, just a new field on AuditEntry.
