## 1. Overview

Phase 2.3c.2 is the SECOND half of the 2.3c skill-system rollout, completing the trio: registry (2.3c.1) + recursion + composition. Two intertwined lifts:

1. **`:llm/call` recursion.** The substrate-side `:llm/call` op is currently a LEAF in the dispatcher (`CANONICAL_AUDIT_WRAPPED_OPS`); when a registered skill's Plan AST contains a `:llm/call` leaf, it WORKS today (2.3c.1 procedural-recall demo proves this), but the audit-chain semantics were never specified for the case where a skill body's `:llm/call` triggers another `:llm/call` deeper in the recursive expansion. 2.3c.2 specifies and audits the recursion explicitly: outer call's prev-hash chain becomes the parent of inner-call audit entries; depth limit + cycle detection + global resource budget enforce safety.

2. **`ComposeWithSkillAction` proposal acceptance.** 2.3b's MCTS expander wrapper (`_searcher.py:361, 369`) REJECTS `ComposeWithSkillAction` at TWO layers (kind-string + isinstance ã FD7 belt-and-braces) because no SkillLibrary was populated. 2.3c.2 lifts both rejections. Expander now legitimately proposes `ComposeWithSkillAction`; MCTS engine's `_apply_compose_with_skill` (`_mcts.py:187-209`) grafts the looked-up skill into the Plan AST during search; winner unparses through 2.3a's `_escalate_plan_body` unchanged. The expander wrapper threads the active SkillLibrary into search context.

**Design discipline lock from kickoff codex consensus pass (2026-05-08):**
- (a) Recursion safety: HARD `MAX_LLM_CALL_DEPTH=3` floor + GLOBAL token-budget soft cap across the full nested call tree (codex NUANCED-HYBRID ã depth-only is bypass vector for big-context calls)
- (b) Composition timing: search-time RESOLUTION + content-hash recording in `:mcts/iteration` provenance; eager grafting OK; pin gives REPLAY-EXPLAINABILITY (audit trail of which skill resolved to what content_hash) ã replay byte-identity itself is provided by the WINNER PLAN AST being content-addressed (codex NUANCED-OTHER; R1-fold I3 corrected: pinning is provenance, NOT byte-identity mechanism)
- (c) Recursion û composition: UNIFIED budget at dispatcher boundary ã both feed same depth + token counter (codex DISAGREE-INDEPENDENT ã composition introduces skills with recursive `:llm/call`, independent limits are bypass loopholes)
- (d) Confidence-based escalation: DEFER to 2.4a (codex AGREE ã out of 2.3c.2 scope)
- (NEW) Cycle detection keyed by **content hash** (not `skill_id` alone) ã codex's missed-flag callout
- (NEW) Audit chain shape decision: linear append + parent pointer (best of both ã see LD5)

### LD1 ã `:llm/call` recursion enabled at dispatcher; per-call DispatcherContext (R0-fold B1+B2: bounds-precision + pre-call token cap)

**Decision (R0-fold B1 amended ã bounds semantics):** `:llm/call` is no longer a leaf ã handlers can themselves dispatch nested `:llm/call`s. Implementation:
- New `DispatcherContext` dataclass (`@dataclass(frozen=False)` ã mutates depth counter)
- Fields: `depth: int = 0`, `token_count: int = 0`, `request_count: int = 0`, `cycle_path: list[str]` (content-hash-keyed; LIST ordered as active-path stack per LD2 R0-fold B3, NOT a global set), `parent_audit_entry_id: str | None = None`, `budget: RecursionBudget`
- ContextVar binding so nested calls inherit the outer context (similar to `_audit_stack.py` ContextVar pattern)
- **Lifetime spans the FULL outer dispatch/decision** (R0-fold I2): DispatcherContext is bound at coder iteration entry (one `coder.run()` call); persists across `:fs/read` / `:shell/exec` / `:git/diff` / etc. interleaved between `:llm/call`s; reset between coder iterations. Lifetime is "one full `_decide` ã `_act` ã `_observe` ã `_decide` cycle within a single `coder.run()` invocation".
- **Depth semantics (R1.1-fold IMPORTANT ã clarified):** `ctx.depth` is the count of CURRENTLY-ACTIVE `:llm/call`s on the stack, INCLUDING the call about to start. Initial state: `ctx.depth = 0` (no active calls). On entry to a `:llm/call`: increment FIRST, then check. So:
  - Outermost call: `ctx.depth` becomes 1 on entry (this is the "depth of THIS call")
  - First nested call: `ctx.depth` becomes 2 on entry
  - Nth nested call: `ctx.depth` becomes N+1 on entry
  - On exit: decrement
  - With `MAX_LLM_CALL_DEPTH = 3`, the allowed depths are {1, 2, 3} ã 3 calls total max (outermost + 2 nested levels)
- **Bounds check on entry (post-increment, > strict):**
  ```python
  ctx.depth += 1                              # increment FIRST
  if ctx.depth > MAX_LLM_CALL_DEPTH:           # > strict; 3 IS allowed
      raise LLMRecursionBudgetExceeded(field="depth", limit=MAX_LLM_CALL_DEPTH, observed=ctx.depth)
  ctx.request_count += 1
  if ctx.request_count > MAX_RECURSIVE_REQUESTS:
      raise LLMRecursionBudgetExceeded(field="requests", ...)
  ```
  On exception OR successful exit: `ctx.depth -= 1; ctx.request_count` is NOT decremented (cumulative across recursion tree per LD4).
- **Token enforcement is LAYERED (R0-fold B2 + R1-fold B1+I1 corrected ã provider-honesty caveat acknowledged)**:
  - **Layer 1 (best-effort early reject):** compute `remaining_token_budget = MAX_RECURSIVE_TOKENS - ctx.token_count`. If `remaining_token_budget <= 0`: raise `LLMRecursionBudgetExceeded(field="tokens", ...)` BEFORE making the call.
  - **Layer 2 (best-effort input estimation):** estimate input tokens via `len(json.dumps(request["messages"])) // 4` (rough; tokenizers undercount for some encodings/role/tool blocks per R1-fold I1). Treated as BEST-EFFORT EARLY REJECT ã if `ctx.token_count + estimated_input > MAX_RECURSIVE_TOKENS`: raise BEFORE call. Estimation MAY undercount; the load-bearing safety is Layer 3 below.
  - **Layer 3 (output cap injection ã provider-honest assumption):** inject `max_tokens` into the request: `request["max_tokens"] = min(request.get("max_tokens", remaining_token_budget), remaining_token_budget)`. This bounds OUTPUT tokens FOR providers that honor the field. **R1-fold B1 caveat:** 2.3c.2 demo uses Anthropic + OpenAI which BOTH honor `max_tokens` per their public APIs. Streaming is OUT of 2.3c.2 scope (non-streaming demo only). For arbitrary future providers, Layer 4 below is the escape hatch.
  - **Layer 4 (post-call hard accounting ã load-bearing for demo-provider scope):** after each call, increment `ctx.token_count` by ACTUAL `response.usage.total_tokens` if present. If `usage.total_tokens` is MISSING from the provider response (R1.1-fold NICE concern): fall back to `estimated_input + injected_max_tokens` as a conservative-overcount substitute. If `ctx.token_count > MAX_RECURSIVE_TOKENS` (over-shoot due to provider not honoring max_tokens OR missing-usage substitute exceeding budget): raise `LLMRecursionBudgetExceeded(field="tokens", ...)` BEFORE the next nested call. This guarantees the budget is enforced for the rest of the recursion tree for demo providers (Anthropic + OpenAI both report `usage.total_tokens`). For arbitrary future providers without `usage` reporting, Layer 4 falls back to the conservative-overcount substitute ã runaway-protection then becomes "best-effort overcount" rather than "exact accounting"; v0.9.x will need provider-specific tokenizer integration for arbitrary providers.

**Honest claim statement:** any single `:llm/call`'s output is bounded by Layer 3's `max_tokens` cap if and only if the provider honors that field. The 2.3c.2 demo's Anthropic + OpenAI providers do; streaming + arbitrary-provider safety is out of demo scope. The CUMULATIVE recursion-tree budget is enforced by Layer 4's post-call accounting, which catches any per-call overshoot before the NEXT nested call. This is "best-effort safety" with a load-bearing recursion-tree cap ã NOT "single-call provider runaway is mathematically impossible".
- Budget exhaustion raises `LLMRecursionBudgetExceeded(field="depth"|"tokens"|"requests", limit=..., observed=...)`

**Resource limits:**
```python
MAX_LLM_CALL_DEPTH: int = 3         # hard cap (demo default; R0-fold N1 wording)
MAX_RECURSIVE_TOKENS: int = 20_000  # soft cap, total across nested tree
MAX_RECURSIVE_REQUESTS: int = 10    # soft cap, total across nested tree
```

Defaults are demo-grade per codex (a); 2.4a dogfood may tune. Bounds-check pattern mirrors 2.3b `MAX_BRANCH_MAX_ITER=50` / `MAX_BRANCH_EXPANDER_K=4`.

**Boundary test obligation (G7 falsifiability):** parametrize tests for `next_depth = MAX_LLM_CALL_DEPTH` (allowed; happy) AND `next_depth = MAX_LLM_CALL_DEPTH + 1` (rejected; raise). Same for tokens/requests. Catches off-by-one in bounds-check direction.

### LD2 ã Content-hash-keyed cycle detection ã ACTIVE-PATH semantics (R0-fold B3: lifetime model + name fix) ã **PARTIAL: Layer A active, Layer B API-only; Layer B runtime wiring v0.9.x rescoped (T9.1-fold, 2026-05-08)**

**Rescope notice (W3 honest-rescope, 2026-05-08 at T9.1-fold):** Codex Impl R1 BLOCKING finding (mean 7.84/min 7.4): "Layer B push/pop is never invoked anywhere in `src/persistence` (only defined in `_recursion.py:320` and unit-tested). If LD2 promises execution-time active-path protection, it currently doesn't happen." Resolution mirrors LD5 W3 pattern: ship Layer B as API surface (push_cycle / pop_cycle helpers + DispatcherContext.cycle_path field + SkillCycleDetected exception) with comprehensive unit tests at T3, but DEFER the execution-time runtime wiring to v0.9.x. Activating Layer B end-to-end requires marking grafted subtrees in `_apply_compose_with_skill` with the skill's content hash + planner-side push/pop on entry/exit of grafted nodes ã substrate-side substantive work that exceeds 2.3c.2's stated scope. **Layer A (search-time static subtree check) is FULLY ACTIVE in 2.3c.2 and covers STRUCTURAL cycles** (the demo's actual cycle exposure surface, since sequential perform-level recursion does not exhibit Python-stack reentrancy). Falsifiable acceptance signal for v0.9.x: integration test where `_escalate_plan_body`'s grafted-skill-body walk pushes/pops `cycle_path` correctly and runtime cycles raise `SkillCycleDetected` end-to-end (currently API-tested only via T3's push/pop unit tests).

**Decision (R0-fold B3 + R1-fold I2 amended ã TWO LAYERS, NOT one): cycle detection has TWO complementary layers covering different cycle classes:**

**Layer A ã search-time STATIC subtree check** (already shipped at `_mcts.py:213-217`): when `_apply_compose_with_skill` is called, the engine checks `if plan.id in skill_ids_in_subtree` and raises `_PlanCycleDetected` if the candidate plan's content-hash already appears as a subtree of the skill being grafted. **2.3c.2 KEEPS this layer unchanged.** Detects STRUCTURAL cycles (skill plan A grafting into a target that already CONTAINS A's plan as a subtree). Layer A operates on Plan AST shape, NOT on runtime call stack.

**Layer B ã execution-time DYNAMIC active-path check** (NEW in 2.3c.2): the DispatcherContext.cycle_path tracks **content hashes** as an ACTIVE-PATH STACK (LIST), not a global set. Specifically:
- On ENTRY to a skill body via execution-time recursive `:llm/call` into a skill body: push the skill's content hash (`plan.id` per `_skill_library.py:_derive_skill_id`) onto `cycle_path`
- On EXIT from the skill body (after execution returns): POP the hash from `cycle_path`
- Cycle check on entry: if the hash is ALREADY in `cycle_path` (currently active up the call chain), raise `SkillCycleDetected(_PlanCycleDetected)` (subclass for clearer error attribution at the runtime layer). This forbids cycles-WITHIN-PATH but ALLOWS reuse after unwind (skill A ã skill B ã return ã skill B again is OK; skill A ã skill B ã skill A is NOT)
- The hash KEY is the FULL `plan.id` (32-hex chars), not the 16-hex-char `skill_id` slice

**Layer ownership clarification (R1-fold I2 + T9.1-fold rescope):** Layer A push/pop is N/A ã it's a SINGLE static check at graft-time, no active stack. Layer B push/pop, when activated, is owned by the execution-time path (planner walking grafted subtrees marked with skill content hash) ã **NOT by `_audit_stack.py` middleware** (the middleware doesn't know which skill body owns a `:llm/call` leaf at this layer). T9.1-fold rescope: Layer B's runtime wiring is deferred to v0.9.x; the API surface (push_cycle / pop_cycle / DispatcherContext.cycle_path) ships as plumbing with unit-test coverage. Search-time `_apply_compose_with_skill` does NOT participate in Layer B (no DispatcherContext at search-time; the engine doesn't run plan-execution semantics during MCTS) ã but it would mark grafted subtrees with the skill's content hash so the planner can push/pop on entry/exit when v0.9.x activates Layer B.

**The two layers cover orthogonal cycle classes:**
- Layer A catches: "skill A's body would, if grafted, contain itself as a subtree" ã STRUCTURAL impossibility (FULLY ACTIVE in 2.3c.2)
- Layer B catches: "execution stack would re-enter currently-active skill A" ã RUNTIME impossibility (API-only in 2.3c.2; runtime wiring v0.9.x rescoped per T9.1-fold)

Layer A fires today; Layer B is API-tested only. Tests G6.1-G6.3 cover the Layer B API surface (push/pop discipline, content-hash keying, sequential reuse) via T3's unit tests in `tests/coder/test_recursion_dispatcher.py`. Pre-existing 2.3b/MCTS tests + T6's `test_g4_d_cycle_detection_layer_a_search_time_static` cover Layer A.

**Exception class fix (R0-fold B3 name correction):** the existing class at `_mcts.py:164` is `_PlanCycleDetected`, NOT `SkillCircularComposition` as the v0 draft erroneously named. Use the ACTUAL existing class. If a SKILL-specific subclass is needed for clearer error attribution, declare `class SkillCycleDetected(_PlanCycleDetected)` in `src/persistence/coder/_recursion.py` (subclass for narrower error type while inheriting existing exception hierarchy).

**Rationale (codex callout + B3 amendment):** content-addressed registry can have semantically-identical skill aliases (different `skill_id` prefixes pointing to byte-identical Plan ASTs via re-registration with different `promotion_id`). skill_id-only cycle detection would (a) reject legitimate updates of the same logical skill across versions, OR (b) allow recursion via aliases that should be detected as cycles. Content-hash keying prevents both failure modes. ACTIVE-PATH semantics (vs global set) correctly model the "function stack" mental model ã a skill that completes and unwinds CAN be re-invoked legitimately later (it's not a permanent ban; only an active-cycle prevention).

**Falsifiable acceptance signal (G6):**
- (G6.1) self-recursion: skill A with body containing `[:compose-with-skill {:skill-id "skill/A..."}]` ã grafting must raise `_PlanCycleDetected` (or `SkillCycleDetected`) on entry to nested A while A is already active
- (G6.2) alias recursion: register skill A with body X; re-register byte-identical body X under different promotion_id ã skill_id_B (same content hash). Skill A composing skill B (same content hash) MUST raise cycle exception even though skill_ids differ
- (G6.3) sequential reuse OK: skill A ã completes ã skill A again (linear, no nesting): MUST NOT raise ã active-path semantics correctly allow post-unwind reuse

### LD3 ã Composition expansion: search-time RESOLUTION + content-hash PROVENANCE for replay-explainability (R0-fold B4: claim narrowed)

**Decision:** when MCTS engine applies `ComposeWithSkillAction(skill_id=X)` via `_apply_compose_with_skill` (`_mcts.py:187-209`):
1. Look up the skill from SkillLibrary at SEARCH time (via the `skill_library: SkillLibrary` parameter threaded through MCTS engine ã already shipped, just populated by 2.3c.2)
2. Compute the looked-up skill's content hash (`plan.id`)
3. **Record the content hash in `:mcts/iteration` provenance** ã emit a new datom field `composed_skill_content_hash: str` alongside the existing `composed_skill_id` field at `_mcts_datoms.py:209`
4. Graft the skill body into the Plan AST eagerly (existing engine behavior)
5. Winner Plan AST is FULLY-RESOLVED (no remaining `[:compose-with-skill ...]` references) ã runs through 2.3a's `_escalate_plan_body` unchanged

**R0-fold B4 corrected claim ã replay byte-identity is provided by the WINNER PLAN AST being content-addressed, NOT by execution-time consultation of the pinned hash.** The original v0 draft over-claimed: "execution-time uses pinned hash to prevent skill drift". With search-time eager grafting, the winner Plan AST is FULLY RESOLVED ã no remaining skill references at execution time, so there's no "fresh lookup vs pinned" choice to make at execution. The actual replay byte-identity comes from:
- Winner Plan AST node hash (content-addressed `plan.id`) ã this IS the replay anchor
- `:mcts/iteration` provenance content_hash ã gives REPLAY EXPLAINABILITY (audit-trail showing which skill_id was looked up + what content_hash that resolved to + which Plan AST was grafted)

**Pinning is INFORMATIONAL/PROVENANCE, not load-bearing for replay byte-identity.** Replay byte-identity is preserved BY the content-addressing of the resolved winner. The pinned hash is the "what skill was this" audit trail for debug/replay tooling.

**Future-proofing for v0.9.x lazy-graft option:** if a future phase adds execution-time skill resolution (lazy-graft), the pinned content_hash in provenance becomes load-bearing for replay (execution must use the pin, not a fresh lookup, to avoid skill drift). 2.3c.2 ships the pin to enable that future without forcing it now.

**Rationale (codex b NUANCED-OTHER + R0-fold B4):** search-time eager grafting is the right design for 2.3c.2 demo (matches existing engine behavior at `_mcts.py:244`); content-hash recording in provenance is the right addition for replay-explainability + future lazy-graft enablement. The corrected framing avoids over-claim.

### LD4 ã UNIFIED dispatcher budget (codex DISAGREE on c)

**Decision:** `:llm/call` recursion AND `ComposeWithSkillAction`-grafted skill execution feed the SAME DispatcherContext budget counter. Specifically:
- When MCTS grafts a skill into the Plan AST, the skill body's `:llm/call` leaves count toward the SAME depth + token + request counter as direct recursive calls
- Cycle detection (LD2) is enforced at the DispatcherContext level ã applies to BOTH composition-driven and recursion-driven skill activation

**Rationale (codex c DISAGREE-INDEPENDENT):** independent limits create bypass loopholes ã N composed skills each with depth-1 recursion would all be locally legal but globally exceed depth=N. Unified budget closes the loophole.

**Test gate (G3 falsifiability):** scenario where MCTS proposes ComposeWithSkillAction whose grafted body contains a recursive `:llm/call` ã verify the recursive call's audit entry inherits the outer DispatcherContext's depth counter (continues at 1, not resets to 0).

### LD5 ã Audit chain: linear append + parent pointer (NEW; R0-fold I3 migration impact) ã **PARTIAL: field shipped, active wiring v0.9.x rescoped (T3-time, 2026-05-08)**

**Rescope notice (W3 honest-rescope, 2026-05-08 at T3 dispatch):** During T3 implementation analysis, a four-way constraint set was discovered: (i) G4's "peak depth=2" assertion + (ii) audit middleware's LIFO `try`/`finally` append ordering + (iii) AuditEntry's frozen + content-hashed invariant + (iv) linear prev_hash chain integrity. These four cannot be simultaneously satisfied for nested-Python-stack `:llm/call` cases without either (a) a two-layer audit-chain redesign (~150-200 LOC; pre-compute "header id" at entry, augment with verdict/result at finally; outer.id then knowable at outer-entry for inner to reference) or (b) outer-finally back-patching with cascading hash recomputation across all entries appended during outer's `try`. Both are phase-scope-creep. **Resolution: T2 ships the AuditEntry field (already done at commit `4dc3fb3`); T3 wires DispatcherContext through the middleware for cycle detection (LD2) + budget enforcement (LD1) but leaves `parent_audit_entry_id` always None at the audit middleware layer. The active wiring of parent_audit_entry_id with non-None values is rescoped to v0.9.x per ôÏ 8.** Falsifiable acceptance signal for the v0.9.x track: integration test where a `:llm/call` audit entry produced from a Python-stack-nested dispatch carries `parent_audit_entry_id` referencing the outer dispatch's audit entry id (currently always None at middleware layer; v0.9.x will populate via the chosen mechanism).

**Decision (rump in 2.3c.2):** AuditEntries are appended LINEARLY to the canonical audit chain (existing semantics ã prev_hash links each entry to its predecessor regardless of nesting). NEW field `parent_audit_entry_id: str | None` is plumbed through the dataclass + canonical content hash + wire form + datom round-trip + spec (T2 shipped). Field defaults to `None`. Middleware does not populate non-None values in 2.3c.2 (rescoped). G3 is reformulated as structural-only ã see ôÏ 4 G3.

```python
# Example chain for outer-then-nested call:
# entry_outer:  prev_hash=PRIOR_HASH, parent_audit_entry_id=None,         op=":llm/call"
# entry_inner:  prev_hash=entry_outer.id, parent_audit_entry_id=entry_outer.id, op=":llm/call"
# entry_after:  prev_hash=entry_inner.id, parent_audit_entry_id=None,        op=":fs/write"
```

**Rationale (codex missed-flag callout):** linear append alone obscures causality (you can't tell which `:llm/call` was a child of which other from the chain shape). Adding parent pointer is cheap (one optional string field) and recovers the parent/child structure for replay/debug. Doesn't break replay byte-identity because parent_audit_entry_id is deterministic given the call structure.

**Migration: AuditEntry shape change (R0-fold I3 ã global drift expected).** Add `parent_audit_entry_id: str | None = None` field to `audit.py:AuditEntry` (currently has `id`, `prev_hash`, `op`, `args_hash`, `verdict`, `result_hash`, `latency_ms`, `recorded_at`). Default `None` preserves field-level backward compatibility with all 2.0a-2.3c.1 audit entries. The chain-hash computation INCLUDES `parent_audit_entry_id` (even when `None`) so prev-hash chain integrity covers it.

**Global chain-hash drift expected (R0-fold I3):** including `parent_audit_entry_id` in the chain-hash computation will produce DIFFERENT entry id values for the same `(prev_hash, op, args_hash, ...)` inputs compared to pre-2.3c.2 entries.
