# Phase 2.5 — v0.9.0a1 External-Alpha GA Tag (design)

**Status:** R0-skipped (release engineering, no substrate-truth risk in design itself); R1 gates mandatory pre-publish.

**Goal:** Ship the `v0.9.0a1` external-alpha GA — annotated git tag on `c3d14d6` (Phase 7 merge), pushed to `origin`, with a GitHub release backed by complete CHANGELOG + new release-notes doc. PyPI publish stays out of scope (deferred to Phase E ~Jul 20).

**Architecture:** Two documentation artifacts (root `CHANGELOG.md` backfill + new `docs/release-notes/v0.9.0a1.md`) feed one git artifact (annotated tag) feeds one GitHub artifact (release). Gate before publish with two codex R1 passes — one on documentation substrate-truth, one on the tag/push commands.

**Tech Stack:** Existing — git, `gh` CLI, `pytest`, `uv build`. No new deps.

## Locked Decisions (LDs)

### LD-1 — CHANGELOG backfill depth: per-phase phase-header + LD summary + ARIS verdict + commits list (matches existing 2.4c/Phase 7 entry shape at `CHANGELOG.md:11-32`)

- **Decision:** For each missing phase, write a sub-section under `## v0.9.0a1 (unreleased)` matching the existing Phase 7 + 2.4c depth (~20-50 lines each). NOT a one-line bullet per phase; NOT a full reproduction of the per-module CHANGELOG content.
- **Reasoning:** The existing two entries (Phase 7 at line 11, Phase 2.4c at line 32) set the visual contract for what a v0.9.0a1 phase entry looks like. New entries must match for the root CHANGELOG to read coherently. Per-module CHANGELOGs already carry the deep detail.
- **Competitors rejected:** (a) Single-line summary per phase — breaks visual coherence with 7+2.4c entries; (b) Full per-module content duplicated into root — duplicates per-module CHANGELOG, drift risk over time.
- **Acceptance signal (falsifiable):** G1 — `grep "^### Phase " CHANGELOG.md | head -20` lists ALL of {Phase 7, 2.4c, 2.4b.1, 2.4b, 2.4a, 2.3d, 2.3c.2, 2.3c.1, 2.3b, 2.3a, 2.2b, 2.2a, 2.1c.x series collapsed-or-expanded, 2.1b, 2.1a} in reverse chronological order BEFORE the `## v0.8.0a1` header at line 154.

### LD-2 — Release-notes doc structure: 6-section public-facing artifact, distinct from positioning doc

- **Decision:** `docs/release-notes/v0.9.0a1.md` with these sections: **Highlights** (cross-link positioning.md 5 capabilities) / **New substrate ops** (`:sys/now`, `:repl/request`, `:repl/response`, `:capability/denied` if shipped, others) / **Breaking changes from v0.8.5a1** / **Migration guide pointer** / **Acceptance** (test counts + ARIS verdicts) / **Known issues + W3-rescoped items**.
- **Reasoning:** Positioning doc is sales narrative ("5 falsifiable proofs"). Release notes are "what's in this version vs the last" — a different audience need.
- **Competitors rejected:** (a) Just cross-link positioning.md (no new doc) — release notes serve a different role than positioning; (b) Pure changelog dump — duplicates root CHANGELOG.
- **Acceptance signal:** G2 — `docs/release-notes/v0.9.0a1.md` exists with the 6 sections; cross-link to positioning.md verified; `gh release view --web` rendering preview clean.

### LD-3 — Tag style: annotated, NOT signed (gpg setup not in place)

- **Decision:** `git tag -a v0.9.0a1 -m "<message>" c3d14d6`. Annotated for the metadata; not signed because no GPG keys are wired into the dev box.
- **Reasoning:** Matches existing tag pattern (`v0.8.5a1` was annotated, not signed). Signing is a Phase E concern when PyPI publish needs it.
- **Competitors rejected:** (a) Lightweight tag — no metadata, can't carry release message; (b) Signed tag — gpg setup is a separate yak-shave, not on critical path.
- **Acceptance signal:** G5 — `git tag -v v0.9.0a1` resolves (not signed but annotated); `git cat-file -t v0.9.0a1` returns `tag`.

### LD-4 — Tag message structure: short title + 5-capability bullet list + cross-link to release notes

- **Decision:** Tag message body is:
  ```
  v0.9.0a1 — Phase 2: persistence-coder MVP

  First external-alpha. 14 phases (2.1a → 2.4c) + Phase 7 orchestrate skill.

  Substrate capabilities:
  - Agent-vs-human identity (Ed25519 audit signing)
  - Governed action (Capability lattice)
  - Audit replay (canonical_audit_stack)
  - Kill switch (REPL pause/resume)
  - Substrate time (:sys/now op)

  Release notes: docs/release-notes/v0.9.0a1.md
  Positioning: docs/release-notes/v0.9.0a1-positioning.md
  ```
- **Reasoning:** Tag message is read by `git tag -l --format` and by `git log` in CI tooling. Short title + 5 highlights + doc pointers gives both humans and downstream tracks something to consume.
- **Competitors rejected:** (a) One-line message — loses context; (b) Full release notes embedded — too long, duplicates `docs/release-notes/v0.9.0a1.md`.
- **Acceptance signal:** G5b — `git tag -l --format='%(contents)' v0.9.0a1` shows the message; line count ≤ 20.

## ARIS gates (R1-only)

**Skip R0** on this design: release engineering with mechanical structure — no substrate-truth risk in the LDs themselves. R0-skip rationale must be documented in the impl plan (FD: "release-engineering R0-skip justified by no behavioral change in any LD").

**R1-A (post-T2, pre-T4) — codex review of documentation substrate-truth:**
- Scope: `CHANGELOG.md` v0.9.0a1 section (post-backfill) + `docs/release-notes/v0.9.0a1.md`
- Question: Do the claims made in the release notes match substrate reality? Specifically:
  - Does each phase's CHANGELOG entry match what's in the per-module CHANGELOGs and git log?
  - Do the "5 substrate capabilities" descriptions match the actual ops + tests shipped?
  - Are any W3-rescoped items misrepresented as shipped?
- Cap ~700 words, codex `--reasoning high` (NOT xhigh), NO web search, file:line citations.
- Verdict: PASS / PASS-WITH-FIXES / FAIL.
- If FAIL → fold inline + R1-A.1 lite confirmation pass.

**R1-B (post-T5 draft, pre-tag push) — codex review of tag message + commands:**
- Scope: tag message text + the exact `git tag` + `git push` + `gh release` commands as drafted.
- Question: Any factual error in the tag message? Any irreversible command flag risk?
- Cap ~300 words.
- Verdict: PASS / FAIL.
- If FAIL → fold; re-run R1-B.
